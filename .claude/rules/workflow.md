# Workflow

- Stage with `git add`, commit with `git commit`
- **Commit after completing each phase or sub-phase** -- do not wait for the user to ask. After finishing a phase (or sub-phase like 7a, 7b), immediately stage the relevant files and commit before moving on to the next task or updating documentation. Uncommitted work is invisible work.
- **`my_notes.org` and `my_notes.md` are READ ONLY** -- read for context, never modify
- **Feature tracking** -- create `docs/tracking/YYYY-MM-DD_HHMM_TOPIC.md` before implementation; update after completion
- **Phase subdivision** -- break large phases into a/b/c sub-phases; track done vs remaining explicitly
- **Standup/dailies interval = creation-to-creation** -- a "day" for standup and dailies purposes is the interval from the creation of the LAST standup/dailies to the creation of the NEXT one, NOT the calendar date. A standup's "Yesterday" covers everything done since the previous standup was created (e.g., if the Mar 1 standup was created at 15:00, then the Mar 2 standup's "Yesterday" covers Mar 1 15:00 through the creation of the Mar 2 standup). Working sessions extend past midnight; the "current" dailies remain the file created at session start (at 01:30 on March 3, the current dailies are still `2026-03-02_dailies.md`). Do NOT create new files just because the clock rolled past midnight.
- **Daily summaries** -- maintain `docs/tracking/standups/YYYY-MM-DD_dailies.md` as a living document throughout the working session. Created by user request at the beginning of a session; update as work progresses (completed items, in-progress work, considerations, blockers). The current session's dailies are Claude's daily report-outs (parallel to the user's standups); prior dailies are read-only records.
- **"Open next working day interval"** -- when the user says "open the next working day interval" (or equivalent), create BOTH documents concurrently: (1) a new standup `docs/standups/standup-YYYY-MM-DD.org` (the user's standup), and (2) a new dailies `docs/tracking/standups/YYYY-MM-DD_dailies.md` (Claude's dailies log). The "Yesterday" section of the `.org` standup covers everything done since the creation of the LAST standup/dailies pair. The dailies `.md` starts with a "Carried forward" section noting in-progress work and pending items, ready for the new session. Both files use the current calendar date.
- **Lessons distillation check** -- when creating new dailies, scan the most recent PIR and the past four dailies for lessons, patterns, or observations that should be distilled into principles documents (`DEVELOPMENT_LESSONS.org`, `PATTERNS_AND_CONVENTIONS.org`, `DESIGN_METHODOLOGY.org`, `DESIGN_PRINCIPLES.org`). Include a "Watching" subsection in the dailies' "Carried forward" for medium-term patterns that are not yet confirmed enough for principles — intermittent observations that need 2-3 more data points before codification. This maintains a medium-term memory over patterns that may only surface across multiple sessions.
- **Commits trigger dailies updates** -- every commit is a signal to update the current session's dailies with: what was done (completed items), design choices made, lessons learned, surprises encountered, and any considerations or blockers. Capture learnings while context is fresh — they feed PIR quality. Don't batch dailies updates; write them alongside each commit.
- **Link commits in documentation** -- when updating dailies, DEFERRED.md, or tracking docs to mark work complete, include a reference to the commit hash (e.g., `(commit abc1234)`). This provides traceability from docs to code and serves as a reminder to commit before documenting.
- **Print commit hash after large changes** -- after committing substantial code changes (new features, multi-file modifications, phase completions), print the commit hash so it's visible in the conversation for reference and traceability.
- **`.org` files are canonical** -- when both `.org` and `.md` versions of a document exist (e.g., `LANGUAGE_VISION.org` + `LANGUAGE_VISION.md`), the `.org` is the canonical source and the `.md` is a generated artifact. ALL edits MUST go to the `.org` file. Content added only to `.md` WILL BE LOST on the next org-mode export. Before editing any principles or tracking document, check if a `.org` version exists and edit that instead.
- **WS-mode validation before marking DONE** -- a phase that adds or modifies user-facing syntax is not complete until its expressions work in a standalone `.prologos` file run through `process-file`. A `process-string-ws` test passing is necessary but not sufficient — top-level scoping and file-level preparse must also be validated. Mark as "DONE (sexp only)" if Level 3 validation has not been performed. See `DESIGN_METHODOLOGY.org` § "WS-Mode Validation Protocol" and `.claude/rules/testing.md` § "Three-level WS validation".
- **Acceptance file as Phase 0** -- for ANY implementation track (not just syntax-changing features), write a `.prologos` acceptance file (`racket/prologos/examples/{date}-{feature}.prologos`) as Phase 0 before starting implementation. The file broadly exercises Prologos features in ideal WS syntax — it serves as both a diagnostic instrument (catching WS-mode regressions that infrastructure changes can introduce) and a living showcase of the language. For syntax-adding features, include commented-out target expressions that are uncommented as phases complete. For infrastructure tracks, the file is a broad safety net: run it before and after each phase via `process-file` to confirm no regressions. The track isn't done until the acceptance file runs with 0 errors.
- **Progress tracker for phased designs** -- when starting implementation from a phased design document, add a Progress Tracker table near the top (under the summary) before writing any code. The table has columns: Phase, Description, Status (emoji only: ⬜ not started, 🔄 in progress, ✅ done, ⏸️ blocked), and Notes (commit hashes, relevant context). Update the tracker as each phase completes — commit the tracker update alongside the phase's code changes. This is the single source of truth for implementation progress against the design.
- **Incremental acceptance validation** -- for phased implementations with an acceptance file, run the acceptance file via `process-file` after *each phase*, not just at the end. The design doc's progress tracker should note which acceptance sections each phase is expected to unlock. A phase is not DONE until its expected acceptance sections pass at Level 3. Late L3 validation causes cascading fixes — catching issues per-phase keeps fixes local.
- **WS Impact section in design docs** -- for features that add or modify user-facing syntax, the Stage 2/3 design doc must include a "WS Impact" section addressing: (1) What forms does the WS reader produce for this syntax? (2) Does preparse need a new pass or handler? (3) Are there keyword/delimiter conflicts with existing syntax? (4) Does `flatten-ws-kv-pairs` or similar normalization apply? This front-loads analysis that otherwise surprises at Level 3 validation.
- **DEFERRED.md triage at track start** -- when beginning a new implementation track, review `DEFERRED.md` for items that are now unblocked by the new track's prerequisites or that have become irrelevant. Remove stale items, promote newly-unblocked items into the current track's scope. Deferred items that sit untriaged go stale.
- **Benchmark after infrastructure phases** -- after completing a phase that changes infrastructure (propagator network, cell allocation, resolution loop, registry management), run the comparative benchmark suite (`racket tools/bench-ab.rkt --runs 10 benchmarks/comparative/`) and compare against the prior phase's baseline. Use `--ref HEAD~1` for quick regression checks. Save results with `--output` for cross-session comparison. The adversarial benchmark (`constraints-adversarial.prologos`) is specifically designed to stress Track 7 infrastructure: prelude loading, per-command cell allocation, registry lookups, polymorphic implicit resolution, and many-small-commands scanning overhead. Micro-benchmarks (`benchmarks/micro/`) are for function-level precision when investigating specific regressions. The full suite (`run-affected-tests.rkt --all`) catches broad regressions but lacks the statistical rigor of `bench-ab.rkt` for A/B comparison. **Concurrency**: independent benchmarks (different tiers, different schedulers) can run up to 3 concurrent processes. Do NOT run competing A/B comparisons concurrently — CPU contention corrupts timing data.
- **Targeted test runs for investigating failures** -- when a test suite run produces failures, investigate by running INDIVIDUAL failing tests (`raco test tests/test-NAME.rkt`), NOT by re-running the full suite. Failure logs are always in `data/benchmarks/failures/*.log`. The full suite is a regression gate, not a diagnostic tool. Running it multiple times to "see more output" wastes 2-3 minutes per run and violates the output capture rule in testing.md.
- **Principles-First Design Gate** -- the ten load-bearing principles (see MEMORY.md § "Principles-First Design Gate") apply at ALL decision points, not just critique rounds:
  - **When drafting designs (Stage 3)**: Each major design choice should state which principle it serves. If a choice doesn't clearly serve any principle, question it. If it conflicts with a principle, name and justify the conflict in the document.
  - **When implementing (Stage 4)**: Mid-flight decisions — "should I use a callback?", "should I add a parameter?", "should I keep the old path alongside the new?" — are filtered through the same principles. When an implementation choice reveals a principle violation the design didn't anticipate, stop and note it. Small violations compound into architectural debt.
  - **When in critique rounds (Stage 3 iteration)**: At least one round must include a **principles challenge** — take each major decision and each workaround/migration strategy and ask "could this be more aligned?" Cataloguing applicability ("✓ we use cells") is not challenging. Challenging is: "the partial retirement maintains an imperative cleanup path — does Correct-by-Construction say this should be structural instead?"
  - **Red-flag phrases that demand principles scrutiny**: "temporary bridge", "belt-and-suspenders", "partial retirement", "keeping the old path as fallback", "adding a parameter/callback to inject X", "scanning/polling for readiness".
- **NTT speculative syntax in propagator designs** -- any design document for propagator network infrastructure should include a section expressing the key constructs in NTT speculative syntax (the syntax from the NTT Syntax Design document). This serves as: (a) design clarity check — does the NTT expression make sense? (b) correctness reference — does the Racket implementation match the NTT declaration? (c) NTT refinement — does the design reveal gaps in the NTT syntax? (d) future implementation guide — when we implement NTT, these examples are test cases. Include a correspondence table mapping NTT constructs to Racket implementations. This practice applies to any track in PM, SRE, BSP-LE, or CIU that adds propagators, lattices, bridges, or stratification changes.
- **Diagnostic protocol when blocked** -- when stuck on a problem after 2-3 failed attempts, stop iterating and follow this protocol:
  1. *Audit the domain*: gather comprehensive data about the full scope of the problem. Don't fix one instance — understand ALL instances. (Track 10: "how many registries ARE there?" not "which registry is this test missing?")
  2. *Hypothesize from data*: form specific, falsifiable hypotheses grounded in the audit. "X because Y" not "something is wrong."
  3. *Test narrowly*: build the smallest test that validates/invalidates each hypothesis. One hypothesis, one test. Don't run the full suite to check a theory about one module.
  4. *Challenge by principles*: ask "is my approach principled, or am I working around a deeper issue?" Every time we're stuck, a principle is being violated — find which one.
  5. *Reframe if still stuck*: question the problem framing itself. "Am I solving the right problem?" If the fix works but the problem persists, the framing is wrong. Return to step 1 with a new lens.
  The anti-pattern this prevents: iterative fix-and-retry (whack-a-mole) where each fix reveals the next failure. The audit (step 1) breaks the cycle by revealing ALL failures at once. Steps 1-3 are diagnostic (understand). Step 4 is evaluative (is the approach right?). Step 5 is strategic (is the problem right?).
- **All tests green before moving on** -- never proceed to the next phase with failing tests. A failing test is a signal, not noise. Investigate immediately. If a failure is genuinely pre-existing and unrelated, document it explicitly with a tracking reference before proceeding.
- **Update Master Roadmap on design docs and PIRs** -- writing a track design document or a PIR is a signal to update `docs/tracking/MASTER_ROADMAP.org`. Link the new design doc and/or PIR in the relevant track's row. Update track status (⬜ → 🔄 → ✅). If the design reveals new tracks or cross-dependencies, add them. The Master Roadmap is the single source of truth for all work — if it's not linked there, it's invisible.
- **Review PIR methodology before writing a PIR** -- before writing any Post-Implementation Review, read `docs/tracking/principles/POST_IMPLEMENTATION_REVIEW.org` to refresh on the 16 questions, the template structure (§1–§9+), format principles, and anti-patterns. The Track 4 PIR was initially written as a sparse 7-section summary; reviewing the methodology revealed it was missing Bugs Found and Fixed, What Went Well/Wrong/Lucky/Surprised, Architecture Assessment, What's Next, Key Files, and cross-references to prior PIRs. A PIR written without consulting the methodology will systematically omit the evaluative and meta-learning sections that make PIRs genuinely valuable. §16 must include a longitudinal survey of the **10 most recent PIRs** (not just 2-3) — the value of longitudinal analysis is in spotting slow-moving trends.
