# Workflow

- Stage with `git add`, commit with `git commit`
- **Commit after completing each phase or sub-phase** -- do not wait for the user to ask. After finishing a phase (or sub-phase like 7a, 7b), immediately stage the relevant files and commit before moving on to the next task or updating documentation. Uncommitted work is invisible work.
- **`my_notes.org` and `my_notes.md` are READ ONLY** -- read for context, never modify
- **Feature tracking** -- create `docs/tracking/YYYY-MM-DD_HHMM_TOPIC.md` before implementation; update after completion
- **Phase subdivision** -- break large phases into a/b/c sub-phases; track done vs remaining explicitly
- **Standup/dailies interval = creation-to-creation** -- a "day" for standup and dailies purposes is the interval from the creation of the LAST standup/dailies to the creation of the NEXT one, NOT the calendar date. A standup's "Yesterday" covers everything done since the previous standup was created (e.g., if the Mar 1 standup was created at 15:00, then the Mar 2 standup's "Yesterday" covers Mar 1 15:00 through the creation of the Mar 2 standup). Working sessions extend past midnight; the "current" dailies remain the file created at session start (at 01:30 on March 3, the current dailies are still `2026-03-02_dailies.md`). Do NOT create new files just because the clock rolled past midnight.
- **Daily summaries** -- maintain `docs/tracking/standups/YYYY-MM-DD_dailies.md` as a living document throughout the working session. Created by user request at the beginning of a session; update as work progresses (completed items, in-progress work, considerations, blockers). The current session's dailies are Claude's daily report-outs (parallel to the user's standups); prior dailies are read-only records.
- **Link commits in documentation** -- when updating dailies, DEFERRED.md, or tracking docs to mark work complete, include a reference to the commit hash (e.g., `(commit abc1234)`). This provides traceability from docs to code and serves as a reminder to commit before documenting.
- **Print commit hash after large changes** -- after committing substantial code changes (new features, multi-file modifications, phase completions), print the commit hash so it's visible in the conversation for reference and traceability.
- **`.org` files are canonical** -- when both `.org` and `.md` versions of a document exist (e.g., `LANGUAGE_VISION.org` + `LANGUAGE_VISION.md`), the `.org` is the canonical source and the `.md` is a generated artifact. ALL edits MUST go to the `.org` file. Content added only to `.md` WILL BE LOST on the next org-mode export. Before editing any principles or tracking document, check if a `.org` version exists and edit that instead.
- **WS-mode validation before marking DONE** -- a phase that adds or modifies user-facing syntax is not complete until its expressions work in a standalone `.prologos` file run through `process-file`. A `process-string-ws` test passing is necessary but not sufficient — top-level scoping and file-level preparse must also be validated. Mark as "DONE (sexp only)" if Level 3 validation has not been performed. See `DESIGN_METHODOLOGY.org` § "WS-Mode Validation Protocol" and `.claude/rules/testing.md` § "Three-level WS validation".
- **Acceptance file before implementation** -- for features that add user-facing syntax, write a skeleton `.prologos` acceptance file (`examples/{date}-{feature}.prologos`) with commented-out target expressions BEFORE starting implementation. Uncomment expressions as phases complete. The feature isn't done until the file runs with 0 errors via `process-file`.
