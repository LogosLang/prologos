4. Trait system and Dependent Type Infrastructure

We have recently designed and implemented a complete Full Trait Resolution infrastructure. This was a large effort, but it was essential for us to complete our numeric tower with subtyping within families, and conversion between families. This allowed us to define general arithmetic functions that can efficiently dispatch to their concretized types, with their type-specific `impl` implementation.

This trait system is easily usable and extendable by the user, so he or she can create their own general functions that dispatch to efficient and proper functions on the concrete types. We think this is a nice way of extending polymorphism and generic functions.

One other innovation on the trait system, before I get to the second part of your question, that I am very proud of is an innovation that we do not think exists in a first class, user extensible manner in any language, and which solves the inheritance problems found in supertrait type designs. The key insight was taking a look traits through the logical lens, and deep experience in Prolog was key here. Through the curry howard correspondence, a Type is a Predicate Statement. Traits are constraints on types. Supertraits are trait Inheritance, which is isomorphic (structurally equal to) to logical implication. `trait Ord: Eq` means "Ord implies Eq"... If Eq changes, it could break Ord. There is a tight coupling, making extensions more challenging and brittle, and conflicts could cause problems down the road. This would make traits a feature only in the hands of careful language implementors.

We found an innovation to this problem, by using Prolog's scalable and composable/reusable approach of conjunctive predicates, and applying that to a novel trait system, which we call `bundle`.
```prologos
;; Simple bundle: conjunction of two traits
bundle Comparable := (Eq, Ord)

;; Bundle of bundles (nested expansion):
bundle Numeric := (Add, Sub, Mul, Comparable)
;; where (Numeric A) → where (Add A) (Sub A) (Mul A) (Eq A) (Ord A)

;; Multi-param/compound constraints in bundles:
bundle Convertible := ([From A B], [Into B A])
;; where (Convertible Int Rat) → where (From Int Rat) (Into Rat Int)

```

The Key Design Properties of Bundles:
1. **Zero runtime overhead**: Bundles are expanded at parse/desugar time. No bundle types, no bundle dicts, no bundle accessors exist at runtime. After expansion, the system sees only flat trait constraints.

2. **No trait coupling**: Traits remain fully independent. `Eq` knows nothing about `Ord` or `Comparable`. Any trait can be used alone without any other trait.

3. **Compositional**: Bundles compose via nesting. `bundle A := (X, Y)`, `bundle B := (A, Z)` expands `B` to `(X, Y, Z)`. Deduplication removes redundancies.

4. **User-definable**: Users create their own domain bundles without special language support: `bundle Serializable := (Show, Read)`.

5. **No diamond problem**: Since bundles are flat expansions, there's no ambiguity about which "path" provides a trait. If `Numeric` includes `Eq` via `Comparable`, and also directly, deduplication in expansion simply removes the duplicate.

We're very proud of this innovation, and hope that it contributes meaningfully to the space of language design. It's working quite well for us, too! ^_^

We also have an auto-implicits declaration, very similar to Idris, which--along with full trait resolution--helps hide the complexity from users.

- Full Example: Async Service with Propagators
```prologos
;; ========================================
;; Schema and Session
;; ========================================

schema Query
  :id QueryId
  :sql String
  :params [Map String Value]

schema QueryResult
  :id QueryId
  :rows [List Row]
  :time-ms Nat

session DatabaseService
  rec Loop
    &>
      | :query
          ? q : Query
          !! result : QueryResult    ;; !! = eventual response
          Loop
      | :close
          end

;; ========================================
;; Server: processes queries async
;; ========================================

defproc db-server : DatabaseService
  ;; Internal cell for tracking pending queries
  cell pending : Lat [Map QueryId Cell]

  loop
    offer
      | :query
          recv q
          ;; Create result cell for this query
          let result-cell = [cell empty]

          ;; Spawn async execution
          spawn
            let rows = [execute-sql q.sql q.params]
            let result = {:id q.id
                          :rows rows
                          :time-ms [elapsed]}
            [put result-cell result]

          ;; Register pending query
          [put pending [assoc [get pending] q.id result-cell]]

          ;; Eventual send: will complete when spawn finishes
          send! [get result-cell]   ;; non-blocking!
          continue

      | :close
          ;; Wait for all pending to complete
          [for-each [vals [get pending]] await]
          stop

;; ========================================
;; Client: fires queries, collects results
;; ========================================

defproc db-client [queries : List Query] : List QueryResult
  let channel = [connect DatabaseService]

  ;; Fire all queries (eventual sends)
  let promises = [map [fn [q]
    select :query channel
    send q channel
    recv! channel    ;; recv! = eventual receive, returns promise
  ] queries]

  ;; Do other work while queries execute
  [log "queries submitted, waiting..."]

  ;; Collect all results
  let results = [map await promises]

  select :close channel
  results
```
