**Session types are not just protocol specifications — they are causal clocks.** Each position in a session type's continuation chain (`!A . ?B . end` → positions [0, 1, 2]) represents a causally-ordered event. This ordering is:

Thank you for this — you're identifying something genuinely deep with the `proc`/`rel` parallel. Let me take each thread in turn.

## On A + D vs. B

Your reasoning is sound and I agree with it. Let me sharpen why:

**B is subsumed by D for session-typed effects.** Timestamps in Architecture B are *opaque annotations* — they carry ordering information but the system has no proof that the timestamps are correct. You assign them, you trust them. Architecture D derives the same ordering from the session type, which is verified by the type system. The ordering is a *consequence* of type safety, not an additional thing to get right.

**B is only needed for unsessioned effects** — but that's exactly where A already works. Top-level `eval` with `SysCap` is sequential (the `defn main` walk is strictly sequential, `spawn` illegal per D18). For that case, A's walk-order execution is correct by construction and trivially simple.

So the architecture becomes:

```markdown
Session-typed IO  →  Architecture D  (ordering from session fidelity)
Unsessioned IO    →  Architecture A  (ordering from sequential walk)
```

No B needed. Two mechanisms, each maximally suited to its domain, no overlap. B would sit in between as a "medium guarantees, medium complexity" option that neither case actually wants.

One caveat: if we ever want *concurrent unsessioned effects* (multiple `main`-like entry points running in parallel), B or D would be needed. But that's a distant scenario, and by then we'd likely want to session-type those interactions anyway, pulling them into D.

## On 9c: Cross-Channel Data Dependencies as Constraint Resolution

This is where your observation gets really exciting. Let me trace the analogy carefully.

### The `rel` Precedent

In the logic engine, `defr` defines relations with logical variables. When you write:

```prologos
defr ancestor [?x ?y]
  &> (parent x y)
  &> (parent x z) (ancestor z y)
```

The solver doesn't know which clause will succeed, which bindings will emerge, or whether `z` even has a consistent value. It maintains multiple hypotheses (via ATMS), propagates constraints (via the PropNetwork), and resolves when enough information accumulates. The ordering of resolution steps doesn't matter — the fixed point is confluent.

The key: **logical dependencies between variables are resolved by propagation to a fixed point**, not by prescribing an execution order.

### The `proc` Analog

Now consider cross-channel data dependencies in a process:

```prologos
proc p a b =
  recv x a            ;; binds x from channel a
  send [f x] b        ;; uses x on channel b
  recv y b            ;; binds y from channel b
  send [g x y] a      ;; uses x and y on channel a
  stop
```

The per-channel ordering is determined by session types:
- Channel `a`: position 0 (recv), position 1 (send) — total order from session
- Channel `b`: position 0 (send), position 1 (recv) — total order from session

But the cross-channel ordering comes from data flow:
- `recv x a` must happen before `send [f x] b` — because `x` flows from a to b
- `recv y b` must happen before `send [g x y] a` — because `y` flows from b to a

These data-flow constraints form **ordering edges in a partial order**. The question: can these edges be computed by propagation, like logical constraints in `rel`?

### The Answer: Yes — Transitive Closure IS a Monotone Fixed Point

Here's the key insight: **computing the transitive closure of a partial order is a monotone fixed-point problem**. Adding edges to a relation can only make the transitive closure larger, never smaller. This is exactly what propagators do.

```markdown
Ordering cell: set of pairs (effect_i, effect_j) meaning "i must precede j"

Initial state:
  Session edges: {(a:0 < a:1), (b:0 < b:1)}      ← from session types
  Data-flow edges: {(a:0 < b:0), (b:1 < a:1)}     ← from data flow analysis

Transitive closure propagator:
  If (i < j) and (j < k), add (i < k)

  Fires, adds: {(a:0 < b:1), (a:0 < a:1)}         ← already present
  Fixed point reached.

Result: total order a:0 < b:0 < b:1 < a:1
```

The propagator network computes the ordering. The effect handler executes in that order. **The ordering computation is convergent (monotone fixed point). The effect execution is sequential (handler).** Same separation as everywhere else in our architecture.

### ATMS for Branching Orders

Now here's where the `rel` analogy goes deep. When there are choices (`proc-case`), different branches create different data-flow patterns:

```prologos
proc p a b =
  recv x a
  case b
    | left  -> send [f x] b ; recv y b ; send y a ; stop
    | right -> send x a ; stop
```

The `left` branch has cross-channel deps `{a:0 < b:1, b:2 < a:1}`. The `right` branch has `{a:0 < a:1}` (no cross-channel dep at all). These are different partial orders.

The ATMS can maintain **both orderings as hypothetical**: one supported by `assumption(left)`, one by `assumption(right)`. The ordering cells hold supported values:

```markdown
Ordering cell under h_left:  {a:0 < b:1, b:2 < a:1, ...}
Ordering cell under h_right: {a:0 < a:1}
```

When the choice resolves (the external endpoint selects `left` or `right`), the ATMS filters to the consistent worldview. The effect handler executes using the ordering from the chosen worldview.

This is *exactly* the logic engine pattern: maintain multiple hypotheses, propagate constraints within each, resolve when forced. The ATMS doesn't choose — it tracks. The choice comes from outside (the other process's selection), and the ordering follows.

### The Proof-Search Connection

You asked: can `proc` run through a similar resolution approach to `rel`?

Yes, and the parallel is structural:

| Aspect | `rel` (Logic Engine) | `proc` (Session Runtime) |
|---|---|---|
| Variables | Logical variables (?x, ?y) | Effect positions (a:0, b:1) |
| Constraints | Unification equations | Ordering edges (≤) |
| Propagation | Unification propagators | Transitive closure propagator |
| Fixed point | All variables bound | Complete partial order |
| Hypotheses | `amb` alternatives | `proc-case` branches |
| Resolution | ATMS worldview selection | Choice resolution |
| Execution | Answer extraction | Effect execution in order |

The propagator network's role is identical in both: compute the fixed point of a set of monotone constraints. The nature of the constraints differs (equality for `rel`, ordering for `proc`), but the computational pattern is the same.

### What This Means for Architecture D

This resolves the open question from §9c in the research doc. Cross-channel data dependencies don't need to be handled by the AST walk (Architecture A) or by external timestamps (Architecture B). They can be resolved *within the propagator network* as ordering constraints, computed to a fixed point by transitive closure propagation.

Architecture D becomes:

```markdown
Layer 1: Session advancement propagators  → per-channel ordering (existing)
Layer 2: Data-flow analysis propagators   → cross-channel ordering edges (NEW)
Layer 3: Transitive closure propagator    → complete partial order (NEW)
Layer 4: ATMS                             → branching orderings per choice (existing)
Layer 5: Effect handler                   → execute in topological order (NEW)
```

Layers 1 and 4 already exist. Layer 2 is data-flow analysis (well-studied, straightforward for our process language). Layer 3 is a single propagator computing transitive closure (monotone, convergent). Layer 5 is the sequential execution handler.

The ordering is fully propagator-native. Only the execution is sequential.

## On 9d: Unsessioned Effects

In the current architecture, top-level `eval` with `SysCap` falls back to A. But there's a subtler observation: **`defn main` is already an implicit session**.

`main` is strictly sequential. Its body walks top-to-bottom. Each IO operation occurs at a well-defined position in that walk. This IS a session — it's just untyped as one. We could formalize:

```markdown
main body ≈ implicit session: !Effect₁ . !Effect₂ . ... . end
```

Where each IO call in `main` is a "send" of an effect descriptor on an implicit system channel. The ordering comes from the syntactic position in `main`'s body — which is exactly what A provides.

If we wanted to make this explicit (bringing unsessioned effects under D), we could infer an implicit session type for `main` from its body. But that's over-engineering for Phase 0 — A is correct and simple for this case.

The important point: A isn't a *compromise* — it's D for the degenerate case of a single-channel sequential session. The main function's sequential walk IS the session. The two architectures are the same thing at different levels of formalization.

## Additional Consideration: The Partial Order Planning Literature

Your intuition about "data dependency as a constraint problem over orderings" connects to a well-studied area in AI planning: [Partial-Order Planning](https://en.wikipedia.org/wiki/Partial-order_planning) (POP).

POP maintains a partial ordering between actions and only commits additional ordering when forced. It uses **causal links** (action A achieves precondition P for action B) and resolves **threats** (action C might interfere with a causal link) by adding ordering constraints (promote C before or demote C after the link).

The correspondence:
- Our data-flow edges = POP causal links
- Our session ordering = POP initial ordering constraints
- Our `proc-case` branches = POP plan alternatives
- Our ATMS = POP's threat postponement (maintain multiple orderings until forced to commit)

POP's principle of **least commitment** — don't order actions until you must — is exactly what we're doing: session types provide necessary ordering, data flow adds necessary ordering, everything else is left concurrent. The propagator network computes the minimal ordering that satisfies all constraints.

*Sources*:
- [Partial-Order Planning (Wikipedia)](https://en.wikipedia.org/wiki/Partial-order_planning)
- [Postponing Threats in Partial-Order Planning](https://www.researchgate.net/publication/221604529_Postponing_Threats_in_Partial-Order_Planning) — Joslin & Pollack
- [From Precedence Constraint Posting to Partial Order Schedules](https://www.ri.cmu.edu/pub_files/2007/5/pcos_AICOM07.pdf) — Policella et al.

## Critique: What Could Go Wrong

Let me be honest about the risks:

1. **Data-flow analysis complexity**: For our simple process language (`proc-send`, `proc-recv`, `proc-case`), data-flow analysis is straightforward — variable binding sites are explicit. But if we add higher-order channel passing or channel polymorphism, data-flow analysis becomes harder. The constraint propagator would need to handle unknown data flows (resolved speculatively via ATMS?).

2. **Cycle detection**: If data-flow edges + session edges form a cycle, that's a deadlock. The transitive closure propagator would detect this as `(a:0 < a:0)` — a contradiction. This is actually a *feature*: the propagator network detects deadlocks automatically as contradictions in the ordering lattice.

3. **Performance**: Computing transitive closure via propagation is O(n³) in the worst case (n = number of effect positions). For typical processes (a handful of operations on 2-3 channels), this is negligible. For large concurrent systems with many channels, we might need incremental/sparse representations.

4. **The "fully native" question**: Even in Architecture D with constraint-resolved ordering, the *execution* of effects is still sequential (the handler). The propagator network computes *what* to do and *in what order*; a separate mechanism *does* it. This is the right separation, but it means "fully propagator-native IO" in the strongest sense (effects fire during propagation) remains out of reach for non-commutative effects. That's fine — it's a fundamental limit, not a design flaw.

## 1. Indexed Session Types for Multiplexed Protocols

This is about the `CapTP-Call {qid : QuestionId}` pattern in the document — where multiple independent sub-conversations share a single connection, each tracked by an index.

**What we have today**: Prologos has *dependent* session types (`!:` / `?:`) which bind a value in the continuation:

```prologos
?: n Nat    ;; receive a Nat, bind it as 'n' in the continuation
!: n Nat    ;; send a Nat, bind it as 'n' in the continuation
```

These are implemented as `sess-dsend` / `sess-drecv` in `sessions.rkt` with de Bruijn indexing, proper substitution through `substS`, and duality support. The type checker in `typing-sessions.rkt` handles them.

**What indexed sessions would require**: The CapTP multiplexing pattern needs something genuinely different — not "the continuation depends on the value you sent/received" but "there are *N* concurrent sub-sessions, each identified by an index, and sending on one sub-session doesn't affect the others."

There are essentially three approaches in the literature:

**(a) Encoding via dependent types (what we can approximate today)**

You can model the index as a dependent send/recv, but you lose the concurrency structure. The session type becomes sequential — each call must complete before the next starts:

```prologos
session CapTP-Call-Seq
  rec Loop
    ?: qid QuestionId
    !! [CTP-Call qid]
    ?? [CapTP-Result qid]
    Loop
```

This captures the *data* dependency (result indexed by question ID) but not the *multiplexing* property (multiple calls in flight simultaneously).

**(b) Parameterized session types (a genuine extension)**

The real solution from the literature is *parameterized* or *indexed* session types, where the session type itself is indexed by a value:

```markdown
S(i) = !! CTP-Call(i) . ?? Result(i) . end
```

And the multiplexed protocol is a *family* of sessions `{S(qid) | qid ∈ active}`. This is what Katsumata's "Parametric Effect Monads" and Atkey's indexed monads point toward — and it's the approach Gay & Hole and Thiemann & Vasconcelos explore for session type indexing.

For Prologos, this would mean extending the session type system with something like:

```prologos
;; Hypothetical syntax — NOT currently implemented
session-family CapTP-Call [qid : QuestionId]
  !! [CTP-Call qid]
  ?? [CapTP-Result qid]
  end
```

Where the multiplexed connection holds a `Map QuestionId (Session CapTP-Call qid)` — a mapping from active question IDs to their current session state.

**(c) Multi-channel sessions (what our process calculus already supports structurally)**

Our `proc-new` / `proc-par` process calculus can already create multiple channels. The multiplexing could be encoded as *dynamic channel creation* — each question ID gets its own channel:

```prologos
;; Each call creates a new sub-channel with its own session type
proc captp-call [main : CapTP-Steady]
  new [sub : CapTP-Call]       ;; fresh channel for this call
    par
      send main sub            ;; delegate the sub-channel
    and
      send sub call-msg
      recv sub result
      stop
```

This is closer to what π-calculus would do — channel passing rather than multiplexing. It's expressive enough, but the runtime model differs from how CapTP actually works (CapTP uses a single connection with integer slot IDs, not separate channels).

**My assessment**: Option (b) — parameterized session types — is the right long-term answer. We already have the dependent type infrastructure to index types by values. The extension would be to allow session type *definitions* to be parameterized and to track a *family* of concurrent session instances. This is a real research contribution but not a massive implementation lift given our dependent type foundations. It's a natural Phase 1+ feature.

For the Endo document as a specification tool, the dependent send/recv encoding (option a) captures the *data-level* invariant correctly (result matches question ID), and the prose captures the *concurrency* property. The session types don't lie about what they verify — they verify the data flow, while the multiplexing is an operational property.

## 2. Bounded Liveness Guarantees

This is about the pin leak observation — that the `Pin-Guard` session has no guarantee that `Unpin` is ever reached. More generally: can a session type enforce that a protocol *eventually terminates* or *reaches a certain state within bounded time/steps*?

**What we have today**: Our session types verify *safety* properties — "bad things don't happen":
- Protocol compliance (you can't send when you should receive)
- Duality (both endpoints agree on the protocol)
- Completeness (`check-session-completeness` detects sessions not consumed to `end`)
- Deadlock detection (the propagator network detects cycles)

The completeness check is actually a *weak* liveness property — it says "if the process terminates, the session must be at `end`." But it can't say "the process *will* terminate."

**What bounded liveness would require**:

Liveness properties are fundamentally different from safety properties. Safety says "nothing bad happens in any prefix of the trace." Liveness says "something good *eventually* happens." Standard session types give you safety. Liveness requires additional machinery.

There are several levels of increasing power:

**(a) Linear typing (what we already have, partially)**

QTT's multiplicity-1 annotation already gives us a weak liveness guarantee for *values*: a linear value *must* be consumed. The pin example in the document used this:

```prologos
? PinToken [1]     ;; acquire: linear
! PinToken [1]     ;; release: MUST happen
```

This is real and implementable today. If `PinToken` is linear, the type checker will reject any code path that drops it without using it. This catches the "crash and forget to unpin" case *at the level of the function that acquires the pin* — if the function's type says it receives a linear pin, it must return or consume it on every code path.

But it doesn't guarantee the *calling* code ever invokes the function. And it doesn't handle the case where a crash happens *between* the acquire and release at the process level (since crashes aren't represented in the type system).

**(b) Fairness constraints (what our strategy system is designed for)**

The session type implementation plan (`docs/tracking/2026-03-03_SESSION_TYPE_IMPL_PLAN.md`) already sketches a `strategy` declaration with `:fairness` parameters:

```prologos
strategy name :fairness :round-robin :fuel 50000
```

This is an operational guarantee, not a type-level one — it says "the scheduler will be fair," which prevents starvation but doesn't give a compile-time proof of bounded progress.

**(c) Timed session types (a genuine extension)**

The literature has several approaches to adding time bounds to session types:

- **Bocchi, Honda, Tuosto (2014)**: Timed multiparty session types with clock constraints. Each message has a deadline.
- **Neykova & Yoshida (2017)**: Session types with explicit timeouts as session-level constructs.
- **Barbanera & de'Liguoro (2015)**: Session types with retries and timeouts.

In Prologos syntax, this could look like:

```prologos
;; Hypothetical — NOT currently implemented
session Pin-Guard-Bounded
  ? PinToken [1]
  timeout 30s                    ;; MUST reach next step within 30s
    +>
      | :name-established ->
          ! PetName
          ! PinToken [1]
          end
      | :timeout ->              ;; if deadline passes, auto-unpin
          ! PinToken [1]
          end
```

This is a hybrid of session types and timed automata. The `timeout` construct introduces a branch that is taken if the bounded time elapses without the normal branch being selected.

**(d) Progress types / termination metrics (dependent types to the rescue)**

The most Prologos-native approach would use *dependent types* to encode the bound directly in the session type:

```prologos
;; Session indexed by a decreasing measure
session Pin-Guard-Measured {fuel : Nat}
  ? PinToken [1]
  rec [fuel]                     ;; recursion bounded by fuel
    +>
      | :step ->
          ?? Progress              ;; some progress toward naming
          Pin-Guard-Measured {pred fuel}  ;; fuel decreases
      | :done ->
          ! PetName
          ! PinToken [1]
          end
```

Since `fuel` is a natural number that strictly decreases at each recursive call, the recursion must terminate. This is the *well-founded recursion* approach — the session type is structurally decreasing, which guarantees termination.

We already have the dependent type infrastructure for this. What we'd need is:
1. Session type definitions that accept type-level parameters (which ties back to the indexed session types question!)
2. A termination checker that verifies the measure decreases at each recursive unfolding

**My assessment**: Bounded liveness is a spectrum. We can deliver increasing levels of guarantee:

1. **Today**: Linear types prevent resource leaks within a single function scope. `check-session-completeness` verifies sessions reach `end` when the process terminates.

2. **Near-term** (small extension): Timeout branches in session types — a `timeout` choice that is taken if the normal protocol doesn't progress. This is syntactic sugar for a choice with an implicit clock, and the session type checker can verify that the timeout branch properly cleans up resources (releases the linear pin).

3. **Medium-term** (parameterized sessions): Fuel-indexed recursive sessions where termination falls out of the well-founded recursion on the decreasing parameter. This requires indexed session types — the same extension needed for the multiplexing question.

4. **Long-term** (timed session types): Full clock constraints à la Bocchi et al., where each message step has a deadline and the type checker verifies that all deadlines are satisfiable. This is a bigger research investment.

**The interesting connection**: Both questions converge on the same extension — *parameterized session types*. Indexed families (for multiplexing) and fuel-decreasing parameters (for bounded liveness) are both instances of "session types indexed by a value from the dependent type level." Our dependent type infrastructure makes this more natural than in most session type systems, where parametricity has to be bolted on separately.

## The CRDT Hypothesis for Formula Retention

The maintainer is pointing at a fundamental problem in distributed capability systems: **revocation under partition**.

Here's the core tension in Endo's current design:

### How Endo handles retention today

The formula graph uses a **single async mutex** (`formulaGraphJobs`) for all mutations, and GC is a centralized sweep that runs on the daemon. When two peers are connected via CapTP:

- **Retention** is tracked by reference counting: `CTP_DROP` decrements the count when a remote reference is no longer needed
- **GC** runs locally on each daemon, using the reference count + local pet store edges to determine reachability
- **Revocation** happens via `cancel` — the host cancels a formula, which cascades through `thatDiesIfThisDies` dependencies

### What breaks under partition

When two peers are *partitioned* (network goes down), they can't coordinate:

1. **Peer A deletes a reference** (removes a pet name, drops a capability) → locally, the formula has no incoming edges → GC wants to collect it
2. **But Peer B still holds a live CapTP import** (`o-N` slot) pointing to that formula → the reference *should* keep it alive
3. **Under partition, Peer A can't send `CTP_DROP`** to tell Peer B to release, and Peer B can't confirm whether the reference is still valid

This creates a dilemma:
- If A **GCs eagerly**, B will get a dangling reference when the partition heals — a *use-after-free* at the capability level
- If A **retains conservatively** (keep everything any peer might reference), leaked references become permanent — no revocation under partition

And crucially for OCap security: **revocation is a security operation**. If Alice grants Bob a capability and later wants to *revoke* it (cancel the formula), she needs that revocation to be *authoritative*, even if Bob is partitioned. But Bob may have already passed the capability to Carol via promise pipelining. Who wins?

### Why CRDTs are the right lens

CRDTs (Conflict-Free Replicated Data Types) solve exactly this class of problem: *state that must converge without coordination*. The key insight is:

**The retention/revocation state for a formula between two peers is a replicated data structure that must make progress on both sides during partition and converge when they reconnect.**

The specific CRDT structure the maintainer likely has in mind is an **OR-Set** (Observed-Remove Set) or **Add-Wins Set** variant for the set of live references:

```markdown
Peer A's view: {ref₁: alive, ref₂: alive, ref₃: revoked}
Peer B's view: {ref₁: alive, ref₂: alive}
               (hasn't seen ref₃ revocation yet)
```

Under partition:
- A can mark `ref₃` as revoked (locally authoritative — A is the granting host)
- B can continue using `ref₁` and `ref₂` (locally available references)
- When partition heals, they *merge* — and the merge semantics determine whether revocation wins or retention wins

### The design space

There's a fundamental choice:

**(a) Revoke-wins (authority-first)**
- Revocation is an *irreversible* operation (like a tombstone in a G-Set)
- Once the granting host cancels a formula, the revocation propagates to all peers on reconnection
- Peers that used the capability during partition had *provisional* access — their actions may need to be rolled back or at least acknowledged as potentially revoked
- This is the OCap-correct choice: the authority that granted a capability should be able to revoke it unconditionally

**(b) Retain-wins (availability-first)**  
- References remain valid as long as any peer holds them
- Revocation only takes effect when all peers acknowledge it (consensus)
- More available under partition, but undermines the revocation guarantee
- Not OCap-correct — you can't revoke a capability if the holder is partitioned

**(c) Hybrid with explicit epoch/version**
- Each formula has a monotonically increasing *epoch*
- Revocation bumps the epoch; the formula at epoch N is a different capability than at epoch N+1
- Peers can use capabilities at their known epoch; on reconnection, they learn the current epoch
- Stale-epoch references are *expired*, not dangling — a clean semantic distinction

### What a session type analysis reveals

This is exactly where session types with linear resources and dependent types are revealing. Let me sketch the problem:

```prologos
;; The current protocol (implicit, no CRDT)
session Retention-Current
  ;; Host grants a capability to Guest via CapTP
  !! Handle                    ;; export the capability
  rec Active
    &>
      | :guest-uses ->
          ?? CTP-Call            ;; guest calls a method
          !! CTP-Return          ;; host returns result
          Active
      | :host-revokes ->
          !! CTP-DROP            ;; host sends drop
          ;; BUG: what if guest is partitioned? CTP_DROP never arrives.
          end
      | :partition ->
          ;; ??? Protocol has no branch for this.
          ;; Both sides are stuck in Active with divergent views.
```

The missing branch is the tell. There's no protocol step for the partition case — it's a *gap* in the state machine. A CRDT-aware protocol would look like:

```prologos
;; Hypothetical CRDT-aware retention protocol
session Retention-CRDT
  !! Handle
  !! RetentionEpoch                ;; initial epoch (monotonic counter)
  rec Active
    &>
      | :guest-uses ->
          ?? [CTP-Call epoch]       ;; call tagged with guest's known epoch
          +>
            | :epoch-valid ->
                !! CTP-Return
                Active
            | :epoch-stale ->
                !! EpochExpired     ;; "your epoch is old, re-sync"
                Active
      | :host-revokes ->
          ;; Bump epoch locally — takes effect immediately
          ! RevocationRecord       ;; persisted locally (CRDT state)
          ;; On reconnection, merge CRDT states
          Active                   ;; stay in Active — revocation is async!
      | :reconnect ->
          ?? PeerRetentionState    ;; receive peer's CRDT state
          ! MergedRetentionState   ;; send merged result
          ;; Revocations from merge take effect now
          Active
      | :disconnect ->
          end
```

The key differences:
1. **Revocation doesn't require the peer to be present** — it's a local operation that produces a CRDT record
2. **Calls are epoch-tagged** — stale references get a clean "expired" response, not a dangling pointer
3. **Reconnection includes a CRDT merge** — both sides reconcile their retention state
4. **The protocol stays in `Active` after revocation** — because the revocation is *local* until merged

### What Prologos can contribute here

This is a beautiful convergence point for Prologos's capabilities:

1. **Session types** model the protocol state machine and expose the missing partition branch
2. **Linear types** can enforce that revocation records are *eventually delivered* (a linear `RevocationToken` that must be consumed by either a merge or a timeout)
3. **Dependent types** can index the protocol by epoch, making stale-epoch calls a *type error*
4. **CRDTs are lattice-based** — and Prologos already has a full lattice trait hierarchy (`Lattice`, `HasTop`, `BoundedLattice`, `GaloisConnection`). A CRDT for retention state would be a specific lattice instance where `join` is the merge operation
5. **Propagator networks** are fundamentally CRDT-like (monotone cell refinement = join-semilattice = CRDT merge). A retention cell in a propagator network would naturally support the convergent semantics

In fact, the propagator network *is* a CRDT in a deep sense — each cell's value moves monotonically upward through a lattice, and `net-cell-write` does `join` on merge. Cross-network propagators (which we've designed but not yet built for distributed execution) would give you the exact distributed convergence semantics that a retention CRDT needs.

So the maintainer's hypothesis is, in my view, *correct and important*. The formula system is missing a CRDT because the current design assumes a centralized daemon with synchronized reference counting. Once you have partitioned peers, you need convergent state — and the right abstraction is a CRDT for the retention/revocation lifecycle of each formula's remote references.

## The Structural Isomorphism

The thing that's been staring at us — and I think you've been circling toward this for a while — is that our propagator-based session type checker is *already* a model checker. It's just not labeled as one. Let me make the correspondence explicit:

| Model Checking | Prologos (today) |
|---|---|
| Kripke structure (states + transitions) | Session type (continuations + send/recv/choice) |
| Transition relation | Session decomposition propagators |
| State labeling | Session cell values (`sess-send`, `sess-end`, etc.) |
| Fixpoint computation | `run-to-quiescence` |
| Safety property (AG¬bad) | Contradiction detection (`sess-top`) |
| Weak liveness (AF end) | `check-session-completeness` |
| Counterexample trace | ATMS derivation chains / `session-op` traces |
| State space branching | `+>` / `&>` choice branches |
| Hypothetical reasoning | ATMS assumptions + `believed` worldview |
| Abstract interpretation | Galois connections (α/γ) across domains |

The reason this works is that **CTL model checking IS lattice fixpoint computation**. The standard algorithms:

- **EF φ** (exists a path eventually reaching φ) = `lfp X. φ ∨ ∃next(X)` — least fixpoint
- **AG φ** (all paths always satisfy φ) = `gfp X. φ ∧ ∀next(X)` — greatest fixpoint  
- **A(φ U ψ)** (on all paths, φ until ψ) = `lfp X. ψ ∨ (φ ∧ ∀next(X))` — bounded liveness as least fixpoint

These are *all* monotone functions over lattices computed to fixpoint. That's literally what `run-to-quiescence` does. The cells refine monotonically, the propagators compute the monotone functions, and the scheduler drives to quiescence (= fixpoint).

So when we added `add-send-prop`, we added a transition-relation propagator. When we added `add-duality-prop`, we added a symmetry constraint. When we added `check-session-completeness`, we added AF(end). These are all model-checking operations expressed in propagator network vocabulary.

## What We're Missing (and What Would Complete It)

The gap is that we're currently limited to *structural* properties — properties that fall out of session type shape-matching. We can check "does this process match this protocol" but we can't yet check *arbitrary temporal properties* over the protocol state machine. The missing pieces:

**1. Temporal property specification language**

Right now, properties are implicit in the session type structure. Making them explicit would look something like:

```prologos
session Pin-Guard
  ? PinToken [1]
  rec Pinned
    +>
      | :name-established -> ! PetName . ! Unpin . end
      | :abandon -> ! Unpin . end

;; Temporal properties (what we'd add):
property pin-never-leaked
  :session Pin-Guard
  :AG [implies pinned [AF unpinned]]    ;; "always: if pinned, eventually unpinned"

property gc-safe-while-pinned
  :session Pin-Guard
  :AG [not [and pinned gc-collecting]]  ;; "never: pinned AND gc running simultaneously"
```

The `:AG`, `:AF`, `:implies` vocabulary maps to CTL formulas, which map to fixpoint computations, which map to propagators. So specifying a temporal property would *literally* generate new propagators that are added to the session-checking network.

**2. A "temporal property" propagator factory**

A function `add-temporal-prop : Network → SessionCell → CTLFormula → Network` that takes a CTL formula and generates the appropriate propagators:

- `AG φ` → greatest fixpoint propagator (starts at top, refines down)
- `EF φ` → least fixpoint propagator (starts at bottom, refines up)
- `A(φ U ψ)` → least fixpoint with guard propagator
- `AG(φ → AF ψ)` → nested fixpoint (the response property)

Each of these is a *specific propagator pattern* — not a new mechanism, but a new way of wiring existing cells and propagators. The machinery is all there.

**3. Branching-aware state exploration**

For CTL (branching-time), we need to explore *all* paths through `+>`/`&>` choices. The ATMS is exactly the right tool here — each choice branch becomes an ATMS assumption, and we check properties under each assumption set. The `solve-all` mechanism already enumerates consistent assumption sets. CTL's universal path quantifier (A) = property holds under all ATMS worldviews. Existential (E) = property holds under at least one worldview.

## The Propagator Network as Universal Substrate

Here's the thing that keeps striking me about the architecture we've built: **every new propagator makes the entire system more capable**. This is not a metaphor. It's a direct consequence of the monotone, composable nature of propagator networks:

- Adding the *duality propagator* gave us protocol compatibility checking for free.
- Adding the *effect ordering propagator* (Architecture D) gives us causal ordering verification for free.
- Adding a *temporal property propagator* would give us LTL/CTL model checking for free.
- Adding a *fairness propagator* would give us bounded liveness guarantees for free.

Each addition composes with all previous propagators through the shared lattice. The ATMS handles the hypothetical branching. The Galois connections handle cross-domain reasoning. The scheduler handles incremental recomputation.

This is why the propagator architecture was the right bet. It's not just a type inference engine — it's a *general-purpose formal verification substrate* that happens to also do type inference.

## Where This Diverges from Classical Model Checking

Two things make the Prologos approach potentially different from classical model checking:

**1. Infinite-state systems via abstraction.** Classical model checking works on finite Kripke structures. Session types with `rec` have potentially infinite unfoldings. But we already have the Galois connection infrastructure for abstract interpretation — `α` maps concrete (potentially infinite) domains to abstract (finite) domains. This means we can model-check infinite-state protocols by abstracting them through a Galois connection to a finite domain, running the fixpoint on the abstract domain, and mapping results back through `γ`. This is the abstract model checking approach (Clarke, Grumberg, Long — LICS 1994), and our infrastructure is built for it.

**2. Incremental verification.** Classical model checkers start from scratch. Our propagator network is incremental — adding a new property or modifying a session type only re-runs affected propagators. This is enormously valuable for iterative development: you write code, the model checker runs in the background, you get feedback in real-time. No "run the model checker for 20 minutes and wait."
