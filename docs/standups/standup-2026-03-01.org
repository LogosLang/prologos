## The World Token Was the Wrong Starting Point

Looking back at the design doc through the lens you're describing, I started from Mercury/Clean's model and worked forward. But Mercury and Clean don't have session types, and they don't have QTT. They needed the World token *because they lacked the richer machinery we already have*. Starting from their model is reasoning from a deficit we don't share.

The World token is a single coarse-grained capability: "you can do I/O." It's the equivalent of giving every function `root` access because the OS only has one permission level. Mark Miller's entire body of work is about why that's the wrong model — the principle of least authority (POLA) demands that you hand out *exactly* the capabilities needed, nothing more.

## Capability Security as a Type-Level Concern

What you're describing — fine-grained capabilities, checked at compile time via QTT + session types — is more fundamental than an I/O library design. It's an *operating system model expressed in the type system*.

Let me think about what this looks like concretely.

In the E language and object-capability model, the key insight is: **an object reference IS the capability**. You can't forge references, you can only receive them by being handed one. Access control isn't a separate ACL system bolted on — it's intrinsic to how references propagate through the system.

In Prologos, the analogue would be: **a linear value IS the capability**. A `:1` value can't be duplicated (no ambient authority amplification), can't be forged (only created by the system or handed to you), and must be consumed (no dangling permissions). QTT already gives us the three levels:

- `:0` — you can *name* the capability (for type-level reasoning) but not *use* it
- `:1` — you have it, you must use it exactly once, you can't share it
- `:w` — unrestricted (this is the "public" or "ambient" capability — use carefully)

This is already more expressive than most capability systems. The `:0` multiplicity is particularly interesting — it's a *phantom capability*, a proof that something exists without granting access. That's exactly what you want for audit trails and provenance without authority leakage.

## Fine-Grained Capabilities vs. World Token

Let me contrast these concretely:

**World token (coarse)**:

read-file  : World :1 -> Path -> <World * Result String IOError>
write-file : World :1 -> Path -> String -> <World * Result Unit IOError>
connect-db : World :1 -> Path -> <World * Result Connection IOError>


Every function gets the same `World`. A function that only needs to read one file could secretly write to another, connect to a database, launch missiles. The type system says "does I/O" but not *what* I/O.

**Fine-grained capabilities**:

read-file  : ReadCap :1 -> Path -> <ReadCap * Result String IOError>
write-file : WriteCap :1 -> Path -> String -> <WriteCap * Result Unit IOError>
connect-db : DbCap :1 -> Path -> <DbCap * Result Connection IOError>


Now the type signature is a *security policy*. A function that takes `ReadCap` provably cannot write or connect to a database. The principle of least authority is enforced by the type checker.

But this raises the ergonomics question you're probing: isn't threading `ReadCap` through every function call painful?

## The Ergonomics Challenge — Learning from QTT

Your analogy to QTT is incisive. QTT does enormous lifting with three symbols (`:0`, `:1`, `:w`) because it's **metadata on things you're already writing**. You don't write separate linearity annotations — you annotate the binders you already have. The cognitive load is marginal because it piggybacks on existing syntax.

The question is: can capability passing follow the same pattern? Can we make capabilities metadata on something you're already writing, rather than extra parameters you thread?

Here's a thought. What if capabilities are metadata on *modules*, not on individual function calls?

```prologos
ns my-app
  :caps [Fs :read, Db :connect]     ;; this module's capability requirements

;; Inside this module, read-file and connect-db just work
;; No capability threading in function signatures
defn process-data [path db-path]
  let data = [read-file [path path]]
  let conn = [connect [path db-path]]
  ...
```

The `:caps` declaration is like a session type preamble — it states what capabilities this module needs. The compiler verifies that the caller provides them. But individual functions within the module don't thread capabilities manually.

This is analogous to how `ns foo` auto-imports the prelude. The capability requirements are declared once and flow through implicitly.

**But** — and this is where the OCaml Eio model is instructive — you lose granularity. If the module declares `:caps [Fs :read, Fs :write]`, every function in that module has both capabilities, even if most only need read. We've reintroduced ambient authority at the module level.

## A Middle Path: Capability Parameters as Implicit Arguments

What if capabilities use the same mechanism as trait constraints — implicit dictionary passing?

```prologos
spec read-file {Fs : ReadCap} : Path -> Result String IOError
spec write-file {Fs : WriteCap} : Path -> String -> Result Unit IOError
```

The `{Fs : ReadCap}` is an implicit erased binder — just like `{A : Type}` or `{Eq A}`. The capability is resolved by the trait resolver at the call site. The caller must have the capability in scope; if they don't, it's a type error.

This is low-friction in the same way trait constraints are. You don't manually thread `ReadCap` — the compiler resolves it. And the capability *is visible in the spec* — so you can read a function's security policy from its type signature.

```prologos
;; This function's type says exactly what it can do:
spec process-data {Fs : ReadCap} {Db : ConnectCap} 
  : Path -> Path -> Result Report IOError

;; This function only reads — you can tell from the type:
spec summarize {Fs : ReadCap} : Path -> Result Summary IOError

;; This function is pure — no capabilities at all:
spec transform : List String -> Report
```

The capability requirements propagate upward through the call chain, exactly like trait constraints do. A function that calls `read-file` automatically requires `ReadCap`. A function that calls both `read-file` and `connect-db` requires both. The compiler infers this.

And here's where it connects to session types: a *session* can declare what capabilities each endpoint is granted:

```prologos
session FileService
  ;; server has ReadCap + WriteCap
  ;; client receives ReadCap only (least authority)
  ...
```

## The SEL4 / Capability Kernel Analogy

seL4's insight is that capabilities are *the only way to name a kernel object*. There's no global namespace. You can't say "open file /etc/passwd" — you can only say "use this capability I was handed."

In Prologos, the analogue would be: I/O functions don't take `Path` (a global name) — they take a capability that *includes* the path authority:

```prologos
;; A FileCap is not just "can read" — it's "can read THIS specific file/directory"
data FileCap : Path -> Type

spec read-file : FileCap p :1 -> Result String IOError
```

Now `FileCap` is *indexed by the path it grants access to*. Dependent types make the capability's scope visible in the type. A `FileCap (path "/tmp")` grants access to `/tmp` and nothing else. This is exactly seL4's model, expressed in the type system.

This is where Prologos's unique combination genuinely offers something no other language does: **dependent types + linear types + session types = capability security with compile-time verification of authority scope**.

## What I Think the Design Doc Should Become

The I/O library design doc is premature. What needs to come first is:

1. **A Capability Security principles document** — articulating POLA, the E/seL4 model, and how QTT + dependent types + session types express it. This becomes a principle in `LANGUAGE_VISION` or its own principles doc.

2. **A design cycle on capabilities as implicit parameters** — working through the ergonomics of `{Fs : ReadCap}` vs. module-level `:caps` vs. explicit threading. What's the QTT-level lightweight notation that makes this low-burden?

3. **Session types design** (as you noted — a separate effort, probably prerequisite) — because session types govern the *protocols* of capability granting and I/O sequencing.

4. **Then** the I/O library falls out naturally as an *application* of the capability system, not a design in its own right.

The I/O library research is still valuable — it maps the problem space and identifies what we need. But the design doc's §4 (the actual type designs) should be rewritten once the capability model is clear.

## The Open Question I'd Put Back to You

The implicit-parameter model (`{Fs : ReadCap}`) is the lightest-weight mechanism I can see that preserves fine-grained authority in function signatures. But it has a tension: implicit parameters are `:0` (erased) by default in our system. Capabilities arguably should be `:1` (linear) — you should have to *give up* a capability, not just *mention* it.

Is there a middle ground? Perhaps capabilities are `:0` (erased, compile-time-only proof) for *authority checking*, but `:1` (linear, runtime) for *authority transfer*? That is: proving you *have* a capability costs nothing at runtime (erased), but *handing* a capability to someone else is a linear operation (the compiler tracks delegation chains).

That would mirror seL4 exactly: using a capability doesn't consume it (you keep your reference), but minting a *derived* capability (with reduced authority) is a tracked operation.

I think this distinction — and whether QTT's three multiplicities are sufficient to express it, or whether we need something richer — is the core design question for the capability layer.

### On Implicit Capabilities — Refinement

The `{Fs : ReadCap}` implicit parameter model has the right ergonomic shape but needs refinement on what "implicit capability" means mechanically.

In our trait system, `{Eq A}` means: "there exists an `Eq` instance for `A`, resolve it from the instance registry." The compiler finds the dictionary and passes it. The user never threads it.

For capabilities, `{Fs : ReadCap}` would mean: "there exists a `ReadCap` in the caller's capability environment." But where does the capability *come from*? Trait instances come from `impl` declarations — they're globally available facts. Capabilities are the opposite: they must be *explicitly granted*, never ambient.

This is the fundamental tension: **traits are ambient (any code can use `Eq Int`), but capabilities must be non-ambient (only code explicitly granted `ReadCap` can use it).**

So the resolution mechanism is different. Traits resolve from a global registry. Capabilities resolve from a *lexical scope* — you have a capability only if someone handed it to you in your call chain.

This actually maps to a pattern we already have: **function parameters**. A capability you receive as a parameter is lexically scoped. The question is whether we can make the *passing* implicit while keeping the *granting* explicit.

Here's a sketch:

```prologos
;; Main entry point — capabilities are granted here
defn main [!fs : Fs, !net : Net]    ;; system grants these at startup
  let report = [process-data (path "/data") (path "/api")]
  [write-file (path "/out/report.txt") report]

;; process-data needs Fs (read) and Net — compiler infers from body
spec process-data {Fs : ReadCap} {Net : HttpCap}
  : Path -> Path -> Result Report IOError
defn process-data [data-path api-path]
  let csv-data = [read-file data-path]           ;; uses ReadCap
  let api-data = [http-get api-path "/endpoint"]  ;; uses HttpCap
  [build-report csv-data api-data]

;; build-report is pure — no capabilities needed, visible in type
spec build-report : CsvData -> ApiData -> Report
```



The capabilities propagate downward through the call chain (from `main` to `process-data`), are resolved implicitly (like trait dictionaries), but are only available if explicitly granted at the entry point. The type signature of every function is a capability manifest — you can read exactly what authority it requires.

On `:0` for Authority Checking vs `:1` for Authority Transfer
This distinction is sharp and I think correct. Let me formalize it:

-- `:0` (erased) capability — a compile-time proof that you have authority. The type checker verifies it, but no runtime token exists. This is the common case: `read-file` needs proof you have `ReadCap`, but doesn't consume it. You can call `read-file` many times with the same `ReadCap`. This is like using an seL4 capability — invoking it doesn't destroy it.

-- `:1` (linear) capability — a runtime token representing transferable authority. Handing a ReadCap `:1` to another function gives away your read access. This is seL4's capability transfer (or E's eventual-send of a capability reference). You can also attenuate — derive a weaker capability from a stronger one:

;; Attenuation: derive a read-only cap from a read-write cap
spec attenuate-read : FsCap :1 -> <ReadCap :1 * FsCap :1>

Wait — that gives back both, which means we're duplicating. Linear attenuation would be:

;; Attenuate: consume full cap, produce read-only cap (irreversible)
spec attenuate-read : FsCap :1 -> ReadCap :1

This is capability narrowing — you give up full access and receive restricted access. The linear type ensures you can't keep the original. This is a powerful pattern that no other language I know of expresses at the type level.

:w (unrestricted) capability — ambient authority. This is what we want to avoid for security-sensitive capabilities. But it's appropriate for things that genuinely should be universally available (e.g., PureCap — the capability to perform pure computation, which everything has).
So QTT's three multiplicities map precisely to three capability modes:

QTT	Capability Meaning	seL4 Analogue
:0	Compile-time authority proof (erased)	Capability slot exists but is type-checked only
:1	Transferable, consumable authority	Capability transfer / delegation
:w	Ambient authority (use sparingly)	Global objects (like the idle thread cap)
The default for capability parameters in specs would be :0 — compile-time proof, no runtime cost. You opt into :1 only when you need to transfer or attenuate authority.

On Hiding Incidental Complexity
The spec/functor metadata model is the right analogy. Today, dependent types are "incidental complexity" that users encounter only when they read spec signatures:

spec map {A B : Type} {C : Type -> Type} {Foldable C} {Buildable C}
  : [A -> B] -> C A -> C B

Most users write defn map [f xs] ... and never think about {Foldable C}. The implicit resolution machinery handles it.

Capabilities should work the same way. Most users write:

defn process [path]
  let data = [read-file path]
  [transform data]

And the compiler infers that process requires {Fs : ReadCap}. The user never writes it unless they want to be explicit (or want to read the inferred signature for audit purposes).

The compiler is the capability auditor. It computes the capability closure of every function — the full set of capabilities transitively required. This is visible on demand (like inferred types) but not syntactically required.

The incidental complexity — "which capabilities does this code need?" — is managed the same way we manage "which trait constraints does this code need?" — by inference.

Where This Gets Interesting: Capability Composition
Capabilities should compose the way traits compose — via bundles:

;; Fine-grained capabilities
trait ReadCap        ;; can read files
trait WriteCap       ;; can write files  
trait HttpCap        ;; can make HTTP requests
trait DbCap          ;; can connect to databases

;; Composite capabilities (bundles)
bundle FsCap = ReadCap + WriteCap
bundle NetCap = HttpCap
bundle AllIO = FsCap + NetCap + DbCap

And capability attenuation is subtyping: if you have FsCap, you can pass it where ReadCap is needed (but not vice versa). This is our existing subtype mechanism applied to capabilities.

What Needs to Happen Next
I think the design doc should be updated to:

Retitle the I/O section as preliminary/research — the type designs in §4 are placeholders pending the capability model
Add a new section (or separate doc) on the Capability Security model — articulating POLA, the :0/:1/:w mapping, implicit resolution, and the main as authority root
Flag session types as a prerequisite — the I/O protocol layer (Phase 5-6 in the current doc) requires session types, which need their own design cycle
Identify the first concrete deliverable — probably: capability traits, implicit capability resolution through the existing trait solver, and a proof-of-concept read-file that requires {Fs : ReadCap} at the type level
The core research question is: can our existing implicit argument resolution (the trait solver) serve as the capability resolution engine, with the constraint that capabilities are lexically scoped (not globally registered)? If yes, the implementation is lighter than it appears. If no, we need a separate resolution mechanism.

### Logic Engine / Propagator Network for Capability Inference

This is the most exciting angle and I think you're onto something deeper than the design doc captured.

The "backward propagation through the call graph" described in §4.6 is **literally describing a propagator network computation**. Let me make this explicit:

**The capability lattice:**
- **Elements**: Sets of capability types (e.g., `{ReadCap}`, `{ReadCap, HttpCap}`, `{FsCap, DbCap}`)
- **Bottom**: `{}` (pure — no capabilities)
- **Join**: Set union (`{ReadCap} ⊔ {HttpCap} = {ReadCap, HttpCap}`)
- **Top**: `{SysCap}` (all capabilities)
- **Monotonicity**: Capability sets only grow — learning that a callee needs `HttpCap` can only ADD to the caller's set, never remove

This is a textbook lattice, and computing capability closures is a textbook **monotonic fixed-point computation** — exactly what the propagator network does.

**The propagator network formulation:**

```markdown
Cell per function:
  cell(main)         : CapabilitySet   ;; starts with {SysCap} (declared)
  cell(process-data) : CapabilitySet   ;; starts with {} (unknown)
  cell(read-file)    : CapabilitySet   ;; starts with {ReadCap} (declared in spec)
  cell(write-file)   : CapabilitySet   ;; starts with {WriteCap} (declared in spec)
  cell(transform)    : CapabilitySet   ;; starts with {} (pure)

Propagators from call edges:
  process-data calls read-file  → propagator: cell(process-data) ⊇ cell(read-file)
  process-data calls write-file → propagator: cell(process-data) ⊇ cell(write-file)
  process-data calls transform  → propagator: cell(process-data) ⊇ cell(transform)
  main calls process-data       → [verification] cell(main) ⊇ cell(process-data)

Run to quiescence:
  cell(read-file)    = {ReadCap}        (declared)
  cell(write-file)   = {WriteCap}       (declared)
  cell(transform)    = {}               (pure)
  cell(process-data) = {ReadCap, WriteCap}  ← propagators fired
  cell(main) ⊇ {ReadCap, WriteCap} ?   ← SysCap >: FsCap >: both ✓
```

The quiescence of the propagator network IS the fixed point of capability inference. Mutual recursion? Handled automatically — the propagator network iterates until stable. Adding a new callee? The cell merges in the new requirements and propagators fire forward.

**Cost/benefit:**

| | Benefit | Cost |
|---|---------|------|
| **Reuse** | Propagator network, lattice traits, quiescence detection already built | Integration surface — another client of the Logic Engine |
| **Correctness** | Lattice-based fixed point handles mutual recursion, cycles, incremental updates automatically | Need CapabilitySet as a Lattice instance (straightforward — it's PowerSet) |
| **Architecture** | Capability inference becomes "just another propagator problem" — same mental model as type inference | Dependency on Logic Engine being stable (Phases 1-3 done, which is sufficient) |
| **Incrementality** | Propagator networks are naturally incremental — editing one function re-propagates only affected cells | Setup overhead for creating the network per module |
| **Verification** | The "main ⊇ transitive closure" check is just reading the final cell values after quiescence | The verification step (checking satisfaction, not just computing closures) needs subtype-aware comparison, which is outside the propagator network itself |

The key insight: **the "backward propagation through call graph" in §4.6 was unconsciously describing propagator behavior**. Making this explicit and using the actual Logic Engine infrastructure would be architecturally cleaner than a hand-rolled fixed-point pass.

The one subtlety: propagators propagate *forward* (callee → caller), but the *verification* goes *backward* (does main's declared set subsume the computed closure?). The propagator network computes the closures; a separate pass checks them. This is the natural two-phase pattern: propagate to quiescence, then read results.

What's your intuition — does framing capability inference as a propagator network problem feel right? It would mean the Logic Engine isn't just for the relational language — it's the *general infrastructure* for all monotonic fixed-point computations in the compiler.

### The `:kind` / Multiplicity Override Question

## Design Consideration: Propagator-Based Capability Inference

### The Core Insight

Capability inference through the call graph is a **monotonic fixed-point computation over a PowerSet lattice**. The Logic Engine — specifically the persistent propagator network (Phase 2) with lattice-based cells — was built for exactly this class of problem. The type inference refactoring (Phase 8) already proved that the propagator network scales to compiler-internal fixed-point computations. Capability inference is a structurally simpler instance of the same pattern.

### The Lattice


CapabilitySet = PowerSet(CapabilityType)

  {SysCap}                                         ← top
     │
  {FsCap, NetCap, DbCap, SpawnCap, ClockCap}
     │
  ... all subsets ...
     │
  {ReadCap}    {WriteCap}    {HttpCap}    ...      ← singletons
     │
    {}                                              ← bottom (pure)


- **Bottom**: `{}` — pure function, no capabilities
- **Join**: Set union — `{ReadCap} ⊔ {HttpCap} = {ReadCap, HttpCap}`
- **Leq**: Subset — `{ReadCap} ⊆ {ReadCap, HttpCap}`
- **Monotonicity**: Capability sets only grow as we discover more callees

This is a straightforward `Lattice` instance — simpler than the `Interval` or `Sign` lattices we already have. We could implement it as a `SetLattice` specialization (we already have `SetLattice` in the lattice library).

### The Propagator Network Architecture

For a module with functions `f₁, f₂, ..., fₙ`:


Phase 1: Build the network
─────────────────────────
For each function fᵢ:
  cell(fᵢ) := new cell with initial value:
    - declared_caps(fᵢ)    if fᵢ has explicit capability params in its spec
    - {}                    if fᵢ has no declared capabilities

For each call edge (fᵢ calls fⱼ):
  add propagator:
    inputs:  [cell(fⱼ)]
    outputs: [cell(fᵢ)]
    fire:    cell(fᵢ) := cell(fᵢ) ⊔ cell(fⱼ)     ;; union the callee's caps into caller

Phase 2: Run to quiescence
──────────────────────────
  run-to-quiescence(network)
  ;; BSP scheduler fires propagators until no cell changes
  ;; Result: each cell(fᵢ) holds the FULL capability closure of fᵢ

Phase 3: Verify
───────────────
For the authority root (main):
  check: cell(main).declared_caps ⊇ cell(main).inferred_caps
  ;; main's explicit params must cover the transitive closure

For each call edge (fᵢ calls fⱼ) where fᵢ has explicit caps:
  check: cell(fᵢ).declared_caps ⊇ cell(fⱼ).inferred_caps
  ;; explicit declarations are also verification points


### How This Compares to the Hand-Rolled Approach

The design doc's §4.6 described a "post-type-checking pass" with a hand-rolled algorithm:

> 1. After all functions are type-checked, collect capability constraints
> 2. Build a call graph from the module's functions
> 3. For each function, compute the transitive union of all callee capability requirements
> 4. Check that each function's explicit or inferred capability set is satisfied by its caller

This IS the propagator network, just described procedurally. The differences:

| Aspect | Hand-rolled pass | Propagator network |
|--------|-----------------|-------------------|
| **Mutual recursion** | Must detect cycles and iterate to fixed point manually | Handled automatically by BSP scheduler |
| **Incrementality** | Recompute from scratch on any change | Write to one cell → only affected propagators re-fire |
| **Backtracking** | Manual state save/restore | Keep old network reference (O(1), structural sharing via CHAMP) |
| **Provenance** | Must build separately | ATMS integration gives it for free — "why does f require ReadCap?" answered by tracing the propagator chain |
| **Correctness argument** | "The loop terminates because capability sets are bounded and monotonic" | Follows from lattice theory — quiescence IS the fixed point |
| **Infrastructure reuse** | New code for capability-specific fixed-point | Reuses `propagator.rkt`, `Lattice` trait, `run-to-quiescence` |

### The ATMS Angle: Provenance for Free

This is where it gets really interesting. If we thread the ATMS (Phase 5) through the capability inference network, every capability requirement gets an **assumption set** — the set of leaf declarations that justify it.

```markdown
cell(process-data) = {ReadCap, WriteCap}
  └── assumption: ReadCap  came from cell(read-file) via call edge at line 23
  └── assumption: WriteCap came from cell(write-file) via call edge at line 25
```

This means the compiler can answer:
- **"Why does `process-data` require `ReadCap`?"** → Because it calls `read-file` (line 23), which declares `{ReadCap}` in its spec.
- **"What would happen if I removed the `write-file` call?"** → `process-data` would only require `{ReadCap}`, not `{WriteCap}`.
- **"Which functions transitively contribute to `main`'s capability requirements?"** → Trace the ATMS dependency tree from `cell(main)`.

This is the "compiler as capability auditor" from the principles document, realized via existing infrastructure. Without the ATMS, we'd need to build provenance tracking from scratch. With it, it's a configuration choice when constructing the network.

### Speculative Resolution via ATMS

There's an even deeper use: **speculative capability resolution**. When the compiler encounters a function whose capability requirements are ambiguous (e.g., it calls a higher-order function whose callback might or might not require capabilities), the ATMS can explore both hypotheses:

```markdown
;; map-with-io takes a function that MAY require capabilities
spec map-with-io : (A -> B) -> List A -> List B

;; If the callback is pure: map-with-io requires {}
;; If the callback requires {ReadCap}: map-with-io requires {ReadCap}
```

The ATMS can create two assumptions (callback-pure, callback-effectful), propagate both, and resolve the ambiguity when the actual callback is known. This is the same speculative type-checking pattern we already use for Church fold attempts and union types (via `save-meta-state`/`restore-meta-state!`), but expressed as ATMS hypotheticals rather than imperative state save/restore.

### Concrete Integration Points

Where this plugs into the existing codebase:

1. **After type checking** (in `driver.rkt`): Build the capability propagator network from the module's typed functions. Each function's spec provides declared capabilities; each function's body provides call edges.

2. **Cell construction** (using `propagator.rkt`): `net-new-cell` with a `SetLattice CapabilityType` merge function. Initial values from specs.

3. **Propagator construction** (using `propagator.rkt`): `net-add-propagator` for each call edge. Fire function: set union.

4. **Quiescence** (using `propagator.rkt`): `run-to-quiescence` — the BSP scheduler we already have.

5. **Verification** (new code, ~50-100 lines): Read final cell values, check subsumption against declared params, emit errors/warnings.

6. **Provenance** (optional, using `atms.rkt`): Thread ATMS assumptions through cell writes. Query assumption sets for error messages and IDE tooling.

### What This Means for the Roadmap

Phase 5 (Capability Inference) in the design doc would change from "hand-rolled post-type-checking pass" to "build a propagator network from the call graph and run to quiescence." The implementation is **simpler** because it reuses existing infrastructure, and **more powerful** because it gets incrementality, backtracking, and provenance for free.

The estimated scope actually *shrinks*: instead of ~200 lines of new fixed-point code, it's ~100 lines of network construction + ~50 lines of verification. The heavy lifting (scheduling, quiescence, CHAMP persistence, ATMS tracking) is already done.

### Open Design Question

One tension: the type inference propagator network (Phase 8 refactoring) lives in the elaborator and runs *during* type checking. The capability inference network would run *after* type checking (it needs the fully-typed call graph). Should these be two separate networks, or could capability inference be wired into the same network as type inference?

My instinct: **separate networks**. The type inference network operates on type-level cells (metavariables, unification constraints). The capability inference network operates on set-level cells (capability sets). They have different lattices, different propagators, and different lifecycles. Sharing a network would complect them unnecessarily. Two networks, each with its own quiescence, is cleaner.

But there's a subtlety: if capability inference influences type inference (e.g., the compiler needs to know a function's capability signature to type-check a caller), then the two networks might need to communicate. This is exactly the **cross-domain propagation** infrastructure from Phase 6c (Galois connections). A cross-domain propagator could bridge the type-inference network and the capability-inference network, allowing capability information to flow into type checking when needed.

---

That's the full design consideration. The key takeaway: the Logic Engine isn't just "nice to have" for capability inference — it's the **architecturally correct** solution. The hand-rolled pass described in §4.6 was reinventing propagator network behavior without the name. Making it explicit gives us incrementality, provenance, and correctness from lattice theory, while reducing implementation scope.

What's your read on this? Particularly interested in your thoughts on the two-network vs. shared-network question, and whether the ATMS provenance angle is worth the integration cost upfront vs. adding it later.

## The Key Insight: IO Library Phase 1 Doesn't Need Phase 7

Looking at the IO library design (`docs/tracking/2026-03-01_1200_IO_LIBRARY_DESIGN.md`), Phase 1 core file I/O works with **simple capabilities**:

```prologos
spec read-file : {fs :0 ReadCap} -> Path -> Result String IOError

```

Dependent capabilities (`FileCap (p : Path)`) are a Phase 2+ IO concern — path-specific access control for fine-grained security. The IO library's actual blocking dependency is:

1. **Opaque FFI types** (`:opaque` for `Handle`, `Path`) — not yet in `foreign.rkt`
2. **The `with-open` macro** — resource lifetime via QTT linearity
3. **Simple capability gating** — already works via Phase 6

## Two Viable Paths

**Path A: IO Library Phase 1 first, then Phase 7**
- Get `read-file`, `write-file`, `read-lines` working with simple `ReadCap`/`WriteCap`
- Build the `:opaque` FFI extension needed anyway
- Then implement Phase 7 and retrofit path-specific access in IO Phase 2
- *Pro:* Delivers usable IO sooner; validates the cap system end-to-end with real IO
- *Con:* May need to revisit API design when adding dependent caps

**Path B: Phase 7 first, then IO Library**
- Build dependent capabilities, then design IO with full path-specific access from day one
- *Pro:* Cleaner IO API; no retrofitting
- *Con:* Delays usable IO; Phase 7 is designed in isolation without a concrete consumer

## Dependent Type Infrastructure Assessment

The good news is the dependent type machinery is **fully sufficient** for Phase 7:

| What Phase 7 Needs | Status | Evidence |
|---|---|---|

1. **CapabilitySet extension** — currently `cap-set (seteq of symbols)`. Dependent caps need `cap-set (seteq of (symbol | type-expr))`. This changes `cap-set-join`, `cap-set-subsumes?`, and how the propagator network handles them.

2. **Dependent resolution** — when checking `{FileCap "/data" :0}`, the resolver must unify the path index against what's in scope. The unification machinery exists; the plumbing to invoke it during capability resolution doesn't.

3. **Minting semantics** — how does `FsCap` narrow to `FileCap path`? The design suggests explicit minting functions: `mint-file-cap : FsCap -> (p : Path) -> FileCap p`. This is straightforward.

These are the real considerations, regardless of sequencing:

**1. CapabilitySet representation for dependent caps**

The current lattice is `PowerSet(Symbol)` — dead simple. With dependent caps, you need to represent `FileCap "/data"` vs `FileCap "/etc"` as distinct elements. Options:
- **Extend cap-set** to hold type expressions (uses existing unification for joins)
- **Flatten to string keys** at inference time (simpler but lossy)
- **Separate tier** for dependent vs simple caps

I'd recommend extending `cap-set` to hold type expressions. The unification machinery handles everything else.

**2. How dependent caps interact with the propagator network**

When `f` calls `g : {FileCap p} -> ...` with a concrete path, what propagates to `f`'s cap cell? Options:
- `FileCap "/data"` — concrete (requires path to be known at analysis time)
- `FileCap ?p` — symbolic (constraint variable; deferred to inference)
- `FsCap` — widened (conservative; loses precision)

For IO library purposes, concrete paths are most useful. For general analysis, symbolic constraints are more powerful but harder.

**3. IO library design tension: World token vs capability passing**

The IO design doc mentions both Mercury's `World :1` token and fine-grained capabilities. These aren't contradictory — you can have `{fs :0 FsCap} -> World -> World` — but the interaction needs explicit design. Dependent caps make this richer: `{fc :0 FileCap "/data"} -> World -> (World, String)`.

This gives you the type-level dependent capabilities for the IO library without the complexity of dependent cap inference.
