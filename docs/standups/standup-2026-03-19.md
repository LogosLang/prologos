---

## What's at play

**System 1: Type-level unification** (`unify.rkt`) — the elaborator's unifier. 37 pattern-match cases in the classifier. Handles Pi, Sigma, App, Vec, Eq, union types, universe levels, HKT normalization, applied metas with Miller's pattern condition. Deeply integrated with the propagator network — postponement, wakeup, readiness propagators, TMS retraction. This is the system that runs during type checking, 123 call sites across 11 files.

**System 2: Solver-level unification** (`relations.rkt`) — the `solve`/`defr` runtime. 5 cases. Logic vars are symbols, terms are keyword-headed lists, substitution is a `hasheq`. No propagator integration at all. Walk, unify-terms, done. This is what our benchmarks exercise.

**The design doc's cell-tree proposal (§2)** is clearly aimed at **System 1** — it talks about decompose-pi/sigma/app, meta cells, zonking, ground-expr?, TMS retraction. These are all type-level concerns. The benchmarks we wrote exercise **System 2**.

That's gap number one: **the benchmarks and the architecture target different systems.**

---

## The gaps

**1. What exactly is PUnify replacing?**

The design doc says "PUnify replaces the current algorithmic unification with structural unification: types as trees of cells." But the fire function in `unify.rkt` already does lazy tree decomposition — `decompose-pi` creates sub-cells and wires propagators between children. The current system *already* builds cell trees emergently. PUnify's proposal is to make the tree primary rather than emergent. The question is: **how much does that actually buy us?**

The 37-case classifier in `classify-whnf-problem` would need to become... what? A constructor-tag comparison? That only works if all types are uniformly represented as tagged n-ary trees. But types aren't uniform — Pi has a binder (codomain depends on domain variable), Sigma has a binder, universe levels have their own lattice, unions are sets. The tree representation has to handle all of this, which means the "simplification" might just move the complexity from the classifier into the tree construction.

**2. The CHAMP structural sharing bet**

The design doc's core claim: "CHAMP structural sharing means comparing two cell-trees that share common subtrees is O(diff), not O(size)." This is true for *data* comparison, but unification isn't comparison — it's *merging with binding*. When you unify `Pi(?A, ?B)` with `Pi(Int, String)`, you're not comparing trees, you're writing `Int` to `?A`'s cell and `String` to `?B`'s cell. The CHAMP sharing wins when two types share subtrees (e.g., two functions both returning `Vec Nat n`), but our benchmarks show that the *bottleneck* is walk chains and substitution growth, not tree comparison.

**3. The solver vs type-level disconnect**

Our adversarial benchmarks stress the solver (`relations.rkt`), which is a flat substitution-based system. The cell-tree architecture targets the elaborator (`unify.rkt`), which already has propagator integration. If PUnify is about improving *type checking* performance, we need type-level benchmarks — things like deeply polymorphic function applications, large trait resolution chains, heavy union type checking. The solver benchmarks tell us about `solve`/`defr` performance, which is useful for the language but isn't what the cell-tree architecture would change.

**4. Lazy vs eager tree creation (Q1 in §10)**

This is still open and it matters a lot. Currently, decomposition is lazy — `decompose-pi` only fires when something needs the domain or codomain separately. Eager tree creation would allocate cells upfront, enabling fan-in readiness from the start but at the cost of many cells that may never be needed. The performance budget says 3-5× more cells per meta. For a typical command with 50 metas, that's 150-250 cells vs 50. We've seen from Track 7 that cell allocation cost is real.

**5. The coexistence question**

§5.3 says "toggle parameter, belt-and-suspenders, gradual migration." But 123 call sites across 11 files means this isn't a clean swap. The current system has *deep entanglement* — the classifier knows about specific AST node types, the decomposition creates specific sub-cell patterns, the zonker knows how to walk specific structures. A parallel implementation means maintaining both for a long transition period.

---

## The options as I see them

**Option A: Full cell-tree architecture (as designed)**
Replace the type representation with cell-trees from the ground up. Big bet, high risk, potentially elegant. But the 37-case classifier suggests types aren't as uniform as the tree model assumes. Binder handling (Pi, Sigma, let) requires special treatment that doesn't map cleanly to "connect children."

**Option B: Targeted cell-tree for the solver only**
Replace `relations.rkt`'s flat substitution model with cell-trees. This is a much smaller scope — 5 cases, no binders, no universes, no HKT. The solver's flat terms (keyword-headed lists) map naturally to tagged n-ary trees. This would directly improve the `solve`/`defr` path that our benchmarks exercise. But it doesn't touch the type-level unifier, which is where the real complexity lives.

**Option C: Incremental improvements to the existing type-level unifier**
Instead of replacing the architecture, address the specific bottlenecks:
- Early termination on mismatch (our benchmarks showed late-failure = success cost)
- Path compression for walk chains (currently O(n))
- Better cell allocation strategy (reuse, pooling)
- Fan-in readiness as an add-on to existing cells (without full tree restructuring)

This is lower risk, lower payoff, but directly measurable against the benchmarks we have.

**Option D: Cell-tree for new constructs only**
Don't migrate existing Pi/Sigma/App decomposition. Instead, use cell-trees for *new* type constructs going forward — row types, session types, refinement types. The existing 37-case classifier stays for current types; new types get the cleaner cell-tree representation. Eventually the old cases could be migrated, but there's no forced timeline.

---

## What I think the real question is

The design doc frames PUnify as an architectural improvement — "the data structure IS the algorithm." That's aesthetically compelling. But our benchmarks revealed that the actual performance bottlenecks are in the *solver* (walk chains, DFS backtracking, per-command overhead), not in type-level unification. The type checker is fast. The solver is where users will feel the pain.

So: **is PUnify about making the type checker architecturally cleaner, or about making the solver faster?** Those lead to very different designs.

## The Three Lattices Already in Play

You already have three lattice implementations for cells, each with a different structure:

1. **`type-lattice.rkt`** — nominally a flat lattice (`bot → T → top`), BUT `type-lattice-merge` calls `try-unify-pure` which recursively decomposes Pi, Sigma, app, etc. So it's **already structural** — it just doesn't know it formally.

2. **`mult-lattice.rkt`** — genuinely flat: `{bot, m0, m1, mw, top}`. Five elements, no internal structure.

3. **`term-lattice.rkt`** — the most interesting one. It's **explicitly tree-structured**: `term-ctor(tag, sub-cells)` where sub-cells are cell-ids. The merge function decomposes by constructor tag, and sub-cell unification is delegated to the propagator network. This is already a cell-as-lattice.

## What "Cell Is a Lattice" Means

Right now, a cell **holds a value from** a lattice. The merge function is the lattice's join operation. But there's a deeper structure hiding here.

Consider a Pi type cell. Its value is `expr-Pi(mult, domain, codomain)`. When `type-lattice-merge` processes two Pi values, it:
- Matches the multiplicity (mult lattice)
- Recursively unifies the domain (type lattice)
- Opens and recursively unifies the codomain (type lattice)

This means the Pi cell's value space isn't really a *flat* lattice — it's a **product lattice**:

```markdown
Pi-Lattice = MultLattice × TypeLattice × TypeLattice(dependent on domain)
```

And the overall type lattice is a **sum** over constructor tags:

```markdown
TypeLattice = FlatLattice(Atom)
            ⊔ Σ_{tag ∈ {Pi, Sigma, App, Eq, Vec, ...}} ∏_{i=1..arity(tag)} TypeLattice_i
```

This is a **recursive lattice**: a tagged sum of products of itself. And that's exactly what a tree is.

## The Lattice Embedding Connection

From the lattice catalog (§ Lattice Embeddings):

> An embedding `e : L₁ ↪ L₂` is an injective lattice homomorphism. It places L₁ as a sublattice of L₂.

Here's where it gets interesting for PUnify. Right now we have:

**Flat view**: `TypeLattice_flat = {bot} ∪ AST_Types ∪ {top}` — a flat lattice where merge does structural comparison

**Tree view**: `TypeLattice_tree` — a recursive product lattice where each node is a cell and merge is pointwise

These are connected by a **Galois connection**:

```markdown
α : Tree → AST     (fold the tree back into a flat AST)
γ : AST → Tree     (unfold an AST into sub-cells on demand)
```

Where `α ∘ γ = id` (fold after unfold recovers the AST) and `γ ∘ α ⊑ id` (unfold after fold may have more structure — sub-cells).

This means **Option C (lazy cell-tree) is actually the γ injection** — you embed the flat lattice into the tree lattice on demand. And `type-lattice-merge` already computes in the tree lattice via `try-unify-pure`; it just doesn't persist the sub-cells.

- The **sub-cells** are projections into the component lattices of the product
- Merge at Ctor level is the **tag check** (sum lattice: same summand required)
- Merge at sub-cell level is **delegated to the network** (product lattice: pointwise)

## Benefits of Thinking Trees-as-Lattices

### 1. Composition Is Free

The lattice catalog documents product, sum, and lifting constructions. If cell-trees are formally recognized as recursive lattice products, you get:

- **Product monotonicity**: If each component merge is monotone, the product merge is automatically monotone. No separate correctness argument needed per constructor.
- **Sum consistency**: Tag mismatch → top. This is just the standard sum lattice join.
- **Lifting**: `bot` = "this cell hasn't been decomposed yet" → lifting the product lattice with a bottom element. This IS the on-demand creation model: a cell starts in the lifted `⊥`, then gets "populated" into the product when first decomposed.

### 2. Lattice-Indexed Families for Dependent Types

The catalog's § Lattice-Indexed Families directly addresses your situation:

> In dependent type theory, the type of a later argument may depend on the value of an earlier one. This creates dependent lattice structures: the lattice used for a cell may depend on the resolved value of another cell.

For Pi types, the codomain lattice depends on the domain value (the opened binder). This makes the Pi lattice a **dependent product**:

```markdown
PiLattice = Σ(m : MultLattice) × Σ(A : TypeLattice) × TypeLattice(A)
```

The `get-or-create-sub-cells` pattern with its fresh fvar for binder opening is computing the **fiber of the dependent product** at the domain point. The `cell-decomps` registry caches these fibers.

### 3. Galois Connections Between Representations

If you ever need to **refine** a cell's representation (e.g., promote from flat to tree, or from tree to tree-with-more-structure), lattice embeddings give you the tool. This could matter for:

- **Widening**: If a cell-tree gets too deep (pathological type), widen by collapsing sub-trees back to flat representation
- **Narrowing**: After widening, narrow back to recover precision in hot paths
- **Domain change**: Promote a type cell from `TypeLattice` to `TypeLattice × MultLattice` when QTT analysis needs multiplicity info at the same node

### 4. The On-Demand Pattern IS Lazy Lattice Projection

The `get-or-create-sub-cells` pattern from `elaborator-network.rkt` is:

```markdown
project : Cell(Σ-tag ∏ Li) → (Cell(L1), Cell(L2), ...)
```

This is **lazy lattice projection**: you don't compute the full product lattice upfront. You compute projections only when a propagator needs a sub-cell. The registry ensures each projection is computed at most once (idempotent).

This maps beautifully: the Option C design (lazy cell-tree) is exactly **lazy evaluation of a recursive lattice product**, where projections are memoized.

## The Key Insight: What Changes

Currently, `type-lattice-merge` does the structural decomposition **inline** — it recurses through the AST, and the result is a single AST value. The structural knowledge is used and discarded.

In the cell-tree model, structural decomposition **persists** — the sub-lattice structure is materialized as cells on the network. This means:

1. **Future unifications reuse the structure**: A Pi cell that's been decomposed once has its domain and codomain cells. Any later unification against that Pi reuses those cells — the decomposition is memoized in the network topology.

2. **Information flows bidirectionally**: If the domain cell gets new information from one source and the codomain from another, both propagate through the Pi cell's product lattice. With flat merge, you'd need to re-unify the whole Pi.

3. **TMS retracts at sub-cell granularity**: Speculation can retract just the codomain solution without touching the domain. With flat merge, you retract the entire Pi value.

## What Doesn't Map Cleanly

### The `ok` and `level` Cases (50% of classifications)

The `ok` case (14%) is "terms are identical" — no lattice decomposition needed. The `level` case (36%) is numeric lattice operations on universe levels — not tree-structured. Together they're 50% of classifications. The tree-as-lattice model adds nothing for these.

### Contravariance

In some type systems, function argument types are contravariant — the "direction" of information flow reverses. This breaks the simple product lattice model. Prologos uses structural unification (not subtyping), so this isn't a current issue, but it's a theoretical limitation if subtype reasoning is ever added to the merge function.

### Non-Deterministic Width

Unlike `term-ctor` which has a fixed arity per tag, union types have variable width. `<Int | String | Bool>` is a 3-way union, while `<Int | String>` is 2-way. The sum-of-products lattice model doesn't directly cover variable-width structures. You'd need a powerset or multiset lattice construction.

---

## Bottom Line: Is This Useful for PUnify?

**Yes, but primarily as a correctness framework, not an implementation change.** The tree-as-lattice perspective gives you:

1. **A proof argument**: If cell-tree operations are product lattice projections and merges, monotonicity of the whole system follows from monotonicity of the components. This is otherwise hard to argue for ad-hoc structural decomposition.

2. **Design vocabulary**: "Get-or-create-sub-cells is lazy projection" is much cleaner than "we cache decomposition." It tells you what invariants to maintain (idempotent, order-preserving, fiber-consistent for dependent types).

3. **A migration path for enrichment**: If you later want cells to carry richer info (e.g., type + multiplicity + provenance), lattice product composition tells you exactly how to extend the merge function.

4. **Confidence that `term-lattice.rkt` is the right model**: The existing `term-ctor(tag, sub-cells)` pattern is formally the right structure. PUnify's type-tree cells should follow the same pattern, parameterized by the 11 type constructor tags instead of the narrowing constructor tags.

What it **doesn't** give you is a performance win or a simpler implementation. The actual code for PUnify Phase 3 (Pi decomposition as sub-cells) looks the same whether you think of it as "ad-hoc structural decomposition" or "lazy product lattice projection." The lattice framing is about correctness assurance and compositional reasoning, not about generating different code.

## On the Musing: GC, Provenance, and Propagator Liveness

The observation about threshold propagators is deeper than it first appears.
