Let me expand on the type sketches, using our `spec`/`:where`/`:forall` metadata pattern as the model. I'll walk through what we'd need to express at each level, with concrete examples from our actual architecture.

---

**Level 0: Lattice — modeled on `trait`/`impl`**

```prologos
;; Lattice as a trait (the theoretical substrate)
trait Lattice {L : Type}
  spec join L L -> L
  spec bot -> L

;; Simple: user defines a sum type, implements Lattice
type Color := red | green | blue | unknown | mixed

impl Lattice Color
  defn join
    | unknown x -> x
    | x unknown -> x
    | x x       -> x        ;; same constructor
    | _ _       -> mixed    ;; different constructors
  defn bot -> unknown

;; With optional top (for contradiction detection)
trait BoundedLattice {L : Type}
  :extends [Lattice L]
  spec top -> L

;; With widening (for termination on infinite-height lattices)
trait WideningLattice {L : Type}
  :extends [Lattice L]
  spec widen L L -> L
```

This uses our existing trait/impl machinery. Nothing new needed syntactically — `Lattice` is just another trait. The lattice laws (commutativity, associativity, idempotency of join) could be expressed as `:where` constraints on the trait itself:

```prologos
trait Lattice {L : Type}
  :where [Commutative join]      ;; join a b = join b a
  :where [Associative join]      ;; join (join a b) c = join a (join b c)
  :where [Idempotent join]       ;; join a a = a
  :where [Identity join bot]     ;; join bot a = a
  spec join L L -> L
  spec bot -> L
```

The `:where` clauses are proof obligations. For simple cases (finite enumerations), the logic engine can verify them by exhaustive case analysis. For complex cases, the user provides a proof or the compiler defers to runtime checking.

**Level 1: Cell — parameterized type**

```prologos
;; Cell as a parameterized type
;; The type system enforces: writes are joins, not replaces
type Cell {L : Type} :where [Lattice L]
  :multiplicity :ω        ;; default: unrestricted reads

;; Linear cell (session channel discipline)
type LinearCell {L : Type} :where [Lattice L]
  :multiplicity :1         ;; exactly one read

;; TMS-tagged cell (speculation-aware)
type TMSCell {L : Type} :where [Lattice L]
  :tagged-by Assumption    ;; entries tagged with assumption IDs
```

Here the `:multiplicity` metadata maps directly to our QTT annotations. `Cell` and `LinearCell` are distinguished by their QTT multiplicity — this is already how we distinguish session channels from shared state. The `TMSCell` variant carries assumption tags, which is what Track 8A delivered.

**Level 2: Propagator — `spec` with `:where Monotone`**

```prologos
;; A propagator's type signature
spec add-prop : Propagator [Cell NatLattice, Cell NatLattice] -> Cell NatLattice
  :where [Monotone add-prop]
  :stratum S0

;; Implementation
defn add-prop [x y]
  [join [read x] [read y]]

;; The Monotone constraint unfolds to:
;; ∀ (a1 a2 b1 b2 : NatLattice).
;;   a1 ≤ a2 → b1 ≤ b2 → add-prop(a1,b1) ≤ add-prop(a2,b2)

;; For structural decomposition (SRE form):
spec decompose-pi : Propagator [Cell TypeLattice] -> [Cell TypeLattice, Cell TypeLattice]
  :where [Monotone decompose-pi]
  :stratum S0
  :form Pi          ;; registers as SRE structural form for Pi
  :components [domain, codomain]
```

The `:form` and `:components` metadata is the SRE form registry information. When the compiler sees `:form Pi :components [domain, codomain]`, it registers this as a polynomial summand: `y^{domain, codomain}` for the Pi case. This is the "types-ready" data model — the typing information is carried as metadata.

Note the multi-output syntax: `-> [Cell TypeLattice, Cell TypeLattice]` for fan-out. This is where polynomial functors matter — the return type is a product of cells, not a single cell.

**Level 3: Network — `schema`/`network` with `embed`/`connect`**

```prologos
;; Network interface as schema (the polynomial functor type)
schema AdderNet
  :inputs  [x : Cell NatLattice, y : Cell NatLattice]
  :outputs [sum : Cell NatLattice]

;; Network implementation
network my-adder : AdderNet
  propagator add-prop x y -> sum

;; Composition
schema CombinedNet
  :inputs  [a : Cell NatLattice, b : Cell NatLattice, c : Cell NatLattice]
  :outputs [result : Cell NatLattice]

network combined : CombinedNet
  embed adder1 : AdderNet
  embed adder2 : AdderNet
  connect a -> adder1.x
  connect b -> adder1.y
  connect adder1.sum -> adder2.x
  connect c -> adder2.y
  connect adder2.sum -> result
```

The `schema` is the polynomial functor type — it specifies positions (outputs) and directions (inputs). The `network` is the implementation. `embed` instantiates a sub-network. `connect` wires outputs to inputs with type checking (lattice types must match).

This mirrors our existing `schema`/`impl` pattern for traits, but applied to networks.

**Level 4: Bridge — `bridge` with α/γ and adjunction**

```prologos
;; Bridge as a first-class construct
bridge TypeToMult : Bridge TypeLattice MultLattice
  :alpha type->mult-alpha      ;; L → M
  :gamma mult->type-gamma      ;; M → L
  :where [Adjunction alpha gamma]

;; The adjunction unfolds to:
;; α ∘ γ ∘ α = α  (lower closure)
;; γ ∘ α ∘ γ = γ  (upper closure)

;; For our actual pure bridge fire functions:
bridge TraitResolution : Bridge TypeLattice TraitLattice
  :alpha [fn [type-cells : [Cell TypeLattice ...]]
            [resolve-trait-key type-cells]]
  :gamma [fn [dict-cell : Cell TraitLattice]
            [extract-type-constraints dict-cell]]
  :where [Adjunction alpha gamma]
  :stratum S0                ;; fires during base propagation
```

The `:where [Adjunction alpha gamma]` is the key — it's a dependent type constraint that the compiler verifies. For simple bridges (finite lattices), exhaustive checking works. For complex bridges, the user provides a proof sketch or the compiler uses the logic engine.

**Level 5: Stratification — declarative configuration**

```prologos
;; Stratum as a type
type Stratum := S-neg1 | S0 | S1 | S2

;; Stratification as declarative configuration
stratification ElabLoop
  :strata [S-neg1 S0 S1 S2]
  :fiber S-neg1 [TypeLattice, SessionLattice, MultLattice]
    :mode retraction     ;; non-monotone
  :fiber S0 [TypeLattice, SessionLattice, MultLattice]
    :mode monotone       ;; all propagators must be Monotone
  :fiber S1 [ReadinessLattice]
    :mode monotone
  :fiber S2 [ActionSet]
    :mode commit         ;; non-monotone: commits + retracts
  :transitions
    S-neg1 -> S0 : barrier
    S0 -> S1 : threshold      ;; Right Kan demand
    S1 -> S2 : dispatch
    S2 -> S-neg1 : retract
  :fuel 100
  :where [WellFounded ElabLoop]
```

This is the most "configuration-like" level — it reads almost like a YAML/TOML declaration but with full type checking. The `:where [WellFounded ElabLoop]` constraint verifies termination. The `:mode monotone` on S0 means the compiler checks that every propagator assigned to S0 (via `:stratum S0` on its spec) has a `Monotone` proof.

**Level 6: Interaction — Kan extensions as metadata**

```prologos
;; Speculative forwarding (Left Kan)
interaction speculative : LeftKan S0 S1
  :forward [partial-fixpoint -> approximate-result]
  :where [Monotone forward]
  :where [Sound forward]     ;; speculative results are lower bounds

;; Demand-driven (Right Kan)
interaction demand-driven : RightKan S0 S1
  :threshold [needed-cells -> targeted-thresholds]
  :where [Minimal threshold]  ;; only demanded cells trigger

;; Combined adjoint pair
interaction pipelined : AdjointPair speculative demand-driven
  :where [Adjoint speculative demand-driven]
```

**Level 7: Parameterized networks via `functor`**

```prologos
;; A network parameterized by lattice type (HKT!)
functor SolverNet {L : Type} :where [Lattice L]
  schema
    :inputs  [constraints : Cell [Set [Constraint L]]]
    :outputs [solution : Cell L]

network type-solver : SolverNet TypeLattice

network mult-solver : SolverNet MultLattice

```

This uses our existing HKT infrastructure (Track 8 B3 delivered `{L : Type}` binders). The `functor` keyword creates a parameterized network — a polynomial functor parameterized by a lattice type. Instantiation fills in the type parameter. This is how users would define generic constraint solvers that work over any lattice.

    ## The Core Tension
    
    β-reduction creates new terms. Each reduction step takes (λx.M) N and produces M[N/x] — a fresh copy of M with N substituted for x. This is allocation-intensive: you're copying the body of the lambda, threading the argument through every occurrence of x.

    Reduction creates. Propagation refines. These are fundamentally different operations.

For a term that needs 50 β-reductions to reach normal form, that's 50 strata — each with its own fixpoint computation, its own quiescence check, its own barrier.     - Edges connect ports of agents (like your propagator connections)
    - Reduction is local graph rewriting: when two agents are connected through their principal ports, they interact — the pair is replaced by a new subgraph

## The Core Typing Problem

A propagator network has several dimensions that need typing:

**The substrate**: what lattice does a cell operate in? What are the join, bottom, top? What is the information ordering?

**The connectivity**: which cells does a propagator read from? Which does it write to? What is the relationship between input types and output types?

**The monotonicity contract**: is a propagator monotone with respect to its input lattices? Does it preserve the ordering? This is the fundamental safety property — a non-monotone propagator in a monotonic layer would break your entire architecture.

**The bridging structure**: when two networks on different lattices are connected by a Galois connection, what are α and γ? Do they form an adjunction? Is the connection a quantale morphism?

**The stratification structure**: when networks are layered, what is the transition between strata? Is it monotone within fibers? Where are the non-monotone barriers? What recovery mechanism applies?

**The interaction pattern**: does a network use speculation (Left Kan)? Demand-driven evaluation (Right Kan)? Both? What are the assumptions being managed?

Each of these is a dimension of the "type" of a network. A fully typed network carries guarantees along all dimensions. An incorrectly composed network — wrong lattice, non-monotone propagator in a monotone layer, Galois connection that isn't actually an adjunction — would be caught by the type system.

## The Building Blocks: A Type Theory for Networks

### Level 0: Lattice Types

The most fundamental type is the lattice. Everything else builds on this.

A lattice type specifies:
- A carrier type (the values the lattice holds)
- A partial order (the information ordering)
- A join operation (how information combines)
- A bottom element (no information)
- Optionally: a top element (contradiction), a meet operation, a widening operator

In Prologos surface syntax, using progressive disclosure:

```prologos
;; Simple: just declare a lattice with a carrier type
lattice NatInfo : Nat | unknown | contradiction

;; More explicit: specify the operations
lattice NatInfo
  carrier: Nat | unknown | contradiction
  join: nat-info-join
  bot: unknown
  top: contradiction

;; Full specification for experts
lattice NatInfo : Lattice
  spec [a : NatInfo, b : NatInfo] : NatInfo
    :where [JoinSemilattice NatInfo]
    :where [BoundedBelow NatInfo]
    :where [BoundedAbove NatInfo]
```

The type system guarantees: anything declared as a `lattice` satisfies the lattice laws. The `join` must be commutative, associative, idempotent, and have `bot` as identity. These can be checked by your dependent type system (the laws are propositions, and the implementations must provide proofs — or the compiler can attempt to verify them automatically via your logic engine).

The progressive disclosure: most users write `lattice NatInfo : Nat | unknown | contradiction` and the compiler infers the join (structural union), bot (unknown), and top (contradiction). Experts can override any of these.

### Level 1: Cell Types

A cell is typed by its lattice:

```prologos
;; A cell holding type information
cell my-type-cell : Cell TypeLattice

;; A cell holding a set of assumptions
cell my-assumptions : Cell (PowerSet Assumption)

;; A cell holding a tropical cost
cell my-cost : Cell TropicalNat
```

`Cell L` is a parameterized type: "a mutable container whose values live in lattice L and can only be updated by lattice join." The type system enforces: you can only write to a `Cell L` with a value of type `L`, and the write is a join, not a replace. (Unless you're in an S(-1) retraction stratum, which has a different type — more on that below.)

### Level 2: Propagator Types

Here's where it gets interesting. A propagator's type should capture:

- What it reads (input cells and their lattices)
- What it writes (output cells and their lattices)
- The monotonicity guarantee

The natural type for a propagator is a **dependent function type between cells**, with a monotonicity constraint:

```prologos
;; A propagator that reads from two Nat cells and writes to one
propagator add-prop : Propagator [Cell NatLattice, Cell NatLattice] (Cell NatLattice)
  :where [Monotone add-prop]

;; The Monotone constraint says:
;; if input1 ≤ input1' and input2 ≤ input2' (in NatLattice ordering)
;; then add-prop(input1, input2) ≤ add-prop(input1', input2') (in NatLattice ordering)
```

This is where your dependent types earn their keep. The `Monotone` constraint is a *dependent type* — it's a proposition about the propagator's behavior that the type system can verify. For simple propagators (lattice join, projection, constant), monotonicity is derivable automatically. For complex propagators, the user provides a proof, or the compiler attempts verification via the logic engine.

For progressive disclosure:

```prologos
;; Simple: just declare inputs and output, monotonicity is checked automatically
propagator add-prop [x : Cell Nat, y : Cell Nat] -> Cell Nat
  [plus (read x) (read y)]

;; The compiler infers:
;;   - reads from x and y
;;   - writes to the output cell
;;   - checks that plus is monotone w.r.t. NatLattice
;;   - if not monotone, type error with explanation
```

### Level 3: Network Types (Polynomial Functors)

A network is a collection of cells and propagators with a typed interface. The interface specifies which cells are "input" (written by the external environment), which are "output" (read by the external environment), and which are "internal" (hidden).

This is exactly a **polynomial functor**. A polynomial functor P(y) = Σ_{i ∈ I} y^{B_i} describes:
- I: the set of "positions" (output cells — one for each output the network exposes)
- B_i: the set of "directions" at each position (input cells — the information the network needs to compute output i)

In Prologos terms:

```prologos
;; A network type: what it exposes
network-type AdderNet
  inputs:  [x : Cell NatLattice, y : Cell NatLattice]
  outputs: [sum : Cell NatLattice]

;; A network implementation
network my-adder : AdderNet
  cell sum : Cell NatLattice
  propagator add-prop [x, y] -> sum
    [plus (read x) (read y)]
```

The polynomial functor structure means: **networks compose by plugging outputs into inputs.** If network A has output `sum : Cell NatLattice` and network B has input `x : Cell NatLattice`, you can compose them by connecting A's sum to B's x. The type system checks that the lattice types match.

```prologos
;; Compose two networks
network combined : CombinedNet
  embed a : AdderNet
  embed b : MultiplierNet
  connect a.sum -> b.x    ;; type-checked: both are Cell NatLattice
```

The `connect` is a cell-tree merge (PUnify). The type system ensures the lattice types are compatible. If they're not — connecting a `Cell NatLattice` to a `Cell BoolLattice` — you get a type error.

For progressive disclosure: most users just define networks with inputs and outputs, and compose them with `connect`. The polynomial functor structure is invisible. Experts can manipulate the polynomial functor directly for meta-programming.

### Level 4: Galois Connections as Bridge Types

A Galois connection between lattices L and M is typed as:

```prologos
;; A bridge between two lattice types
bridge TypeToSession : GaloisConnection TypeLattice SessionLattice
  alpha: type-to-session-alpha    ;; L -> M (abstraction)
  gamma: session-to-type-gamma    ;; M -> L (concretization)
  :where [Adjunction alpha gamma] ;; α ∘ γ ∘ α = α, γ ∘ α ∘ γ = γ
```

The `Adjunction` constraint is a dependent type — it's a proposition about the relationship between α and γ. The compiler can verify it for simple bridges (where α and γ are defined in terms of lattice operations) or require a proof for complex ones.

Bridges compose: if you have a bridge L ↔ M and a bridge M ↔ N, the composite bridge L ↔ N is automatically derived (composition of adjunctions is an adjunction). The type system verifies this.

```prologos
;; Compose bridges
bridge TypeToEffect : GaloisConnection TypeLattice EffectLattice
  via: [TypeToSession, SessionToEffect]
  ;; compiler verifies: composition of adjunctions is an adjunction
```

For progressive disclosure:

```prologos
;; Simple: just define the two directions
bridge TypeToSession
  forward:  [type-val -> session-extract type-val]
  backward: [session-val -> type-inject session-val]
  ;; compiler checks adjunction laws automatically
```

### Level 5: Stratification as Type-Level Control Flow

Strata are typed by their **fiber type** (what lattice the stratum operates in) and their **transition type** (how information moves between strata).

```prologos
;; A stratum type
stratum S0 : Stratum
  fiber: TypeLattice × SessionLattice × MultLattice
  mode: monotone     ;; all operations within S0 are monotone

stratum S-1 : Stratum
  fiber: TypeLattice × SessionLattice × MultLattice
  mode: retraction   ;; non-monotone: can decrease cell values

;; A stratification: ordered sequence of strata with typed transitions
stratification ElabLoop : Stratification
  strata: [S-1, S0, S1, S2]
  transitions:
    S-1 -> S0 : NonMonotone (retraction barrier)
    S0  -> S1 : Monotone    (threshold detection)
    S1  -> S2 : Monotone    (action dispatch)
    S2  -> S-1: NonMonotone (resolution commitment)
  :where [WellFounded ElabLoop]
  :where [FuelBounded ElabLoop 100]
```

The type system guarantees: within each monotone stratum, all propagators are monotone (Level 2 typing). Non-monotone transitions are explicitly marked. The stratification is well-founded (no infinite loops without fuel bounds). The entire Layered Recovery Principle is encoded in the type.

For progressive disclosure:

```prologos
;; Simple: just declare strata and the compiler infers transition types
stratification MyLoop
  monotone: [propagation, readiness]
  barrier:  [commitment]
  ;; compiler checks: monotone strata contain only monotone propagators
  ;; compiler checks: barriers are well-founded
```

### Level 6: Fixpoint Types (Inductive and Coinductive)

This is where lfp and gfp enter.

An inductive propagator type (lfp) says: "this network converges from below. Start at ⊥, propagate upward, reach a least fixpoint." This is your standard propagator semantics.

A coinductive propagator type (gfp) says: "this network converges from above. Start at ⊤, propagate downward, reach a greatest fixpoint." This is dual — useful for liveness properties, infinite structures, and your WF-LE bilattice.

```prologos
;; Inductive: least fixpoint (standard)
network my-solver : Network
  fixpoint: lfp    ;; start at bot, go up
  ;; guarantees: all propagators are inflationary (output ≥ input in lattice)

;; Coinductive: greatest fixpoint
network my-liveness : Network
  fixpoint: gfp    ;; start at top, go down
  ;; guarantees: all propagators are deflationary (output ≤ input)

;; Mixed: your WF-LE bilattice
network my-wf-solver : Network
  fixpoint: mixed [lfp truth-ordering, gfp knowledge-ordering]
  ;; the bilattice has two orderings; lfp on one, gfp on the other
```

The type system ensures: an lfp network contains only inflationary propagators. A gfp network contains only deflationary propagators. A mixed network specifies which ordering each propagator respects. Putting a deflationary propagator in an lfp network is a type error.

### Level 7: Interaction Patterns (Kan Extensions)

Speculative evaluation (Left Kan) and demand-driven evaluation (Right Kan) are typed as interaction patterns between strata:

```prologos
;; Speculative forwarding: stratum n+1 gets early approximation from n
interaction speculative : LeftKan S0 S1
  forward: [partial-fixpoint -> approximate-result]
  :where [Monotone forward]
  ;; guarantees: speculative results are sound lower bounds

;; Demand-driven: stratum n computes only what n+1 needs
interaction demand-driven : RightKan S0 S1
  threshold: [needed-cells -> targeted-thresholds]
  :where [Minimal threshold]
  ;; guarantees: only demanded cells trigger forwarding

;; Combined: adjoint pair
interaction pipelined : AdjointPair (LeftKan S0 S1) (RightKan S0 S1)
  ;; compiler verifies: left and right are adjoint
  ;; guarantees: speculative + demand-driven = waiting for full fixpoint
```

For progressive disclosure:

```prologos
;; Simple: just declare the interaction style
interaction between S0 and S1 : speculative + demand-driven
  ;; compiler generates the appropriate forwarding propagators
```

## The Operad Structure: Composing All of This

The multi-input nature of propagators means the composition of all these pieces is best described by an **operad**. An operad captures "operations with multiple inputs and one output, that compose by plugging outputs into inputs."

In your system, the operad has:
- **Colors** (types): lattice types, cell types, network interface types
- **Operations** (multi-morphisms): propagators (multiple input cells, one output cell), bridges (two lattices, two functions), stratification transitions (one stratum to another)
- **Composition**: plugging outputs into inputs, type-checked by lattice compatibility

The operad structure gives you a clean answer to: "what does it mean to compose a bridge with a stratification with a network with a Kan extension?" Each of these is an operation in the operad. Composition is well-defined when the types match. The type system checks the types.

For the user, this means: **any well-typed composition of the building blocks produces a correct network.** You can't accidentally put a non-monotone propagator in a monotone stratum (the types prevent it). You can't connect a NatLattice cell to a BoolLattice cell (the types prevent it). You can't compose a bridge that isn't an adjunction (the dependent types verify the laws).

## The Type-Theoretic Foundation

What makes all of this work as a coherent type system rather than a grab-bag of annotations:

**Lattice types are types.** `Lattice L` is a type class (trait) that any type can implement. The lattice laws are propositions in your dependent type system. Implementing `Lattice` for a type means providing `join`, `bot`, and proofs of the laws.

**Monotonicity is a type-level property.** `Monotone f` where f : L → M is a proposition: for all x ≤ y in L, f(x) ≤ f(y) in M. Your dependent type system can express this. Your logic engine can attempt to verify it automatically.

**Networks are types.** A network type is a polynomial functor — a dependent type that specifies the input/output interface. Network composition is function composition of polynomial functors, type-checked by the dependent type system.

**Bridges are types.** A Galois connection is a pair of functions with a dependent type constraint (the adjunction laws). The type system verifies the laws.

**Stratifications are types.** A stratification is a dependent type that specifies the strata, their fiber types, their transitions, and the well-foundedness condition. The type system verifies all conditions.

**Everything composes via the operad structure.** The operad's composition law is: well-typed compositions of well-typed pieces produce well-typed wholes. This is the "correct by construction" guarantee you want.

**QTT linear types govern resource usage.** A cell with multiplicity :1 (linear) must be read exactly once — this is your session type channel discipline. A cell with multiplicity :ω (unrestricted) can be read any number of times — this is ordinary shared state. The QTT annotations on cells tell the type system how the cell can be used, and the type system enforces it.

**On operads and fan-out.** You're right that operads describe single-output operations. A propagator with fan-out (multiple output cells) generalizes this. The correct generalization is a **PROP** (PROducts and Permutations) — a symmetric monoidal category where morphisms have multiple inputs AND multiple outputs. Morphisms compose by wiring outputs to inputs, with symmetric monoidal structure handling the routing (which output connects to which input).

But even PROPs may not be the most apt framing. David Spivak's work on **polynomial functors as the natural language for dynamical systems** is closer to what we actually have. His framework — developed extensively in "Polynomial Functors: A Mathematical Theory of Interaction" — describes systems with typed interfaces (positions = outputs, directions = inputs) that compose by wiring. A propagator network is exactly such a system. The polynomial functor `p(y) = Σ_{i∈I} y^{B_i}` says: "I have output positions indexed by I, and at each position i, I need inputs from B_i." Composition of polynomial functors corresponds to wiring networks together.

The advantage of the polynomial functor framing over PROPs: polynomial functors naturally handle **dependent** fan-out — where the number and type of outputs can depend on the input values. This captures propagators like structural decomposition, where decomposing a Pi produces two sub-cells (domain, codomain), but decomposing an App produces two different sub-cells (function, argument). The fan-out structure depends on the constructor tag. PROPs have fixed arity; polynomial functors can have data-dependent arity.

**On fibrations between strata.** The stratification is a **Grothendieck fibration** over the stratum poset. Each fiber is a network type (a polynomial functor). The fibration structure captures:
- **Reindexing**: how a network in stratum S0 "appears" in stratum S1 (via threshold abstraction)
- **Cartesian lifts**: how information in S1 uniquely determines what happened in S0 (the readiness cell is a cartesian lift of the threshold transition)
- **Opcartesian lifts**: how committing in S2 pushes forward into S(-1) retraction

The bifibration (both fibration and opfibration) captures bidirectional information flow between strata. Our 2026-03-21 categorical analysis found that only the type checker genuinely inhabits the opfibration structure — but with the SRE, ALL systems would inhabit it, because the SRE makes structural reasoning uniform across domains.

**On dependent types providing the discipline.** This is where our existing infrastructure earns its keep:

- **Monotonicity** is `∀ (x y : L). x ≤ y → f(x) ≤ f(y)` — a Pi type with a proof obligation. Our logic engine can attempt to verify it for simple cases (lattice join, projection, constant functions). For complex propagators, it's a `:where` clause on the `propagator` spec.
- **Adjunction laws** are dependent propositions: `α ∘ γ ∘ α = α` and `γ ∘ α ∘ γ = γ`. Expressible as `:where [Adjunction α γ]` with the trait system providing the proof.
- **Well-foundedness** is an inductive type — our Nat-based structural recursion already handles this pattern. Stratification well-foundedness is provable when each stratum's lattice is finite or has a widening operator.

The progressive disclosure stack maps directly: Layer 1 users never write proofs (the compiler infers monotonicity for simple cases). Layer 4 users write categorical specifications with explicit proof terms.

**On whether Prologos-expressed types should be first-priority.** 

This is the key engineering question. There are two possible orderings:

**(A) Types-first**: Express the network type system in Prologos. Use it to verify our own infrastructure. Build the SRE with typed forms. Then build further infrastructure on the typed SRE.

**(B) Infrastructure-first**: Build the SRE and further infrastructure in Racket. Annotate with typing information (lattice tags, monotonicity class) but don't verify it yet. Build the network type system later, retrofitting types onto the existing infrastructure.

I think **(A) is the principled answer but (B) is the practical one**, with a crucial middle path:

**(C) Type-informed infrastructure**: Build the SRE in Racket, but design its form registry data model to carry the typing information from day one. Each form registration includes lattice type, component types, monotonicity class — as data, not enforced. When the network type system is built in Prologos, it reads the same data and enforces it. No retrofit needed because the data was always there.

This is what I mean by "types-ready" — the Racket infrastructure carries the type annotations as structured data, and the Prologos type system reads that data when it's ready to enforce it. The transition from "annotated but unchecked" to "checked" is a toggle, not a rewrite.

**On performance.** The types are checked at definition time (when a propagator or network is declared), not at runtime (when propagation fires). The type checker verifies that propagators are monotone; those propagators include the type checker's own propagators. This circularity is resolvable by bootstrapping: verify a minimal trusted core, then use that core to verify the rest.
