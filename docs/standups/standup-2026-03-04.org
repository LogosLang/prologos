## The Core Insight

In our session types, after every operation there's a **continuation** — a type that describes "what comes next." We already use this for recursion:

```prologos
session FileRead
  +>
    | :read-line -> ? String? . FileRead    ;; continuation = self (recursion)
    | :close     -> end                      ;; continuation = end (termination)
```

But there's nothing in the formalism that says the continuation *must* be `self` or `end`. It's a **type position**. So what if the continuation is... a different session type?

```prologos
session FileRead
  +>
    | :read-line -> ? String? . FileRead
    | :close     -> end
    | :upgrade   -> SecureFileRead           ;; continuation = a DIFFERENT protocol!
```

That `:upgrade` branch **transitions the protocol**. You were speaking `FileRead`, and now you're speaking `SecureFileRead`. The type checker verifies the transition. The channel is the same physical channel — only the protocol changes.

## Protocol Composition: Sequencing

This gets really powerful when you think about sequential composition. Consider real-world network protocols — they're almost always layered phases:

```prologos
;; Phase 1: negotiate protocol version
session Handshake
  ! Version
  ? Version
  end

;; Phase 2: authenticate
session Auth
  ! Credentials
  ? [Result Token AuthError]
  end

;; Phase 3: do actual work
session FileOps
  +>
    | :read  -> ! Path . ? [Result String IOError] . FileOps
    | :write -> ! Path . ! String . ? [Result Unit IOError] . FileOps
    | :done  -> end
```

Each of those is a complete, self-contained protocol. But what you're realizing is: **what if we compose them?**

```prologos
;; HTTPS-like: handshake, then authenticate, then operate
session SecureFileSession
  ! Version . ? Version .                      ;; Handshake phase
  ! Credentials . ? [Result Token AuthError] . ;; Auth phase
  FileOps                                       ;; Operations phase (named type!)
```

The continuation after auth is `FileOps` — a whole other session type. The composed protocol IS a type. And if `FileOps` has internal recursion and branching, all of that is inherited. You've built a complex protocol by snapping together simpler pieces.

## Protocol Branching: Mode Transitions

It gets even wilder when you combine this with choice. A protocol can *branch* into different sub-protocols:

```prologos
session DatabaseSession
  +>
    | :query    -> QueryProtocol        ;; switch to query sub-protocol
    | :admin    -> AdminProtocol        ;; switch to admin sub-protocol
    | :close    -> end

session QueryProtocol
  ! String . ! [List String] .              ;; SQL + params
  ? [Result [List Row] DbError] .
  DatabaseSession                            ;; ...and back to main protocol!

session AdminProtocol
  +>
    | :vacuum   -> ? [Result Unit DbError] . DatabaseSession
    | :backup   -> ! Path . ? [Result Unit DbError] . DatabaseSession
    | :drop-all -> ! Confirmation . DatabaseSession
```

Look at what's happening: `DatabaseSession` branches to `QueryProtocol`, which does its thing, and then the continuation goes *back to* `DatabaseSession`. You've got **mutual recursion between protocol types**. The protocol graph forms a state machine where named session types are states and operations are transitions.

## The Really Mind-Bending Part: Capabilities Compose Too

In Prologos, each session type can carry capability requirements. When you compose protocols, the capabilities compose:

```prologos
session Handshake {net :0 TlsCap}
  ! ClientHello . ? ServerHello . end

session Auth {net :0 AuthCap}
  ! Credentials . ? Token . end

session FileOps {fs :0 ReadCap}
  +>
    | :read -> ! Path . ? String . FileOps
    | :done -> end

;; Composed: capability requirements are the UNION
session SecureFileSession {net :0 TlsCap, net :0 AuthCap, fs :0 ReadCap}
  Handshake . Auth . FileOps
```

The composed protocol's authority requirements are automatically the union of its constituent protocols. The compiler can verify this. At runtime, `:0` capabilities are erased — zero cost.

**Protocols compose. Capabilities compose. Types verify the whole thing.**

## Has This Been Done Before?

Short answer: the *theory* supports it, but nobody has made it a user-facing design feature the way you're envisioning.

**What exists in the literature:**

- **Toninho, Caires, Pfenning (2011)** — Session types as propositions via Curry-Howard for linear logic. In their framework, composing sessions is just the **cut rule** — the most fundamental operation in logic. So theoretically, yes, session composition is as old as linear logic itself. But their work is deeply theoretical — process calculi, not programming languages.

- **Polymorphic/parameterized session types** (Caires et al., 2013; various) — Session types that take type parameters, including continuation parameters. This gives you open-ended protocols: `session Handshake[K] = ! Version . ? Version . K` where `K` is the "what comes after." This is the formal machinery that makes composition work cleanly. But again, primarily in calculi, not in user-facing languages.

- **Nested session types** (Das et al., 2021) — Session types that can nest, giving more expressive protocols. Related but focused on nesting rather than sequential composition.

- **Context-free session types** (Padovani, 2017; Thiemann & Vasconcelos, 2016) — Extend session types beyond regular languages to context-free, which gives more expressive composition. Powerful but adds significant complexity.

- **Scribble** (Honda, Yoshida et al.) — A protocol description language for multiparty session types. Scribble has some notion of protocol "modules" that can be composed, and it's the closest practical tool. But it's a specification language, not a programming language with types.

**What's potentially novel in Prologos:**

1. **Protocols-as-types with named composition** — Using named session types as continuations in other session types, with the language's type system verifying the whole thing. Most implementations treat session types as monolithic specifications, not as composable building blocks.

2. **Capability composition tracking** — The combination of session composition with capability composition, where the compiler automatically infers the authority union. I'm not aware of another system that does this.

3. **Progressive disclosure** — Tier 1 users never see any of this. Tier 2 users see simple `with-open` brackets. Only Tier 3 users compose protocols. The protocol composition is available but never forced.

4. **Propagator-based runtime** — The propagator network scheduling composed protocols is architecturally novel. Each phase of the composed protocol can be a different propagator, and the scheduler handles transitions.

## The Design Question: `end`-Splicing

There's one real design subtlety. If `Handshake` is defined with `end`:

```prologos
session Handshake
  ! Version . ? Version . end
```

Then `Handshake` is a *complete* protocol. To use it as a composable prefix, you'd need to "splice out" the `end` and replace it with the next phase. Two approaches:

**Approach A: Parameterized session types (continuation passing)**

```prologos
;; Open-ended: K is "what comes next"
session Handshake[K]
  ! Version . ? Version . K

;; Closed: Handshake that ends
type HandshakeDone = Handshake[end]

;; Composed: Handshake then Auth then FileOps
type SecureFile = Handshake[Auth[FileOps]]
```

This is the theoretically clean approach — session types are higher-kinded, parameterized by their continuation. But it adds complexity.

**Approach B: Implicit `end`-replacement (`.` as sequencing operator)**

```prologos
;; Both defined as complete protocols
session Handshake
  ! Version . ? Version . end

session Auth
  ! Credentials . ? Token . end

;; `.` between session types means "replace end with next"
session SecureFile = Handshake . Auth . FileOps
```

The `.` operator between named session types does the `end`-substitution automatically. Simpler surface syntax, but the semantics need to be clear: `.` on session types is not the same as `.` within a session type (which separates action from continuation). Though arguably they ARE the same operation at different levels — "do this, then do that."

**Approach C: The continuation position just works (what we have)**

Actually... looking at our current system, if you just write a named session type in continuation position, it already works without any new mechanism:

```prologos
session SecureFileSession
  ! Version . ? Version .           ;; inline the handshake actions
  ! Credentials . ? [Result Token AuthError] .  ;; inline auth
  FileOps                            ;; named type as continuation
```

You inline the phases you want and use the named type at the end. The named type in continuation position is already legal — it's just a type. No new mechanism needed. The cost is that you manually inline the prefix phases rather than referencing them by name.

---

So yes — you're thinking about this correctly, and it *is* a big deal. Composable protocols-as-types means you can build a **protocol library** — small, verified, reusable protocol building blocks that snap together like Lego. Handshake protocols, auth protocols, compression negotiation protocols, each defined once, composed freely, with capabilities tracking automatically.

The existing theory supports it. But making it a *first-class design feature* in a practical language with nice syntax, automatic capability composition, and progressive disclosure? That's where Prologos would be doing something genuinely new.

  Yeah, let's walk through this carefully, because I think the answer is "what we have is more powerful than it first appears, and the end-splicing question is narrower than it seems."

## What Already Works: Named Types in Continuation Position

The core mechanism is simple. In any session type, after the `.`, there's a continuation — and that continuation is a **type**. Any session type name is a valid type. So:

```prologos
session Reading
  +>
    | :read-line -> ? String? . Reading  ;; continuation = self (recursion)
    | :switch    -> Writing              ;; continuation = different session type!
    | :done      -> end                  ;; continuation = terminal

session Writing
  +>
    | :write-line -> ! String . Writing
    | :switch     -> Reading                   ;; ...and back again
    | :done       -> end
```

No new machinery. `Writing` in continuation position just resolves to the `Writing` session type. The type checker already does this — it's the same mechanism as recursive self-reference, just with a different name. You've got a **state machine** where named session types are states and operations are transitions.

This gives you three composition patterns for free:

### Pattern 1: Phase Transition (Sequential)

```prologos
session Handshake
  ! Version . ? Version . Authenticated       ;; after handshake, become Authenticated

session Authenticated
  ! Credentials
  ? [Result Token AuthError]
  +>
    | :ok   -> Ready                           ;; success → operational phase
    | :fail -> end                              ;; failure → done

session Ready
  +>
    | :query -> ! String . ? [Result Rows DbError] . Ready
    | :close -> end
```

The protocol flows: `Handshake → Authenticated → Ready → (loop) → end`. Each phase is a named type. Each transition is verified. The **author** of each session type decides what comes next — `Handshake` says "after me comes `Authenticated`," not the caller.

### Pattern 2: Mode Branching

```prologos
session DatabaseSession
  +>
    | :query  -> QueryMode
    | :admin  -> AdminMode
    | :close  -> end

session QueryMode
  ! String . ! [List Param] .
  ? [Result [List Row] DbError] .
  DatabaseSession                              ;; return to main protocol

session AdminMode
  +>
    | :vacuum -> ? [Result Unit DbError] . DatabaseSession
    | :backup -> ! Path . ? [Result Unit DbError] . DatabaseSession
```

Branch into a sub-protocol, do work, return to the parent. This is **mutual recursion** between session types — `DatabaseSession` branches to `QueryMode`, which returns to `DatabaseSession`. The protocol graph can be arbitrarily complex.

### Pattern 3: Inline Prefix + Named Continuation

```prologos
session SecureFileSession
  ;; Inline the handshake and auth operations...
  ! Version . ? Version .
  ! Credentials . ? [Result Token AuthError] .
  ;; ...then transition to the named operational protocol
  FileOps
```

You write out the prefix phases inline and use a named type as the final continuation. This is the most direct form of "do these things, then become that protocol."

## So What CAN'T You Do Without End-Splicing?

One specific thing: you can't write this —

```prologos
;; Three complete, self-contained protocols:
session Handshake
  ! Version . ? Version . end

session Auth
  ! Credentials . ? Token . end

session FileOps
  +> | :read -> ! Path . ? String . FileOps
     | :done -> end

;; HYPOTHETICAL: compose them with `.`
session Composed = Handshake . Auth . FileOps
```

This would require the `.` operator between named types to mean "take `Handshake`, find every `end` in it, replace with `Auth`; then take that result, find every `end`, replace with `FileOps`." That's end-splicing.

## Why End-Splicing Gets Complicated

It seems simple — just substitute `end` → next phase. But consider:

**Multiple exit points:**
```prologos
session MaybeAuth
  +>
    | :auth -> ! Credentials . ? Token . end    ;; exit point 1
    | :skip -> end                               ;; exit point 2

MaybeAuth . FileOps
;; Both `end`s become FileOps? Probably yes — "after MaybeAuth finishes
;; (by any path), do FileOps." That's actually reasonable.
```

OK, that case works. But:

**Recursion with partial exits:**
```prologos
session RetryLoop
  ! Request . ? Response .
  +>
    | :retry -> RetryLoop     ;; recurse (no end)
    | :accept -> end          ;; exit

RetryLoop . NextPhase
```

The `:accept` branch's `end` should become `NextPhase`. But what about `:retry → RetryLoop`? That recursive reference points to the *original* `RetryLoop` — whose `:accept` branch still says `end`. For the composition to be correct, the recursive reference needs to point to the *spliced* version.

This means the `.` operator isn't a simple textual substitution — it needs to create a **new recursive type** where the recursive self-references also reflect the spliced continuation. That's:

```markdown
rec X. ! Request . ? Response .
  +> | :retry -> X              ;; X = the whole spliced thing
     | :accept -> NextPhase     ;; end replaced
```

Doable, but it's real type-level machinery. You're transforming recursive type definitions.

**Ambiguity with mixed patterns:**
```prologos
session Weird
  +>
    | :a -> OtherProtocol    ;; continuation is ALREADY a named type
    | :b -> end              ;; continuation is end

Weird . NextPhase
;; Replace end → NextPhase in :b branch. Fine.
;; But :a already goes to OtherProtocol... leave it? 
;; What if OtherProtocol also has `end`? Do we splice THAT too?
```

Do you splice transitively? Only at the top level? This is where the semantics get genuinely ambiguous.

## The Tradeoff, Honestly

**Without end-splicing (what we have):**
- Protocol authors decide continuations at definition time: `Handshake` says "after me comes `Authenticated`"
- Protocols that need to be reusable prefix phases are defined *without* `end` — they name their continuation directly
- Protocols with `end` are *complete* — they're terminal, not composable prefixes
- This distinction is **explicit and intentional**, not an accident
- No new machinery, no ambiguity, no recursive type transformation

**With end-splicing:**
- Any complete protocol can be retroactively composed with any other
- Beautiful syntax: `Composed = A . B . C`
- But: non-trivial type machinery, ambiguous edge cases, and arguably **less clear** — when you read `Handshake`, you don't know if it's terminal or a prefix without checking who's splicing it

## The Deeper Point

I actually think the "without end-splicing" design is **better**, not just simpler. Here's why:

When `Handshake` says `end`, it means "I'm done, the connection is finished." When `Handshake` says `. Authenticated`, it means "I'm a phase, and after me comes authentication." These are **different protocols with different meanings**. A handshake-then-close and a handshake-then-authenticate are not the same handshake with different suffixes — they're different commitments.

The named-continuation approach makes protocol composition **intentional**. The author of `Handshake` explicitly declares what comes next. This is a design choice, not a mechanical operation. And in a language about correctness through types, that intentionality matters.

If you want reusable phases, you define them as open-ended:

```prologos
;; Reusable: author explicitly says "something comes after me"
session Handshake
  ! Version . ? Version . Authenticated

;; Terminal: author explicitly says "I'm the end"
session HandshakeDone
  ! Version . ? Version . end
```

Two types, two intentions, both clear. No machinery needed to distinguish them.

So to directly answer your question: Approach C — what we already have — gives you protocol composition through named types in continuation position, mutual recursion between protocols, and branching into sub-protocols. The end-splicing question is really about whether you can *retroactively* compose protocols that were defined as terminal. And I'd argue: if they were defined as terminal, respecting that is a feature, not a limitation.
