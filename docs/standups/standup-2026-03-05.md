### The Core Discovery

**Session types are not just protocol specifications — they are causal clocks.** Each position in a session type's continuation chain (`!A . ?B . end` → positions [0, 1, 2]) represents a causally-ordered event. This ordering is:

1. **Intrinsic** — derived from the session structure, not imposed externally
2. **Sound** — follows from session fidelity (a theorem of the Caires-Pfenning / Wadler Curry-Howard correspondence between linear logic and session types)
3. **Compositional** — multiple channels compose via vector clocks (per-channel positions)
4. **Monotone** — session advancement is monotone, so position derivation is monotone, so it lives happily in the propagator network

### The Galois Connection

I formalized the connection as a Galois connection `(α, γ)`:

- **α : Session → EffectPosition** — extracts the causal position from a session cell's value
- **γ : EffectPosition → Session** — reconstructs the remaining protocol from a position
- Both are monotone; the adjunction guarantees soundness

At a deeper level, both the session lattice and the effect position lattice are **effect quantales** (lattice-ordered monoids with sequential composition). The Galois connection is a **quantale morphism** — preserving both ordering and sequential composition.

---

This is a great question that sits at the intersection of several design threads you've already been working through. Let me trace through what exists, what the tensions are, and where I think the interesting design space lives.

## Current State

Right now, facts are represented as **flat positional lists**. The `fact-row` struct in `relations.rkt:69` is simply:

```racket
(struct fact-row (terms) #:transparent)
;; terms: a list of values, e.g. '("Alice" "Bob")
```

A `variant-info` holds a list of `fact-row`s alongside a list of `param-info`s (name + mode). So the "column names" exist in the parameter metadata, but the fact data itself is just a positional vector. The solver does positional unification against these lists — iterate through `(fact-row-terms fr)`, unify term-by-term against the goal arguments.

The `schema` form in the vision document introduces named, typed columns (`:parent String`, `:child String`), and the vision explicitly says schemas provide "positional ordering" — the field declaration order establishes the canonical mapping. But at runtime, fact data is still flat lists.

## The Triangle You're Describing

You're identifying three vertices:

1. **Positional tuples** — what Prolog uses, what we have now. Compact, fast to unify, but opaque: `("Alice" "Engineering" 95000)` means nothing without the schema nearby.

2. **Keyed maps** — what your Prolog-with-databases workflow uses (`table_name({column_key: Val, ...})`). Self-describing, composable, but heavier and potentially slower for the hot unification path.

3. **Typed records** — what schemas aspire to be. Named fields, type-checked, but with the efficiency of positional access.

The tension: positional is fast but fragile; keyed is robust but heavy; records are ideal but usually require compile-time resolution.

## How to Square It

I think the answer is already implicit in your schema design, but making it explicit reveals the right representation:

### The Schema *is* the Rosetta Stone

A `schema` declaration establishes a **bijection between positions and keys**:

```prologos
schema Employee
  :name   String    ;; position 0
  :dept   Department ;; position 1
  :salary Nat       ;; position 2
```

This means the underlying runtime representation can be **positional vectors** (fast, compact, cache-friendly) while the *surface syntax* supports **both positional and keyed access**. The schema mediates:

- **Storage**: flat vector `#("Alice" Engineering 95000)` — essentially a database row
- **Authoring (positional)**: `|| "Alice" Engineering 95000N` — tabular, concise
- **Authoring (keyed)**: `|| {:name "Alice" :dept Engineering :salary 95000N}` — self-documenting
- **Access (keyed)**: `alice.name` or `:name` in goal patterns
- **Unification**: positional, against the vector, always

The key insight: **keyed syntax is a compile-time convenience that erases to positional access**. The schema provides the mapping; after elaboration, everything is positional. This is exactly what databases do — SQL lets you say `WHERE name = 'Alice'` but the storage engine uses column offsets.

### What About Schema-less Relations?

Bare `defr` without a schema annotation works like traditional Prolog — positional only, untyped. This is fine for quick prototyping and small relations. The moment you want keyed access or database interop, you add a schema.

### The Database Import Use Case

Your Prolog workflow of `table_name({column_key: Val, ...}).` maps beautifully:

```prologos
;; Import: each table gets a schema
schema Employees
  :id     Nat
  :name   String
  :dept   String
  :salary Nat

;; Facts loaded from CSV/SQL/JSON — keyed or positional
defr employees : Employees
  || 1N "Alice" "Engineering" 95000N
     2N "Bob"   "Marketing"   72000N
     3N "Carol" "Engineering" 110000N

;; Or bulk-loaded from file
defr employees : Employees
  :source "data/employees.csv"
```

The schema does double duty: it's the type contract *and* the column mapping for import/export. A CSV header row, a SQL `CREATE TABLE`, a JSON schema, and a Prologos `schema` all describe the same thing — a named sequence of typed fields. The schema is the interop surface.

### PVec vs Flat Vector vs CHAMP Map

For the underlying runtime representation, the options are:

| Representation | Pros | Cons |
|---|---|---|
| **Flat Racket vector** | Fastest access, minimal allocation | Not persistent, no structural sharing |
| **PVec (RRB tree)** | Persistent, shareable, log-time access | Overkill for small fixed-width rows |
| **CHAMP map** | Keyed access, persistent | Heaviest, hash overhead per lookup |
| **Struct** | Fastest, named fields | No dynamism, one struct per schema |

My analysis: for **fact storage**, flat vectors (or even Racket structs generated per-schema) are the right choice. A fact row is small (typically 3-15 fields), fixed-width (schema determines it), and accessed many times during unification. The hot path is "unify position i of fact with goal argument i" — this wants O(1) indexed access, not hash lookup.

PVec and CHAMP are great for *user-facing collections* (lists, maps in the functional language), but using them for the internal fact representation would be paying persistent-data-structure overhead for data that doesn't need persistence — facts are immutable once asserted.

**The representation I'd suggest**: compile-time-generated Racket structs per schema (fastest) with a fallback to flat vectors for schema-less relations. The struct fields correspond to schema columns. The schema provides the name→position mapping for keyed syntax. This is essentially what Ecto changesets do — the schema generates a struct, and the struct is the runtime representation.

### Keyed Pattern Matching in Goals

The really interesting ergonomic question is: can you use keyed patterns in relational goals?

```prologos
;; Positional (current)
(employees id name "Engineering" salary)

;; Keyed (proposed) — only bind what you care about
(employees {:dept "Engineering" :name name :salary salary})

;; Mixed — positional with keyword override
(employees _ name :dept "Engineering" _)
```

The keyed pattern form is a big ergonomic win for wide relations. If your `employees` table has 15 columns, positional patterns require 15 arguments with `_` for every column you don't care about. Keyed patterns let you mention only the columns you need — which is exactly what SQL's `SELECT ... WHERE` does.

At compile time, with the schema in hand, `{:dept "Engineering" :name name}` desugars to a positional pattern with logic variables in the right positions and `_` everywhere else. Zero runtime overhead.

### The Graph-Query Power

The combination of schemas + keyed patterns + tabling + rule composition gives you something very powerful for your database use case:

```prologos
;; Schema per table (generated from DB metadata)
schema Employees  :id Nat :name String :dept String :salary Nat
schema Departments  :name String :budget Nat :head String

;; Facts loaded from DB
defr employees : Employees
  :source "db://company/employees"

defr departments : Departments
  :source "db://company/departments"

;; Composable graph queries — this is where Prologos shines
defr high-earners-in [?dept ?name ?salary]
  &> (employees {:dept dept :name name :salary salary})
     (> salary 90000N)

defr over-budget-depts [?dept ?total ?budget]
  &> (departments {:name dept :budget budget})
     (aggregate (employees {:dept dept :salary s}) [sum s] total)
     (> total budget)
```

Each `defr` is a reusable, composable, tabled view. You can build them up incrementally, test them individually, and compose them freely — something that requires CTEs and subqueries in SQL.

## Summary of the Design

1. **Storage**: flat vectors (or generated structs), positional, compact
2. **Schema**: the bijection between positions and named/typed keys
3. **Authoring**: dual syntax — positional `||` for tabular data, keyed `{...}` for clarity
4. **Goals**: keyed patterns desugar to positional at compile time
5. **Interop**: schema is the import/export contract (CSV, SQL, JSON all map)
6. **Schema-less**: falls back to positional-only Prolog-style (no keyed access)

The schema is the load-bearing element. It resolves the positional-vs-keyed tension by being the compile-time dictionary that lets you *write* keyed and *execute* positional. No runtime overhead for the keyed ergonomics. No loss of self-documentation. And the same schema serves as the database interop surface.
