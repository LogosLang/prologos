## What We Already Have (and what it points toward)

The existing `decompose-pi/sigma/app` in `elaborator-network.rkt` already creates cell trees:

```markdown
unify(?A, Pi(x : Nat, Vec Nat x))

→ decompose:
  ?A-domain-cell ←→ Nat-cell
  ?A-mult-cell   ←→ m1-cell  
  ?A-codom-cell  ←→ Vec-cell
    → further decompose:
      ?A-codom-elem-cell ←→ Nat-cell
      ?A-codom-len-cell  ←→ x-cell
```

Each `←→` is a bidirectional unify propagator. This IS structural unification — the network topology mirrors the type structure. But the creation of this topology is **lazy and algorithmic**: the `make-unify-propagator` fire function contains pattern-matching logic that decides when and how to decompose.

## Your Insight: Make the Structure Primary

What I hear you saying is: instead of the unification *algorithm* creating the cell tree on demand, the cell tree should be the **representation** of the type from the start. A meta `?A` isn't a single cell — it's a **potential tree** that unfolds as structure is learned. Unification becomes connecting two trees at their roots and letting structural propagation flow.

The CHAMP property you're pointing at is key: if both trees share a subtree (e.g., both have `Nat` in the domain position), CHAMP structural sharing means that subtree is literally the same memory. Comparing the trees is O(diff) — you only traverse the parts that differ. For real programs where most unifications involve types that share common structure (e.g., `List Nat` unified with `List ?B` — the `List` constructor is shared, only the element type differs), this could be very fast.

## The PVec Connection

You mentioned PVec-as-tree — I think the connection is: a type like `Pi (x : A) B` is structurally a 3-tuple `(Pi, A, mult, B)`. If we represent this as a CHAMP-backed persistent tree of cells, the nesting is captured by the data structure itself:

```markdown
cell-tree for Pi(x : Nat, Vec Nat x):
  root: Pi (constructor tag)
  children: [
    Nat        (domain — leaf, ground)
    m1         (mult — leaf, ground)
    cell-tree  (codomain — subtree)
      root: Vec
      children: [Nat, x]
  ]
```

This is a persistent tree where each node is either a ground value (leaf) or a cell (variable, may refine). CHAMP structural sharing across unification pairs means: the shared `Nat` leaf between domain and `Vec` element is literally pointer-equal.

## What This Means Architecturally

If types are cell-trees from the start, several things simplify:

1. **Unification becomes tree alignment**: connect two cell-trees at corresponding positions. No algorithmic decomposition needed — the structure IS the decomposition.

2. **Occurs check becomes cycle detection**: instead of traversing the type expression looking for a meta-id, you check whether connecting two cells would create a cycle in the tree topology. The network structure makes this explicit.

3. **Zonking becomes tree traversal**: reading the solution for a meta is traversing its cell-tree, reading each cell's value. No separate `zonk` pass needed — the tree IS the zonked value.

4. **Ground-check becomes tree completeness**: `ground-expr?` is "are all cells in this tree non-bot?" — which is exactly what the fan-in readiness propagator already checks.

## The Design Question

The question you're raising — "how exactly to map the parts of the structure" — is about the cell-tree layout. I see two approaches:

**A. Constructor-tagged n-ary trees**: Each type constructor has a known arity (Pi=3, Sigma=2, App=2, Vec=2). The cell-tree has one cell per constructor argument. Metas start as single cells (unknown constructor) and expand when the constructor is learned.

**B. Flattened path-indexed cells**: Each position in the type is indexed by a path (e.g., `[domain]`, `[codomain, elem]`). A meta `?A` has a root cell; when solved to `Pi(Nat, Vec Nat x)`, cells are created for each path. Unification connects cells at matching paths.

Approach A is more natural (mirrors the AST), but approach B might compose better with CHAMP (paths as hash keys — O(log32 n) lookup).

## What I'd Want to Explore

Before committing to this design, the audit should investigate:

1. **How many unification sites create compound types?** If most unifications are meta-vs-ground (common), the cell-tree for the ground side is fully known at creation time. The tree only "unfolds" on the meta side.

2. **What's the overhead of pre-creating cell trees vs lazy decomposition?** Currently `decompose-pi` only fires when both sides have structure. If we pre-create trees, we're allocating cells for structure that might never be inspected.

3. **How does this interact with the `with-enet-reads` bridge?** If zonking is tree traversal, the bridge disappears — reads are just cell reads, already pure.

This feels like the right direction for Track 8. The audit should ground it with concrete numbers (how many decompositions per command, what fraction are lazy vs eager), and the design should compare cell-tree unification against the current algorithm on specific benchmarks.

%%% Constraint propagation through shared variables
?- f(A, g(A, B), h(B, C), i(C, A)) = f(1, g(X, 2), h(Y, 3), i(Z, W)).
%% A=1, B=2, C=3, X=1, Y=2, Z=3, W=1

%%% Circular constraint chain
?- p(A, B, C, D) = p(X, X, Y, Y), p(A, B, C, D) = p(1, W, W, Z).

?- diamond(top(X), left(X, A), right(X, B), bottom(A, B, Y)) = diamond(top(1), left(L, 2), right(R, 3), bottom(2, 3, Y)).

?- outer(inner(A, B), inner(B, C), inner(C, A)) = outer(inner(1, X), inner(X, Y), inner(Y, Z)).

%%% Cross-referencing lists
?- [pair(A,B), pair(B,C), pair(C,A)] = [pair(1,X), pair(X,Y), pair(Y,Z)].

%%% Non-trivial propagation (not all same)
?- triple(A, B, C) = triple(f(X), g(X,Y), h(Y)), triple(A, B, C) = triple(f(1), g(1,2), h(Z)).
%% A=f(1), B=g(1,2), C=h(2), X=1, Y=2, Z=2

%%% Structure copying via shared variables
?- copy(Original, Duplicate) = copy(tree(L,R), tree(L,R)), Original = tree(leaf(1), node(2, leaf(3))).
%% Duplicate = tree(leaf(1), node(2, leaf(3)))

%%% Bidirectional list zipper
?- zip([A,B,C], [X,Y,Z], Zipped) = zip([1,2,3], Ys, [pair(1,a), pair(2,b), pair(3,c)]).
%% A=1, B=2, C=3, X=a, Y=b, Z=c, Ys=[a,b,c], Zipped=[pair(1,a),pair(2,b),pair(3,c)]

%%% Transpose-like constraint
?- matrix(row(A,B), row(C,D)) = matrix(row(X,Y), row(Y,X)), A = 1, D = 2.
%% A=1, B=2, C=2, D=2, X=1, Y=2... wait, this fails! B must equal Y, D must equal X, so B=Y, D=X, but also B=Y=X=D, so 1=2 fails.

%%% Working transpose constraint
?- matrix(row(A,B), row(C,D)) = matrix(row(P,Q), row(R,S)), transposed(row(A,C), row(B,D)) = transposed(row(P,R), row(Q,S)).
%% Forces A=P, C=R, B=Q, D=S (consistent but shows the constraint pattern)

%%% Function application pattern matching both ways
?- apply(func(Param, Body), Arg, Result) = apply(func(x, add(x,x)), 5, add(A,A)), Param = A, Arg = 5.
%% Param=x, A=x, Body=add(x,x), Arg=5, Result=add(x,x)

%%% Mutual recursion structure
?- even(A, odd(B, even(C, odd(D, nil)))) = even(X, odd(X, even(Y, odd(Y, nil)))), A = 0, C = 1.
%% A=0, B=0, C=1, D=1, X=0, Y=1

%%% Graph edge bidirectional constraint
?- graph([edge(A,B), edge(B,C), edge(C,A)]) = graph([edge(1,X), edge(X,Y), edge(Y,1)]).
%% A=1, B=X, C=Y, and Y=1 so C=1, and then edge(C,A)=edge(1,1) so we need edge(Y,1)=edge(1,1), Y=1, so C=1, X=B, edge(B,C)=edge(X,1), so B=X and C=1... B=X, and edge(A,B)=edge(1,X), A=1, B=X. Consistent: A=1,B=X,C=1,Y=1

%%% Substitution/environment lookup
?- subst(var(X), env(X, Val, _), Result) = subst(var(foo), env(foo, 42, nil), R).
%% X=foo, Val=42, Result=R (R unconstrained until we add:)
?- subst(var(X), env(X, Val, _), Val) = subst(var(foo), env(foo, 42, nil), R).
%% X=foo, Val=42, R=42

%%% Palindrome structure constraint
?- palindrome([A,B,C,B,A]) = palindrome([1,2,X,Y,Z]).
%% A=1, B=2, C=X, Y=2, Z=1

%%% Deep bidirectional with nested lists
?- f([A|As], [B|Bs], [[A,B]|ABs]) = f([1,2], [a,b], Zipped), f(As, Bs, ABs) = f([2], [b], Rest).
%% A=1, As=[2], B=a, Bs=[b], Zipped=[[1,a]|ABs], ABs=Rest, and f([2],[b],Rest)=f([2],[b],[[2,b]]) so Rest=[[2,b]]
%% Final: Zipped = [[1,a],[2,b]]

## The Narrowing System — What It Is

The term "narrowing" in Prologos actually refers to **two different things** that share a name but operate at different levels:

**1. Ground narrowing (type-tag dispatch)** — what the PIR surprise is about. When you write `[+ 1 2]`, the reducer inspects the arguments, infers a concrete type tag (`'Int`), and uses that to look up the **exact monomorphic implementation** (`prologos::core::arithmetic::Int--Add--add`) directly from the impl registry. No trait constraint created. No propagator network involvement. No stratified resolution. Just: inspect arguments → lookup FQN → call function. This is essentially **monomorphization at reduction time** — the generic `+` is resolved to the concrete `int+` before the type system even has to think about it.

**2. Relational narrowing (Curry-style)** — the FL narrowing infrastructure from Track 3 with definitional trees, needed analysis, and propagator-driven evaluation. This is the theoretically deep part — narrowing as lazy instantiation of logic variables guided by definitional tree structure.

The surprise from the adversarial benchmark is that **ground narrowing handles virtually all arithmetic dispatch**. The stratified resolution loop's 29 cycles are driven by implicit resolution (dictionary passing for `{A : Num}` binders), not by trait dispatch for concrete operations.

## Why It's So Effective

The key insight is that in a dependently-typed language with type inference, **most arithmetic expressions have ground arguments by the time they're reduced**. The elaborator has already solved the type metas. By the time `reduction.rkt` sees `[+ 1 2]`, the arguments are concrete literals — type-tag inference is trivially successful.

The impl registry lookup is O(1) — it's a hash lookup of `"Int--Add"` → FQN. The constraint lattice building + refinement is O(candidates), but there are only ~5-10 numeric types (Nat, Int, Rat, Posit8, Posit32). The entire narrowing path for `[+ 1 2]` is **sub-microsecond**.

Compare with what trait resolution would require:
1. Create trait-constraint cell (constraint store write)
2. Register wakeup entries for type-arg metas
3. Wait for S1 scanning to detect readiness
4. Execute S2 resolution (monomorphic lookup → dictionary → solve dict-meta)
5. Re-enter S0 for propagation

That's 5 network operations for something narrowing does with a hash lookup.

## What This Means For Us

**Good news**: Polymorphic dispatch for concrete types is essentially solved. When arguments are ground, we get C++-style static dispatch performance without a compilation pass. The prelude's generic arithmetic (`+`, `-`, `*`, `/`) "just works" efficiently for any registered numeric type.

**The implication for Track 7**: The readiness propagator infrastructure (Phase 8) that we built to replace O(total) scanning with O(changed) readiness detection is **architecturally important** (extensibility, correct-by-construction) but **not the performance bottleneck**. Narrowing already short-circuits the hot path. The stratified loop is exercised primarily by implicit resolution — the `{A : Num}` dictionary passing that happens when polymorphic functions are applied.

**The implication for the language**: We're effectively doing **JIT monomorphization** — but at the language level, not the compiler level. No specialization pass needed. The definitional tree + type-tag inference gives us what Rust's monomorphization and Haskell's specialization pragmas achieve, but as a runtime dispatch optimization.

## Room For Improvement

**1. Multi-argument narrowing**: Currently `infer-narrowing-type-tag` infers from *any* ground argument. For `[+ 1 ?x]`, it infers `'Int` from `1` and narrows. But for operations where the first argument is unground and the second is ground (`[+ ?x 2]`), does it check both? If not, that's a missed narrowing opportunity.

**2. Target-type narrowing**: When the expected return type is known (e.g., `def x : Int := [+ a b]`), the target type could narrow even when arguments are unground. The code has `target-expr` threading but I'd want to verify it's actually used for narrowing.

**3. Union-type narrowing**: `[+ v 3]` where `v : <Int | String>` currently falls back to speculation. But narrowing could resolve the `Int` branch eagerly and only speculate for `String` (which would fail). This is a micro-optimization but could matter for union-heavy code.

**4. Cross-trait narrowing**: If `[+ a b]` where `a : Nat` narrows `Add` to `Nat--Add`, could we propagate this to also narrow `Eq` constraints on the same type? Currently each trait narrows independently. A "narrowing cascade" across related traits could reduce constraint counts.

**5. Narrowing as a propagator**: Currently narrowing fires during reduction (imperative call). The propagator-first architecture suggests it should be a cell value — when a type-tag cell refines, a narrowing propagator fires and writes the resolved FQN to a dispatch cell. This would compose with the readiness infrastructure from Track 7 and enable the LSP to observe narrowing decisions. This is probably Track 10+ territory.

The bottom line: narrowing is doing the heavy lifting for polymorphic dispatch, and it's doing it well. The trait system is the general fallback for when narrowing can't determine the answer — which is primarily implicit resolution (dictionary passing), not arithmetic dispatch. Our architecture is in a good place.
