## §13 / §18.8 — Capability Integration

This is the area that needs the most design work. You're right that it should be integral, not bolted on. Let me articulate where I think the integration points are:

**The core question: What does a capability-gated session look like?**

The CAPABILITY_SECURITY.md doc establishes that capabilities are `:0` (erased proof) for authority checking and `:1` (linear transfer) for delegation. In session types, there are two scenarios:

**Scenario A: The session itself requires a capability to establish.**

```prologos
;; Only processes holding a NetCap can open this session
session HttpService {net :0 NetCap}
  ? HttpRequest
  ! HttpResponse
  end
```

Here the `{net :0 NetCap}` is a capability constraint on the session type itself — you need network authority to participate in this protocol. This is analogous to `spec read-file {fs :0 ReadCap} : ...` in the functional world. The erased proof gates *opening* the channel.

**Scenario B: Capabilities are delegated within the session.**

```prologos
session CapDelegation
  ? (path : Path)
  !: cap FileCap path :1    ;; dependent send of a linear capability
  end                        ;; after sending, the sender no longer has FileCap for path
```

This is where session types and capabilities deeply interleave — the protocol describes authority transfer. The `:1` on the capability ensures it's transferred, not duplicated.

**Scenario C: Session operations require capabilities for external effects.**

```prologos
defproc file-server : FileAccess [self] {fs :0 FsCap}
  path := self ?
  self ! [read-file path]   ;; read-file needs ReadCap; fs subsumes it
  stop
```

Here the process implementation has capability requirements, separate from the session protocol. The session type doesn't mention capabilities — the process implementation does.

**My proposal for the design:** We should support all three scenarios but with different syntactic weight:

- **Scenario C** (most common): Process-level capability constraints. Written as `{cap :0 CapType}` on the `defproc` header, exactly like `spec` constraints. No new syntax needed.
- **Scenario A** (less common): Session-level capability constraints. Written as `{cap :0 CapType}` on the `session` header. Same syntax, applied to session declarations.
- **Scenario B** (advanced): Capability delegation within protocols. Uses `!:` dependent send with the capability type. This is just dependent sessions applied to capability values.

The key insight from CAPABILITY_SECURITY.md: **capability-checking (`:0`) is the common case; capability-transfer (`:1`) is the rare case.** So most session types won't mention capabilities at all — they'll be on the `defproc`, not the `session`. The session type describes the *conversation*; capabilities gate *who can have that conversation* and *what effects the implementation can perform*.

This keeps capabilities "integral but out of the way" — present in the type system, visible in signatures, but not cluttering protocol declarations unless authority transfer is part of the protocol itself.

## Propagator-as-Scheduler: The Deep Argument

### Why This Isn't Just "Use What We Built"

The surface argument — "we have propagator infrastructure, so let's use it for process scheduling" — is pragmatic but undersells what's actually happening. The deeper argument is about **substrate unification**:

In every existing language, the process execution model is a *separate mechanism* from the type checking / constraint solving model. Erlang's BEAM scheduler is a different thing from its type system (such as it is). Go's goroutine scheduler is separate from the compiler. Rust's Tokio is a runtime library with no connection to the borrow checker. Even Haskell's green threads (forkIO) are a runtime system that the type checker knows nothing about.

In Prologos, propagator-as-scheduler means **type checking and process execution share the same computational substrate**. A session type constraint propagator and a process execution propagator live in the same network. The implications:

1. **A protocol violation at runtime produces the same ATMS derivation chain as a protocol violation at compile time.** The error message infrastructure is unified. You don't have separate "type error" and "runtime error" mechanisms for sessions — it's all constraint failure with derivation trees.

2. **Session type inference and process execution are the same operation at different lattice levels.** At compile time, the cells hold type-level information (session type lattice). At runtime, the cells hold value-level information (actual messages). The propagators are the same; the lattice changes. This is a Galois connection between static and dynamic.

3. **Backtracking is free.** Propagator networks with persistent cells give O(1) backtracking. For processes, this means: if a protocol branch fails (a `match` on a received value hits an unexpected case), you can backtrack to the choice point without rewinding global state. No existing process model offers this natively. Erlang requires explicit state management through gen_server; Go has no backtracking; Rust's async has no notion of it.

### What Processes Look Like as Propagators

Let me make this concrete. Consider:

```prologos
defproc greeter : Greeting [self]
  self ! "hello"
  name := self ?
  stop
```

In a traditional model, this is a sequential thread that blocks on `self ?`. In propagator-as-scheduler:

```markdown
Cell: self.out        (outgoing message cell)
Cell: self.in         (incoming message cell)  
Cell: self.session    (current session state cell)

Propagator P1: 
  watches: self.session
  fires when: self.session = Send(String, S)
  effect: write "hello" to self.out, advance self.session to S

Propagator P2:
  watches: self.in, self.session  
  fires when: self.session = Recv(T, S) AND self.in has a value
  effect: bind name := self.in, advance self.session to S

Propagator P3:
  watches: self.session
  fires when: self.session = End
  effect: mark process as stopped
```

`run-to-quiescence` fires P1 immediately (the session state cell starts at `Send String...`). P1 writes to `self.out`. The dual endpoint's propagators detect the message and fire. Eventually a response arrives in `self.in`, which lets P2 fire. When the session reaches `End`, P3 fires and the process terminates.

**There is no scheduler.** There is no thread. There is no blocking. There is only: cells, propagators, and `run-to-quiescence`. The "scheduling" emerges from the data dependencies — propagators fire when their input cells have enough information. This is inherently dataflow-driven, which is the right model for message-passing concurrency.

### Subsumption: Green Threads and Coroutines as Special Cases

Your intuition that this subsumes the other models is correct, and here's why:

**Green thread** = a linear chain of propagators where each fires exactly once and triggers exactly one successor. The chain is the "thread." If you constrain propagator topology to be linear-sequential, you recover green threads.

**Coroutine** = a propagator that writes to an output cell and then waits on an input cell. The yield/resume protocol is a two-cell handshake. Cooperative multitasking = propagators that voluntarily stop firing after one step.

**Erlang mailbox** = a cell with a list-valued lattice (messages accumulate monotonically, never retract). Pattern matching on the mailbox = a propagator that fires when a matching message appears. Selective receive = propagator with a guard condition.

**E-style eventual send** = write to a cell (non-blocking). Promise = a cell that doesn't have a value yet. `@promise` (your proposed deref syntax) = a propagator that blocks until the cell has a value. The E turn-based event loop IS `run-to-quiescence`. This is the deep OCapN connection — Prologos's propagator model naturally implements E-style programming.

The key: you can build all of these ON the propagator substrate, but you cannot build the propagator substrate on top of any of them. The other models are topologically constrained; propagators are the general case.

### Genuine Challenges to Address

Let me be honest about where this gets hard:

**1. Fairness.** In a propagator network, if one cell gets many updates rapidly, its propagators fire repeatedly while other cells' propagators starve. Traditional schedulers guarantee fairness through round-robin or priority queues. We need a fairness mechanism:

- **Option A**: Priority annotations on propagators. Session-type propagators get default priority; user processes can specify `:scheduler :round-robin` or `:scheduler :priority` (your `solver`-style configuration idea).
- **Option B**: Time-sliced quiescence. Run for N propagator firings, then yield even if not quiescent. This is a "fuel" mechanism for the network itself, analogous to the reduction fuel limit we already have.
- **Option C**: Cell-level fairness. Round-robin across cells, not propagators. Each scheduling tick, pick a different cell to process first.

I think **Option A + B combined** is the right starting point. Default round-robin with configurable fuel limits.

**2. Long-running computation.** Propagators are cooperative — they run to completion. A process step that does a heavy computation (e.g., `self ! [expensive-computation x]`) blocks the entire network while `expensive-computation` runs. Solutions:

- **Fuel-limited reduction** (already exists): `expensive-computation` respects the reduction fuel limit and yields if it exceeds it.
- **Segmented execution**: Break the computation into a chain of propagators. First propagator computes step 1 and writes to an intermediate cell; second propagator continues from there. This is CPS-at-the-propagator-level.
- **Escape to OS thread**: For genuinely CPU-bound work, spawn an OS thread and write the result back to a cell when done. This is the I/O integration pattern.

**3. I/O integration.** This is where capabilities become essential. Pure propagators are effect-free; I/O is effectful. The bridge:

```
Cell: io-request    (write: "read file /tmp/foo")
Cell: io-response   (populated asynchronously by the I/O subsystem)
Capability: fs :0 FsCap  (gates access to the io-request cell)

Propagator: IO-bridge
  watches: io-request
  fires when: io-request has a new request AND capability check passes
  effect: performs actual I/O, writes result to io-response
```

The IO-bridge propagator is special — it's the boundary between the pure propagator network and the effectful external world. **This is exactly where capabilities gate.** Only propagators holding the right capability can trigger IO-bridge propagators. The capability system and the execution model compose naturally.

**4. True parallelism.** A single propagator network is sequential (one propagator fires at a time). For multi-core parallelism:

- **Partitioned networks**: Each OS thread runs its own propagator network. Cross-partition communication via shared cells with atomic writes. This is the "multiple schedulers" approach.
- **Work-stealing**: Propagators from the ready queue can be stolen by idle threads. Requires careful cell synchronization.
- **Hierarchical**: A top-level network manages OS-thread-level propagators; each thread has its own sub-network for process-level propagators.

For Phase 0, single-threaded is fine. The architecture doesn't preclude parallelism later — the cell abstraction can be swapped from "mutable box" to "concurrent cell" without changing propagator code.

### The Unique Research Contribution

What's genuinely unprecedented here:

1. **No existing language uses propagator networks as the primary process scheduling mechanism.** The closest relative is Oz/Mozart's dataflow variables, but those don't have the lattice-theoretic foundation or ATMS integration.

2. **Session types + propagator scheduling + ATMS derivation = unified verification.** Static protocol checking and dynamic protocol execution use the same mechanism. No existing system has this.

3. **Backtracking in process execution** via persistent cells. If a protocol hits a dead end, the network can explore alternative choice branches without explicit programmer intervention. This connects to the logic programming leg of Prologos — `proc-solve` (proof search within protocols) becomes natural rather than bolted-on.

4. **E/OCapN as a special case**, not a separate runtime. Eventual sends, promises, and promise pipelining fall out of the propagator model. Mark Miller's work implemented atop the same substrate as the type checker.

## Capability Integration: The Session-Boundary Model

### The Core Insight: Channels ARE Capabilities

Let me start from a principle already in CAPABILITY_SECURITY.md: *"A reference IS the capability."* In the ocap model, if you have a reference to something, you have the authority to use it. No separate permission check.

For internal Prologos channels, this means: **the channel endpoint is the capability.** If a process holds `self : Greeting`, it has the authority to send and receive on that channel. The session type constrains *what* it can do (send a String, receive a String, end). No separate capability is needed. The channel reference was obtained legitimately (via `new`, via parameter passing, via delegation) — that's sufficient.

This is important because it means the common case is clean:

```prologos
;; No capabilities mentioned. The channel IS the authority.
defproc greeter : Greeting [self]
  self ! "hello"
  name := self ?
  stop
```

Capabilities enter the picture at exactly one place: **the boundary between Prologos and the external world.**

### The Three Boundaries

There are three kinds of boundary where capabilities gate access:

**Boundary 1: Opening external connections.**

A process wants to talk to something outside Prologos — a network socket, a file, a database, an OS service. This requires a capability:

```prologos
;; 'open' creates a channel to an external resource
;; The capability gates the creation, not the communication
defproc file-reader [output : DataStream] {fs :0 FsCap}
  ch := open "/tmp/data.csv" : FileProtocol {fs}
  data := ch ?
  output ! data
  close ch
  stop
```

`open` is capability-gated. Once the channel `ch` exists, using it (`ch ?`, `ch !`) requires no further capability checks — the session type handles the rest. This is exactly like `open()` in Unix: the capability check happens at open time; subsequent read/write on the file descriptor is authorized by possession of the descriptor.

**Boundary 2: Accepting incoming connections.**

A process listens for external connections — a network server, an IPC endpoint:

```prologos
;; 'listen' accepts incoming connections
;; Capability gates who can bind to a port
defproc http-server {net :0 NetCap}
  ch := listen 8080 : HttpProtocol {net}
  req := ch ?
  ch ! [handle-request req]
  rec
```

Again, the capability gates the `listen` operation. Once the channel exists, the session type governs.

**Boundary 3: Spawning processes.**

Creating a new process is itself an authority-bearing operation — you're allocating computational resources:

```prologos
;; spawn requires a SpawnCap (or is implicitly gated by the parent's authority)
defproc supervisor {spawn :0 SpawnCap}
  worker := spawn worker-proc : WorkerProtocol {spawn}
  worker ! initial-task
  ...
```

### Capability Delegation Through Sessions

The advanced case: a protocol explicitly transfers authority from one process to another. This is where capabilities and session types deeply compose:

```prologos
session CapDelegation
  ?: path Path                    ;; dependent receive: bind path
  !: cap ReadCap path :1          ;; dependent send: linear capability for that path
  end

defproc authority-granter : CapDelegation [self] {fs :1 FsCap}
  path := self ?
  read-cap := attenuate fs path   ;; narrow FsCap to ReadCap for specific path
  self ! read-cap                 ;; send the capability (linear: we give it up)
  stop
```

After `self ! read-cap`, the granter no longer has `read-cap` — it was `:1` (linear), so sending it consumed it. The receiver now holds it. QTT enforces this at compile time.

Note how `!:` (dependent send) is essential here — the capability's type (`ReadCap path`) depends on the value `path` that was received earlier. This is why dependent sessions matter for capability security: the authority being delegated is scoped to a specific resource.

### The Ergonomic Design

Your requirement: "integral but out of the way." Here's how the three tiers achieve this:

**Tier 1 (99% of code): No capabilities visible.**

Pure internal communication. Channel references are capabilities. Session types constrain the protocol. The programmer never writes a capability annotation:

```prologos
session Counter
  rec . +> | :inc -> ! Nat . rec | :done -> end

defproc counter : Counter [self]
  offer self
    | :inc -> self ! current . rec
    | :done -> stop
```

**Tier 2 (boundary code): Capabilities on process headers.**

When a process touches the external world, its `defproc` header declares what authority it needs. This is identical to how `spec` declares capability constraints on functions:

```prologos
defproc web-handler : HttpProtocol [self] {net :0 NetCap, db :0 DbCap}
  req := self ?
  result := [query-db req]    ;; query-db requires DbCap; resolved from {db}
  self ! result
  stop
```

The `{net :0 NetCap, db :0 DbCap}` is erased at runtime (`:0`). It's a compile-time proof that the process was granted these authorities. The programmer sees it in the signature and knows exactly what this process can touch. Least privilege is manifest.

**Tier 3 (delegation code): Capabilities in the protocol.**

When the protocol itself involves authority transfer, the session type mentions capabilities explicitly. This is rare — most protocols exchange data, not authority:

```prologos
session FileGrant
  ?: path Path
  !: cap FileCap path :1       ;; the session type shows authority transfer
  ? Unit                        ;; client signals done
  end
```

This tier is visible in the `session` declaration, not just the `defproc`. It tells the reader: "this protocol transfers authority." That's important information — it changes the security properties of the interaction.

### How This Composes with Propagators

In the propagator execution model, capabilities gate specific cells:

```markdown
Internal cell (message passing):
  - No capability check. Any propagator attached to the cell can read/write.
  - The cell's existence IS the authorization.

I/O-bridge cell (external boundary):
  - Capability check on the propagator that bridges to the external world.
  - The IO-bridge propagator holds the capability proof.
  - Other propagators can write requests to the IO cell, but the bridge
    only fires if the capability is satisfied.

Capability-transfer cell:
  - Cell holds a linear value (capability).
  - Write consumes the capability from the sender's context.
  - Read grants the capability to the receiver's context.
  - QTT propagator verifies linearity.
```

The beauty: capability checking is just another propagator constraint. It lives in the same network as session type checking, QTT checking, and process scheduling. There's no separate "capability runtime" — it's all cells and propagators.

### Capability Attenuation in Sessions

The CAPABILITY_SECURITY.md doc describes attenuation as a linear operation. In session context, attenuation happens naturally through the delegation pattern:

```prologos
;; Parent has full FsCap
defproc supervisor {fs :1 FsCap, spawn :0 SpawnCap}
  ;; Attenuate to read-only before delegating
  let (fs-remaining, read-only) := split-cap fs :read

  ;; Spawn child with narrowed capability
  new [my-ch child-ch] : WorkProtocol
  spawn (proc : WorkProtocol [ch] {r :0 ReadCap}
    data := [read-file "/config" {r}]
    ch ! data
    stop) read-only

  ;; Supervisor keeps fs-remaining (still has write access)
  result := my-ch ?
  ...
```

`split-cap` is the attenuation operation: consume one capability, produce two narrower ones. QTT tracks the linearity. The child process only gets `ReadCap` — it literally cannot write, because it doesn't have the type.

### The `open` / `connect` / `listen` Vocabulary

For external boundary operations, I'd propose these channel-creation primitives:

| Operation | Meaning | Capability |
|---|---|---|
| `new [a b] : S` | Internal channel pair | None (channels are caps) |
| `open path : S {cap}` | Open local resource | FsCap, DbCap, etc. |
| `connect addr : S {cap}` | Connect to remote endpoint | NetCap |
| `listen port : S {cap}` | Accept incoming connections | NetCap |
| `spawn proc {cap}` | Create new process | SpawnCap (or parent authority) |

All of these return channel endpoints. Once you have the endpoint, communication is uniform — `chan !`, `chan ?`, `select`, `offer`. The capability check is at creation time, not at use time.

This is the seL4 model: capabilities gate *access to kernel objects* (channels, processes, memory). Once you have a capability-derived handle, operations on that handle are authorized by possession. Session types add the additional constraint that operations must follow the protocol.

### Open Questions for Phase II on Capabilities

1. **Capability minting**: Who creates the root capabilities? In seL4, it's the initial process. In Prologos, the entry point (`main`) receives its capabilities from the runtime. How does the runtime decide what to grant? Configuration file? Command-line flags? A capability manifest?

2. **Capability revocation**: Can a granted capability be revoked? In pure ocap, no — if you gave it away, it's gone. But in practice, revocation is sometimes needed (user logged out, permission changed). Options: revocable capabilities (a wrapper that can be invalidated) or time-bounded capabilities (expire after duration).

3. **Capability composition**: If a process has `FsCap` and `NetCap`, can it compose them into a `TransferCap` (read a file and send it over the network)? Or must each operation be independently authorized? The principle of least privilege says: don't compose unless necessary. But ergonomics say: don't force the programmer to thread five separate capabilities through every function.

4. **The `:0` / `:1` boundary**: Most capability uses are `:0` (erased proof). But `open` and `connect` consume a capability *use* (not the capability itself). Is `open` the right model — `open` checks the capability but doesn't consume it? Or should `open` produce a channel AND consume a "use token"? This connects to the seL4 distinction between invoking a capability (reusable) and transferring a capability (one-shot).

5. **Schema + capability interaction**: When a session sends a value gated by a selection (`! Employee * EmployeeSummary`), is the selection a form of capability attenuation on data? The receiver can only access `:name` and `:dept`, not `:salary`. This is data-level least privilege, enforced by the type system. How formal should this connection be?

### 1. `main` as Powerbox — Confirmed

This is the right model. `main` is the root of the authority tree. It receives capabilities from the runtime (what the OS/environment grants the process), and it delegates attenuated capabilities to child processes:

```prologos
defproc main [args : List String] {fs : FsCap, net : NetCap, spawn : SpawnCap}
  ;; main has full authority. It attenuates before delegating.
  let read-only := attenuate fs :read
  let local-net := attenuate net :localhost

  spawn web-server {local-net}
  spawn file-watcher {read-only}
  ...
```

The runtime is the only entity that can mint capabilities. `main` is the first process that receives them. Everything else gets them by delegation. No manifest file — the authority chain is in the code, visible in types, auditable by the compiler.

Your point about manifests is well-taken — an external manifest is a second source of truth that can drift from the code. The type signatures ARE the manifest. If a process needs `NetCap`, it says so in its header. The compiler verifies the chain from `main` down.

### 2. Revocation — Deferred (Noted)

Agreed. For single-machine application development, the linear ownership model is sufficient. A capability, once delegated, is consumed — you can't use it again. That's revocation-by-design for the common case. True revocation (invalidating a previously-granted capability) is a distributed systems concern. We'll return to it when we address distributed channels and multi-node Prologos.

### 3. Overauthorization — Compiler Warning

Right. The compiler should warn when:
- A process receives more capabilities than it uses (dead authority)
- A `:w` multiplicity is used on a capability type (ambient authority)
- A capability is passed through without attenuation when narrowing is possible

Union composition is allowed but warned:

```prologos
;; Compiler warning: composite capability — consider whether both are needed
defproc handler {io : FsCap | NetCap}
  ...

;; Better: separate, narrower capabilities
defproc handler {fs : FsCap, net : NetCap}
  ...
```

The warning is informational, not an error. Sometimes composite authority is the right choice. The point is to make it visible.

### 4. `open` as Double-Boundary — Confirmed

The flow:

```markdown
Process → open (capability check) → IO-bridge cell → IO propagator (capability verified again) → External world
```

Two checkpoints. The first (`open`) is compile-time — does this process have the right to create this channel? The second (IO propagator) is runtime — is this operation still valid when it actually executes? The double-boundary is defense in depth. The compile-time check catches most errors; the runtime check catches anything that slipped through (e.g., a capability that was valid when the process was compiled but the resource no longer exists).

### 5. Schema × Capability Composition — The Design Challenge

This is the interesting one. Let me restate the question concretely:

When a session sends a schema-typed value gated by a selection, we have:

```prologos
session EmployeeService
  ? (query : String)
  ! Employee * EmployeeSummary     ;; send Employee, gated by EmployeeSummary selection
  end
```

The `* EmployeeSummary` gates field access — the receiver can only read `:name` and `:dept`, not `:salary`. This IS a form of data-level capability attenuation. The question is: how do these two concepts — field-level access control (selection) and system-level authority (capabilities) — compose in the syntax and the type system?

Let me lay out three scenarios where they interact:

**Scenario A: Selection as data attenuation, capability as system attenuation. Independent.**

```prologos
;; The session protocol gates WHAT data fields are visible
session EmployeeService
  ? (query : String)
  ! Employee * EmployeeSummary
  end

;; The process header gates WHAT system resources are accessible
defproc employee-server : EmployeeService [self] {db :0 DbCap}
  query := self ?
  emp := [db-lookup query {db}]    ;; DbCap needed for database access
  self ! emp                        ;; EmployeeSummary gates field access for receiver
  stop
```

Here they're completely independent. The capability (`DbCap`) gates the side effect (database access). The selection (`EmployeeSummary`) gates the data shape (which fields the receiver sees). They don't interact syntactically. This is clean but maybe misses something.

**Scenario B: Selection gates outgoing data, capability gates incoming authority.**

```prologos
session FileAccess
  ?: path Path
  !: handle FileHandle path * ReadOnly    ;; send a handle, gated to read-only view
  ? Unit
  end
```

Here `* ReadOnly` on the send isn't a schema selection — it's a capability attenuation on the handle. The `FileHandle path` is the full handle; `* ReadOnly` narrows it. This reuses the `*` operator for a different kind of gating — data shape (selection) vs authority shape (capability attenuation).

**Scenario C: They compose — the selection IS the capability attenuation.**

This is where it gets philosophically interesting. Consider:

```prologos
schema Employee
  :name String
  :dept Department
  :salary Int          ;; sensitive field

selection PublicInfo from Employee
  :requires [:name :dept]

session EmployeeQuery
  ? String
  ! Employee * PublicInfo     ;; the selection IS the attenuation
  end
```

The receiver gets an `Employee` value but can only access `:name` and `:dept`. From a capability perspective, the `PublicInfo` selection IS a data capability — it's an unforgeable, typed restriction on what the receiver can do with the value. The receiver literally cannot call `.salary` on the result because the type system says the selection doesn't include it.

**This is where I think the synthesis lives:** selections and capabilities are the same pattern at different levels.

| Level | Full Authority | Attenuated Authority | Mechanism |
|-------|---------------|---------------------|-----------|
| System | `FsCap` | `ReadCap` | Capability attenuation |
| Data | `Employee` | `Employee * PublicInfo` | Selection gating |
| Channel | `FileAccess` (full protocol) | subtype (fewer branches) | Session subtyping |

All three are: "you have a reference to something, but you can only use part of it." The type system enforces the restriction. The runtime never even constructs the full-authority view for the restricted party.

**The syntactic question:** Should these use the same `*` operator, or should capabilities have a different marker?

I think `*` works for both because the semantics are the same — it's a Sigma type (dependent pair) where the second component constrains how the first can be used:

```prologos
! Employee * PublicInfo        ;; data × selection constraint
!: cap FileCap path * ReadOnly ;; capability × attenuation constraint
```

The type system distinguishes them: `PublicInfo` is a selection (gates field access), `ReadOnly` is a capability attenuation (gates operations). The `*` means "this value comes with this constraint."

But here's where ergonomics get tricky. What if you need BOTH? A capability-bearing value with a field-access restriction:

```prologos
;; Send a database handle that's read-only AND limited to certain tables
!: db DbHandle * ReadOnly * TableSubset
```

Now `*` is chaining multiple constraints. Is that clear? Or does it look like nested Sigma types?

**My proposal for syntactic clarity:**

Use `*` for the primary constraint (the one most relevant to the protocol). Use `where` for additional constraints:

```prologos
;; Primary constraint is the selection (data shape)
! Employee * PublicInfo

;; Primary constraint is the capability attenuation
!: cap FileCap path * ReadOnly

;; Multiple constraints
!: db DbHandle * ReadOnly where TableSubset
```

Or — and this might be simpler — lean into the fact that attenuated capabilities are just types. `ReadOnlyDbHandle` is a type. `PublicEmployee` could be a type (or a type alias for `Employee * PublicInfo`). Then the protocol just names the type:

```prologos
;; The attenuation is in the type, not the protocol
!: db ReadOnlyDbHandle
! PublicEmployee

;; Where:
type ReadOnlyDbHandle := DbHandle * ReadOnly
type PublicEmployee := Employee * PublicInfo
```

This pushes the complexity into type definitions and keeps the protocol declarations clean. The protocol says WHAT is sent. The type definition says WHAT CONSTRAINTS apply. Decomplected.

I think this is the right direction: **selections and capability attenuations are type-level concepts that compose through the type system, not through special protocol syntax.** The session declaration just names the type. The type carries the authority restrictions.

## Response to External Critique

The critique is well-structured but misses several key aspects of our existing infrastructure. Let me respond point by point, grounding in the specifics of what we've already built.

### Concern 1: Non-Determinism in Propagator Execution — **Misdiagnosis**

The critique asks: *"when a process does `offer self | :granted -> ... | :denied -> ...`, how does the propagator network 'choose'?"* and suggests we need to fork the network or add an external choice operator.

**This conflates two different things.** Choice in session types is not non-determinism in the propagator network — it's *external input resolution*.

Here's what actually happens:

1. Process A reaches `offer self | :granted -> Pa | :denied -> Pb`. Its session state cell holds `Offer {:granted → S1, :denied → S2}`. Two branch propagators are created, each with a guard: P-granted watches `(self.session = S1)`, P-denied watches `(self.session = S2)`.

2. Process B (the dual endpoint) executes `select other :granted`. This writes `:granted` to a *selection cell* — a perfectly monotonic cell write (from ⊥ to `:granted`). This is standard lattice-theoretic: the cell goes from "no selection" to "selected :granted", which is a join on a flat lattice of labels.

3. A *selection propagator* on the selection cell fires, advances `self.session` from `Offer {...}` to `S1` (the `:granted` continuation).

4. P-granted's guard is now satisfied, so it fires. P-denied never fires.

No forking. No non-determinism. No "parallel universes." The propagator network is doing exactly what it always does — monotonic cell writes trigger guarded propagators. The "choice" comes from *external input* (the other endpoint's `select`), which is a cell write. This is identical to how our existing type inference works: a metavariable cell starts at ⊥, gets resolved by unification (a cell write), and propagators that depend on it fire.

The critique's confusion is understandable — they're thinking of choice as a *search* operation (like Prolog backtracking). But session type choice is *communication* — one endpoint decides and tells the other. That's a cell write.

For **internal choice** (`+>`), the process itself writes to the selection cell — `select self :accept` writes `:accept` monotonically. Same mechanism.

We should add a subsection to §10 making this explicit, with the cell-state diagram. But the concern is resolved — no new mechanisms needed.

### Concern 2: Deadlock Freedom — **Partially Valid, Proportionally Addressed**

The critique is right that multi-channel deadlock is real and their example is correct. But they overstate the urgency and understate what our infrastructure already provides.

**What we have**: The propagator network inherently detects circular dependencies. If Process A is waiting on cell X (which depends on cell Y's update from Process B) while Process B is waiting on cell Y (which depends on cell X's update from Process A), `run-to-quiescence` terminates with *neither* propagator having fired. This IS deadlock detection — the network quiesces without completing. The ATMS can then report the minimal conflict set showing the circular dependency.

**What we're proposing**: Single-session processes are deadlock-free by construction (Caires-Pfenning guarantees this structurally). Multi-channel processes (§17's orchestrator) are where deadlock can occur. The Balzer-Pfenning priority approach assigns a partial order to channels — you must acquire channels in order. This maps to propagator cell ordering, which we can enforce.

The critique suggests restricting multi-channel processes to a "safe subset." We should not do this — it would prevent the orchestrator pattern (§17) which is our primary multi-party approach. Instead, deadlock checking should be a propagator-level analysis in Phase S4, leveraging the same ATMS infrastructure we already use for type error reporting.

I'd move deadlock checking from "future" to Phase S4 as the critique suggests, but clarify that the *mechanism* is already present — we just need to formalize the channel ordering constraint.

### Concern 3: Dependent Sessions Semantics — **Already Solved Infrastructure**

The critique asks: *"When checking `? Vec String n`, what is n?"* and suggests extending the session lattice.

**We already have this.** Prologos has full dependent types with metavariables, unification, and two-phase zonking (CLAUDE.md documents this). `sessions.rkt` already implements `substS` — dependent substitution for session types. Sprint 8 added `sess-meta` for session continuation inference via metavariables.

Here's what happens with `?: n Nat . ? Vec String n`:

1. The `?:` propagator creates a fresh meta `?n` for the received value
2. Cell(self.session) advances to `Recv (Vec String ?n) . S`
3. When the actual receive fires at runtime, `?n` gets resolved (unified) to the concrete value — say `5`
4. The session lattice cell now holds `Recv (Vec String 5) . S`
5. Standard session type checking proceeds

The session lattice doesn't need extension because dependent values flow through the *type* lattice, not the session lattice. The session lattice tracks protocol structure (Send/Recv/Choice/End). The types *within* Send/Recv are handled by the existing dependent type infrastructure. These are cross-domain bridge propagators (§15.6: Session ↔ Type).

However, the critique is right that §15.2 should *show* this interaction. I'd add a diagram showing how the session lattice and type lattice interact at dependent boundaries. Good documentation feedback.

### Concern 4: Strategy Decomplection Premature — **Partially Agree**

The critique says we're "designing tuning knobs before we know what needs tuning."

**Fair point**, but with a nuance: we're not committing to implementation — we're establishing the *principle* and reserving the keyword. The strategy keyword design in §11 is deliberately minimal (four properties). The important decision is that strategy is **decomplected from session type definition** — that principle should be stated now even if the exact property vocabulary changes.

I'd keep §11 but add a note: "The property vocabulary (`:fairness`, `:fuel`, etc.) is provisional and will be refined based on execution experience. The principle — execution strategy is orthogonal to protocol specification — is the design commitment."

### Concern 5: Error Handling Verbosity — **Agree, Move `throws` Earlier**

The critique is right that explicit branches are verbose and that `throws` desugaring is mechanical. The transformation is:

```markdown
session S throws E = at every step, wrap with &> | :normal -> ... | :error -> ? E . end
```

This is a preparse-level transformation — the same kind of desugaring we already do for implicit maps, dot-access, and mixfix. Moving `throws` to Phase S3 (elaboration) is reasonable and would significantly improve ergonomics. I'd accept this.

### Concern 6: Implicit `self` Transition Cost — **Disagree**

The critique suggests always requiring `[self : Greeting]` to make the transition to multi-channel smoother.

This violates progressive disclosure. The transition from single-channel to multi-channel is a *conceptual* upgrade — the programmer is moving from a simpler model to a more complex one. Adding `[self : Greeting]` to every single-channel process is ceremony for the 99% case to smooth a 1% transition. The mechanical change (find-replace `self` references, add explicit bindings) is trivial compared to the conceptual change of understanding multi-channel coordination.

Our precedent: `defn foo [x] body` doesn't require type annotations because inference handles it. `defproc greeter : Greeting` likewise doesn't need to spell out what's inferrable from the session type.

### Concern 7: `@` Collision Risk — **No Collision in Prologos**

The critique worries about `@[]` for vectors. **We don't use this syntax.** Our vector literal syntax is `'[1 2 3]` (quote-bracket). Our PVec literal syntax uses constructors. `@` is unused in the Prologos reader (verified against `grammar.ebnf`). There is no collision.

The languages mentioned (Rust decorators, email in strings) are irrelevant to our syntax space. `@` is clean for promise deref, and now also `@>` for promise pipeline resolution.

### Concern 8: Duality — **Good Documentation Point**

The critique correctly identifies that duality handling should be more explicit in the document. `new [c1 c2] : S` gives `c1 : S` and `c2 : dual(S)` implicitly. When a `defproc` implements a session type, it implements the provider side by default; the dual side is what the client sees.

I'd add a subsection under §9 or §5 making this explicit. Not a design issue, just incomplete documentation.

### Concern 9: `.` Continuation Collision — **Already Resolved**

The critique worries about `.` collision with map access. **We already resolved this**: module paths migrated from `.` to `::` (dot-access design, commits documented in MEMORY.md). Map access `.` is reader-level (`user.name` → `[map-get user :name]` via `$dot-access` sentinel). Inside `session` bodies, `.` is unambiguously continuation because the parser knows the context — `session` establishes a parsing mode. The grammar will make this explicit.

### Concern 10: Branch Syntax — **Intentionally Both Forms**

The critique notes inconsistency between `| :label -> single-step` and `| :label` + indented block. This is intentional and consistent with the rest of Prologos:

- `match` supports both `| pattern -> expr` and `| pattern` + indented body
- `defn` supports both `| clause -> body` and `| clause` + indented body
- `type` supports both inline and multi-line variants

Both forms desugar to the same AST. The choice is the programmer's — short inline for simple branches, indented blocks for complex ones. This is documented but could be more explicit.

### Concern 11: `shared` Keyword — **Fair, Add to Operator Table**

`shared` should be in the operator table (§3) and explained more clearly. It's the session-level marker for `:w` multiplicity services. I'd add it.
