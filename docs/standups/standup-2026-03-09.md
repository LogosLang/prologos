## How Traits Work in Prologos Today

### The Dictionary-Passing Transform

Traits in Prologos are implemented via **dictionary passing** — the classic Haskell/GHC approach, but with some interesting wrinkles from your type system.

When you write:

```prologos
trait Eq {A}
  eq? : A A -> Bool
```

`process-trait` in `macros.rkt:6059` generates:

1. **A `deftype`** — `(deftype (Eq A) (A A -> Bool))` — the trait *is* a type. Specifically, `Eq Nat` is a type whose inhabitants are functions `Nat Nat -> Bool`. For multi-method traits, it's a nested Sigma (product) of the method types.

2. **Accessor functions** — `Eq-eq? : {A :0 Type} -> Eq A -> A A -> Bool` — a function that takes a dictionary and projects out the method. For single-method traits, the accessor is identity (the dict *is* the function).

When you write:

```prologos
impl Eq Nat
  defn eq? [a b] [nat-eq a b]
```

`process-monomorphic-impl` (`macros.rkt:6837`) generates:

1. **A method helper** — `Nat--Eq--eq? : Nat Nat -> Bool` — the concrete implementation
2. **A dictionary definition** — `Nat--Eq--dict : Eq Nat := Nat--Eq--eq?` — for single-method traits, the dict *is* the helper function
3. **A registry entry** — `impl-key "Nat--Eq"` → `impl-entry` mapping to `Nat--Eq--dict`

### How Constraints Become Metavariables

When the elaborator encounters a `where (Eq A)` clause on a function, it:

1. Creates an **implicit lambda parameter** with name `$Eq-A` (the `$` prefix marks dict params — `is-dict-param-name?` at `elaborator.rkt:116`)
2. Allocates a **metavariable** for the dictionary
3. Registers a `trait-constraint-info` in the `current-trait-constraint-map` (`metavar-store.rkt:233`), tagging that meta with `(trait-name: 'Eq, type-arg-exprs: [meta for A])`
4. Builds a **wakeup index** — reverse mapping from type-arg metas to dict metas, so when `A` gets solved (say to `Nat`), the system knows to try resolving `Eq A` → `Eq Nat`
5. Records **cell-ids** for the type-arg metas (P3a in `metavar-store.rkt:242`) — this is the propagator connection

When the function body uses `eq?`, `resolve-method-from-where` (`elaborator.rkt:161`) looks up the active `current-where-context`, finds the `where-method-entry` for `eq?`, and emits `(Eq-eq? A $Eq-A)` — applying the accessor to the type variable and the dictionary parameter.

### The Resolution Pipeline

After type-checking, `resolve-trait-constraints!` (`trait-resolution.rkt:305`) walks the constraint map:

1. **Zonk** each type-arg meta (resolve any solved metas)
2. **Check groundness** — all type args must be fully solved
3. **Try monomorphic resolution** — look up `"Nat--Eq"` in the impl registry
4. **Try parametric resolution** — pattern-match against parametric impl entries (e.g., `impl Eq (List A) where (Eq A)`)
5. **Solve the dict meta** — `solve-meta!` writes the dictionary expression (e.g., `Nat--Eq--dict` or `(List--Eq--dict A $Eq-A)`)

Parametric resolution is recursive — resolving `Eq (List Nat)` triggers resolution of the `where (Eq Nat)` constraint inside the parametric impl.

### Bundles: Composition Without Inheritance

Bundles (`macros.rkt:5268`) are purely a **constraint shorthand**:

```prologos
bundle BoundedLattice := (Lattice, HasTop)
```

This registers a `bundle-entry` with `params: '(A)` and `constraints: '((Lattice A) (HasTop A))`. When you write `where (BoundedLattice A)` in a spec, the bundle expands into `where (Lattice A) (HasTop A)` — two separate dict params, two separate constraints, two separate resolutions.

This is the key insight you identified: bundles are **conjunctions of constraints**, not subtypes. `BoundedLattice` isn't "a type that inherits from Lattice and HasTop" — it's "the constraint that both `Lattice A` and `HasTop A` are satisfied." Set intersection of the constraint space, not subtype inclusion.

### Dict Params Use `mw` (Not `m0`)

In QTT, type parameters are erased (`m0`) — they exist only at compile time. But dict params use `mw` (unrestricted multiplicity) because the dictionary is a runtime value that the function body actually uses. If dict params were `m0`, any method call in the body would be a QTT violation ("using erased variable at runtime"). This is handled in `elaborator.rkt:794` — when a lambda binder is detected as a dict param name, the elaboration context adjusts multiplicities.

### Are Traits First-Class? — Not Yet, But They're Close

Currently, traits are:
- **Types** at the Prologos level — `Eq Nat` is a type, inhabited by dictionaries
- **Not first-class values** — you can't write `let t = Eq` and pass `t` around; you can't abstract over traits as values
- **Compile-time entities** — the trait registry (`current-trait-registry`) is a mutable Racket-side hash populated during macro expansion

Dictionaries *are* first-class values (they're just Sigma pairs or bare functions). But the *trait itself* (the mapping from name → methods → type) lives in the macro expander, not the runtime.

### What First-Class Traits Would Mean

Making traits first-class means reifying the trait-as-type-function into a runtime value:

```prologos
;; Today: Eq is a macro-expanded type constructor
;; Eq Nat  ==>  (Nat Nat -> Bool)

;; First-class: Eq is a value of kind (Type -> Type)
def my-constraint : Type -> Type := Eq

;; Or even: traits as values you can pass to functions
spec require-equality : <(T : Type -> Type) -> T Nat -> Nat Nat -> Bool>
defn require-equality [dict a b] [T-eq? dict a b]  ;; ← but how do you project?
```

The fundamental tension: trait dispatch requires knowing the *structure* (which methods, what types) at compile time for dictionary projection. If `T` is an unknown trait, you can't generate the accessor code.

There are three ways to resolve this:

**1. Traits as records with row polymorphism.** A trait becomes a row-polymorphic record type. Projection works because row polymorphism maintains structural knowledge:

```prologos
;; Trait is a row: { eq? : A A -> Bool | r }
;; First-class: pass any record that has an eq? field
spec flexible-compare : {r : Row} -> { eq? : A A -> Bool | r } -> A -> A -> Bool
```

This connects to your schema system — schemas are record types with known fields.

**2. Traits as dependent type-level functions.** A trait is a `Type -> Type` function that you can pass around, but projection requires dependent pattern matching on the trait value:

```prologos
;; TraitOf : (methods : List MethodSpec) -> Type -> Type
;; Eq = TraitOf [MethodSpec "eq?" (A A -> Bool)]
```

This is the "universe of traits" approach — traits become data in a type-level language.

**3. Traits as propositions (the Curry-Howard connection).** `Eq A` is a *proposition* — "there exists an equality comparison for A." The dictionary is the *proof*. First-class traits means first-class propositions — which you already have, because your type system is dependently typed. A `where (Eq A)` constraint is just a Pi binding over the proof type:

```prologos
;; where (Eq A) is sugar for:
Pi ($Eq-A :w (Eq A)) -> ...

;; First-class: abstract over the proposition itself
Pi (C : Type -> Type) -> Pi (dict : C A) -> ...
```

The problem: without knowing `C`'s structure, you can't project methods from `dict`. You'd need some form of **structural type matching** or **reflection**.

### Traits and Propagators: The Connection That Already Exists

The trait system already touches the propagator network in two places:

1. **Cell-state-driven resolution** (P3a, `metavar-store.rkt:228`): Type-arg metas for trait constraints have corresponding propagator cells. When a type cell refines (via unification), the wakeup index triggers trait resolution. This is already a propagator-adjacent pattern — "when this cell changes, try to resolve this constraint."

2. **Trait constraints as lattice elements**: A trait constraint `Eq ?A` sits in the meta store with type-arg metas. As type inference progresses (cells refine from `type-bot` toward ground types), the trait constraint becomes increasingly solvable. This is monotonic refinement — the same pattern as propagator cells.

### Trait Dispatch as Narrowing: The Bridge to Mixfix

Here's where it gets really interesting. Your mixfix system (`macros.rkt:4406`) works like this:

3. `add` and `mul` are trait methods (from `Add` and `Mul` traits) — so after Pratt parsing, the elaborator resolves them via trait dispatch

The chain: **mixfix syntax → Pratt parse → trait method → dictionary lookup → concrete function.**

Now, the question you're driving at: what if the operand types are unknown? If you have `.{x + y}` and `x : ?A`, `y : ?B`, then:
- The Pratt parser emits `(add x y)` — fixed, no type info needed
- `add` resolves to `Add-add` — the trait accessor
- The elaborator creates a trait constraint `Add ?A` with a dict meta
- Type inference refines `?A` — when it lands on a ground type, trait resolution fires

4. Run forward-checking (your `global-constraints.rkt` machinery)
5. Prune instances whose interval constraints contradict

This is trait dispatch as search — and it maps directly onto your existing infrastructure:
- The **narrowing DFS** handles the search tree
- **Interval domains** handle the numeric constraint propagation
- **Global constraints** (`all-different`, `element`) provide the pruning

## How Traits Constrain Types in Prologos

The mechanism is precise and worth stating clearly:

**A trait declaration creates a type-level function. A trait constraint creates a proof obligation.**

When you write `trait Eq {A}`, `process-trait` (`macros.rkt:6059`) generates a `deftype` — so `Eq A` is literally a type. For single-method traits like `Eq`, `Eq Nat` *is* the type `Nat Nat -> Bool`. For multi-method traits, it's a nested Sigma (product) of the method types. The trait "constrains" a type `A` by demanding that a value of type `Eq A` exists — a dictionary, which is a witness that `A` has the required operations.

A `where (Eq A)` clause on a function signature becomes an implicit Pi binding:

```markdown
spec foo : {A :0 Type} -> {$Eq-A :w (Eq A)} -> A -> A -> Bool
```

The elaborator (`elaborator.rkt:100-156`) detects the `$`-prefixed dict param name, parses it into `(trait-name: Eq, type-vars: [A])`, builds a `where-method-entry` for each trait method, and populates `current-where-context`. When the function body mentions `eq?`, `resolve-method-from-where` emits `(Eq-eq? A $Eq-A)` — the accessor applied to the type variable and the dictionary.

So the constraint is:

1. **At the type level**: an additional Pi parameter of type `Eq A`
2. **At the term level**: a dictionary value passed at every call site
3. **At resolution time**: a metavariable that the trait resolver must solve to a concrete dictionary expression

The dict param uses `mw` (unrestricted multiplicity), not `m0` (erased), because the dictionary is a *runtime* value — the function body actually calls methods through it. This is the QTT connection: trait constraints are computationally relevant witnesses, not erased propositions.

## Bundles as Constraint Conjunction

Your insight is exactly right and it's implemented cleanly. `bundle BoundedLattice := (Lattice, HasTop)` registers a `bundle-entry` with `constraints: '((Lattice A) (HasTop A))`. When expanded, it becomes two separate dict params, two separate constraint metas, two separate resolutions. The bundle is a **logical AND** — a conjunction in the constraint space.

This is fundamentally different from inheritance. In a subtype hierarchy, `BoundedLattice <: Lattice` would mean "every BoundedLattice IS a Lattice." In the bundle model, `BoundedLattice A` means "both `Lattice A` AND `HasTop A` are satisfied." The distinction matters because:

- Conjunction is commutative and associative — `(Lattice, HasTop)` = `(HasTop, Lattice)`
- Conjunction is extensible without coherence problems — adding a new trait to a bundle doesn't create a diamond
- Conjunction is flat — no depth, no method resolution order, no linearization

In Prolog terms: each line of a predicate body is a constraint on the goal. `parent(X, Y) :- mother(X, Y).` and `parent(X, Y) :- father(X, Y).` are disjunctions (OR — two clauses). But within a clause body, `grandparent(X, Z) :- parent(X, Y), parent(Y, Z).` the comma is conjunction (AND). Bundles are the conjunctive case — they compose constraints, not alternatives.

```prologos

**Level 2: Traits as records with method-level access.** If trait dictionaries carried their structure at the type level (via row types or a method signature index), you could project dynamically:

This is essentially what you get with row polymorphism — the type `{ eq? : A A -> Bool | r }` lets you project `eq?` without knowing the full trait. It connects directly to your schema system: a schema is a row type for data, a trait could be a row type for operations.

**Level 3: Traits as propositions in the Curry-Howard sense.** Your type system is dependently typed. `Eq A` is a proposition ("A has decidable equality"). The dictionary is a proof. `where (Eq A)` is universal quantification over proofs. First-class traits at this level means first-class propositions — quantifying over *which property* is satisfied:

```prologos
spec parametric-over-property :
  {P : Type -> Type} -> {A : Type} -> P A -> A -> Bool
```

This is already expressible in your Pi calculus — `P` is just a type-level function. The challenge is that without knowing `P`'s structure, you can't do anything useful with the `P A` value. You'd need either:
- **A universe of traits** (reify trait structure as data)
- **Structural type matching** (pattern match on the dictionary's type to discover methods)
- **A `HasMethod` constraint** (another trait-level constraint: "P has a method named `check` of type `A → Bool`")

The third option is the most practical and it's essentially row polymorphism at the trait level.

### What First-Class Traits Buy You

1. **Higher-order trait abstraction**: Functions that are generic over which trait they require. This is useful for test frameworks ("run this property-based test for any type that satisfies *some* trait"), serialization frameworks ("serialize any type that has *some* encoding"), and meta-programming.

2. **Trait-level computation**: Compute which traits are needed based on runtime or type-level conditions. `if condition then require Eq else require Ord` — conditional constraints.

3. **Reflection**: Enumerate the methods of a trait at runtime. Useful for REPL tooling, documentation generation, and the kind of structural introspection that makes exploratory programming powerful.

4. **The narrowing connection**: If traits are values in the search space, the narrowing engine can search over *which trait instance to use* — not just which value to assign.

### The Current Gap

### Trait Dispatch as a Propagator Cell

Model it directly in your existing propagator infrastructure:

```markdown
trait-dispatch-cell:
  value: lattice of possible impl entries
  bot:   all impls possible (unconstrained)
  top:   no impl satisfies constraints (contradiction)
  join:  intersection (add constraint → fewer candidates)
```

This is a **finite domain** — the set of registered impls for a trait. A `trait-dispatch-cell` for `Add ?A` starts at `{Add Nat, Add Int, Add Rat, Add (List A), ...}` and narrows as type information flows in.

The propagators connecting it:

1. **Type→Dispatch propagator**: When the type cell for `?A` refines (e.g., from `type-bot` to `Nat`), remove impls that don't match. `?A = Nat` → `{Add Nat}`.

2. **Dispatch→Type propagator** (the novel direction): When the dispatch cell narrows (e.g., only `Add Nat` and `Add Int` remain), constrain the type cell. `{Add Nat, Add Int}` → `?A ∈ {Nat, Int}` (a union type or finite domain).

3. **Method→Constraint propagator**: Each remaining impl candidate has concrete method implementations. The `add` method for `Add Nat` has type `Nat → Nat → Nat`. The `add` for `Add Rat` has type `Rat → Rat → Rat`. This constrains the return type of `.{x + y}`.

4. **Result→Dispatch propagator**: If the result is known (e.g., `.{x + y} = 5N` where `5N : Nat`), the return type constrains which impls are viable: only impls whose `add` returns `Nat`. This eliminates `Add Rat`, `Add Int`, etc.

This is **bidirectional** — exactly what propagators are for. And it connects directly to your narrowing search: at each DFS node, the trait-dispatch propagator either determines a unique impl (proceed) or enumerates candidates (branch).

### The Mixfix-Narrowing Integration

```prologos

;; Elaboration:

;;   2. add → Add-add ?A $Add-?A → trait constraint

;;   Cell α: type of x (?A)         — starts at type-bot

## What Level 3 genuinely gives beyond Level 2

**Level 2** lets you narrow through trait-dispatched operators and get bidirectional type↔dispatch flow. But the trait system remains *opaque* — the propagator network knows "there exists an `Add Nat` impl" but can't reason about *what a trait is* at the type level. Traits are still Racket-side registry entries, not Prologos values.

**Level 3** makes traits first-class in the Curry-Howard sense. Given that Prologos already treats types as propositions (dependent types), QTT multiplicities as resource evidence, and session types as protocol specifications — trait dictionaries as *proof witnesses* is the natural completion. `Eq Nat` isn't just "a type whose inhabitants are equality functions" — it's the *proposition* "Nat has decidable equality," and the dictionary is the *evidence*.

This enables things Level 2 can't:

1. **Quantifying over properties**: `{P : Type → Type} → P A → ...` — "for any property P that A satisfies." This is genuinely new expressivity. You could write a function that takes *any* trait dictionary and does something with it (e.g., a generic serialization framework, a property-testing harness).

2. **Trait combinators as values**: Instead of `bundle Num := (Add Sub Mul)` being syntactic sugar that expands at the call site, `Num` could be a *computed value* at the type level. Trait combination becomes type-level programming, not macro expansion.

3. **Proofs about traits**: With dependent types, you could state and prove things like "if `A` has `Ord`, then `List A` has `Ord`" — not as an impl registration side effect, but as a type-level theorem.

4. **Trait-parametric propagators**: In the propagator network, a cell could be parameterized over *which trait constraint* it's tracking. Instead of hardcoding "this is an Add dispatch cell," you'd have "this is a dispatch cell for trait P," where P is a variable.

## Why I still recommended Level 2 first — the real blocker

The honest reason isn't "Level 3 is too hard" in general. It's one specific thing: **method projection**.

If you have `dict : P A` and don't know what `P` is, you can't *do* anything with `dict`. You can pass it around, store it, return it — but you can't call `eq?` on it because you don't know `P` has a method called `eq?`. The dictionary is a black box.

To make Level 3 *useful* (not just expressible), you need one of:

- **Row polymorphism on trait methods**: `{P : Type → Type, P has { eq? : A → A → Bool | r }}` — structural access to methods by name. This is a significant type system extension (row types).
- **A universe of traits**: Reify trait structure as data (`trait-methods : Trait → List Method`), then do type-level computation over it. Requires a reflection/universe mechanism.
- **HasMethod constraints**: `where (HasMethod P "eq?" (A → A → Bool))` — like Haskell's `HasField` for records, but for traits. Requires a constraint system extension.

Each of these is a substantial piece of type system infrastructure that Prologos doesn't have yet. Level 2 needs *none* of them — it works entirely within the existing type system, just adding a new lattice to the propagator network.

## The path-dependency concern

Here's what I think your real question is: "Will building Level 2 create assumptions that make Level 3 harder later?"

I don't think so, for a structural reason: **Level 2 is a subset of Level 3**. Everything in Level 2 (dispatch cells, bidirectional propagators, conditional activation) would still exist in a Level 3 world. The dispatch lattice doesn't assume traits are opaque — it just doesn't *require* them to be transparent. Adding row polymorphism or a trait universe later would give the type system more information to feed *into* the dispatch cells, not replace them.

The design principle to preserve is: **don't bake in the assumption that trait names are atoms.** If the dispatch cell stores `impl-key` symbols today, make sure the abstraction layer allows those to become type-level-computed values later. That's a small API discipline, not an architectural constraint.

- Dispatch cells should be parameterized by a trait *identifier* that could later become a type-level value (not just a symbol)

## The Session Types Precedent

The pattern you're describing is: **first-class-ness creates affordances you can't anticipate**. ## The Design Principle

Yes, this should absolutely be captured. The principle is something like:

> **First-Class by Default.** When introducing a language construct — types, multiplicities, session protocols, traits, propagator cells — make it a first-class citizen: a value that can be stored, passed, returned, abstracted over, and computed upon. The marginal cost of first-class-ness is typically modest (reification + type rules), but the option value is unbounded. First-class constructs compose with every other first-class construct, creating a combinatorial surface for emergent capabilities. Prologos's history confirms this: session types as first-class types yielded composable protocol continuations — a capability that wasn't designed but *fell out* of the architecture.

This connects to several existing principles — homoiconicity (code-as-data), the Curry-Howard thread, and the propagator network philosophy (every domain is a lattice, every lattice is a cell, every cell participates in the same network). It's the *meta-principle* behind all of those.

## Opening Up Level 3: What's Actually Needed?

I was too hand-wavy about Level 3's requirements. Let me be specific about the three paths and what each actually costs in Prologos's current infrastructure.

### Path A: HasMethod Constraints (Most Achievable)

This is the **minimum viable Level 3**. Analogous to Haskell's `HasField` for records.

```prologos
;; Built-in constraint:
;; HasMethod : (Type → Type) → Symbol → Type → Constraint

spec apply-method {P : Type → Type} {A : Type}
  where (HasMethod P "eq?" <A A -> Bool>)
  [P A] A A -> Bool
```

What it needs:
1. **A `HasMethod` built-in constraint type** — the resolver looks up the trait registry (which *already stores* method names and types in the `trait-method` struct at macros.rkt:6222). This is a query against existing data.
2. **A `project` operation** — extracts a method from a dictionary by name. For single-method traits (dict IS the function), this is identity. For multi-method traits, it's record field access.
3. **Constraint resolution integration** — `HasMethod Eq "eq?" T` checks the registry, unifies `T` with the method's type, done.

The key realization: **the trait registry already has all the information `HasMethod` needs**. The `trait-meta` struct stores methods. The resolver is a lookup + unification. No new type theory.

Cost: moderate. It's a new constraint type with a custom resolver, plus a projection primitive. The infrastructure for custom constraint resolution already exists (it's what trait constraint resolution does). This is *within the current type system* — no new type-theoretic machinery.

1. **Row kinds** — a new kind `Row` alongside `Type`
2. **Row variables** — type variables of kind `Row`, with row unification (Rémy-style)
3. **Record types with row tails** — `{ field : T | r }` syntax
4. **Row constraints** — lacks/has constraints for field uniqueness
5. **Interaction with dependent types** — row types in Pi, Sigma contexts

This is a *substantial* type system extension, but it's also useful far beyond traits — it gives you extensible records, polymorphic variants, and modular effects. It's the kind of feature that's worth doing for its own sake.

## What's Essential vs. Incidental

You're asking exactly the right question. Let me separate theory from convention.

### Essential (grounded in type theory)

There are really only **three** essential operations for first-class traits:

1. **Trait abstraction** — quantifying over *which property* is satisfied. In type theory, this is second-order universal quantification: `∀P. ∀A. P(A) → ...`. This is Pi over predicates. Every dependently-typed language with universe polymorphism already has the substrate for this — `P : Type → Type` is just a function at the type level.

2. **Method projection** — given evidence `d : P A` and knowledge that `P` has a method with a certain signature, extracting that method from the dictionary. This is *elimination* for record/existential types. The essential content is: "this evidence contains a function of this shape, and I can get it out."

3. **Evidence construction** — building a dictionary for a specific type, possibly from sub-evidence. This is already what `impl` does. The essential content is: "here is a witness that type A satisfies trait P."

That's it. Everything else — `HasMethod`, row variables, trait universes, `project` — is *mechanism for expressing* these three operations. The mechanism is incidental; the operations are essential.

### Incidental (borrowed from other languages)

- **`HasMethod P "eq?" T`** — Haskell's `HasField` pattern. Uses a *string* to name the method. This is incidental — method identity could be structural (position in a record), symbolic (a reified name), or type-based (the signature itself).
- **Row polymorphism** `{ eq? : A → A → Bool | r }` — PureScript/OCaml pattern. Elegant but specific to structural typing. Prologos traits are *nominal* (you declare `trait Eq`, not a structural method set).
- **Trait universe** `TraitCode = EqCode | ...` — Agda/Idris reflection pattern. Requires a closed enum of traits, which conflicts with open extension.

The lesson from `spec` is: **express essential operations through structured metadata on existing keywords**. The "Pi Types as Implementation Detail" principle already says dependent types are the *substrate*, not the surface. The same should apply to trait abstraction.

## What's striking about `?n:PosInt:Even`

The `:` chain is doing something theoretically unified that *looks* like three different things but is actually one thing:

```prologos
?xs:Ord        ;; trait constraint on the TYPE of ?xs
?n:PosInt      ;; type constraint on the VALUE of ?n  
?n:PosInt:Even ;; type + predicate constraint
```

Naively these seem like different mechanisms — "the type satisfies a trait" vs "the value inhabits a type" vs "the value satisfies a predicate." But in a dependently-typed language, **they're all the same thing**: propositions that narrow a domain.

- `PosInt` is a type. Its inhabitants are positive integers. Constraining `?n:PosInt` says "the domain of ?n is the set of positive integers."
- `Ord` is a trait — `Type → Type`. Constraining `?xs:Ord` says "the type of ?xs must have an ordering." This is a proposition about the type, witnessed by a dictionary.
- `Even` could be a dependent predicate — `Nat → Type`. `Even n` is the proposition "n is even." Constraining `?n:Even` says "?n must satisfy evenness."

In all three cases, the `:` is introducing a **domain restriction**. The compiler can distinguish which kind by looking at the *kind* of the constraint:

| Constraint kind | What it means | Elaboration |
|---|---|---|
| `C : Type` | Value inhabits this type | Type annotation on the logic variable |
| `C : Type → Type` | Type of variable satisfies this trait | Dispatch cell in the propagator network |
| `C : A → Type` | Value satisfies this dependent predicate | Proof obligation / constraint cell |

The elaborator already does kind-directed dispatch — this is the same mechanism. The `:` chain is just syntactic sugar for "add these constraints to the propagator network for this variable."

## Reflection: Every Line Is a Type

The Prolog insight you're describing — "every line is a type" — is exactly the right lens. Each relational clause IS a constraint on the space of possibilities. Each conjunction narrows the venn diagram. The programmer is sculpting a *shape* in possibility space, not writing sequential instructions.

What Prologos is doing is making this *literal*. In traditional Prolog, "every line is a type" is a metaphor — the programmer sees it that way, but the runtime is doing chronological backtracking with no awareness of the constraint geometry. The constraints are procedurally ordered because the engine doesn't understand their structure.

With propagator networks + dispatch cells + first-class traits, the constraint geometry becomes *the actual execution model*. Each constraint is a cell in the network. The narrowing is genuinely order-independent — information flows bidirectionally until quiescence. The "most constrained first" heuristic isn't something the programmer worries about; it's something the network discovers through propagation.

And the `?n:PosInt:Even` syntax makes this philosophy surface-visible. Each `:` adds a circle to the venn diagram. The programmer is literally writing constraint geometry, and the runtime executes it as constraint geometry. No gap between intent and mechanism.

### The `[]` bracket choice

I think you're right that `[]` works for parameterized constraints:

```prologos

?xs:SortedBy[compare]:NonEmpty
?p:Satisfies[prime?]
```

The `?` prefix already signals "relational variable." The `[]` signals "functional evaluation / application" — which is correct, because `Between[50 100]` is constructing a constraint value. You're applying a constraint constructor to arguments. The paradigm blurring is intentional and well-motivated: constructing the constraint is functional, using it is relational.

And it composes with the chain naturally. `?n:Between[50 100]:Even` reads left-to-right as progressive narrowing: start with ?n, constrain to the interval [50,100], further constrain to even values. Each `:` is conjunction. The propagator network processes it as simultaneous constraints, but the syntax reads as a narrative of refinement.

### Where the lines blur — and why that's good

The LF-narrowing work already blurs functional and relational: a functional equation `f(x) = y` becomes a relational constraint that can be solved in any direction. First-class traits blur another boundary: type-level reasoning and value-level reasoning. The dispatch cell is simultaneously a type system concept (which impl to use) and a constraint programming concept (what domain does this variable range over).

With `?n:PosInt:Even`, you're blurring a third boundary: type annotations and domain constraints. In the functional world, `x : Int` classifies a value. In the relational world, `?x:Int` constrains a search space. Same syntax, different operational meaning, unified semantics.

These aren't accidental blurrings — they're *the point*. Each boundary that dissolves reveals that two "different" things were always the same thing seen from different angles. Types ARE constraints. Traits ARE domain refinements. Functional evaluation IS relational narrowing. The language just makes it visible.

## Summary: Where We've Landed

### 1. First-Class by Default (Design Principle — committed `6825d38`)

Language constructs should be first-class citizens by default. First-class-ness creates combinatorial affordances that can't be predicted. The session types precedent: composable protocol phases emerged from session types being first-class, not from deliberate design.

### 2. Traits Are Types-Refining-Types

The essential nature of a trait: **a function from Type to Type that refines a type with additional structure**. `Eq : Type → Type` — given a type `A`, `Eq A` is the refined type "A with decidable equality." The dictionary is the evidence of refinement.

This is not how Haskell or Rust thinks about traits. It's grounded in dependent type theory: traits are type families, dictionaries are proof terms, trait constraints are propositions. Nothing borrowed, everything derived from the type-theoretic substrate Prologos already has.

### 3. Three Levels Are Cumulative, Not Alternative

- **Level 1**: Trait names are type constructors (`Eq : Type → Type`). Can be stored, passed, applied at the type level. Nearly free — add to tycon table.
- **Level 2**: Trait dispatch is a propagator cell. Finite-domain lattice of impl candidates, narrowing monotonically. Four bidirectional propagator types. Integrates into the existing propagator network.
- **Level 3**: Traits as first-class propositions. Quantify over *which trait*. Method projection via `:method` / `:has` metadata (the HasMethod mechanism, surfaced the Prologos way — structured metadata, not raw type theory). Enables trait-polymorphic programming.

Each level builds on the prior. No level precludes the next. The design target is Level 3; the implementation path goes through Levels 1 and 2.

### 4. The HasMethod Blocker Is Smaller Than It Appeared

The "method projection" problem — given `dict : P A` where P is unknown, how to extract methods — doesn't require row types or a trait universe. The **HasMethod constraint** (surfaced as `:method` or `:has` metadata on `spec`) is:
- A query against the existing trait registry (which already stores method names and signatures)
- A projection primitive (identity for single-method traits, field access for multi-method)
- Comparable in complexity to adding a new built-in trait, not a new type system feature

This belongs on the main roadmap, not deferred to a separate design session.

### 5. Unified Constraint Syntax for the Relational World

The `:` chain on logic variables (`?n:PosInt:Even`, `?xs:Ord`, `?n:Between[50 100]`) unifies three kinds of constraint through one syntax:

| Constraint kind | Example | What narrows |
|---|---|---|
| Type membership | `?n:PosInt` | Value domain (inhabitants of PosInt) |
| Trait satisfaction | `?xs:Ord` | Type domain (types with Ord impl) → dispatch cell |
| Dependent predicate | `?n:Even` | Value domain (values satisfying Even) |

The compiler distinguishes them by kind. The propagator network doesn't care — all three are cells that narrow domains monotonically. The syntax reflects the unified semantics.

Parameterized constraints use `[]`: `?n:Between[50 100]:Even`. `?` signals relational variable, `[]` signals functional construction of the constraint, `:` signals conjunction.

### 6. The Dispatch Cell Generalizes to Constraint Cell

The original design doc proposed `dispatch-lattice.rkt` specifically for trait impl candidate sets. The relational constraint syntax reveals this is one instance of a more general pattern: **any finite-domain constraint on a variable is a cell in the propagator network**. The module design should reflect this generality:

- Trait dispatch cells (impl candidate sets)
- Type membership cells (inhabitants of a type)
- Predicate satisfaction cells (values satisfying a dependent predicate)
- Parameterized constraint cells (Between, LessThan, etc.)

All share: bot = unconstrained, join = intersection, top = contradiction, monotonic narrowing.

### 7. Surface Syntax Follows "Prologos Way"

Complex type theory hides behind structured metadata:

```prologos
;; Trait abstraction — no raw Pi types in surface
spec apply-eq {A : Type} A A -> Bool
  :over P
  :method P eq?

;; Relational constraints — no separate constraint declarations
defr sum-evens [?xs:Nat:Even ?total:Nat]
  (fold add zero ?xs ?total)

;; Functional constraints — existing where clause
spec sort {A : Type} where (Ord A) [List A] -> [List A]
```

Each context gets the ergonomic surface that fits its paradigm, all elaborating to the same propagator network cells.

### 8. Order-Independent Constraint Logic

The philosophical commitment: constraints are genuinely order-independent. The programmer draws circles (each clause, each `:` constraint), the propagator network computes the intersection. No chronological backtracking ordering. "Most constrained first" emerges from propagation dynamics, not programmer discipline.

First-class traits enable this for trait-dispatched operations: instead of the programmer ordering trait resolution (current batch model), the network resolves traits as part of the same fixpoint as type inference and value narrowing.

## Current State

## The Triangle You're Describing

You're identifying three vertices:

1. **Positional tuples** — what Prolog uses, what we have now. Compact, fast to unify, but opaque: `("Alice" "Engineering" 95000)` means nothing without the schema nearby.

2. **Keyed maps** — what your Prolog-with-databases workflow uses (`table_name({column_key: Val, ...})`). Self-describing, composable, but heavier and potentially slower for the hot unification path.

3. **Typed records** — what schemas aspire to be. Named fields, type-checked, but with the efficiency of positional access.

The tension: positional is fast but fragile; keyed is robust but heavy; records are ideal but usually require compile-time resolution.

## How to Square It

I think the answer is already implicit in your schema design, but making it explicit reveals the right representation:

### The Schema *is* the Rosetta Stone

A `schema` declaration establishes a **bijection between positions and keys**:

```prologos
schema Employee
  :name   String    ;; position 0
  :dept   Department ;; position 1
  :salary Nat       ;; position 2
```

This means the underlying runtime representation can be **positional vectors** (fast, compact, cache-friendly) while the *surface syntax* supports **both positional and keyed access**. The schema mediates:

- **Storage**: flat vector `#("Alice" Engineering 95000)` — essentially a database row
- **Authoring (positional)**: `|| "Alice" Engineering 95000N` — tabular, concise
- **Authoring (keyed)**: `|| {:name "Alice" :dept Engineering :salary 95000N}` — self-documenting
- **Access (keyed)**: `alice.name` or `:name` in goal patterns
- **Unification**: positional, against the vector, always

The key insight: **keyed syntax is a compile-time convenience that erases to positional access**. The schema provides the mapping; after elaboration, everything is positional. This is exactly what databases do — SQL lets you say `WHERE name = 'Alice'` but the storage engine uses column offsets.

### What About Schema-less Relations?

Bare `defr` without a schema annotation works like traditional Prolog — positional only, untyped. This is fine for quick prototyping and small relations. The moment you want keyed access or database interop, you add a schema.

### The Database Import Use Case

Your Prolog workflow of `table_name({column_key: Val, ...}).` maps beautifully:

```prologos
;; Import: each table gets a schema
schema Employees
  :id     Nat
  :name   String
  :dept   String
  :salary Nat

;; Facts loaded from CSV/SQL/JSON — keyed or positional
defr employees : Employees
  || 1N "Alice" "Engineering" 95000N
     2N "Bob"   "Marketing"   72000N
     3N "Carol" "Engineering" 110000N

;; Or bulk-loaded from file
defr employees : Employees
  :source "data/employees.csv"
```

The schema does double duty: it's the type contract *and* the column mapping for import/export. A CSV header row, a SQL `CREATE TABLE`, a JSON schema, and a Prologos `schema` all describe the same thing — a named sequence of typed fields. The schema is the interop surface.

### PVec vs Flat Vector vs CHAMP Map

For the underlying runtime representation, the options are:

| Representation | Pros | Cons |
|---|---|---|
| **Flat Racket vector** | Fastest access, minimal allocation | Not persistent, no structural sharing |
| **PVec (RRB tree)** | Persistent, shareable, log-time access | Overkill for small fixed-width rows |
| **CHAMP map** | Keyed access, persistent | Heaviest, hash overhead per lookup |
| **Struct** | Fastest, named fields | No dynamism, one struct per schema |

My analysis: for **fact storage**, flat vectors (or even Racket structs generated per-schema) are the right choice. A fact row is small (typically 3-15 fields), fixed-width (schema determines it), and accessed many times during unification. The hot path is "unify position i of fact with goal argument i" — this wants O(1) indexed access, not hash lookup.

PVec and CHAMP are great for *user-facing collections* (lists, maps in the functional language), but using them for the internal fact representation would be paying persistent-data-structure overhead for data that doesn't need persistence — facts are immutable once asserted.

**The representation I'd suggest**: compile-time-generated Racket structs per schema (fastest) with a fallback to flat vectors for schema-less relations. The struct fields correspond to schema columns. The schema provides the name→position mapping for keyed syntax. This is essentially what Ecto changesets do — the schema generates a struct, and the struct is the runtime representation.

### Keyed Pattern Matching in Goals

The really interesting ergonomic question is: can you use keyed patterns in relational goals?

```prologos
;; Positional (current)
(employees id name "Engineering" salary)

;; Keyed (proposed) — only bind what you care about
(employees {:dept "Engineering" :name name :salary salary})

;; Mixed — positional with keyword override
(employees _ name :dept "Engineering" _)
```

The keyed pattern form is a big ergonomic win for wide relations. If your `employees` table has 15 columns, positional patterns require 15 arguments with `_` for every column you don't care about. Keyed patterns let you mention only the columns you need — which is exactly what SQL's `SELECT ... WHERE` does.

At compile time, with the schema in hand, `{:dept "Engineering" :name name}` desugars to a positional pattern with logic variables in the right positions and `_` everywhere else. Zero runtime overhead.

### The Graph-Query Power

The combination of schemas + keyed patterns + tabling + rule composition gives you something very powerful for your database use case:

```prologos
;; Schema per table (generated from DB metadata)
schema Employees  :id Nat :name String :dept String :salary Nat
schema Departments  :name String :budget Nat :head String

;; Facts loaded from DB
defr employees : Employees
  :source "db://company/employees"

defr departments : Departments
  :source "db://company/departments"

;; Composable graph queries — this is where Prologos shines
defr high-earners-in [?dept ?name ?salary]
  &> (employees {:dept dept :name name :salary salary})
     (> salary 90000N)

defr over-budget-depts [?dept ?total ?budget]
  &> (departments {:name dept :budget budget})
     (aggregate (employees {:dept dept :salary s}) [sum s] total)
     (> total budget)
```

Each `defr` is a reusable, composable, tabled view. You can build them up incrementally, test them individually, and compose them freely — something that requires CTEs and subqueries in SQL.

## Summary of the Design

1. **Storage**: flat vectors (or generated structs), positional, compact
2. **Schema**: the bijection between positions and named/typed keys
3. **Authoring**: dual syntax — positional `||` for tabular data, keyed `{...}` for clarity
4. **Goals**: keyed patterns desugar to positional at compile time
5. **Interop**: schema is the import/export contract (CSV, SQL, JSON all map)
6. **Schema-less**: falls back to positional-only Prolog-style (no keyed access)

The schema is the load-bearing element. It resolves the positional-vs-keyed tension by being the compile-time dictionary that lets you *write* keyed and *execute* positional. No runtime overhead for the keyed ergonomics. No loss of self-documentation. And the same schema serves as the database interop surface.
