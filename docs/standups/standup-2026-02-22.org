#+begin_src markdown
Good — this is worth thinking through carefully. Let me lay out the considerations.

## The mainstream precedent is strong and consistent

Every language with mixed numeric types follows the same rule: approximate dominates exact. `2 + 3.0 → 5.0` in Python, Ruby, JavaScript, C, Java, Haskell, Common Lisp, Clojure, Julia. No mainstream language returns an exact integer when one operand is a float. Users don't just expect this — they rely on it unconsciously. Deviating from it would be a genuine source of bugs for anyone coming from another language.

The reason every language converges here is semantic: once you've entered the approximate domain, you can't get back out. `3.14` has already lost information (it's the *nearest* Posit32 to 157/50, not 157/50 itself). Claiming the result is exact after mixing with an approximate value would be dishonest.

So I think your instinct is right: Posit dominance as the default for generic operators (`+`, `-`, `*`, `/`) is the correct design.

## On detecting exact forms in Posits

You're right to dismiss this. The problems compound:

1. **Efficiency**: Every arithmetic operation would need a post-hoc check — "is this Posit32 bit pattern exactly representable as a Rat?" That's a decode + rationality check on every operation, not just at boundaries.

2. **Semantic incoherence**: Some Posit values *are* exactly representable as rationals (powers of 2, small integers), but most aren't. So `1.0 + 1.0` could return `2 : Int` while `1.0 + 1.1` returns `Posit32`. The return type of `+` would depend on the *runtime values*, not the *types*. This violates a core principle of static typing — types should be predictable at compile time.

3. **You'd still need dominance as fallback**, as you noted. So you'd be implementing the complex path *and* the simple path, with the complex path firing unpredictably. Worst of both worlds.

4. **Posit-specific wrinkle**: Posits have a tapered precision regime. A Posit32 can represent some very large integers exactly (near the center of the projective circle) but not others. The boundary between "exact" and "approximate" in Posit space is not intuitive. Users would be surprised by which operations "recovered" exactness and which didn't.

## On opt-in Posit dominance

The idea I sketched in the audit was something like:

```prologos
ns my-module :posit-dominance    ;; module-level opt-in

;; Or pragma-level:
{-# POSIT-DOMINANCE #-}
```

Here are the tradeoffs:

**Pro opt-in:**
- Conservative — no implicit precision loss without consent
- A function's type signature tells you everything (no hidden coercions)
- Libraries don't surprise consumers with unexpected widening
- Aligns with the "explicit is better than implicit" philosophy that exact-arithmetic communities value

**Con opt-in (and I think these are decisive):**
- **It fractures the language into two dialects.** Library A written with `:posit-dominance` and Library B without it would have different semantics for `[+ x y]` on the same types. This is exactly the kind of braiding/complecting that Prologos's design philosophy opposes.
- **Nobody will opt in.** Pragmas that aren't the default are effectively dead features. The people who most need mixed-type convenience (scientists, data analysts, numerical programmers) are the least likely to know about a pragma.
- **It undermines the generic operator story.** The whole point of Phase 2 (`+`, `-`, `*`, `/`) is "write once, works for all numeric types." If mixed types are rejected by default, the generic operators are generic in name only — you still need manual conversion at every boundary.
- **The type-specific keywords already serve as the strict opt-out.** `int+` and `p32+` are already there for when you want to enforce exact type matching. Adding a pragma creates a *third* level of control (type-specific keywords / strict generic / permissive generic) that's harder to explain and teach.

## My recommendation

Make Posit dominance the **default** behavior for generic operators (`+`, `-`, `*`, `/`), with these properties:

1. **Generic operators coerce**: `[+ 42 3.14] → Posit32`. The Int is widened to Posit32 before the operation. The return type is always the join of the operand types.

2. **Type-specific keywords remain strict**: `[int+ 42 3.14]` is a type error. `[p32+ x y]` requires both args to be Posit32. This is your escape hatch, and it already exists — no new machinery needed.

3. **Informational notes, not warnings**: When cross-family coercion happens, emit a note (as sketched in Phase 3c). Not a warning, not an error — just information. This respects the user's choice to use generic operators while keeping them informed. Notes can be silenced per-module if desired.

4. **No pragma, no dialect split.** One language, one set of rules. The rules are: generic operators are permissive, type-specific keywords are strict. Simple to explain, simple to learn.

The principle at work: **the syntax encodes the user's intent**. Writing `[+ a b]` says "I want addition, figure out the types." Writing `[int+ a b]` says "I want Int addition specifically." Posit dominance in the first case is not the language making a decision behind the user's back — it's the language honoring the semantics of the generic operation the user asked for.

## Question 3: Multi-Arity Specs with Metadata

### Current State

Today, multi-arity specs use `|` (which becomes `$pipe`) to separate branches:

```prologos
;; WS mode
spec greet
  | Nat -> Nat
  | Nat Nat -> Nat

;; sexp mode
(spec greet ($pipe Nat -> Nat) ($pipe Nat Nat -> Nat))
```

Each branch is a fully independent type signature. The `spec-entry` struct stores them as a list of type-datums, with `multi?` set to `#t`. Implicit binders, where-constraints, and docstrings are all shared across all branches — they attach once to the spec-entry, not per-branch.

The question is: when you add `:properties`, `:examples`, etc., what's the relationship between the metadata and the branches?

### The Option Space

**Option A: Metadata is function-level (the current recommendation)**

```prologos
spec greet
  | Nat -> String
  | String Nat -> String
  :doc "Greet someone"
  :examples
    - [greet 42N] => "hello 42"
    - [greet "world" 3N] => "hello world world world"
  :properties
    - :name "greet-nat-positive"
      :forall {n : Nat}
      :holds [str-length [greet n] > 0N]
```

Metadata comes after *all* branches. Properties quantify over whatever they like and don't know which branch they test — the dispatch mechanism handles it.

**Option B: Metadata is per-branch**

```prologos
spec greet
  | Nat -> String
    :examples
      - [greet 42N] => "hello 42"
    :properties
      - :name "greet-nat-positive"
        :forall {n : Nat}
        :holds [str-length [greet n] > 0N]
  | String Nat -> String
    :examples
      - [greet "world" 3N] => "hello world world world"
    :properties
      - :name "greet-repeat-length"
        :forall {s : String} {n : Nat}
        :holds [eq? [str-count-substr [greet s n] s] n]
```

Each branch carries its own metadata. This is more specific — the property about Nat input only applies to the Nat branch.

**Option C: Both levels**

```prologos
spec greet
  :doc "Greet someone"                ;; function-level
  | Nat -> String
    :examples                          ;; branch-level
      - [greet 42N] => "hello 42"
  | String Nat -> String
    :examples
      - [greet "world" 3N] => "hello world world world"
  :properties                          ;; function-level (applies across all)
    - :name "non-empty"
      :forall {n : Nat}
      :holds [str-length [greet n] > 0N]
```

### Critique

**Option A** is simplest. It matches how `spec-entry` already works — one metadata per function. Properties that mention specific argument types implicitly target the right branch via the type checker. But it has a real limitation: you can't write a property like `:forall {s : String} {n : Nat} :holds ...` without the system needing to figure out that this is testing the second branch. If the branches don't overlap in arity, dispatch is trivial. If they do overlap in arity (say, both take one arg but different types), the property's `:forall` types guide dispatch — which is actually fine because the forall'd types will only match one branch.

**Option B** is more precise but creates a structural problem: the implicit map syntax works by detecting keyword-headed tails under a `def`/`spec` head. If metadata is per-branch, the parser needs to understand that a `|` introduces a sub-scope with its own keyword block. That's doable but adds nesting complexity. It also leads to repetition — `:doc` would need to go somewhere function-level anyway, so you end up with Option C.

**Option C** is the most expressive but also the most complex to parse. You need two levels of scope: function-level keywords before/after the branches, and branch-level keywords nested inside each branch. The WS reader's implicit-map syntax would need to distinguish "this keyword block belongs to the branch" vs "this keyword block belongs to the whole spec." That's tractable via indentation (branch-level keywords are indented under the `|`), but it's the most involved parsing change.

### My actual recommendation

**Option A with a refinement.** Properties already have `:forall` binders. Those binders determine which branch the property exercises. The type checker doesn't need to be told — it infers it from the quantified types. And for examples, the call syntax itself picks the right branch:

```prologos
spec greet
  | Nat -> String
  | String Nat -> String
  :examples
    - [greet 42N] => "hello 42"              ;; obviously exercises branch 1
    - [greet "world" 3N] => "world world world"  ;; obviously exercises branch 2
```

There's no ambiguity because the example *is* a concrete call. Same for properties — the `:forall` types determine which branch fires. Per-branch metadata is theoretically more precise but practically redundant: the types already carry the dispatch information.

The one place where Option B genuinely adds value is per-branch `:doc` strings — "what does this branch mean?" But that's better served by comments above each `|` line, which is already idiomatic.

## Question 4: Could `:properties` Subsume Implicit Binders?

This is a really interesting question. Let me first make sure the intuition for implicit binders is clear, then explore the subsumption idea.

### When do you need implicit binders today?

Implicit binders (`{A B : Type}`) serve one purpose: **introducing type variables that don't appear in positional argument types but need to exist in the type signature.** In practice:

```prologos
;; Without implicit binders — A is inferred from the argument:
spec length [List A] -> Nat
;; The type checker sees (List A) and infers A must be a type variable.
;; No {A : Type} needed — A appears in a position that constrains it.

;; WITH implicit binders — needed when A is "free":
spec empty {A : Type} [List A]
;; Without {A : Type}, the system doesn't know what A is.
;; There's no argument position to infer it from.

;; HKT needs it for kind annotation:
spec gmap {A B : Type} {C : Type -> Type} (Seqable C) -> (Buildable C) -> [A -> B] -> [C A] -> [C B]
;; {C : Type -> Type} tells the kind checker C is a type constructor, not a type.
```

The honest answer is: **you need explicit implicit binders when the type checker can't infer either the existence or the kind of a variable from the function's argument/return types.** That's two cases:

1. **Type variables with no constraining position** (like `empty : {A : Type} -> List A`)  
2. **Higher-kinded type variables** where the kind matters (like `{C : Type -> Type}`)

For most everyday code, you don't need them. The type checker will infer `A` from `[List A] -> Nat` just fine. They start mattering when you write generic/abstract code.

### The subsumption question

You're asking: could `:properties` replace or absorb the need for `{A B : Type}`? Let me think about what that would look like.

The idea would be that instead of:

```prologos
spec sort {A : Type} [List A] -> [List A]
  :where (Ord A)
```

You'd write something like:

```prologos
spec sort [List A] -> [List A]
  :where (Ord A)
  :properties
    - :name "idempotent"
      :forall {xs : [List Nat]}
      :holds [eq? [sort [sort xs]] [sort xs]]
```

And the system would infer `{A : Type}` from context — `A` appears in `[List A]`, so it must be a type variable. The `:where (Ord A)` constrains it further.

**This already works today for simple cases.** You can write:

```prologos
spec sort [List A] -> [List A]
  :where (Ord A)
```

...and the implicit binder `{A : Type}` is unnecessary because `A` appears in a position the type checker can infer. The `{A : Type}` is only needed when it *doesn't* appear in such a position.

So the real question is: **does `:properties` provide additional constraining positions for free type variables?**

The answer is: **not really, and it shouldn't.** Here's why:

Properties are a separate semantic layer. They're assertions *about* the function, not *part of* its type. A property like:

```prologos
:forall {xs : [List Nat]}
:holds [eq? [sort [sort xs]] [sort xs]]
```

This is a *use* of `sort` at a *specific* type (`Nat`). It doesn't help infer the general type of `sort`. In fact, the property's `:forall {xs : [List Nat]}` is intentionally more specific (monomorphic) than the spec's polymorphic type — it's testing `sort` at `Nat`, not defining `sort`'s type.

### The deeper question: should we need fewer implicit binders?

I think your real intuition is: "the language should figure this out for me more often." And I agree. There are a few directions:

**Direction 1: Better inference of free type variables.** If `A` appears in the type signature (even just in `[List A] -> [List A]`), the system can see it's unbound and auto-introduce `{A : Type}`. This already works for simple cases. The only time it fails is:
- Higher-kinded variables (`{C : Type -> Type}`) where the *kind* can't be inferred
- Variables that truly don't appear anywhere in the signature

Both are rare in practice. Most Prologos code that uses `{A : Type}` is being unnecessarily explicit.

**Direction 2: Kind inference from `:where` clauses.** If you write `spec gmap ... :where (Seqable C)`, and `Seqable` is declared as a trait over `{C : Type -> Type}`, the system could infer `C`'s kind from the where clause alone. This would eliminate the most annoying case — having to write `{C : Type -> Type}` just because you use a HKT trait. This is feasible and is basically what `propagate-kinds-from-constraints` already does internally.

**Direction 3: `:properties` as *documentation of intent*, not type information.** The properties tell the reader (and the test system) what the function's contract is. They don't feed back into the type system at Phase 2 (property testing). At Phase 3 (verification), they would become proof obligations — but those obligations are *derived from* the types, not *constitutive of* them.

### Summary of my view

`:properties` can't and shouldn't subsume implicit binders because they live at a different semantic level: properties are *about* the function, implicit binders are *part of* its type. But the pain point you're identifying — "when do I need these braces?" — is real and is better addressed by making the type inferencer smarter (directions 1 and 2 above), so that `{A : Type}` becomes almost never needed for kind-`Type` variables, and `{C : Type -> Type}` becomes unnecessary when a `:where` clause pins the kind. At that point, the only time you'd write explicit implicit binders is when you want to be pedantic or when the system truly can't figure it out — which would be very rare in practice.

Does this clarify the intuition? I think the direction worth pursuing is: "make `{...}` implicit binders optional in more cases via better inference," not "make `:properties` do the job of `{...}`."

## `:implicits` as a Metadata Key

The idea is to lift implicit binders out of the type signature and into the metadata map. Compare current vs proposed:

**Current:**
```prologos
spec gmap {A B : Type} {C : Type -> Type} (Seqable C) -> (Buildable C) -> [A -> B] -> [C A] -> [C B]
```

**With `:implicits` + `:where`:**
```prologos
spec gmap [A -> B] -> [C A] -> [C B]
  :implicits {A B : Type} {C : Type -> Type}
  :where (Seqable C) (Buildable C)
```

The type signature now reads as pure function shape: "takes a function A→B and a C A, gives you a C B." The machinery — what's implicit, what's constrained — lives in the metadata where it doesn't clutter the primary thing you want to read.

This stacks nicely with simpler cases too:

```prologos
;; Today:
spec sort {A : Type} [List A] -> [List A]
  :where (Ord A)

;; With :implicits:
spec sort [List A] -> [List A]
  :implicits {A : Type}
  :where (Ord A)

;; Or, if inference handles {A : Type} automatically (which it already does here):
spec sort [List A] -> [List A]
  :where (Ord A)
```

The beauty is that all three forms coexist. Inline `{A : Type}` still works for quick one-liners. `:implicits` is there when you want the signature clean. And when inference is good enough, you skip both. Progressive disclosure at its best.

For the HKT case — which is the main pain point — `:implicits` is a real readability win:

## Reusable Properties: The `property` Keyword

This is where it gets interesting. You're right that the parallel to `bundle` is direct:

| Concept | For methods | For propositions |
|---|---|---|
| Single declaration | `trait` | individual `:holds` clause |
| Named group | `bundle` | `property` |
| Composition | `bundle` includes `trait`s/`bundle`s | `property` includes `property`s |
| Attachment | `:where (Ord A)` | `:properties (sortable-laws A)` |

### Basic `property` Declaration

```prologos
property sortable-laws {A : Type}
  :where (Ord A)
  - :name "idempotent"
    :forall {xs : [List A]}
    :holds [eq? [sort [sort xs]] [sort xs]]
  - :name "length-preserving"
    :forall {xs : [List A]}
    :holds [eq? [length [sort xs]] [length xs]]
  - :name "ordered-output"
    :forall {xs : [List A]}
    :holds [sorted? [sort xs]]
```

A `property` is a named, parameterized conjunction of propositions. Each clause is a universally quantified boolean assertion. The `:where` constrains what types the property is meaningful for.

### Usage in `spec`

```prologos
spec sort [List A] -> [List A]
  :implicits {A : Type}
  :where (Ord A)
  :properties (sortable-laws A)
```

The `(sortable-laws A)` reference is like `(Ord A)` in a `:where` — it's a named requirement, parameterized by type variables from the spec.

### Composition via `:includes`

Here's where the conjunctive-clause perspective shines:

```prologos
;; Base algebraic properties
property semigroup-laws {A : Type}
  :where (Add A)
  - :name "associativity"
    :forall {x y z : A}
    :holds [eq? [add [add x y] z] [add x [add y z]]]

property monoid-laws {A : Type}
  :where (Add A) (AdditiveIdentity A)
  :includes (semigroup-laws A)
  - :name "left-identity"
    :forall {x : A}
    :holds [eq? [add additive-identity x] x]
  - :name "right-identity"
    :forall {x : A}
    :holds [eq? [add x additive-identity] x]

property group-laws {A : Type}
  :where (Add A) (AdditiveIdentity A) (Neg A)
  :includes (monoid-laws A)
  - :name "left-inverse"
    :forall {x : A}
    :holds [eq? [add [neg x] x] additive-identity]
  - :name "right-inverse"
    :forall {x : A}
    :holds [eq? [add x [neg x]] additive-identity]
```

`group-laws` includes `monoid-laws` which includes `semigroup-laws`. Flattening: `group-laws` is the conjunction of all six properties. This is exactly how `bundle Num` includes `Add`, `Sub`, `Mul`, `FromInt` — a bundle is a conjunction of trait constraints, a property is a conjunction of propositions.

### The Logic Programming Reading

From the sequent calculus perspective:

```
property P {A} :where (C₁ A) (C₂ A) :includes (Q A) = 
  Γ, C₁(A), C₂(A) ⊢ Q(A) ∧ p₁(A) ∧ p₂(A) ∧ ...
```

Each `:holds` clause is a proposition. `:includes` is conjunction. The whole `property` is a universally quantified sequent: "for all A satisfying C₁ and C₂, the conjunction of all these propositions holds."

When you attach it to a spec with `:properties (P A)`, you're asserting that your implementation satisfies that sequent. At Phase 2 (QuickCheck), each conjunct becomes a test. At Phase 3 (verification), each conjunct becomes a proof obligation.

The composability is clean because conjunction is associative and commutative — the order and nesting of `:includes` don't matter; the result is always a flat set of propositions.

### Functor/Monad Laws

This is where it becomes really satisfying:

```prologos
property functor-laws {F : Type -> Type}
  :where (Functor F)
  - :name "identity"
    :forall {xs : [F A]}
    :holds [eq? [fmap id xs] xs]
  - :name "composition"
    :forall {f : [A -> B]} {g : [B -> C]} {xs : [F A]}
    :holds [eq? [fmap [>> f g] xs] [fmap g [fmap f xs]]]

property applicative-laws {F : Type -> Type}
  :where (Applicative F)
  :includes (functor-laws F)
  - :name "identity"
    :forall {xs : [F A]}
    :holds [eq? [ap [pure id] xs] xs]
  - :name "homomorphism"
    :forall {f : [A -> B]} {x : A}
    :holds [eq? [ap [pure f] [pure x]] [pure [f x]]]

property monad-laws {M : Type -> Type}
  :where (Monad M)
  :includes (applicative-laws M)
  - :name "left-identity"
    :forall {a : A} {f : [A -> [M B]]}
    :holds [eq? [bind [pure a] f] [f a]]
  - :name "right-identity"
    :forall {ma : [M A]}
    :holds [eq? [bind ma pure] ma]
  - :name "associativity"
    :forall {ma : [M A]} {f : [A -> [M B]]} {g : [B -> [M C]]}
    :holds [eq? [bind [bind ma f] g] [bind ma [fn [a] [bind [f a] g]]]]
```

And a trait could reference its laws:

```prologos
trait Functor {F : Type -> Type}
  :laws (functor-laws F)
  spec fmap {A B : Type} [A -> B] -> [F A] -> [F B]
```

The `:laws` key on `trait` says "any instance of this trait is expected to satisfy these properties." When you write `instance Functor List`, the system knows to check `(functor-laws List)`. At Phase 2, that means generating `List`-specialized QuickCheck tests. At Phase 3, proof obligations.

### Parameterized Properties (Higher-Order)

Properties can also be parameterized over functions, not just types. This covers the case where you want to state "any function satisfying this spec should have these properties":

```prologos
;; Properties about any sorting function, not just our `sort`
property sorting-properties {A : Type} {f : [List A] -> [List A]}

  - :name "idempotent"

  - :name "permutation"
    :forall {xs : [List A]}

  - :name "ordered"
    :forall {xs : [List A]}
    :holds [sorted? [f xs]]

  :implicits {A : Type}
  :where (Ord A)
  :properties (sorting-properties A sort)

spec merge-sort [List A] -> [List A]
  :implicits {A : Type}
  :where (Ord A)
  :properties (sorting-properties A merge-sort)
```

Both `sort` and `merge-sort` share the same property suite. This is exactly the reusability you're after — define the contract once, attach it to any implementation.

### Composition Example: Collection Properties

```prologos
property insertable-laws {C : Type -> Type} {A : Type}

  - :name "insert-increases-size"
    :forall {xs : [C A]} {x : A}

```

Small, focused properties compose into larger ones. You test `(collection-laws PVec Nat)` and get all the sub-properties for free.

One design question: what if `monad-laws` includes `applicative-laws` which includes `functor-laws`, and both `monad-laws` and `functor-laws` have a property named `"identity"`? The simplest answer: **fully qualified names**. ### The Connection to Traits More Broadly

There's a beautiful symmetry emerging:

```
trait     : Type → Set of method signatures     (what you must implement)
bundle    : conjunction of traits                (shorthand for multiple traits)
property  : Type → Set of propositions           (what must be true)
  includes: conjunction of properties            (shorthand for multiple properties)

spec      : function → type + metadata
  :where  : trait constraints                    (what the types must support)
  :properties : property constraints             (what the function must satisfy)

instance  : Type → method implementations        (how you implement)
  (auto)  : Type → property evidence             (how you prove / test)
```

Traits say what operations exist. Properties say what laws they obey. Today only the first is checked; the design gives us a path to checking the second.

What we're proposing is a *named abstraction with declared structure* — the operations, laws, documentation are first-class. `abbrev` makes you think of `typedef` — mechanical shortening. `functor` makes you think of structure-preserving mappings — which is what these actually are.

  `type` clashes with `deftype` (which creates algebraic data types with constructors). A `functor` doesn't create constructors; it names a type with structure.

  There is a potential naming collision: `functor` (keyword) vs `Functor` (the trait for `fmap`). But this is actually consistent:

  - `Functor` (capitalized) is a trait — it says "this type supports `fmap`"
  - `functor` (keyword) declares a type abstraction — it says "this is a named type concept"

  The relationship is: a `functor` declaration might *itself* be a `Functor` instance. A `functor Xf {A B}` that has a `:compose` and `:identity` could automatically be checked for `Functor` laws. The keyword names the concept; the trait constrains it.

  This parallels how `property` (keyword) declares propositions, while individual `:holds` clauses are propositions. Same word, different grammatical role (keyword vs noun).

  ### Category-Theoretic Metadata Keys

  The keys themselves are where CT maps to approachable language:

  | CT concept | Key | Meaning in plain English |
  |---|---|---|
  | Object mapping | `:unfolds` | "What this actually expands to under the hood" |
  | Morphism composition | `:compose` | "How to chain two of these together" |
  | Identity morphism | `:identity` | "The do-nothing version" |
  | Functor laws | `:laws` | "What rules these always follow" |
  | Natural transformation | `:transforms` | "How to convert between different functors" (future) |
  | Adjunction | `:adjoint` | "What functor is this paired with" (far future) |

  Only `:unfolds` is always needed. The rest are progressive. A user who never thinks about category theory writes:

  ```prologos
  functor Xf {A B : Type}
    :doc "A transducer"
    :unfolds <(S :0 Type) -> [S -> B -> S] -> S -> A -> S>
  ```

  A user who cares about algebraic structure adds `:compose` and `:identity`. A user who wants verification adds `:laws`. The CT is always there but only surfaces when you reach for it.

  ### The Deeper Point: Pi Types as Implementation Detail

  Your phrase "never to have to see Pi or Sigma types leak up to the surface" crystallizes something important about Prologos's design philosophy. Dependent types are the *implementation substrate* — they give the type system its power. But the surface language should express intent in domain terms:

  - "This function sorts lists" → `spec sort [List A] -> [List A]`
  - "Sorting requires ordering" → `:where (Ord A)`
  - "Sorting is idempotent" → `:properties (sortable-laws A)`
  - "A transducer transforms reductions" → `functor Xf {A B}`
  - "Transducers compose" → `:compose xf-compose`

  The Pi types, Sigma types, universe levels, multiplicity annotations — those exist in `:unfolds`, in the elaborator, in the type checker. They're available when you need them (via `??` holes, via `:unfolds`, via error messages with `--expand-types`). But the *primary interface* is domain vocabulary expressed through keywords and metadata.

  This is the logical endpoint of "Open Metadata, Closed Semantics" from the principles doc: the metadata describes what things *mean*, the type theory provides the formal backing.

  ### How `functor` Composes with `spec`, `property`, and `trait`

  Everything ties together:

  ```prologos
  ;; The type abstraction
  functor Xf {A B : Type}
    :doc "A transducer: transforms A-reductions into B-reductions"
    :compose xf-compose
    :identity id-xf
    :laws (transducer-fusion-laws A B)
    :unfolds <(S :0 Type) -> [S -> B -> S] -> S -> A -> S>

  ;; Reusable properties (using the functor name!)
  property transducer-fusion-laws {A B C : Type}
    - :name "compose-associativity"
      :forall {f : [Xf A B]} {g : [Xf B C]} {h : [Xf C D]}
      :holds [eq? [xf-compose [xf-compose f g] h] [xf-compose f [xf-compose g h]]]
    - :name "left-identity"
      :forall {f : [Xf A B]}
      :holds [eq? [xf-compose id-xf f] f]
    - :name "right-identity"
      :forall {f : [Xf A B]}
      :holds [eq? [xf-compose f id-xf] f]

  ;; Function specs using the functor name
  spec xf-compose [Xf A B] -> [Xf B C] -> [Xf A C]
    :implicits {A B C : Type}
    :doc "Compose two transducers (apply first argument first)"
    :properties (transducer-fusion-laws A B C)

  spec into-list [Xf A B] -> [List A] -> [List B]
    :implicits {A B : Type}
    :doc "Run a transducer over a list, producing a new list"
    :examples
      - [into-list [map-xf suc] '[1N 2N 3N]] => '[2N 3N 4N]
  ```

  The `functor` names the type. The `property` states its laws. The `spec` uses both. Everything is readable. The Pi types are present exactly once, in `:unfolds`, consulted only when necessary.

  ### What About `functor` for Non-CT Uses?

  Not every type abbreviation has algebraic structure. Is `functor` the right keyword for `functor FilePath :unfolds String`?

  Two options:

  ,**A.** Use `functor` for everything, treat zero-metadata as the degenerate case. A "constant functor" in CT terms — maps nothing, just assigns a name. Slightly weird naming but maximally simple (one keyword).

  ,**B.** Have `abbrev` for plain synonyms, `functor` for structured abstractions. Like how `trait` is for single methods and `bundle` is for groups. Clear separation of intent.

  I'd lean toward **A** for consistency — `functor` is the one keyword for "I'm naming a type concept." If you provide `:compose`, it has algebraic structure. If you don't, it's just a synonym. Same progressive disclosure pattern as everything else. And honestly, calling `FilePath` a functor is technically correct (constant functor from 0 to Type), even if it sounds grandiose.

  ### One Open Question: Is `:unfolds` Always Required?

  For simple synonyms, `:unfolds` is the only key and feels mandatory:

  ```prologos
  functor FilePath
    :unfolds String
  ```

  But what about the case where the `functor` is purely described by its structure, and you don't want to expose the representation at all? An opaque type:

  ```prologos
  functor Handle {A : Type}
    :doc "An opaque reference to a managed resource"
    :compose handle-chain
    ;; no :unfolds — representation is hidden
  ```

  This would be an *abstract type* — exists only through its declared operations. Whether to support this or require `:unfolds` is a design choice. I'd say: require `:unfolds` for Phase 1, consider opaque functors later.
