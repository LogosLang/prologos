### [Schema as Protocol: GraphQL and the Schema-Session Interaction](../research/2026-03-02_SCHEMA_AS_PROTOCOL_RESEARCH.md)

### 🤖

```markdown
Good — important findings. Note: `select` is already used as a process keyword for session internal choice (`(select channel label continuation)`), so we can't reuse it for schema selection. `selection` as a keyword is available though.

Now let me work through the surface syntax variations systematically. Let me present several approaches with concrete examples.

---

Here are the variations I want to walk through. I'll ground each in a common scenario: a `User` schema used across functional, relational, and process contexts.

### The Common Schema (all variations share this)

```prologos
schema Address
  :street  String
  :city    String
  :state   StateCode
  :zip     ZipCode

schema User
  :id       UserId
  :name     String
  :email    Email
  :address  Address
```

---

### Variation A: Named `selection` with `requires`/`provides` blocks

Schema defines shape. `selection` is a separate named form. `requires` and `provides` are sub-clauses:

```prologos
;; Named selections — reusable
selection MovieTimesReq from User
  requires :id
  requires :address { :zip }

selection PlaceOrderReq from User
  requires :name
  requires :email
  requires :address { :street :city :state :zip }

selection UserResponse from User
  provides :id
  provides :name
  provides :email
  provides :address { * }

;; Composition: include other selections
selection FullOrderReq from User
  includes PlaceOrderReq
  requires :id

;; Usage in spec
spec get-times : User & MovieTimesReq -> List MovieTime
defn get-times [user]
  lookup-times user.id user.address.zip

;; Usage in session
session OrderService
  ? req : User & PlaceOrderReq
  ! conf : OrderConfirmation
  end

;; Usage in relation
defr user : User & UserResponse
  || 1N "Alice" "alice@co.com" {:street "123 Main" :city "SF" :state "CA" :zip "94102"}
```

**Pros**: Clean separation. Named selections are reusable. `requires`/`provides` are natural English. The `&` operator reads as "a User *that* satisfies MovieTimesReq." Follows the `spec`/`defn` pattern of "declaration shape separate from definition."

**Cons**: Verbose for simple cases. Two new keywords (`selection`, `includes`). Nested selection syntax (`{ :zip }`) needs design.

---

### Variation B: Selection as schema refinement (no new top-level keyword)

Instead of a separate `selection` keyword, selections are schemas that refine other schemas:

```prologos
schema MovieTimesReq := User
  requires :id
  requires :address { :zip }

schema PlaceOrderReq := User
  requires :name :email
  requires :address { * }

schema UserResponse := User
  provides :id :name :email
  provides :address { * }

;; Usage
spec get-times : MovieTimesReq -> List MovieTime

;; Session — schemas used directly, no & needed
session OrderService
  ? req : PlaceOrderReq
  ! conf : OrderConfirmation
  end
```

**Pros**: No new top-level form. Selection IS a schema (just a constrained one). `schema X := Y` reads as "X is a Y with these constraints." Simpler — the `&` operator is replaced by just naming the refined schema. Follows the existing `deftype` pattern where `:=` introduces definitions.

**Cons**: Blurs the schema/selection distinction (a selection looks like a schema). Can't use the *same* User type with *different* selections in the same session without naming every combination. `MovieTimesReq` is-a `User` subtype, which may have unintended implications.

---

### Variation C: Inline selection (GraphQL-style, no named form)

No named selections. The selection is always inline where you use the schema:

```prologos
;; Inline selection with braces
spec get-times : User { :id, :address { :zip } } -> List MovieTime
defn get-times [user]
  lookup-times user.id user.address.zip

;; Session — inline selection
session OrderService
  ? req : User { :name, :email, :address { * } }
  ! conf : OrderConfirmation
  end

;; For reuse, bind a selection to a name via def
def MovieTimesReq := User { :id, :address { :zip } }

spec get-times : MovieTimesReq -> List MovieTime
```

**Pros**: Maximum conciseness for one-off use. Directly mirrors GraphQL query syntax. No new top-level forms. The `{ }` braces after a schema type name naturally read as "this subset of fields." Progressive disclosure: start inline, name it when you reuse it.

**Cons**: Inline selections in long session types become noisy. No `requires`/`provides` distinction (everything is a flat field list). The `{ }` syntax after a type name could conflict with map literal syntax or brace-params. Harder to compose — you'd need set operations on inline selections.

---

### Variation D: Hybrid — named form + inline shorthand

Named `selection` for reuse, inline `{ }` for one-offs. `requires`/`provides` in named form:

```prologos
;; Named selection for reuse (Approach B style)
selection MovieTimesReq from User
  requires :id
  requires :address { :zip }

selection UserResponse from User
  provides :id :name :email
  provides :address { * }

;; Inline shorthand for one-offs (GraphQL style)
spec quick-lookup : User { :id, :address { :zip } } -> Info

;; Both compose with &
spec get-times : User & MovieTimesReq -> List MovieTime

;; Session uses named selections for protocol clarity
session OrderService
  ? req  : User & PlaceOrderReq
  ! resp : User & UserResponse
  end

;; Inline in session when ad-hoc
session QuickService
  ? req : User { :id }
  ! resp : Bool
  end
```

**Pros**: Best of both worlds. Named selections for shared contracts; inline for one-offs. `requires`/`provides` available for the directional distinction in named form. Progressive disclosure: `schema` → inline `{ }` → named `selection` → `requires`/`provides`.

**Cons**: Two syntaxes for the same concept (inline vs named). Must define which is canonical for the sexp backend.

---

### Variation E: `requires`/`provides` as standalone spec-like annotations

Rather than a new `selection` form, use `requires`/`provides` as standalone annotations that attach to an existing schema, like `spec` attaches to `defn`:

```prologos
;; Schema defines shape
schema User
  :id       UserId
  :name     String
  :email    Email
  :address  Address

;; requires/provides as standalone forms (like spec is to defn)
requires MovieTimesReq : User [:id :address.zip]
requires PlaceOrderReq : User [:name :email :address]
provides UserResponse  : User [:id :name :email :address]

;; Deep nested requirement
requires DeepReq : User [:id :address.zip :address.city]

;; Composition
requires FullReq : User [PlaceOrderReq :id]

;; Usage in spec
spec get-times : MovieTimesReq -> List MovieTime
defn get-times [user]
  lookup-times user.id user.address.zip

;; Usage in session
session OrderService
  ? req  : PlaceOrderReq
  ! resp : UserResponse
  end

;; For ad-hoc: schema with inline vec of keys
spec quick-lookup : User [:id :address.zip] -> Info
```

**Pros**: `requires` and `provides` are top-level forms — maximally visible. The `name : Schema [fields]` syntax is concise and reads naturally: "MovieTimesReq requires from User: id and address.zip." Dot-path syntax for nested fields (`address.zip`) avoids nested braces. Composition by including other selection names in the field list. Consistent with the `spec name : Type` pattern.

**Cons**: Dot-path (`address.zip`) is clean but doesn't support "all of address" as naturally (would need `:address.*` or `:address`). The `[field-list]` syntax using vec brackets is concise but may not scale to complex nested requirements.

---

### My Observations on the Design Space

1. **`requires`/`provides` as keyword names** — these are excellent. They're directional (input vs output), natural English, and with the module rename to `imports`/`exports`, there's zero collision.

2. **The `selection` keyword vs. making it a schema variant** — Variation A (`selection ... from ...`) and Variation E (`requires Name : Schema [fields]`) are the cleanest separations. Variation B (schema refinement) conflates schema and selection.

3. **The `&` operator** — useful for composing schema + selection in type positions, but if selections produce subtypes (as in Variation B/E), you can just use the selection name directly as the type. The `&` becomes optional convenience.

4. **Nested field syntax** — three contenders:
   - Brace nesting: `:address { :zip :city }` (tree-like, mirrors schema structure)
   - Dot paths: `:address.zip` (flat, concise, like GraphQL)
   - Wildcard: `:address { * }` or `:address.*` (select all sub-fields)

5. **The progressive disclosure ladder** works best in Variation D:
   - Level 0: `schema Point :x Int :y Int` (all required, use as-is)
   - Level 1: `User { :id :name }` (inline selection)
   - Level 2: `selection MovieTimesReq from User` (named, reusable)
   - Level 3: `requires`/`provides` with directional semantics

What's your instinct on which variation(s) feel right? I'm particularly curious about:
- Named form keyword preference: `selection` vs standalone `requires`/`provides`
- Nested field syntax: braces vs dot-paths
- Whether the `&` operator is needed or if named selections as types are sufficient
