**The termination argument for layered quiescence:**

Each layer individually terminates:
- **S(-1)**: Believed-assumptions set can only shrink. Finite assumptions → finite retractions. ✓ Structural.
- **S0**: Each meta cell transitions at most once (⊥ → solved). Finite metas = finite writes. ✓ Structural.
- **L1**: Each readiness propagator fires at most once per dependency becoming non-⊥. ✓ Structural.
- **L2**: Each resolution fires at most once per ready constraint. ✓ Structural.

**The cross-layer feedback is where termination becomes non-obvious.** L2 resolves a trait → calls `solve-meta!` → writes to S0 → S0 propagates → L1 detects new readiness → L2 resolves again. The bound is: each cycle must solve at least one meta, and metas are finite. So the cycle count is bounded by the number of unsolved metas.

**But** — and this is the critical gap — trait resolution can *create new metas*. Resolving `Add Int` is concrete (no new metas). But resolving `Num a` may create dictionary metas for `Add a`, `Sub a`, `Mul a` — which themselves may need resolution. If resolution creates metas that create constraints that create metas...

In Haskell, this is exactly why `UndecidableInstances` exists as a pragma — type class resolution is undecidable in general. GHC uses a fuel-based escape hatch (context reduction stack depth limit, default 100). The termination *guarantee* doesn't exist; they use fuel.

**Your principle demands better than fuel.** You're asking: can we make the prop-net a total computation model — one where termination is structural, not fuel-dependent?

I think the answer is yes, but it requires a **well-founded measure** on the resolution depth. Specifically:

**Structural termination via type-depth ordering**: Every trait resolution either:
1. Produces a concrete dictionary (no new metas) — terminal
2. Creates new metas at a *strictly smaller type depth* than the triggering constraint

If we can show that type depth decreases on every resolution step, then the resolution chain is well-founded (it bottoms out at atomic types, which resolve concretely). This is the same argument that makes structural recursion on inductive types total — the recursive call is on a strictly smaller argument.

The type-depth measure would be: `depth(Num a) > depth(Add a)` (bundle decomposes to components), `depth(Add (List a)) > depth(Add a)` (type constructor decreases). If every instance declaration satisfies this decreasing-depth property, resolution terminates structurally.

This gives us a **decidability condition on instance declarations**: an instance is admissible iff its superclass/component constraints have strictly smaller type depth than the head. This is strictly stronger than Haskell's Paterson conditions but weaker than requiring all instances to be ground — it's the sweet spot where resolution is provably total.

**For the prop-net, this means:**

The prop-net itself is structurally terminating (finite lattices, monotone propagators, finite cells). The *only* source of potential non-termination is L2 resolution creating new cells. If we impose the well-founded measure on instance declarations (checked at registration time — correct by construction!), then:

1. Each resolution step decreases type depth
2. Type depth is a natural number (well-founded)
3. Therefore resolution chains are finite
4. Therefore cross-layer feedback cycles are finite
5. Therefore layered quiescence terminates

**The fuel becomes a defense-in-depth, not the primary guarantee.** It catches bugs in the termination argument, not expected non-termination. This is the correct role for fuel — a safety net, not a design crutch.

**On "Gödel Completeness" as a design principle:**

I think this is a principle that belongs in DESIGN_PRINCIPLES.org alongside the propagator-first and correct-by-construction principles. The prop-net should be a **decidable computation model**:

- Every lattice has finite height (structural)
- Every propagator is monotone within its stratum (structural)
- Every cross-stratum feedback has a well-founded decreasing measure (verified at registration)
- Fuel exists as defense-in-depth, not as the termination mechanism

This rules out certain computations on the network (anything that could diverge), which is exactly the point — the network is a *safe space* where all computation provably terminates. User-level computation (general recursion, IO) happens outside the network. The network is the kernel of decidable type-level computation.

This is genuinely novel — I'm not aware of any propagator network implementation that provides termination guarantees beyond fuel. The combination of stratified semantics + well-founded measures + structural monotonicity gives you something stronger than what Radul & Sussman described.

Want me to capture this as a principle and update the design doc with the termination argument?

**The hierarchy of termination guarantees:**

There's a spectrum, not a binary:

1. **Structural (Tarski)**: Finite lattice + monotone propagators → fixpoint in bounded steps. The lattice height IS the bound. No fuel needed. This covers S0 (type propagation), effect propagation, QTT multiplicity — all finite lattices.

2. **Well-founded measure**: Cross-stratum feedback decreases a well-founded ordering. Each cycle makes "structural progress" — type depth decreases, constraint count decreases, etc. Termination follows from well-foundedness. This covers L2→S0 feedback (trait resolution with decreasing depth).

3. **Widening-guaranteed convergence**: The concrete domain is infinite, but a widening operator forces convergence to an abstract fixpoint in bounded steps. You lose precision but gain termination. Narrowing (lattice sense) then recovers precision without threatening termination (it's bounded by the number of widening steps). This covers abstract interpretation over integers, sizes, resource bounds.

4. **Coinductive productivity**: The computation is infinite (a stream, a session, an ongoing process), but each *observation* terminates. Productivity replaces termination — the guarantee is "you can always get the next element," not "the computation eventually stops." The gfp dual of inductive termination.

5. **Fuel bound**: Decidable by fiat. The computation stops because we force it to. Informative about safety, not about structure.

Levels 1-4 tell you *why* things terminate. Level 5 tells you *that* they terminate. Both are decidable. The principle is: pursue 1-4, accept 5 pragmatically, always ask "can we move this from fuel to structure?"

**The IO Bridge as monadic propagation:**

This is where the architecture gets interesting. The IO bridge captures effects as pure values — descriptions of intent, not executed side effects. The propagator network processes these descriptions. This is structurally a graded monad:

- The effect annotation (the grade) is a lattice value propagated through the network
- `bind` sequences effect descriptions; the grade combines via the effect quantale's tensor
- The propagator network is the "evaluator" operating on monadic values

The key insight: because effect descriptions are *values* in the network (not side effects), they're subject to the same termination guarantees as any other cell value. The effect lattice has finite height (declared effects are finite), so effect propagation terminates structurally (Level 1). The IO bridge propagator that processes effect descriptions is a monotone function on a finite lattice.

This means we can reason about effectful computations *within* the decidable propagator framework. The effects themselves are Gödel Complete — their composition, ordering, and constraint checking all terminate structurally. Only the *execution* of effects (actual IO) happens outside the network, in the imperative world.

**Infinite domains through abstraction:**

Here's where gfp/lfp and widening expand what the network can handle:

**Inductive (lfp)**: "Build up from nothing." Nat, List, Tree — finite structures. Propagation over these terminates because each inductive value has finite depth. The structural recursion principle gives you a measure that decreases on each step. This is Level 2.

**Coinductive (gfp)**: "Tear down from everything." Streams, sessions, processes — potentially infinite structures. You can't guarantee *termination*, but you can guarantee *productivity*: each observation (head of stream, next message in session) is computable in finite time. The propagator network can handle this by computing one "layer" of the coinductive structure per quiescence cycle. Each cycle terminates (Level 1 within the cycle); the number of cycles may be infinite (but each produces a result). This is Level 4.

**Abstract interpretation**: For domains like integers where the concrete lattice has infinite height, widen to a finite abstract domain. The propagator network operates on the abstract domain (finite height → Level 1 termination). The Galois connection between concrete and abstract ensures soundness. Narrowing refines the result — and because narrowing is bounded by widening steps, it's also Level 1.

**Narrowing (functional-logic)**: The existing narrowing system in Prologos reduces the search space by unifying with constructors. Each narrowing step reduces the set of possible values — this is a well-founded measure on the size of the search space. Level 2.

**What this means for the architecture:**

The propagator network isn't limited to finite lattices. It can handle:
- Finite domains (structural, Level 1)
- Infinite inductive domains (well-founded measure, Level 2)
- Infinite coinductive domains (productivity, Level 4)
- Infinite abstract domains (widening, Level 1 on the abstract lattice)

The *combination* of these gives you a surprisingly expressive decidable computation model. You can reason about streams (coinductive) of abstract integer ranges (widened) with effect annotations (finite lattice) — all within the network, all with termination guarantees.

**The honest gap:**

The place where Level 5 (fuel) is genuinely needed is **open-ended trait resolution with user-defined instances**. A user could write:

```markdown
trait Foo A
impl Foo [List A] where Foo A
```

This creates a resolution chain: `Foo [List [List [List ...]]]` — infinite descent. The decreasing type-depth condition catches this at registration time (the impl's constraint `Foo A` has smaller depth than the head `Foo [List A]`... wait, actually it does decrease). Let me think more carefully...

Actually, the Paterson-like condition DOES catch this: `Foo A` has depth 1, `Foo [List A]` has depth 2. The constraint's depth (1) is strictly less than the head's depth (2). So this instance is admissible, and resolution terminates because depth decreases.

The dangerous case is:
```markdown
trait Bar A
impl Bar [F A] where Bar [G A]   ;; depth doesn't decrease if F and G are at same level
```

Where `G A` has the same or greater depth as `F A`. The Paterson condition rejects this. If we enforce it, trait resolution is Level 2 (well-founded on type depth). If we don't (Haskell's `UndecidableInstances`), we need Level 5 (fuel).

The pragmatic answer: enforce the condition by default (Level 2 guarantee), offer an opt-out pragma for advanced users that falls back to fuel (Level 5). The user makes an explicit choice to leave the decidable fragment.

I think this should be captured as something like:

> **Decidability-First Propagation**: Every computation hosted on the propagator network should come with a termination guarantee — structural proof (Tarski, well-founded measure, widening convergence) preferred, fuel bound as pragmatic fallback. The network is a decidable computation model; general recursion and unbounded computation happen outside it. When a new propagator or lattice is added, its termination argument must be stated explicitly.

This is a design discipline, not a theorem — it's a commitment to *thinking about termination* at every level, and choosing the strongest guarantee available. The fuel fallback ensures pragmatism; the discipline ensures we don't default to fuel when a structural argument exists.

**§4: Bridging persistent-registry-network and elab-network**

The two networks serve different lifecycle concerns (persistent vs per-command), but during elaboration a propagator in the elab-network may need to read a registry value from the persistent network. The question is whether a Galois connection or bridge propagator is the right abstraction.

I think these are *not* orthogonal — there's a natural bridge pattern. When `register-macros-cells!` currently creates ephemeral cells seeded from parameters, what it's really doing is creating shadow cells in the elab-network that mirror the persistent registry cells. Track 5 already designed this exact pattern for definition cells: shadow cells in the file's network mirror the module network's cells.

The bridge would be: a one-way propagator from the persistent registry cell to a shadow cell in the elab-network. The shadow cell receives writes from the persistent cell (Galois connection: the persistent cell is the lower adjoint, the shadow is the upper adjoint). During elaboration, all reads go through the shadow cell. Writes go to the persistent cell (which propagates to the shadow). This is the same cross-network bridge pattern we use for module→file definition shadowing.

The cost question: one bridge propagator per registry cell per command (29 propagators). But these are static wiring — created once per command, fired once (initial sync). The ongoing cost is zero because registry cells change infrequently during a single command. This is actually *cheaper* than the current approach of recreating 29 cells and seeding them from parameters, because the bridge is lazy (only fires when the source cell changes) rather than eager (always copies all 29 values at command start).

So yes — a Galois connection simplifies things. It unifies the two-network architecture with the existing cross-network bridge pattern from Track 5, rather than inventing a new mechanism.

**§4: Gauss-Seidel vs BSP scheduling**

Gauss-Seidel processes one layer to fixpoint before the next. BSP (Bulk Synchronous Parallel) processes all layers simultaneously but synchronizes at barriers. The question is which is favored when.

**Gauss-Seidel advantages**: Each layer sees a *complete* fixpoint from the layer below. This is important when the higher layer's semantics depend on completeness — S1 readiness detection only makes sense when S0 has quiesced (otherwise you'd detect "ready" constraints that aren't actually ready because more type info is still propagating). The mathematical argument: stratified fixpoint semantics requires each stratum to compute its lfp before the next stratum observes it.

**BSP advantages**: Exploits parallelism. If S0 and L1 can make independent progress (L1 readiness checks on constraints whose deps are already ground, while S0 is still solving other metas), BSP could be faster. The synchronization barrier ensures consistency at well-defined points.

**CALM theory insight**: By CALM (Consistency As Logical Monotonicity), monotone computations can run without coordination. Within S0, all propagators are monotone — they can safely run in parallel (BSP within a stratum). But the S0→L1 transition is NOT monotone from L1's perspective — L1 needs S0's *complete* fixpoint to make correct readiness decisions. So the inter-stratum ordering requires coordination (Gauss-Seidel between strata).

The natural hybrid: **BSP within each stratum, Gauss-Seidel between strata**. This is actually what `run-to-quiescence` already does for S0 — it processes all dirty propagators in rounds (BSP-like) until fixpoint. Extending this to the full layered scheduler means: run S(-1) propagators BSP to fixpoint, then S0 BSP to fixpoint, then L1 BSP to fixpoint, then L2 BSP to fixpoint. The Gauss-Seidel ordering is only at the layer transitions.

We can reason about this a priori (CALM gives us the theoretical foundation), but measuring against an adversarial constraint graph would validate the theory. An adversarial graph would be one with deep trait resolution chains (L2→S0 feedback), many constraints with shared dependencies (tests L1 propagator count), and nested speculation (tests S(-1) retraction). I'd want this as a Phase 0 benchmark.

**§9.3: Fan-in propagators**

This is a real design space. Currently the proposal has one readiness propagator per constraint per dependency. For a constraint with 3 dependencies, that's 3 propagators, each checking "am I non-⊥?" and then "are ALL deps non-⊥?". The redundancy: each propagator independently checks the full dependency set.

A **fan-in propagator** would be: one propagator per *constraint* (not per dependency) that watches ALL dependency cells simultaneously. When any input changes, it checks the full readiness condition.

```racket
;; Fan-in: ONE propagator watching N cells
(net-add-propagator!
  (list dep-cell-1 dep-cell-2 dep-cell-3)  ;; all deps
  (lambda (v1 v2 v3)
    (when (and (not (bot? v1)) (not (bot? v2)) (not (bot? v3)))
      (cell-write! ready-queue (list action)))))
```

vs the current proposal:

```racket
;; Per-dep: N propagators, each checking ALL deps
(for ([dep (in-list deps)])
  (net-add-propagator!
    (list dep)
    (lambda (v)
      (when (and (not (bot? v)) (all-deps-non-bot? meta-id))
        ...))))
```

The fan-in approach: N constraints × 1 propagator = N total. The per-dep approach: N constraints × avg-deps propagators = N×D total. For 50 constraints averaging 2 deps, that's 50 vs 100 propagators.

The trade-off: fan-in propagators fire on ANY input change, even if the constraint isn't ready yet (one dep solved but others still ⊥). Per-dep propagators also fire on any change but are individually cheaper (fewer inputs to check). The net work is similar — fan-in does one check per input change, per-dep does one check per input change. But fan-in has fewer propagator objects (less overhead in the scheduler's dirty set management).

Our network already supports multi-input propagators (`net-add-propagator!` takes a list of input cell IDs). So fan-in is mechanically straightforward. I think we should use fan-in — fewer propagator objects, same total work, cleaner mental model. The propagator taxonomy would be:

| Pattern | Inputs | Outputs | When to use |
|---------|--------|---------|-------------|
| Transform | 1 cell | 1 cell | Simple lattice map |
| Fan-in | N cells | 1 cell | Readiness, conjunction, aggregation |
| Fan-out | 1 cell | N cells | Broadcasting, shadow cells |
| Bridge | 1 cell (net A) | 1 cell (net B) | Cross-network Galois connection |

**§13.2: Which option for greatest observability?**

This is a principles-driven question. The answer is: **whatever keeps the information on the network as cell values**. If warnings are cell values with assumption tags, provenance annotations, and source locations, then tooling (LSP, error reporter, IDE) can observe them through cell reads. If warnings are in a side-channel (parameter, log), tooling needs a separate observation path.

This argues for the full assumption-tagged cell treatment (not a simpler parameter-based approach). The cost is slightly higher (tagged entries), but the observability is maximal — and observability is a core principle.

**§13.3: On-network queueing**

You're right to push on this. In the current design, the ready-queue cell accumulates action descriptors, and L2 reads/clears it. The "clearing" is the issue — it's a non-monotone operation (removing entries from the queue after processing). Where does the clearing happen?

If the ready-queue is a cell with list-append merge, entries accumulate monotonically. L2 reads the queue. But after processing, the entries need to be removed so they're not processed again. This is a non-monotone operation on the queue cell.

**On-network queueing approach**: The ready-queue cell uses assumption-tagged entries. When L2 processes an entry, it retracts the entry's assumption (marking it as "consumed"). S(-1) cleans it up on the next cycle. This keeps queueing entirely on-network:

- L1 readiness propagator fires → writes action descriptor to ready-queue cell (tagged with a fresh assumption)
- L2 resolution propagator reads ready-queue → processes each entry → retracts each entry's assumption
- S(-1) cleans retracted entries from the ready-queue cell on the next cycle

No off-network clearing. The queue lifecycle is: write (L1) → read (L2) → retract (L2) → clean (S(-1)). All on-network, all through the existing assumption/retraction mechanism.

This is a nice instance of the propagator taxonomy — the ready-queue cell is a **channel cell** (write-read-consume pattern), implemented through assumption tagging rather than mutable state. The "taxonomy of propagators" you're thinking about would include:

| Cell type | Write pattern | Read pattern | Lifecycle |
|-----------|---------------|-------------|-----------|
| Value cell | Monotone merge (lattice) | Current value | Permanent within stratum |
| Accumulator cell | List-append / hash-union | Full collection | Retractable by assumption |
| Channel cell | Tagged write | Read + retract assumption | Produce-consume |
| Shadow cell | Mirror from source (bridge) | Same as source | Follows source lifecycle |

The channel cell is the queueing pattern — and it's implementable entirely through the existing assumption-tagged cell + S(-1) retraction mechanism. No new infrastructure needed.

**§13.4: Snapshot boxes**

Agreed — 1 box per snapshot context. If the same snapshot serves all consumers, 1 box. If different modules need different views, each gets its own box pointing to its CHAMP subtree. The CHAMP structural sharing means this is cheap even with multiple boxes.
