OK, here's my thinking on this. There are several interconnected questions, so let me unpack them layer by layer.

## What `solve` Returns: The Spectrum

The core tension is between **simplicity for the common case** and **richness for the advanced case**. There are really four levels of information a solve can return:

### Level 0 — Just bindings
```prologos
(solve [ancestor "alice" ?who])
;; => Seq (Map Keyword Value)
;; => '[{:who "bob"} {:who "carol"} {:who "dave"}]
```
This is what 90% of use sites want. Clean, composable with `map`/`filter`/`fold`.

### Level 1 — Bindings + metadata per answer
```prologos
;; Each answer carries: which clause derived it, how many steps, etc.
;; => Seq (Answer Value)
;; where Answer has :bindings, :clause-id, :depth, :cost
```

### Level 2 — Bindings + derivation tree (provenance)
```prologos
;; Each answer carries a tree of rule applications
;; ancestor("alice", "carol")
;;   ├── parent("alice", "bob")        [fact, clause 1]
;;   └── ancestor("bob", "carol")      [derived, clause 2]
;;       └── parent("bob", "carol")    [fact, clause 1]
```

### Level 3 — Full ATMS view (support sets, worldviews, nogoods)
```prologos
;; Each answer carries its ATMS support set — the minimal
;; assumptions required for this answer to hold
;; Useful for: belief revision, explanation, "what-if" analysis
```

## The Design Question: Where Do Options Live?

You raised the right question. There are three places provenance/reporting controls could live:

### Option A: On the solver definition
```prologos
solver verbose-solver
  :execution  :parallel
  :provenance :full         ;; always track derivation trees
  :reporting  :detailed     ;; Level 2 answers
```
**Pro**: Self-documenting, easy to switch globally by shadowing `default-solver`.
**Con**: Rigid. Can't ask for provenance on one query without changing the solver for all queries.

### Option B: On `solve-with` as inline options
```prologos
(solve-with :provenance :full [ancestor "alice" ?who])
```
**Pro**: Per-query control. Most explicit.
**Con**: Verbose for the common case. Mixes solver dispatch config with output format config.

### Option C: Solver merge — options extend the solver
```prologos
;; The solve-with form can take a solver AND inline overrides
(solve-with default-solver :provenance :full [ancestor "alice" ?who])

;; Or equivalently, an option map merges with the solver:
(solve-with [solver-with default-solver {:provenance :full}]
  [ancestor "alice" ?who])
```
**Pro**: Composable. Base solver defines defaults; per-call overrides extend them.
**Con**: Need a merge semantics.

**My recommendation is Option C** — with the important twist that `solve` (bare) stays simple, and the merge is just map-merge semantics (which we already have):

```prologos
;; Common case: bare solve, Level 0 return
(solve [ancestor "alice" ?who])
;; => Seq (Map Keyword Value)

;; Advanced case: merge provenance option into solver
(solve-with default-solver :provenance :full [ancestor "alice" ?who])
;; => Seq (Answer {:bindings Map, :derivation DerivationTree})
```

The solver config is just a Map under the hood. `solve-with solver-name :key val ...` merges `{:key val}` into the solver's config map. This means:

- Solver definitions set **defaults**
- Inline options **override** per-query
- No new mechanism needed — it's map merge

## The Return Type: Polymorphic on Provenance Level

This is the crux. The return type of `solve` should depend on the provenance level:

```prologos
;; Level 0 (default): Seq (Map Keyword Value)
;; - Just bindings. Lightweight. Composable.

;; Level 1 (:provenance :summary): Seq (Answer Value)
;; - Answer is a record with :bindings + lightweight metadata

;; Level 2 (:provenance :full): Seq (Answer Value)  
;; - Answer.derivation is a DerivationTree

;; Level 3 (:provenance :atms): Seq (Answer Value)
;; - Answer.support is the ATMS support set (assumption set)
;; - Answer.worldview is the worldview that produced this answer
```

For the type system, `solve` always returns `Seq A` where `A` depends on the provenance level. The default `A = Map Keyword Value`. With provenance, `A = Answer Value` where `Answer` is:

```prologos
;; In lib/prologos/core/solver.prologos
deftype Answer {V : Type}
  :bindings    (Map Keyword V)
  :clause-id   (Option Keyword)       ;; which clause produced this
  :depth       Nat                     ;; derivation depth
  :derivation  (Option DerivationTree) ;; present when :provenance :full
  :support     (Option (Set Keyword))  ;; present when :provenance :atms

deftype DerivationTree
  :goal     Keyword                    ;; relation name
  :args     (List Value)               ;; instantiated arguments
  :rule     Keyword                    ;; clause identifier
  :children (List DerivationTree)      ;; sub-derivations
```

## A Cleaner Ergonomic: `solve` vs `explain`

Actually, I think there's an even better decomposition. Instead of overloading `solve` with provenance options, what if we have **two forms** that make the intent explicit:

```prologos
;; solve: I want answers. Always returns Seq (Map Keyword Value).
;; Zero ceremony. The 90% case.
(solve [ancestor "alice" ?who])

;; explain: I want to understand the derivation. Returns Seq Answer.
;; Provenance tracking is always on. The 10% case.
(explain [ancestor "alice" ?who])

;; explain-with: same merge semantics as solve-with
(explain-with default-solver :provenance :atms [ancestor "alice" ?who])
```

This gives us:

| Form | Returns | Provenance | Use Case |
|---|---|---|---|
| `solve` | `Seq (Map Keyword Value)` | Off | Computation |
| `solve-with` | `Seq (Map Keyword Value)` | Off | Custom solver |
| `explain` | `Seq (Answer Value)` | Full | Debugging, audit |
| `explain-with` | `Seq (Answer Value)` | Configurable | Advanced debugging |

The **key insight**: `solve` never pays provenance overhead. `explain` always includes it. No runtime flag-checking, no polymorphic return type confusion. The types are monomorphic per form.

And `solve-with` still supports the full merge semantics for solver config (execution, strategy, threshold, tabling, timeout), just not provenance. Provenance is the `explain` family's domain.

## What Do You Think?

The main design decisions are:

1. **Solver merge (Option C)** — `solve-with solver-name :key val ...` merges inline options into the solver config. Composes naturally.

2. **`solve` / `explain` split** — clean separation between "give me answers" and "show me why." Monomorphic return types per form.

3. **`Answer` as a deftype** — when you use `explain`, you get a proper structured record, not just a map. Derivation trees, support sets, clause IDs are all typed fields.

4. **`default-solver` remains shadowing-based** — module-level override, no global mutation.

The question I'd want your take on: does the `solve` / `explain` split feel right, or would you rather `solve` itself be able to return richer data when asked? The tradeoff is simplicity of mental model (two forms, each does one thing) vs. fewer keywords to learn (one form, configurable output).

  ## `explain` returns answers WITH provenance — absolutely right

  Your point that the what and the why should be bundled is spot-on. Separating them and forcing a zip later would be bad ergonomics and also lose the structural guarantee that answer N's provenance IS answer N's provenance. The `Answer` type should be a single record:

  ```prologos
  ;; explain always returns this:
  ;; Seq (Answer Value)
  ;; where each Answer carries BOTH
  ;;   .bindings  : Map Keyword Value     ;; the what
  ;;   .derivation : DerivationTree        ;; the why
  ;;   .support   : Option (Set Keyword)   ;; the assumptions (if ATMS active)
  ```

  And you're right — if someone only wants the bindings from an `explain` result, they just map `.bindings` over the seq. The provenance is there but you pay no ceremony ignoring it. Data is data.

  ## On the separate top-level form

  Your pattern recognition is correct — `solver` → `solve`, `spec` → type-checks — these top-level configuration forms have been a good pattern. The question is whether provenance config merits its own.

  I think the answer is **no, but for an interesting reason**.

  A `solver` configures the *execution engine*: how to traverse the search space. Provenance is *what to observe during that traversal*. These feel orthogonal at first, which is why a separate form seems appealing. But here's the thing: provenance tracking has real execution costs. Turning on `:derivation :full` means the solver must build tree nodes at every rule application. Turning on `:support :atms` means the solver must maintain support sets even when it otherwise wouldn't. The provenance level *changes what the solver does*, not just what it reports afterward.

  So provenance options genuinely belong on the solver config. They're execution parameters, not post-hoc formatting choices.

  This means:

  ```prologos
  solver default-solver
    :execution   :parallel
    :strategy    :auto
    :tabling     :by-default
    :provenance  :none          ;; the default: no provenance overhead
    :timeout     :none

  solver debug-solver
    :execution   :sequential    ;; sequential for reproducible traces
    :strategy    :auto
    :provenance  :full          ;; derivation trees + support sets
    :timeout     10000
  ```

  And the dispatch is clean:

  ```prologos
  ;; solve ignores :provenance even if the solver has it set
  ;; (or more precisely, solve's return type is always Seq Map regardless)
  (solve [ancestor "alice" ?who])
  ;; => Seq (Map Keyword Value)

  ;; explain reads :provenance from the solver (defaults to :full if unset)
  (explain [ancestor "alice" ?who])
  ;; => Seq (Answer Value) — uses default-solver, forces provenance on

  ;; explain-with: override solver + inline merge
  (explain-with debug-solver [ancestor "alice" ?who])
  (explain-with default-solver :provenance :atms [ancestor "alice" ?who])
  ```

  The key semantic: `explain` is *not* a different solver. It's `solve` with provenance forced on. If the solver already has `:provenance :full`, great — `explain` uses it. If the solver has `:provenance :none`, `explain` overrides that to `:full`. `solve` always overrides to `:none` regardless. The forms control the *observation mode*, the solver controls the *execution strategy*.

  This means no separate top-level form needed. The solver carries all config. `solve` vs `explain` is the lightweight toggle on whether you want observation or not.

  ## The inline map merge — yes, for both families

  Both `solve-with` and `explain-with` accept `solver-name :key val ...` with map merge semantics:

  ```prologos
  ;; All of these work:
  (solve-with debug-solver [goal])                           ;; named solver
  (solve-with default-solver :timeout 5000 [goal])           ;; merge override
  (solve-with :execution :sequential :timeout 5000 [goal])   ;; anonymous config, merges into default-solver

  (explain-with debug-solver [goal])                         ;; named solver
  (explain-with default-solver :provenance :atms [goal])     ;; upgrade provenance level
  ```

  The elaboration rule: if `solve-with` / `explain-with` gets a bare solver name, use that. If it gets `:key val` pairs after the name, merge them over the solver's config. If it gets `:key val` pairs with no name, merge over `default-solver`.

  ## Why this works better than a separate form

  If we had a separate `explainer` / `witness` / `lens` form, you'd write:

  ```prologos
  explainer my-explainer
    :provenance  :full
    :derivation  :tree
    :support     :atms

  (explain-with debug-solver my-explainer [goal])
  ```

  Now you're passing *two* configs to one call. What happens when they conflict? If `debug-solver` has `:provenance :summary` and `my-explainer` has `:provenance :full`, who wins? You need merge precedence rules between two different config types. That's accidental complexity.

  Keeping it on one config (the solver) and using `solve` vs `explain` as the observation toggle avoids all of this. One source of truth for execution parameters. Two verbs for "do I want the why or not."

  ## Summary

  | Form | Returns | Provenance | Solver source |
  |---|---|---|---|
  | `solve` | `Seq (Map Keyword Value)` | Always off | `default-solver` |
  | `solve-with` | `Seq (Map Keyword Value)` | Always off | Named + merge |
  | `explain` | `Seq (Answer Value)` | Forced on (default `:full`) | `default-solver` |
  | `explain-with` | `Seq (Answer Value)` | On, level from solver/merge | Named + merge |

  Solver carries `:provenance` as a config key. `solve` ignores it. `explain` respects it (defaulting to `:full`). One top-level form (`solver`), two verb families (`solve`, `explain`), inline map merge for both. No separate provenance config form — the cost stays proportional to the concept count.

  Good — the `{...}` syntax is a much cleaner parse. Here's my understanding of the full design:

## Solve/Explain Design Proposal

### Core Principle

Two verb families, one config form. `solve` and `explain` are observation-mode toggles over the same solver execution engine. The solver carries all configuration, including provenance options. The difference is purely what you get back.

### Return Types

**`solve`** — returns `Seq (Map Keyword Value)`. Bare bindings. Provenance tracking is off regardless of what the solver config says. This is the fast path: zero observation overhead, maximum composability with functional collection ops.

**`explain`** — returns `Seq (Answer Value)`. Each `Answer` bundles the bindings AND the provenance together as a single record. Provenance tracking is forced on (defaulting to `:full` if the solver doesn't specify a level). The what and the why are never separated.

```prologos
deftype Answer {V : Type}
  :bindings    (Map Keyword V)
  :derivation  DerivationTree
  :depth       Nat
  :support     (Option (Set Keyword))   ;; present when :provenance :atms
```

### The Four Forms

```prologos
;; 1. Bare solve — dispatches to default-solver, returns Seq Map
(solve [ancestor "alice" ?who])

;; 2. Solve-with — named solver, optional map merge, returns Seq Map
(solve-with debug-solver [goal])
(solve-with default-solver {:timeout 5000} [goal])
(solve-with {:execution :sequential :timeout 5000} [goal])

;; 3. Bare explain — dispatches to default-solver, forces provenance, returns Seq Answer
(explain [ancestor "alice" ?who])

;; 4. Explain-with — named solver, optional map merge, returns Seq Answer
(explain-with debug-solver [goal])
(explain-with default-solver {:provenance :atms} [goal])
(explain-with {:provenance :summary :timeout 5000} [goal])
```

### `solve-with` / `explain-with` Arity

The `{...}` map literal makes the grammar unambiguous. The `-with` forms have exactly three shapes:

| Shape | Meaning |
|---|---|
| `(-with solver-name [goal])` | Use named solver as-is |
| `(-with solver-name {overrides} [goal])` | Merge overrides into named solver |
| `(-with {overrides} [goal])` | Merge overrides into `default-solver` |

No variadic keyword parsing. The `{...}` is always a single syntactic unit. The `[goal]` is always last. The parser sees at most three children after the keyword.

### Solver Config — Single Top-Level Form

`solver` carries everything: execution strategy, provenance level, tabling, timeout. No separate "explainer" form.

```prologos
solver default-solver
  :execution   :parallel
  :threshold   4
  :strategy    :auto
  :tabling     :by-default
  :provenance  :none          ;; default: no provenance cost
  :timeout     :none

solver debug-solver
  :execution   :sequential
  :strategy    :auto
  :provenance  :full          ;; derivation trees always
  :timeout     10000
```

Provenance options on the solver are:
- `:none` — no tracking (default)
- `:summary` — clause-id + depth, no tree
- `:full` — complete derivation tree
- `:atms` — derivation tree + ATMS support sets per answer

### Interaction Between `solve`/`explain` and `:provenance`

- **`solve`** always behaves as if `:provenance :none`, regardless of what the solver says. Even if you write `(solve-with debug-solver [goal])` where `debug-solver` has `:provenance :full`, you get `Seq Map` back. The provenance config is ignored — `solve` means "I want answers, not explanations."

- **`explain`** reads `:provenance` from the (possibly merged) solver config. If the solver says `:none` or doesn't specify, `explain` defaults to `:full`. So `(explain [goal])` with `default-solver` (which has `:provenance :none`) still gives you full derivation trees — `explain` means "I want the why."

- **`explain-with`** respects whatever `:provenance` level the merged config produces. So `(explain-with default-solver {:provenance :summary} [goal])` gives you lightweight metadata without full trees.

This means `solve` and `explain` are truly about the *caller's intent*, not the solver's config. The solver's `:provenance` key is a *default level for explain*, not a gate on whether provenance happens.

### `default-solver` Shadowing

Unchanged from our earlier design. `default-solver` is a module-level binding. Users shadow it in their namespace:

```prologos
;; This module's solve/explain calls all use this solver
solver default-solver
  :execution  :sequential
  :provenance :full
  :timeout    5000
```

No global mutation. Normal name resolution.

### Map Merge Semantics

The `{overrides}` map does a shallow merge over the solver's config. Last-writer-wins per key. This is just our existing map semantics — `{:timeout 5000}` merged over `default-solver`'s config produces a new config where `:timeout` is `5000` and everything else is inherited. The merge produces a transient solver config for that one call — it's not a named entity and doesn't persist.

---

That's my understanding. The design has five moving parts: one top-level form (`solver`), two bare verbs (`solve`, `explain`), two `-with` variants, and `{...}` map merge for inline overrides. Everything else — provenance levels, execution strategy, tabling — lives on the solver config as keyword options.

## The Relational Language: Surface Syntax and Fusion with the Functional Language

Here is my understanding of the Relational Language's surface syntax and how it interfaces with the Functional (Command) Language, grounded in Prologos's principles.

---

### 1. The Core Duality: `defn`/`fn` ↔ `defr`/`rel`

The Relational Language is not an embedded DSL or a macro library — it is a co-equal paradigm alongside the Functional Language, with a symmetric keyword structure:

| Concern | Functional | Relational |
|---------|-----------|------------|
| Named definition | `defn` | `defr` |
| Anonymous form | `fn` (lambda) | `rel` (anonymous relation) |
| Application | `[f x y]` — evaluate, return a value | `(R ?x ?y)` — constrain, return a stream of substitutions |
| Delimiter sigil | `[...]` (brackets) | `(...)` (parentheses) |

The **anonymous relation** (`rel`) is the key innovation. Just as `fn` creates a first-class functional value that can be bound, passed, and composed, `rel` creates a first-class relational value with the same compositional affordances. No existing logical or logic-functional language has a true anonymous relation that participates in the language as a lambda does.

The bracket/paren distinction is not incidental syntax — it reflects a genuine semantic boundary. `[f x y]` says "evaluate this command, produce a value." `(R ?x ?y)` says "constrain this search space, produce a stream of substitutions." The reader dispatches differently on the opening delimiter; the s-expression representation preserves the distinction (`($app f x y)` vs `($rel-app R ?x ?y)`).

---

### 2. `defr` — Named Relations with `&>` Clause Syntax

A `defr` defines a named relation with one or more clauses. Each clause is introduced by `&>`, which reads as "and also, by this rule." Within a clause, goals are **implicitly conjoined** by whitespace juxtaposition.

```prologos
defr ancestor [?x ?y]
  &> (parent ?x ?y)                        ;; clause 1: base case
  &> (parent ?x ?z) (ancestor ?z ?y)      ;; clause 2: recursive
```

Design principles at work:
- **`&>` is visually distinct** from functional operators (`|>`, `>>`, `|`). The `&` evokes conjunction/AND; each `&>` introduces an alternative (OR) whose internal goals are conjoined (AND).
- **Tabled by default**: All `defr` relations are automatically tabled (SLG-style memoization). This ensures left-recursive rules terminate, redundant computation is eliminated, and completeness is guaranteed. Opt-out is via `:tabled false` on the spec.
- **Self-specifying**: A relation's clauses *are* its specification (the Curry-Howard perspective). However, `spec` can *optionally* enhance a `defr` — for instance, providing mode annotations, lattice answer modes, documentation, or ergonomic type constraints on logic variables. The `spec` is not required, but it's available for the same progressive enrichment that functions enjoy.

---

### 3. Logic Variables: `?var`, `-var`, `+var`

Logic variables are distinguished by a `?` prefix. This is the default "free" mode — the variable may be bound or unbound, and unification determines its value.

```prologos
(ancestor ?who "carol")    ;; ?who is a logic variable
[map inc xs]               ;; xs is a functional variable (no ?)
```

The `?` prefix serves four purposes simultaneously:
1. **Visual distinction** from functional variables (immediate human readability)
2. **Reader-level dispatch** (the reader emits `($logic-var who)`)
3. **Scope delimitation** (logic vars are scoped to their enclosing `rel`/`defr`)
4. **Mode annotation carrier** (see below)

Mercury-inspired mode annotations provide optimization hints:

| Mode | Symbol | Meaning |
|------|--------|---------|
| Free | `?var` | May be bound or unbound |
| In   | `-var` | Must be bound on entry |
| Out  | `+var` | Unbound, will be bound by goal |

Modes are *optional* — the default `?` works for all cases. When provided, modes let the solver choose more efficient strategies (index on bound arguments, generate rather than check unbound arguments). This follows the progressive disclosure principle: beginners write `?var` everywhere; experts add `-var`/`+var` for performance.

---

### 4. The Two Bridges: `solve`/`explain` and `is`

The fusion between paradigms requires two directional bridges — one to enter the relational world from functional code, one to call functional code from relational goals.

#### 4a. Functional → Relational: `solve` and `explain`

`solve` is the primary bridge from functional code into the relational world. It evaluates a relational goal and returns results as a functional value:

```prologos
;; solve returns: Seq (Map Keyword Value) — bare bindings
let results := (solve [ancestor "alice" ?who])
;; => '[{:who "bob"} {:who "carol"} {:who "dave"}]
```

The return type is `Seq (Map Keyword Value)` — a lazy sequence of substitution maps. This integrates naturally with the functional collection system: results can be mapped, filtered, folded, taken, piped through `|>`. The logic variables become keyword keys (`:who`, not `?who`) in the result maps.

`explain` is the observation-mode bridge. It returns `Seq (Answer Value)`, where each `Answer` bundles the bindings (the *what*) with provenance (the *why*) — derivation trees, clause IDs, ATMS support sets:

```prologos
;; explain returns: Seq (Answer Value) — bindings + provenance bundled
(explain [ancestor "alice" ?who])
;; Each Answer has:
;;   .bindings   : Map Keyword Value     — the what
;;   .derivation : Option DerivationTree — the why
;;   .depth      : Nat
;;   .support    : Option (Set Keyword)  — ATMS support
```

The key design principle: **bindings and provenance are never separated**. If you want the *why*, you always see *which answer* it explains. This avoids the ceremony of zipping or mapping the what and the why together after the fact.

The progressive disclosure hierarchy:
- **Level 0**: `(solve [goal])` — bare bindings, zero overhead
- **Level 1**: `(solve-with my-solver [goal])` — named solver config
- **Level 2**: `(explain [goal])` — bindings + derivation tree
- **Level 3**: `(explain-with my-solver {:provenance :atms} [goal])` — full ATMS support sets
- **Level 4**: Module-level `solver default-solver` shadowing — ambient config change

Both `-with` forms accept `{...}` explicit map literals for override merges:
```prologos
(solve-with default-solver {:timeout 5000} [goal])
(explain-with {:provenance :atms :timeout 5000} [goal])
```

The `{...}` syntax is unambiguous in the parser (cannot be a goal, cannot be a solver name), and follows the existing map-merge semantics — shallow merge, overrides win per key.

#### 4b. Relational → Functional: `is`

`is` evaluates a functional expression and binds the result to a logic variable. It is the bridge from relational goals into the functional world:

```prologos
defr factorial [?n ?result]
  &> (= ?n 0) (= ?result 1)
  &> (> ?n 0)
     (is ?n1 [sub ?n 1])         ;; functional expression [sub ?n 1]
     (factorial ?n1 ?partial)    ;; recursive relational goal
     (is ?result [mul ?n ?partial])
```

The functional expression inside `is` is a full Prologos expression — type-checked, trait-resolved, subject to all the guarantees of the functional core. It just happens to be invoked from within a relational context.

---

### 5. `rel` — The Anonymous Relation (First-Class Relational Lambda)

Just as `fn [x] body` creates a first-class functional value, `rel [?x ?y] &> ...` creates a first-class relational value:

```prologos
;; Anonymous relation — first-class, bindable, passable
let reachable := (rel [?from ?to ?path]
  &> (= ?from ?to) (= ?path '[?to | nil])
  &> (edge ?from ?mid)
     (reachable ?mid ?to ?rest)
     (= ?path '[?from | ?rest]))

;; Use in functional context via solve
match (solve [reachable start end ?p])
  | [some path] -> [just path]
  | none        -> nothing
```

This is where the paradigm fusion is most visible: a functional `defn` creates a local `rel`, binds it to a name via `let`, and then queries it via `solve`. The relational computation happens inside a functional scope, and the results flow back as functional values. The bracket/paren distinction makes the paradigm boundaries instantly visible at the character level.

---

### 6. The `solver` Form — Configurable Metadata Entity

`solver` defines a configuration object that parameterizes how relational queries execute:

```prologos
solver default-solver
  :execution   :parallel     ;; BSP parallel execution
  :threshold   4             ;; parallelize when ≥4 runnable propagators
  :strategy    :auto         ;; PropNetwork → ATMS on first amb
  :tabling     :by-default   ;; all defr predicates tabled
  :provenance  :none         ;; no provenance overhead (solve ignores this; explain uses it)
  :timeout     :none
```

This follows the established Prologos pattern: structured keyword metadata attached to operations (cf. `spec`'s `:doc`, `:examples`, `:properties`). The solver is not a fixed runtime — it's an inspectable, composable value.

`default-solver` is a module-level binding, shadowed by normal name resolution. A module that wants different defaults simply redefines `default-solver` locally — no global mutation, no side effects.

---

### 7. How the Principles Are Embodied

| Principle | How the Relational Language embodies it |
|-----------|----------------------------------------|
| **Bracket sigils as paradigm markers** | `[...]` functional, `(...)` relational — visible, mechanical, unambiguous |
| **Progressive disclosure** | `solve` → `solve-with` → `explain` → `explain-with` → `solver` shadowing; modes optional; spec optional on defr |
| **Homoiconicity** | Every relational form has a canonical sexp: `($rel-app ...)`, `($clause-sep)`, `($logic-var ...)` |
| **Decomplection** | Solver config is decoupled from query form; provenance is decoupled from bindings (solve vs explain); execution strategy is decoupled from tabling |
| **Correctness through types** | Logic variables scoped to their relation; `solve` returns typed `Seq (Map Keyword Value)`; `is` evaluates type-checked functional expressions |
| **Spec as living interface** | `:tabled`, `:answer-mode`, `:strategy` metadata on specs — same progressive metadata escalation |
| **No trait hierarchies** | Solver configs compose via map merge, not inheritance. `{...}` overrides are flat conjunctive merge |
| **Keyword symmetry** | `defr`/`rel` mirror `defn`/`fn`; `solver` mirrors `spec`/`trait` as named metadata entity |
| **AI-first design** | Derivation trees as first-class data; `explain` surfaces machine-checkable justifications; provenance via ATMS support sets |

---

### 8. Seamless Fusion: The Complete Picture

The functional and relational languages fuse at two points and one configurable seam:

1. **`solve`/`explain`**: Relational → Functional (results as lazy sequences of maps/answers)
2. **`is`**: Functional → Relational (evaluation of functional expressions within relational goals)
3. **`solver`**: The configurable seam — how the relational engine runs, what it records, when it parallelizes

A complete program can freely intermix both paradigms:

```prologos
ns pathfinder

;; Functional: define a type and data
spec find-all-paths : Graph Node Node -> [List (List Node)]
defn find-all-paths [graph start end]
  ;; Relational: define search space locally
  let path := (rel [?from ?to ?p]
    &> (= ?from ?to) (= ?p '[?to | nil])
    &> (edge-in graph ?from ?mid)
       (path ?mid ?to ?rest)
       (= ?p '[?from | ?rest]))
  ;; Bridge back: solve, then pipe through functional transforms
  |> (solve [path start end ?p])
    map [fn [bindings] [map-get bindings :p]]
    filter [fn [p] [< [length p] 10]]
    into-list
```

The brackets tell you which world you're in at every point. The `solve` call is the bridge. The `|>` pipe is pure functional. The `rel` is pure relational. They compose without ceremony.
