The crucial finding: **there aren't separate networks for sessions, effects, and IO.** They're all sub-graphs within a single `prop-network`, differentiated by their lattice merge functions and connected by cross-domain bridges (Galois connections). The `prop-network` itself is CHAMP-based — immutable, structural sharing, O(1) snapshots. This means the infrastructure for what you're asking about already exists at the primitives level.

---

**On your point 1 — lifecycle generalizability:**

There *is* a pattern across the domain-specific sub-graphs:

| Domain | Lifecycle | Lattice | Bridge mechanism |
|--------|-----------|---------|-----------------|
| Session | `sess-bot → send/recv/... → sess-top` | `session-lattice-merge` | cross-domain propagator (α/γ) |
| Effect | `eff-bot → eff-position → eff-top` | unidirectional α from session | `add-session-effect-bridge` |
| IO | `io-bot → io-opening → io-open → io-closed → io-top` | `io-lattice-merge` | Gauss-Seidel scheduler |
| Module (proposed) | `mod-loading → mod-loaded → mod-stale → mod-reloading` | ? | ? |

The session and IO lattices encode protocol ordering — you can't `recv` before `send` in a session, can't `read` before `open` in IO. A module lifecycle lattice would encode: you can't reference a definition before it's loaded, and staleness is monotonic (once a dependency changes, you're stale until reloaded).

But here's where I think the generalization gets interesting: the module lifecycle isn't really about a *single cell's* lattice progression. It's about a *collection of cells* (the module's definitions) having a collective state. Session protocols track one channel's state. Module networks track many definitions' states. The lifecycle is better modeled as **a property of the network itself** — a "network status" cell that monitors all the definition cells within it.

So: a module network could have a single `mod-status` cell with lattice `loaded | stale | reloading`, where propagators from dependency edges write `stale` when a source changes. This gives the LSP a single cell to watch per module. The ordering effect ("don't use definitions from a stale module") becomes a propagator guard.

**On your point 2 — maximum value from cross-module, minimum recompute:**

This is where CHAMP structural sharing becomes the key enabler. Currently `module-info` caches an `env-snapshot` as a flat hasheq. With persistent networks:

Instead of materializing the snapshot, the module cache holds a **network reference**. When file `foo.prologos` imports module `bar`, it doesn't copy `bar`'s definitions into its own network. Instead, it wires cross-network read propagators: "when I look up `bar::map`, read from `bar`'s network's `map` cell."

Minimum recompute then falls out naturally:
- Module `bar` changes → `bar`'s network is rebuilt → `bar::map` cell gets new value
- Cross-network propagator fires → marks `foo`'s `uses-bar-map` dependency edge as stale
- Only definitions in `foo` that transitively depend on `bar::map` are marked for re-elaboration
- Everything else in `foo` is untouched

The cross-network propagator is just `net-add-cross-domain-propagator` with identity α/γ (same type domain, different network). The infrastructure exists — it's used for session→effect bridges already. We'd be applying it to module→file bridges.

**On your point 3 — `global-env-snapshot` as network reference:**

Agreed, worth designing for now. The change is: `module-info`'s `env-snapshot` field changes from `hasheq` to a network handle (or a pair: network + cell-id-map). Consumers that import a module get a live reference rather than a dead copy.

The backwards-compatibility path: `global-env-snapshot` can return a `module-network-ref` struct that implements the same hasheq interface (via `prop:dict` or a wrapper) but reads from cells underneath. Existing code that does `(hash-ref snapshot name)` keeps working but actually reads from the module's live cells.

---

**On your Q1 revisit — shared structure vs. separate networks:**

Given that networks are CHAMP-based, "separate network per module" and "shared pointers to common structure" aren't mutually exclusive — they're the same thing at the CHAMP level. Two module networks that share common definitions (e.g., both import the prelude) share CHAMP nodes for those cells. The networks are logically separate (different `prop-network` structs) but physically sharing.

But your ATMS question opens a deeper possibility. Consider:

A test file does `(parameterize ([current-fuel 100]) ...)`. The prelude module was loaded with default fuel. The test's definitions might elaborate differently under different fuel. Today this doesn't matter because module loading caches the first load's result. But with persistent networks and incremental re-elaboration, we face a question: **is the module's elaboration result parameterization-dependent?**

If yes, then different parameter contexts are literally different *assumptions* about the world. ATMS handles this perfectly:
- Assumption A₁: "fuel = default" → prelude definitions D₁
- Assumption A₂: "fuel = 100" → prelude definitions D₂ (possibly identical, possibly different)
- A consumer under A₂ reads definitions tagged with A₂'s support set

The TMS cells already support this — Track 4 put TMS cells in the network for exactly this kind of assumption-tagged values. The module's definition cells could be TMS cells where the support set includes the parameter context assumption.

**But** — and this is important — I think this is the right *future architecture* but premature for Track 5. Here's why:

1. Today, module caching is deterministic. Same module source → same definitions. Parameters like `current-fuel` affect *how much work* the elaborator does, not *what definitions* it produces (modulo fuel exhaustion errors). The only parameter that actually changes definition semantics would be something like a conditional compilation flag, which doesn't exist yet.

2. Making every module definition a TMS cell adds Track 4 machinery to every module load. The depth-0 fast path helps, but it's still overhead for a scenario that doesn't yet exist (parameter-dependent module elaboration).

3. The ATMS approach shines when you have *multiple simultaneous parameter contexts* wanting to share a module — that's an LSP scenario (different open files with different configs). Batch compilation has one context.

**Recommendation**: Design the module network API so definition cells *can be* TMS cells (the slot exists, the merge function is compatible), but instantiate them as plain cells in Track 5. Track 6 or the LSP track can flip the switch to TMS cells when multi-context sharing is needed. The network structure is identical — only the cell type changes.

This is the belt-and-suspenders principle applied forward: design for TMS, implement with plain cells, upgrade is a one-line change per cell creation.

---

**On phase ordering (point 4 from earlier):**

Given this discussion, I think the phases should be reordered. The current design puts write consolidation first (Phases 1-2), then networks (Phases 3-4), then edges (Phase 5). But if we're designing for persistent module networks with cross-module edges from the start, the architecture is:

1. **Phase 1**: Module network infrastructure (the new primitive — persistent network per module, network-ref in module-info, mod-status cell)
2. **Phase 2**: `global-env-remove!` + failure cleanup (needs network awareness)
3. **Phase 3**: Per-module network activation (remove `#f` overrides, module loading writes to cells)
4. **Phase 4**: Cross-module dependency edges (the high-value deliverable)
5. **Phase 5**: Write consolidation + validation (clean up the now-redundant env-threading)
6. **Phase 6**: Performance validation + PIR

This puts the architectural foundation first and the mechanical cleanup last, which matches the "first-one-is-the-hardest" learning from Track 3.

Track 5 introduces three new mechanisms:
1. `module-network-ref` (new struct wrapping an existing `prop-network`)
2. Shadow cells (new pattern for cross-network reads)
