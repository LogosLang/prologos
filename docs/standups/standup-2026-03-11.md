## LSP Architecture: Key Design Questions

### 1. Where does the LSP server live?

There are three realistic options:

**A) Racket-native LSP server** — Write the server in Racket, wrapping your existing pipeline with JSON-RPC. This is the path of least resistance since all the type checking, elaboration, and error machinery is already in Racket. Racket has `json` and TCP libraries. Idris 2, Lean 4, and Agda all take this approach (server in the implementation language).

**B) Thin bridge process** — A lightweight Node.js or Python process handles LSP protocol, spawns/communicates with a Racket subprocess via stdin/stdout or a custom protocol. More plumbing but lets you use battle-tested LSP libraries (`vscode-languageserver` for Node).

**C) Hybrid** — Tree-sitter handles syntax-level features (highlighting, folding, outline) client-side, while Racket handles semantic features (diagnostics, hover types, go-to-def) via a simpler custom protocol.

My recommendation: **Option A** for the long term, possibly starting with a thin variant of **B** if you want rapid iteration on the VSCode side. The Racket pipeline is where all the intelligence lives — keeping it in-process avoids serialization overhead for type queries.

### 2. What's the biggest gap?

The pipeline currently processes whole files in one shot. For LSP to feel responsive, you need:

- **Source locations on core AST**: Surface AST (`surf-*`) preserves srclocs, but core AST (`expr-*`) drops them. You'd need an elaboration-time map from expressions to locations/types — this is the single most impactful infrastructure change.
- **Incremental-ish processing**: Full re-elaboration on every keystroke won't scale. At minimum, you'd want module-level caching (only re-check the changed file, keep cached module-infos for dependencies). Your `current-module-registry` already supports this pattern.

### 3. VSCode Extension Architecture

For VSCode specifically, the extension would have two layers:

**Syntax layer** (immediate, no server needed):
- TextMate grammar or tree-sitter grammar for highlighting
- Language configuration (brackets, comments, auto-closing)
- Snippets for common forms (`defn`, `spec`, `match`, etc.)

**Semantic layer** (requires LSP):
- Diagnostics (errors/warnings on save or on-type)
- Hover (show inferred type at cursor)

- Signature help (from `spec` declarations)

## The Fundamental Shift: Opaque Pipeline → Transparent Proof Object

Right now, state flows through Racket parameters. Parameters are *black holes* — you can read the current value, but you can't ask:

- **How did this value get here?** (provenance)
- **What assumptions does it rest on?** (support set)
- **What else would change if this value were different?** (dependency)
- **What combinations of inputs are known to be inconsistent?** (nogoods/learned clauses)

When state flows through ATMS-backed propagator cells, every piece of derived information carries its *justification*. The compilation isn't just producing a result — it's producing a **certificate** that explains *why* that result holds, and *under what assumptions*.

## What "Whole-System Information Surveillance" Unlocks

### 1. Provenance → Error Explanation as Derivation Chains

We already have a taste of this — Phase D's ATMS nogoods give us derivation chains for speculation failures. But that's just speculation sites. If *all* elaboration state flows through cells:

- "Type mismatch at line 12" becomes "type mismatch at line 12 *because*: user annotated `x : Nat` at line 3 (assumption A₁), trait `Add` requires `Num` at line 7 (assumption A₂), and `{A₁, A₂}` is a recorded nogood — `Nat` doesn't implement `Num`."
- Every constraint, every trait resolution, every unification has a support set. You can trace any failure to its root causes through the assumption graph.
- This is the difference between GCC's "in instantiation of template..." walls of text and a structured derivation that a tool can navigate.

### 2. Assumption Retraction → What-If Analysis (Model Checking)

ATMS retraction lets you ask counterfactual questions without re-running elaboration:

- **"What if this definition had a different type?"** — Retract the assumption for `def foo : T`, write a new value under a new assumption, re-propagate. Only the transitive dependents recompute.
- **"Does this property hold under all valid assumption sets?"** — Enumerate the consistent environments (the ATMS already knows which assumption combinations are inconsistent via nogoods). Each consistent environment is a "world." Model checking is exploring these worlds.
- **Session type protocol conformance**: A session type specifies a protocol. Checking conformance is asking "under the assumption that this process follows this protocol, are there any reachable inconsistencies?" That's exactly ATMS nogood discovery over the session-type constraint cells.

The propagator network becomes a *constraint system that can be explored*, not just evaluated.

### 3. Nogoods as Learned Clauses → Connection to SAT/SMT/CDCL

Each ATMS nogood says "these assumptions together are inconsistent." This is *exactly* what CDCL SAT solvers do — they learn conflict clauses during search and use them to prune future exploration.

Right now, the elaboration pipeline discovers inconsistencies (type mismatches, failed trait resolution, linearity violations) but forgets the *structure* of why they're inconsistent. With full cell-based flow:

- The nogood database accumulates across the entire compilation — it's a growing set of *learned facts* about the program.
- An ATP system could query this: "given the current nogoods, is there an assignment of metavariables that satisfies all constraints?" That's a satisfiability problem over the ATMS.
- Dependency-directed backjumping (the GDE/ATMS's original purpose) means when elaboration hits a contradiction, it doesn't just fail — it *explains* the minimal set of assumptions that caused the failure and jumps back to the relevant decision point.

### 4. The Self-Hosting Convergence

Here's where it gets really interesting for Prologos specifically. The language already unifies:

- **Dependent types** — compile-time and runtime blur
- **Session types** — protocol conformance checking (a model checking problem)
- **Linear types (QTT)** — resource usage verification (constraint tracking)
- **Logic programming** — constraint solving IS the computation model
- **Propagators** — the computation substrate

If the *compiler's internal reasoning* flows through the same propagator infrastructure that it *provides to users*, then:

- The compiler can **explain itself** using the same mechanisms programs use to explain their reasoning
- The compiler's elaboration trace is a **proof object** in the same sense that a dependent type is a proof
- User-level propagator programs and compiler-level propagator networks are the **same substrate** — the gap between "writing a program" and "reasoning about a program" collapses

This is deeper than the usual self-hosting story ("compiler written in its own language"). It's: **the compiler uses the same reasoning mechanisms it provides to users, and both are inspectable through the same interface.**

### 5. Runtime Verification via Assumption Certificates

If compilation artifacts carry their support sets (the assumptions under which they were elaborated), you get a basis for runtime verification:

- "This module was compiled assuming module X provides trait instance Y" → verify at link/load time
- "This function's type was inferred under assumption set S" → if S changes (hot reload, LSP edit), you know exactly which downstream results are invalidated
- Session type monitors can verify at runtime that the assumptions the type-checker relied on still hold

### 6. Incremental Everything

The dependency graph isn't just for LSP file-change invalidation. It's for:

- **Incremental verification**: change a lemma, re-verify only the dependents
- **Incremental optimization**: change a definition, re-optimize only the affected call sites
- **Incremental testing**: change a module, re-test only the dependent tests (we already have `run-affected-tests.rkt` doing this at the file level — cells could make it definition-level)

The re-scope I proposed was correct *tactically* — we shouldn't force batch-mode reads through cells just for LSP. But your question reframes it: the value isn't "the LSP needs cells." The value is **the entire system becomes a transparent, queryable, verifiable proof object.**

The question isn't "should we read from cells?" It's "what questions do we want to be able to ask?" — and then the cell migration follows from the answer.

This is a great question that sits at the intersection of several design threads you've already been working through. Let me trace through what exists, what the tensions are, and where I think the interesting design space lives.

## Current State

Right now, facts are represented as **flat positional lists**. The `fact-row` struct in `relations.rkt:69` is simply:

```racket
(struct fact-row (terms) #:transparent)
;; terms: a list of values, e.g. '("Alice" "Bob")
```

A `variant-info` holds a list of `fact-row`s alongside a list of `param-info`s (name + mode). So the "column names" exist in the parameter metadata, but the fact data itself is just a positional vector. The solver does positional unification against these lists — iterate through `(fact-row-terms fr)`, unify term-by-term against the goal arguments.

The `schema` form in the vision document introduces named, typed columns (`:parent String`, `:child String`), and the vision explicitly says schemas provide "positional ordering" — the field declaration order establishes the canonical mapping. But at runtime, fact data is still flat lists.

## The Triangle You're Describing

You're identifying three vertices:

1. **Positional tuples** — what Prolog uses, what we have now. Compact, fast to unify, but opaque: `("Alice" "Engineering" 95000)` means nothing without the schema nearby.

2. **Keyed maps** — what your Prolog-with-databases workflow uses (`table_name({column_key: Val, ...})`). Self-describing, composable, but heavier and potentially slower for the hot unification path.

3. **Typed records** — what schemas aspire to be. Named fields, type-checked, but with the efficiency of positional access.

The tension: positional is fast but fragile; keyed is robust but heavy; records are ideal but usually require compile-time resolution.

## How to Square It

I think the answer is already implicit in your schema design, but making it explicit reveals the right representation:

### The Schema *is* the Rosetta Stone

A `schema` declaration establishes a **bijection between positions and keys**:

```prologos
schema Employee
  :name   String    ;; position 0
  :dept   Department ;; position 1
  :salary Nat       ;; position 2
```

This means the underlying runtime representation can be **positional vectors** (fast, compact, cache-friendly) while the *surface syntax* supports **both positional and keyed access**. The schema mediates:

- **Storage**: flat vector `#("Alice" Engineering 95000)` — essentially a database row
- **Authoring (positional)**: `|| "Alice" Engineering 95000N` — tabular, concise
- **Authoring (keyed)**: `|| {:name "Alice" :dept Engineering :salary 95000N}` — self-documenting
- **Access (keyed)**: `alice.name` or `:name` in goal patterns
- **Unification**: positional, against the vector, always

The key insight: **keyed syntax is a compile-time convenience that erases to positional access**. The schema provides the mapping; after elaboration, everything is positional. This is exactly what databases do — SQL lets you say `WHERE name = 'Alice'` but the storage engine uses column offsets.

### What About Schema-less Relations?

Bare `defr` without a schema annotation works like traditional Prolog — positional only, untyped. This is fine for quick prototyping and small relations. The moment you want keyed access or database interop, you add a schema.

### The Database Import Use Case

Your Prolog workflow of `table_name({column_key: Val, ...}).` maps beautifully:

```prologos
;; Import: each table gets a schema
schema Employees
  :id     Nat
  :name   String
  :dept   String
  :salary Nat

;; Facts loaded from CSV/SQL/JSON — keyed or positional
defr employees : Employees
  || 1N "Alice" "Engineering" 95000N
     2N "Bob"   "Marketing"   72000N
     3N "Carol" "Engineering" 110000N

;; Or bulk-loaded from file
defr employees : Employees
  :source "data/employees.csv"
```

The schema does double duty: it's the type contract *and* the column mapping for import/export. A CSV header row, a SQL `CREATE TABLE`, a JSON schema, and a Prologos `schema` all describe the same thing — a named sequence of typed fields. The schema is the interop surface.

### PVec vs Flat Vector vs CHAMP Map

For the underlying runtime representation, the options are:

| Representation | Pros | Cons |
|---|---|---|
| **Flat Racket vector** | Fastest access, minimal allocation | Not persistent, no structural sharing |
| **PVec (RRB tree)** | Persistent, shareable, log-time access | Overkill for small fixed-width rows |
| **CHAMP map** | Keyed access, persistent | Heaviest, hash overhead per lookup |
| **Struct** | Fastest, named fields | No dynamism, one struct per schema |

My analysis: for **fact storage**, flat vectors (or even Racket structs generated per-schema) are the right choice. A fact row is small (typically 3-15 fields), fixed-width (schema determines it), and accessed many times during unification. The hot path is "unify position i of fact with goal argument i" — this wants O(1) indexed access, not hash lookup.

PVec and CHAMP are great for *user-facing collections* (lists, maps in the functional language), but using them for the internal fact representation would be paying persistent-data-structure overhead for data that doesn't need persistence — facts are immutable once asserted.

**The representation I'd suggest**: compile-time-generated Racket structs per schema (fastest) with a fallback to flat vectors for schema-less relations. The struct fields correspond to schema columns. The schema provides the name→position mapping for keyed syntax. This is essentially what Ecto changesets do — the schema generates a struct, and the struct is the runtime representation.

### Keyed Pattern Matching in Goals

The really interesting ergonomic question is: can you use keyed patterns in relational goals?

```prologos
;; Positional (current)
(employees id name "Engineering" salary)

;; Keyed (proposed) — only bind what you care about
(employees {:dept "Engineering" :name name :salary salary})

;; Mixed — positional with keyword override
(employees _ name :dept "Engineering" _)
```

The keyed pattern form is a big ergonomic win for wide relations. If your `employees` table has 15 columns, positional patterns require 15 arguments with `_` for every column you don't care about. Keyed patterns let you mention only the columns you need — which is exactly what SQL's `SELECT ... WHERE` does.

At compile time, with the schema in hand, `{:dept "Engineering" :name name}` desugars to a positional pattern with logic variables in the right positions and `_` everywhere else. Zero runtime overhead.

### The Graph-Query Power

The combination of schemas + keyed patterns + tabling + rule composition gives you something very powerful for your database use case:

```prologos
;; Schema per table (generated from DB metadata)
schema Employees  :id Nat :name String :dept String :salary Nat
schema Departments  :name String :budget Nat :head String

;; Facts loaded from DB
defr employees : Employees
  :source "db://company/employees"

defr departments : Departments
  :source "db://company/departments"

;; Composable graph queries — this is where Prologos shines
defr high-earners-in [?dept ?name ?salary]
  &> (employees {:dept dept :name name :salary salary})
     (> salary 90000N)

defr over-budget-depts [?dept ?total ?budget]
  &> (departments {:name dept :budget budget})
     (aggregate (employees {:dept dept :salary s}) [sum s] total)
     (> total budget)
```

Each `defr` is a reusable, composable, tabled view. You can build them up incrementally, test them individually, and compose them freely — something that requires CTEs and subqueries in SQL.

## Summary of the Design

1. **Storage**: flat vectors (or generated structs), positional, compact
2. **Schema**: the bijection between positions and named/typed keys
3. **Authoring**: dual syntax — positional `||` for tabular data, keyed `{...}` for clarity
4. **Goals**: keyed patterns desugar to positional at compile time
5. **Interop**: schema is the import/export contract (CSV, SQL, JSON all map)
6. **Schema-less**: falls back to positional-only Prolog-style (no keyed access)

The schema is the load-bearing element. It resolves the positional-vs-keyed tension by being the compile-time dictionary that lets you *write* keyed and *execute* positional. No runtime overhead for the keyed ergonomics. No loss of self-documentation. And the same schema serves as the database interop surface.
