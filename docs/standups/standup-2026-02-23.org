** Research And Design Mix-Fix Syntax `.{}`
*** Design parameters for `.{}`
#+ATTR_ORG: :tag prompt
#+begin_src quote
  I'm thinking of syntax to easily support and integrate mixed-fix syntax using `.{}`

  If there is a wildcard in the form, it should work just as our partail funtion syntax

  `map [* _ _] xs` is the same as `map .{_ * _} xs`

  The intent is to provide a light-weight alternative, in part to newcomers, but also--even as an experienced Lisper adapt at unfolding paren-trees--it still takes a second longer for the human to parse heavily nested arithmetic. I'm inspired by the `expr` sublanguage in TCL--another homoiconic language without heavy paren-groupings.

  I would like for us to do a comprehensive research on the design of mixfix precedent tables and design. I'm familiar with both Maude and Prolog's user-defined operator precedence, and would likewise like to enable user-definable infix precedence. I'm thinking to support this feature through a new key in `spec` ":mixfix" whose value is a Map {:assoc _ :}, with the convention of this map being in the compact EDN-like syntax. I know we want to support a key for associativity (:left | :right), and another key (perhaps with the name `:precedence`) for the number ordering? (I think)

  Again, the design of this is contingent on what we find during our extensive web-search research. Is there a clear pattern that other mix-fix languages take? Are there know gotchas? Or lessons learned? How can we take the best learnings on developing a sane precedence table that allows for users to--as intuitively as possible (perhaps we have excellent documentation and recommendations for them?)--create their own operator precedence within our custom Mixfix Syntax.

  I would like for our own library functions to use the same system that a user would in defining common mixfix operator precedences as an intended netx step towards implementation.

  But first! We're in a research and design phase, following the successful patterns of our Ergonomics Audit and design, and Extended spec design from yesterday (found in our `tracking` directory). Let us create a research and design document that compares and contrasts different language's approaches, highlights key learnings, offers critiques, and recommends options.

  Then I shall review and we can discuss how we want to go ahead with implementation.

  Your output from this should be "Mixfix Syntax Design.org" in our `tracking` directory.

-- Open questions
  1. *Should =.{...}= support statement-like forms?* E.g., =.{x = y + 1}= for assignment. Recommendation: No in Phase 1. Keep =.{...}= purely for expressions. Assignment remains =def x [add y 1]=.

2. *Should we support =.{...}= in =match= patterns?* E.g., =match n | .{_ + 1} => ...=. This requires extending patterns with infix sugar. Defer to Phase 4.

3. *What about =do= notation inside =.{...}=?* E.g., =.{x >>= fn [a] .{a + 1}}=. This is syntactically possible but may be too clever. Prefer =do= blocks for monadic code.

4. *Should =functor= =:compose= auto-register a mixfix symbol?* E.g., =functor Xf ... :compose xf-compose= could auto-generate =:mixfix {:symbol >> :group composition}=. Appealing but adds coupling.
