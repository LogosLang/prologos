## The MapEntry Question

This is the most consequential decision in the audit, and I think you're right to lean toward Clojure's approach. Let me explain why it's actually a better fit for Prologos than either of the other two options.

The fundamental problem is kind mismatch: traits require `{C : Type -> Type}` but Map is `Type -> Type -> Type`. There are three ways to resolve this:

**HKT partial application** (`Map K` as `Type -> Type`) is technically possible but deeply entangled with the type system. You'd need the trait resolver to recognize that `Map K` partially applied to `V` produces `Map K V`, and then thread the `K` constraint through everywhere. This leaks into the trait resolver's internals and creates weird asymmetries --- `Seqable (Map K)` sequences over values, losing keys. Not great.

**Separate KV traits** doubles your surface area for one type. Every generic function needs two versions. This is the "heavyweight" solution that violates the progressive disclosure principle.

**MapEntry** is elegant because it resolves the kind mismatch *at the data level*, not the type system level. A `MapEntry K V` is itself `Type -> Type` when you fix `K` --- but actually, you don't even need to think about it that way. The key insight from Clojure is: a map *is* a collection of entries. `(seq {:a 1 :b 2})` gives you `([:a 1] [:b 2])`. The entries are the elements.

For Prologos, I'd suggest `MapEntry` as a lightweight deftype:

```prologos
deftype MapEntry (K : Type) (V : Type)
  mk-entry : K -> V -> MapEntry K V
```

Then `Map K V` implements `Seqable` not as `{C : Type -> Type}` over `V`, but by having a standalone `map-seq : Map K V -> LSeq (MapEntry K V)` that produces entries. The trick: you DON'T need Map to satisfy the HKT trait directly. You just need the bridge to be ergonomic and the generic functions to work *through* it.

One thing I want to flag: `Pair` already exists and does exactly what `MapEntry` would do. The question is whether to reuse `Pair K V` (simpler, no new type) or introduce a distinct `MapEntry K V` (clearer intent, can have specialized printing like `[:a 1]` instead of `(pair :a 1)`). Clojure distinguishes MapEntry from Vector for good reason --- it lets `first` on a MapEntry mean "key" and `second` mean "value" with clear semantics, while a generic Pair is just positional.

My recommendation: new `MapEntry` type with `entry-key` and `entry-val` accessors, plus dot-access support (`e.key`, `e.val`). This gives you a clean entry abstraction that can eventually participate in pattern matching too.

## `reduce` with Identity from `spec` Metadata

This is a beautiful idea and aligns perfectly with the spec-as-documentation philosophy. The codebase already uses `reduce` as the name for left-fold in `prologos::data::list`, so the naming is aligned.

The spec metadata system already supports arbitrary keyword keys --- anything not in the recognized set gets stored as-is in the metadata hash. So `:identity` would work today at the storage level:

```prologos
spec add : {A : Type} where (Add A) A -> A -> A
  :identity 0
  :doc "Addition with additive identity"
```

But making `reduce` *pull* the identity at runtime is the interesting design challenge. There are two layers:

**Layer 1: Multi-arity `reduce`**
```prologos
;; 3-arg: explicit identity (expert)
reduce + 0 @[1 2 3]        ;; => 6

;; 2-arg: identity pulled from spec (ergonomic)
reduce + @[1 2 3]           ;; => 6 (reads :identity from +'s spec)
```

The 2-arg version needs the elaborator to look up the spec for the first argument, extract `:identity`, and insert it. This is a compile-time operation --- the spec metadata is available during elaboration. The mechanism would be similar to how `where` constraints auto-insert dicts: the elaborator sees `reduce` with 2 args, checks if arg-1 has a spec with `:identity`, and rewrites to the 3-arg form.

**Layer 2: What if there's no `:identity`?**
Two options: (a) type error at compile time ("reduce requires either an explicit identity or a function with :identity in its spec"), or (b) use the `reduce1` pattern (fold over non-empty, error on empty). I'd lean toward (a) for the 2-arg form --- it's a clear error message, and it nudges users to annotate their functions properly.

**The deeper question**: Should `:identity` be on the *function's* spec, or on a *trait method* spec? Consider: `Add` has method `add`, and `AdditiveIdentity` has method `additive-identity`. The identity *already exists* in the trait system. Rather than duplicating it into spec metadata, `reduce` could pull the identity from the trait:

```prologos
;; reduce on + looks up: + resolves to Add.add,
;; Add is bundled with AdditiveIdentity in Num,
;; so identity = additive-identity for this type
reduce + @[1 2 3]
```

This is more principled but harder to implement (requires the elaborator to trace from a function back to its trait, then to a sibling trait in a bundle). The spec `:identity` approach is simpler and more general --- it works for any function, not just trait methods. I'd suggest starting with `:identity` on specs, and potentially connecting it to the trait identity system later.

## Generic `into` with Target Collection Syntax

You already have `into-list`, `into-vec`, `into-set` functions and an `Into` trait (auto-derived from `From`). The gap is a *unified* `into` that takes a target collection indicator.

The Clojure pattern is `(into [] coll)` where `[]` is an empty vector literal serving as the "target type." In Prologos:

```prologos
into @[] @[1 2 3]        ;; => @[1 2 3] : PVec Nat  (no-op)
into '[] #{3 1 2}        ;; => '[1 2 3] : List Nat
into #{} '[1 2 1 3]      ;; => #{1 2 3} : Set Nat
into {} '[mk-entry :a 1, mk-entry :b 2]  ;; => {:a 1 :b 2} : Map Keyword Nat
```

This is actually tricky to type. The target `@[]` is a `PVec A` with `A` unconstrained, and the source `#{3 1 2}` is a `Set Nat`. The return type should be `PVec Nat`. The signature would need to bridge two collection types:

```prologos
spec into : {A : Type} {C1 C2 : Type -> Type}
            where (Seqable C1) (Buildable C2)
            [C2 A] -> [C1 A] -> [C2 A]
```

The first argument is the "target exemplar" (empty collection of the target type), and the second is the source. The implementation: `from-seq (to-seq source)` using `C1`'s Seqable and `C2`'s Buildable.

But there's an issue: the first argument `[C2 A]` is a *value* (an empty collection), not just a type indicator. You're allocating an empty PVec just to tell the system "I want a PVec." This feels wasteful.

Alternative: make `into` take a *constructor tag* rather than a value. This is where it connects to your `functor` keyword vision --- `PVec` as a first-class type constructor:

```prologos
into PVec #{3 1 2}      ;; PVec as a type tag, not a value
```

But this requires type constructors as runtime values, which is a bigger lift. For Phase 1, the Clojure pattern (empty collection as target) works fine. The empty collections are trivially cheap to construct, and the ergonomics are good.

## Efficient Generics: The Dispatch Story

You said "efficient implementations always," and this is where the rubber meets the road. The current generic path is: `to-seq` (materialize to LSeq) -> operate -> `from-seq` (rebuild). Three traversals.

There are two levels of efficiency to pursue:

**Level 1: Specialization at the trait level** (medium effort, big win).
Instead of all generic ops going through LSeq, make each collection's trait instance do the work directly:

```prologos
;; Current PVec Foldable instance:
;; fold f init pvec = fold f init (pvec-to-list pvec)  -- converts to List first!

;; Better PVec Foldable instance:
;; fold f init pvec = <native RRB-tree traversal>  -- single pass
```

This means the *trait instances themselves* are efficient, so generic dispatch via `where (Foldable C)` automatically gets the efficient version. No conversion. The generic `reduce` calls `fold` from the Foldable dict, and the dict's implementation is native. This is the big architectural win: push efficiency into the instances, not the generic wrappers.

For Foldable, this requires new AST primitives (`expr-pvec-fold`, `expr-set-fold`) that traverse the RRB-tree / CHAMP trie directly. For Functor/map, same pattern (`expr-pvec-map`). It's the 14-file pipeline touch per new primitive, but each primitive eliminates the conversion overhead for its type.

**Level 2: Elaborator specialization** (harder, bigger win).
When the elaborator can see that `map inc @[1 2 3]` has a concrete `PVec` argument, it bypasses the generic path entirely and emits the type-specific call. This is like how `+` on two `Int` literals compiles to `int+` directly instead of going through the `Add` trait.

This is the "type-specific dispatch for novices" you mentioned. From the user's perspective, they write `map inc @[1 2 3]` and get the fastest possible implementation. Under the hood, the elaborator sees PVec, resolves the Seqable+Buildable dicts, and the dict implementations are native. The user never touches a `pvec-map` --- but the performance is identical.

The good news: Level 1 is sufficient. If the trait instances are efficient, then generic dispatch *is* efficient dispatch. You don't need a separate specialization pass. The trait resolver already fills in the concrete implementation; if that implementation is native, you're done.

---

## Surface Syntax and Dependent Types

Your strong goal --- "never let dependent types leak into surface extractions" --- is well-served by the current architecture. The `spec` keyword already hides Pi types behind a surface syntax:

```prologos
;; User writes:
spec map : {A B : Type} where (Seqable C) (Buildable C) [A -> B] -> [C A] -> [C B]

;; Elaborator sees:
;; Pi (A :0 Type) (Pi (B :0 Type) (Pi (C :0 (Type -> Type))
;;   (Pi (_ :w (Seqable C)) (Pi (_ :w (Buildable C))
;;     (Pi (_ :w (-> A B)) (Pi (_ :w (C A)) (C B)))))))
```

The user never writes the Pi/Sigma chain. `{...}` is implicit binders, `where (...)` is dict binders, `[...] -> [...]` is explicit arrow.

The gap I see is for *return types* that involve dependent computation --- like `into` returning a different collection type. But for the collection use case, the types are straightforward enough that `spec`+`where` covers it.

The `functor` keyword is currently metadata-only (no elaboration, no typing). For the collections story, you don't actually need `functor` --- the `trait`/`impl`/`bundle` system handles the abstraction. `functor` would be useful for higher-kinded type *declarations* (declaring that `PVec` is a functor in the category-theoretic sense), but the practical dispatch goes through traits.

---

## Pulling It Together: What I'd Refine in the Audit

Based on this discussion, here's what I'd change:

1. **Decision 2 (Map)**: Strengthen the MapEntry recommendation. Introduce `MapEntry K V` as a proper deftype (not reusing Pair). Add `map-seq : Map K V -> LSeq (MapEntry K V)` and `map-from-seq : LSeq (MapEntry K V) -> Map K V` as bridge functions. This is cleaner than the current "standalone functions" proposal.

2. **Decision 1 (Generic ops)**: Path C is confirmed, but add the efficiency note: generic dispatch is only as fast as the trait instances. Phase 3's native implementations become Phase 2 prerequisites for the types where performance matters (PVec fold, at minimum).

3. **Add `reduce` as the primary name**: 2-arg with `:identity` lookup, 3-arg with explicit identity. Keep `foldr` for right-folds. Drop `fold` as a standalone name.

4. **Add generic `into`**: Clojure-style empty-collection-as-target for Phase 1. The signature uses `Seqable` + `Buildable`. Consider `into` as part of the `Collection` bundle's expected surface.

5. **Phase 2 should include at least one native trait instance** (PVec Foldable via direct RRB traversal) as a proof that the generic dispatch path can be efficient. Otherwise "efficient generics" remains aspirational.

6. **Type-specific dispatch is automatic**: Clarify in the audit that when trait instances are efficient, generic dispatch IS type-specific dispatch. The elaborator doesn't need a separate specialization pass --- the trait resolver already selects the concrete implementation.

Shall I update the audit document with these refinements, or do you want to discuss any of these points further first?
