## Design Discussion: Collection Generics — Eliminating the LSeq Intermediary

Extended back-and-forth design session analyzing the current generic collection
architecture and identifying three layers of improvement.

### Problem Statement

Generic collection operations (`map`, `filter`, `reduce`, etc.) currently route
through LSeq as a universal hub:

```
Collection A → to-seq → LSeq A → transform → LSeq B → from-seq → Collection B
```

For PVec/Set this means 5 intermediate allocations (`PVec → List → LSeq → LSeq →
List → PVec`). Meanwhile, native fold primitives that operate directly on each
collection's internal structure already exist (`pvec-fold`, `set-fold`,
`map-fold-entries`) — they just aren't wired into the generic dispatch path.

### Three-Layer Architecture

**Layer 1: Fold+Build** — Add `Reducible` trait (left-fold) + `Buildable.conj`.
- Reducing ops (`reduce`, `length`, `any?`, `all?`, `find`) fold natively via Reducible.
- Transforming ops (`map`, `filter`) stay on Seqable+Buildable for now.
- Key insight: `conj` has different semantics per collection (List: prepend, PVec: append, Set: insert), so order-preserving ops can't use reduce+conj without reversal.

**Layer 2: Trait-level specialization** — Clause-style constraint matching (like Haskell instance overlap).
- `impl Seqable PVec` already optimized via `pvec-to-list → list-to-lseq`
- With specialization: `impl Foldable PVec` could use native `pvec-fold` directly
- Removes remaining List intermediary for PVec folding

**Layer 3: Transducer fusion** — O(n) pipeline fusion via composable transducers.
- `|> coll (map-xf f) (filter-xf p) (into target)` = single-pass O(n)
- Already have transducer infrastructure; needs wiring into generic dispatch

### Key Design Decisions (Layer 1)

1. **Which ops switch to Reducible?** Only value-reducing ops, NOT collection-producing ops.
2. **List `conj` = `cons`** (O(1) prepend). Produces reversed results with left-fold.
3. **`reduce1` stays on Seqable** — needs head access for first element.
4. **Argument order**: Reducible uses `(B → A → B)` (acc-first), unlike Foldable's `(A → B → B)` (element-first).
5. **Type parameter ordering**: Reducible's `Pi [A :0 Type] Pi [B :0 Type]` where A=element, B=accumulator. This differs from List's `reduce` which has B=accumulator first.

### Three Open Issues Discussed

1. **Trait-level specialization mechanism** — how to match `(Foldable PVec)` more specifically than `(Foldable C)`. Deferred to Layer 2.
2. **Map HKT problem** — Map is `Type → Type → Type` (2-param), traits expect `Type → Type`. Can't directly implement `Seqable Map`. Need partial application (`Map K`) or adapter types.
3. **MapEntry sufficiency** — `Map K V` could implement `Seqable (Map K)` over `MapEntry K V` elements. But `MapEntry` as the only sequence element type is limiting for key/value-only iteration. Discussed `map-keys-seq`, `map-vals-seq` alternatives.

## Considerations

- Sexp-mode `impl` with inline `defn` has pre-existing parsing limitation
  (paren-shape loss). Trait/impl forward-reference tests use WS mode in stdlib.
- Mutual data types (A references B's ctor type, B references A's ctor type)
  need intra-bucket topological sort — not addressed yet.
- `test-reduction-perf-02-01.rkt` at 33.6s is slightly over 30s threshold
  but only has 2 tests, can't split further.
- **List `conj = cons` reversal**: Layer 2 specialization needed before `map`/`filter` can use Reducible. Current Seqable+Buildable path preserves order.
- **Map HKT**: 2-type-param constructors can't implement 1-type-param traits directly. Needs partial application or adapter approach (Layer 2+).
- **lseq-fold type arg swap**: A=accumulator, B=element in `lseq-fold` (reversed from Reducible convention). Instance wraps with `lseq-fold B A` to compensate.
