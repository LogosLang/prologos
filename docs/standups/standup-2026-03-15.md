## The Key Discovery: Meta-Info Reads Are Already Cell-Based

Looking at the actual production code paths:

```racket
;; meta-solved? — PRODUCTION path (when network exists)
(define cid (prop-meta-id->cell-id id))    ;; CHAMP lookup #1: id-map
(let ([v (read-fn (unbox net-box) cid)])   ;; CHAMP lookup #2: cells
  (and (not (prop-type-bot? v)) (not (prop-type-top? v))))
```

`meta-solved?` and `meta-solution` **already read from per-meta cells in the propagator network**, not from the meta-info CHAMP. Each meta gets its own cell at creation time (in `fresh-meta!` → `elab-fresh-meta`), initialized to `type-bot`. The cell value IS the lattice element.

The meta-info CHAMP stores a `meta-info` struct with fields: `id, ctx, type, status, solution, constraints, source`. But in production:
- **status/solution** → derivable from cell value (bot=unsolved, non-bot=solved, the value IS the solution)
- **ctx, type, source** → immutable, set at creation, never modified
- **constraints** → already live in constraint cells (Track 1)

The meta-info CHAMP is **largely redundant with the per-meta cells**. The mutable parts are shadows of what's already in the network. The immutable parts don't need speculation rollback.

## What This Means for the Design

The original design proposed "put the meta-info CHAMP in a network cell" (a CHAMP-inside-a-cell). This is the wrong move. It adds indirection on a structure that's either redundant or immutable. Instead:

**The per-meta cells are already the right abstraction.** Each meta is already a cell in the type lattice. The question is: can we make these existing cells speculation-aware?

## Lattice Embedding Direction

Your PVec intuition points at something important. A `PVec @[A B C D | Rest]` decomposes into a product lattice of individual cells. Similarly, the metavar store is already a product lattice — one cell per meta, each in the type lattice (⊥ = unsolved, concrete type = solved, ⊤ = contradiction).

Speculation is a **lattice operation on this product**, not a temporal save/restore:

- **Entering speculation** = creating a sub-lattice (a "pocket universe" — the research docs already use this term with Galois connections)
- **Computing** = monotone operations within the pocket
- **Succeeding** = projecting results back via γ (concretization)  
- **Failing** = discarding the pocket (it never touched the global lattice)

The ATMS's TMS cells implement exactly this: each cell can hold multiple assumption-tagged values. Reading under a worldview filters to the relevant sub-lattice. Retraction projects the sub-lattice away.

## The Architectural Insight

Rather than:
```markdown
Option A: Put meta-info CHAMP in a network cell (adds indirection, wrong abstraction)
```

The direction should be:
```markdown
Make existing per-meta cells TMS-aware (adds assumption-tagging to what's already there)
```

Per-meta cells already exist. They already have type lattice values. They're already read by `meta-solved?`/`meta-solution`. Making them TMS cells means:

1. `fresh-meta!` creates a TMS cell (value tagged with current assumption)
2. `solve-meta!` writes to TMS cell under current assumption
3. `meta-solved?` reads TMS cell filtered by current worldview
4. On failure: retract assumption → reads see pre-speculation values
5. No save/restore of the meta-info CHAMP at all (immutable metadata stays; mutable state is in TMS cells)

The read path cost: currently `net-cell-read` does 1 CHAMP lookup → returns value. With TMS, `atms-read-cell` does 1 hash-ref → iterates supported-values (newest first) → returns first value whose support ⊆ believed. For the common case (1 speculation depth), this is still effectively O(1).

## But There's a Structural Question

Right now, TMS cells live in the ATMS struct (`atms-tms-cells`), which is OUTSIDE the propagator network (`prop-network-cells`). These are two separate cell stores:

- **prop-network cells**: monotonic, standard merge, captured by network box snapshot
- **ATMS TMS cells**: assumption-tagged, worldview-filtered, NOT captured by network snapshot

If per-meta cells move from prop-network to ATMS TMS cells, they exit the network snapshot. This means the network box save/restore no longer captures meta state — which is the goal (no save/restore for metas), but it means we need TMS retraction to actually work correctly for all meta operations.

The alternative: **integrate TMS into the prop-network itself**. A cell in the network can be either monotonic (standard merge) or TMS (assumption-tagged). Network snapshot captures both. This keeps "one network, one snapshot" while adding TMS capability.

**1. TMS cell representation — designing for rich queries.**

The question is: what data structure lets us efficiently answer both the hot-path query ("what value does this cell have under the current worldview?") and the tooling/provenance queries ("what assumptions support this value?", "what values exist under alternative worldviews?", "show me the derivation tree of how this cell evolved")?

A flat list of `supported-entry` is the minimal structure. But you're right to ask about trees. Let me think about what the actual query patterns are:

- **Hot path (type checker)**: "give me the value visible under the current believed set" — this is a filter-and-return-first
- **Error reporting (current)**: "what assumptions led to this nogood?" — this queries the ATMS control plane, not cell values directly
- **GDE (Track 9)**: "for this unsolved meta, what values were attempted under which assumptions, and why did each fail?" — this needs the *full history* of a cell: every value ever written, its support set, and whether that support set is nogood
- **Tooling/IDE**: "show me the speculation tree for this expression" — this is a tree query: root = unconditional value, branches = speculative writes, each branch has sub-branches for nested speculation
- **Formal modeling**: "what is the proof certificate that this value holds?" — this needs the support set traced back to axioms (user annotations, language rules)

The tree structure emerges naturally from nested speculation. When speculation nests:

```markdown
depth 0: cell has value ⊥ (unconditional)
  depth 1 (H1): cell solved to Int
    depth 2 (H2): cell solved to Nat (widening attempt)
    depth 2 (H3): cell solved to Int (stays)
  depth 1 (H4): cell solved to String
```

That's a tree. Each node is a `(value, assumption-set, children)` triple. The hot-path query is "find the deepest leaf whose full assumption path ⊆ believed." The provenance query is "give me the path from root to the current value."

**Concrete data structure proposal:**

```racket
(struct tms-entry
  (value        ;; the lattice element
   assumption   ;; assumption-id that introduced this entry (or #f for unconditional)
   parent       ;; tms-entry | #f (the entry this branched from)
   timestamp)   ;; monotonic counter for ordering
  #:transparent)
```

A TMS cell's value is `(listof tms-entry)` — but the `parent` links form a tree. The list is the flattened frontier (all "leaf" entries). The hot path scans the frontier. Tooling traverses parent links for provenance.

**Why not a nested tree struct?** Because the hot path needs to scan leaves without tree traversal. A flat frontier with parent backpointers gives O(k) scan for hot path (k = frontier size, typically 1-3) and O(d) traversal for provenance (d = speculation depth, typically 1-2). A nested tree would require O(d) traversal even for the hot path.

**Alternative**: An indexed structure — `hasheq assumption-id → tms-entry` — gives O(1) lookup by assumption. **3. Depth-0 fast path design.**

My concern was about entry ordering in the list. But thinking more carefully: the clean design is not "guarantee ordering" but rather **store the unconditional value separately**:

```racket
(struct tms-cell-value
  (base       ;; the unconditional value (depth-0 value, always visible)
   entries)   ;; (listof tms-entry) — speculative entries (may be empty)
  #:transparent)
```

At depth 0, `entries` is empty and we return `base` directly — zero scanning cost, no type-tag check beyond `null?` on the entries list. During speculation, entries accumulate. The hot path becomes:

```racket
(if (null? entries)
    base  ;; fast path: no speculation active
    (tms-scan entries believed base))  ;; slow path: filter
```

This is clean: the common case (no speculation) pays effectively zero overhead. The speculation case pays the scan. And `base` is always the "ground truth" that speculation branches from — good for provenance too (it's the root of the tree).
