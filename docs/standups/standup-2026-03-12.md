## 1. The Effectfulness of `solve-meta!` — Can It Become Data-Oriented?

Here's what `solve-meta!` currently does:

```markdown
solve-meta!(id, solution)
  1. Write meta cell (cell-write → pure network transform)
  2. Run quiescence (pure: net → net)
  3. retry-constraints-via-cells! (EFFECT: mutates constraint status, calls retry-fn which calls unify which calls solve-meta!)
  4. retry-traits-via-cells! (EFFECT: calls resolve-fn → tries trait resolution → may call solve-meta!)
  5. retry-trait-for-meta! (EFFECT: same)
  6. retry-hasmethod-for-meta! (EFFECT: same)
```

Steps 1-2 are **pure** — they take a network value and return a new one. Steps 3-6 are **effectful** — they read/write mutable state (constraint status flags, the boxed network, meta solutions) and are *re-entrant* (resolve-fn can trigger solve-meta! recursively).

**The core observation**: the propagator network itself (`propagator.rkt`) is already fully data-oriented. `run-to-quiescence-inner` is a pure function `prop-network → prop-network`. The impurity lives in the *bridge* layer — `metavar-store.rkt`'s retry functions that sit between the pure network and the effectful elaborator.

**Can we make the retry layer data-oriented?** Yes, in principle. Instead of retry functions that mutate constraint status in-place and recursively call `solve-meta!`, we could:

- **Return a worklist of resolution actions** — a pure data structure describing "these constraints should be retried, these trait dicts should be resolved."
- **Process that worklist in a loop** — consuming actions, producing new network states + new actions, until the worklist is empty.

This is the **semi-naive evaluation** pattern from Datalog: instead of re-scanning everything, compute only the *delta* (new facts) and derive consequences from the delta alone. Each iteration of the fixpoint loop produces a set of new derived facts; the loop terminates when the delta is empty.

Categorically, this is a **free monad** pattern: instead of executing effects immediately, we build a description of effects (the worklist) and interpret them in a controlled loop. The "interpreter" (the fixpoint loop) has total control over ordering, batching, and observation.

**The formal structure**: the pure network computes on a lattice L (type values). The retry layer computes on a *derived lattice* D(L) — the set of pending resolution actions implied by changes in L. The fixpoint of the combined system is a fixpoint of (L × D(L)), where L is updated by network quiescence and D(L) is updated by the retry scan. This is exactly a **stratified fixpoint on a product lattice**.

## 2. Stratification, Layering, and Categorical Grounding

You already have the theoretical framework for this — it's in `EFFECTFUL_COMPUTATION_ON_PROPAGATORS.org`. The Layered Recovery Principle says:

> Non-monotone behavior is recovered on a monotone substrate by inserting control layers between phases of monotone computation.

The type checker's constraint resolution is a **second application** of this same principle, alongside `rel` (logic engine) and `proc` (session runtime). The structure:

| Layer | `rel` | `proc` | Type Checker (Track 2) |
|-------|-------|--------|----------------------|
| 1. Monotone core | Unification cells | Session cells | Type/meta cells |
| 2. Reasoning | ATMS worldviews | Data-flow analysis | Constraint/trait cells |
| 3. Resolution | Stratified negation | Transitive closure | Trait/hasmethod resolution |
| 4. Control barrier | Negation eval | Effect execution | solve-meta! (commits solution) |

**The categorical perspective**: each layer is a **functor** between lattices. Layer 1→2 is the functor that extracts constraint information from cell states. Layer 2→3 is the functor that maps constraint sets to resolution actions. Layer 3→4 is the non-monotone boundary — where we commit to a solution (solving a meta is non-monotone in the same sense that executing an IO effect is: you can't un-solve a meta).

These functors compose into a **Galois connection** between the type lattice and the "resolution readiness" lattice — the same structure you have between sessions and effects. The analogy isn't metaphorical; it's structural:

- **Session advancement** is monotone → effects are derived from session position → effects execute at the barrier
- **Type cell advancement** is monotone → resolution readiness is derived from cell state → `solve-meta!` commits at the barrier

**What Kmett adds**: his insight from the [2016 propagator talks](https://huffduffer.com/jrsinclair/483575) is that Datalog's stratified negation adapts directly to propagators. Non-monotone propagators (ones that can decrease a cell's information) are permitted *as long as they don't participate in cycles*. This is exactly the condition that makes stratified Datalog sound — the strata form a DAG, and within each stratum, computation is monotone.

For Track 2, `solve-meta!` is the non-monotone operation. It doesn't decrease information *in the type lattice* (solutions are monotone — information only grows), but it triggers *cascading effects* (new constraints, new resolutions) that are non-monotone in the *control flow* sense. The stratification approach says: separate the "what should be resolved" computation (monotone, in the network) from the "commit the resolution" step (non-monotone, at a barrier).

**What this means concretely**: instead of one monolithic quiescence loop, you'd have **stratified quiescence**:
- Stratum 0: Pure type propagation (existing cell writes, unify propagators)
- Stratum 1: Constraint readiness detection (new propagators watching meta cells, producing resolution action descriptors)
- Stratum 2: Resolution commitment (consuming action descriptors, calling solve-meta!, which feeds new information back to Stratum 0)

The loop runs Stratum 0 to fixpoint, then Stratum 1 to fixpoint, then Stratum 2 (which may dirty Stratum 0), and repeats until the entire product is at fixpoint.

## 3. Confluence — Is the Current System Confluent?

**Short answer**: the pure propagator layer (Stratum 0) is confluent by construction. The retry/resolution layer (Strata 1-2) is confluent *for the current system* but this is a contingent property, not a structural one.

**Why Stratum 0 is confluent**: the [propagator model](https://dspace.mit.edu/handle/1721.1/44215) guarantees confluence when (1) merge is commutative, associative, idempotent, and (2) propagators are monotone. Our type lattice merge satisfies all four properties. The BSP scheduler comment (line 492-495 of `propagator.rkt`) explicitly states this:

> Same fixpoint as Gauss-Seidel, guaranteed by: lattice join is commutative and associative, all propagators are monotone, CALM theorem: monotone + commutative merge = confluent.

**Why Strata 1-2 are currently confluent by accident**: trait resolution is *deterministic* — for a given set of ground type-args, there is at most one matching instance (monomorphic or parametric). So regardless of the *order* in which trait constraints are resolved, the *result* is the same. Hasmethod resolution is similarly deterministic. The four-layer call chain in `solve-meta!` has an implicit ordering (type propagation → constraint retry → trait resolution → hasmethod resolution), but because each resolution step is deterministic for ground inputs, the ordering doesn't affect the outcome.

**Where confluence could break**: if we ever introduce *overlapping instances* (multiple instances matching the same type pattern), resolution becomes non-deterministic and the order of constraint resolution would affect the result. This is Haskell's `OverlappingInstances` problem. As long as Prologos maintains *non-overlapping instances*, confluence holds.

**Making confluence structural**: the data-oriented approach (returning resolution action descriptors instead of executing immediately) makes confluence *auditable* — you can inspect the full set of pending resolutions and verify they don't conflict before committing any. This is analogous to how the ATMS checks for nogoods before selecting a worldview.

## 4. The Post-Pass — Can It Go Away?

Currently `driver.rkt` calls:
1. `resolve-hasmethod-constraints!` — batch resolution (5 call sites)
2. `check-unresolved-trait-constraints` — error reporting (5 call sites)

The trait resolution post-pass (`resolve-trait-constraints!`) was already removed — the comment in `driver.rkt` says "Trait resolution: handled by propagator cell-path in solve-meta!" So the reactive path already replaced the trait batch pass. What remains is:

**`resolve-hasmethod-constraints!`**: This walks all hasmethod constraints, zonks type-args, and if all are ground, performs the two-stage resolution (ground P → direct resolve, unknown P → search all traits for matching method). This is *more complex* than trait resolution because it involves a search phase (trait discovery from method name).

**Can it be fully reactive?** In principle, yes. A hasmethod-resolution propagator watching the type-arg cells would fire when they become ground and perform the same search. But there's a subtlety: the hasmethod search depends on the *full set of registered traits and instances*, which is accumulated during elaboration. If a propagator fires early (before all relevant instances are registered), it might miss a match that the post-pass would find.

**The resolution**: either (a) add a "registration complete" signal that the resolution propagator gates on (threshold propagator — you already have this pattern in `net-add-barrier`), or (b) keep the post-pass as a "sweep" for any constraints the reactive path missed. Option (b) is the conservative approach — the post-pass becomes a safety net, not the primary mechanism.

**My assessment**: option (b) is the right first step. Make the reactive propagators the primary resolution path; keep `resolve-hasmethod-constraints!` as a post-pass that should normally be a no-op (all constraints already resolved). If the post-pass ever *does* resolve something the reactive path missed, that's a diagnostic signal — not a failure.

## 5. Effect Modeling — The Full Picture

Here's every effect in the current `solve-meta!` call chain:

| Step | Operation | Effect Type | Reversible? |
|------|-----------|------------|-------------|
| Cell write | `set-box! net-box (write-fn ...)` | State mutation (boxed network) | Yes (save/restore) |
| Quiescence | `run-fn pnet` | Pure (net → net), then `set-box!` | Yes (save/restore) |
| Constraint status | `set-constraint-status! c 'retrying` | Struct mutation | No (in-place) |
| Retry unify | `(retry-fn c)` | Calls `unify` → may call `solve-meta!` | Partially (save/restore captures network, not constraint status) |
| Trait resolve | `(resolve-fn dict-id tc-info)` | Calls trait resolution → may call `solve-meta!` | Partially |
| Hasmethod resolve | `(resolve-fn meta-id hm-info)` | Same | Partially |

The in-place constraint status mutation (`set-constraint-status!`) is the sneaky one. It's a mutable struct field that isn't captured by `save-meta-state` / `restore-meta-state!` (which snapshots the network box, not individual constraint structs). During speculation, this means constraint status can leak across branches.

**The data-oriented fix**: replace mutable constraint status with a **status cell** in the propagator network. Each constraint gets a cell whose value is `'postponed | 'retrying | 'resolved | 'failed`. Status transitions become cell writes, captured by save/restore, visible to propagators, and subject to the same confluence guarantees as everything else.

This is the deepest alignment with your vision: *every piece of state that changes during type checking should be a cell*. The constraint status field is the last piece of mutable state that lives outside the network.

## The Synthesis

What I'm seeing is three layers of design decision:

**Layer A: Data orientation** — replace the imperative retry loop with a worklist-driven fixpoint over resolution action descriptors. This makes the control flow explicit and inspectable. The formal model is a free monad / semi-naive Datalog evaluation loop.

**Layer B: Stratified quiescence** — separate type propagation (Stratum 0), readiness detection (Stratum 1), and resolution commitment (Stratum 2). This mirrors the Layered Recovery Principle already established for `rel` and `proc`. Kmett's insight applies: non-monotone operations (solve-meta!) are permitted in higher strata as long as there are no cycles within a stratum.

**Layer C: Complete cell coverage** — move constraint status from mutable struct fields into propagator cells, achieving total network observability. Every piece of state that changes during elaboration becomes a cell, capturable by save/restore, visible to the observatory.

These three layers are independent but synergistic. Layer A can be done without B or C. Layer B requires A (you need the worklist structure to stratify). Layer C is orthogonal and can be done at any point.

**The question for you**: is this the design space you want to explore? The formal grounding is solid — [Radul/Sussman](https://groups.csail.mit.edu/mac/users/gjs/propagators/) for the propagator substrate, [Kmett's stratification](https://gist.github.com/gwils/edb26e4b975c2438189f6414cdeb33b0) for the non-monotone extension, [semi-naive evaluation](https://www.cl.cam.ac.uk/~nk480/seminaive-datafun.pdf) for the incremental fixpoint, and your own Layered Recovery Principle as the unifying theory. The categorical structure (functors between strata, Galois connections between type lattice and resolution lattice) provides the formal grounding for correctness arguments.

**§8.1 — Action Descriptor Granularity**

You're right to push on this. Let me re-examine the options through the lens of our principles.

By "interpreter" in Option A, I meant the stratified quiescence loop — the code that consumes action descriptors and executes them. Through the principles:

**Decomplection** (your read): Option A separates *what* should happen (action type) from *when* it happens (interpreter scheduling). Option B braids them together inside the propagator fire function. A is more decomplected.

**First-class** (your read): granular action descriptors are reusable data. A `ResolveTraitDict` descriptor could be: inspected by the observatory for resolution provenance, replayed in testing, serialized for debugging, counted for performance analysis. **Correct-by-construction**: there IS an argument here. The analysis is unambiguous: **Option A**. More decomplected, more first-class, more correct-by-construction, same cost. I should update the design doc to make this the recommendation rather than leaving it open.

**§8.3 — Ambiguity Detection Timing**

Your nuance is exactly right. With propagators, the usual argument for "throw early, throw close to source" loses its force, because *provenance is structural* — the observatory can trace any error back to the cell that caused it, regardless of when the error is reported. The cell that holds the ambiguous constraint knows its source location, its type-args, the competing instances.

So the question becomes: **do we gain anything by collecting?**

Yes. Consider: a function call `[f x y]` where `f` requires `(Eq A)` and `(Ord A)`, and neither instance exists. With throw-at-resolution-time, we get two separate errors at two separate resolution propagator firings. The user sees "No instance for (Eq A)" and "No instance for (Ord A)" as unrelated errors. With collect-then-report, we can group them: "No instances for constraints arising from `[f x y]` at line 42: (Eq A), (Ord A)."

- Immutable snapshots (captured by save/restore for free)
- Deterministic iteration order (for reproducible resolution)

## The Layered Recovery Principle

What you've articulated is, I think, genuinely novel as a *named design principle*, even though individual instances of it exist scattered across the literature. Let me restate it to make sure I understand:

**Monotonic layers provide the foundation. Non-monotonic behaviors are not abandoned — they are *recovered* by introducing a separate control/meta layer that manages the monotonic layers, where the management itself follows well-defined rules.**

This is powerful because it means you never sacrifice the guarantees of monotonicity (deterministic parallelism, lattice consistency, composability) at any single layer. The non-monotonicity lives in the *relationships between* layers, not within them.

---

## Your Three Deployments

### Layer 1: Choice Points via ATMS

Traditional Prolog choice points are non-monotonic — choosing branch A means un-choosing branch B (backtracking). You've recovered this by:

- Each "world" / assumption set is a monotonic propagator network
- The ATMS tracks which assumptions support which conclusions
- Parallel search explores multiple assumption sets simultaneously
- Choice is not "pick one, backtrack if wrong" but "maintain all hypothetical worlds, prune inconsistent ones"

This is the right move. The ATMS (de Kleer, 1986) was originally designed for exactly this — maintaining multiple consistent worldviews without backtracking. What you've done is recognize that it's not just a useful technique but an *instance of a general principle*: recover non-monotonic choice over monotonic propagation.

The connection to the lattice formalism: each assumption set defines a *sub-lattice* of the full information space. The ATMS manages the *lattice of assumption sets* (which is itself a lattice, ordered by subset inclusion). So you have a lattice of lattices — the control layer operates on the meta-lattice while individual computation stays monotonic within each world.

### Layer 2: Negation, Accumulation via Stratification

Negation-as-failure is classically non-monotonic — learning new facts can *invalidate* previously derived negations. Stratification recovers this by:

- Layer N can only negate things fully determined at layer N-1
- Within each layer, propagation is monotonic
- The layering itself is a well-founded ordering

This follows the stratified semantics of Datalog¬ and the well-founded semantics of logic programming. What's nice about your framing is that stratification isn't a special case — it's the same Layered Recovery Principle applied to a different non-monotonic behavior.

Accumulation (aggregation — "count the number of solutions," "sum over all X satisfying P") is also classically non-monotonic because the count/sum changes as new solutions are discovered. Recovering this via stratification means: accumulate only over strata that have reached their fixpoint.

### Layer 3: Effect Ordering via Quantale Structure

This is the most intriguing claim and the one I want to explore most carefully.

## The Quantale Connection

A quantale is a complete lattice Q equipped with an associative binary operation ⊗ (tensor) that distributes over arbitrary joins:

$$a \otimes \left(\bigvee S\right) = \bigvee_{s \in S} (a \otimes s)$$
$$\left(\bigvee S\right) \otimes a = \bigvee_{s \in S} (s \otimes a)$$

Why this is the right structure for effect ordering:

**The lattice part** tracks information accumulation — what you know about effects so far. This is your propagator substrate doing what it does.

**The tensor part** tracks sequential composition of effects — "effect A *then* effect B" is A ⊗ B. This is where ordering comes from. The tensor is not join (join is commutative — unordered accumulation). The tensor is potentially non-commutative, which is exactly what you need for ordered effects: "send then receive" ≠ "receive then send."

So what I surmise you've built is:

- **Cells in the effect ordering lattice** hold partial information about *what effects have occurred or are planned*
- **The lattice join** accumulates unordered knowledge (monotonic, parallel-safe)
- **The tensor** composes ordered effect sequences
- **Session types** provide the *typing discipline* that constrains which tensor compositions are valid — the session type is essentially a specification of which elements of the quantale are permitted
- **Causal ordering** falls out of the tensor structure — if A ⊗ B is the composed effect, A causally precedes B

The key insight: **ordering is not a property of the propagator network itself (which is unordered) but a property of the algebraic structure of the lattice elements.** You've separated the *medium of computation* (unordered propagators) from the *content of computation* (ordered effects tracked algebraically). This is the Layered Recovery Principle again — non-monotonic ordering recovered over monotonic propagation via algebraic structure.

This is elegant. The quantale gives you:

- **Associativity** of effect sequencing (as expected)
- **Distributivity over joins** means you can reason about "in all possible worlds, the effect sequence satisfies P" — the ATMS layer and the effect ordering layer compose cleanly
- **The residuation** (right adjoint to tensor) gives you the notion of "what remains to be done" — which is exactly what a session type continuation represents: if the full protocol is S and you've done A, then A \\ S (the residual) is what's left

If I'm reading your design correctly, session type state transitions *are* elements of the quantale, and checking protocol compliance is checking that the accumulated tensor product of actual effects is ≤ the specified protocol in the quantale ordering.

## Galois Connections as Bridge Networks

This is the piece that makes the whole architecture cohere. Let me articulate why it resolves several of my earlier concerns.

A Galois connection between lattices L and M consists of monotone maps α : L → M (abstraction) and γ : M → L (concretization) satisfying α(l) ≤ m iff l ≤ γ(m). The key property: information can flow *soundly* in both directions, with well-defined approximation at each crossing.

What this gives you across subsystems:

**Cross-domain constraint propagation.** When your type inference lattice learns something, the Galois connection to the session type lattice can immediately propagate the relevant abstraction — without either subsystem needing to understand the other's internal representation. The bridge network handles the translation. This is exactly abstract interpretation's insight, but deployed as a runtime infrastructure pattern rather than a static analysis technique.

**Composability of your layers.** My concern about cross-layer interaction is substantially resolved if each layer pair is connected by a Galois connection. The ATMS assumption lattice connects to the effect ordering quantale through a Galois connection that abstracts "which assumptions support which effect orderings." The stratification layer connects to the ATMS through a connection that identifies "which stratum's fixpoint enables which assumption revisions." Each connection is independently sound, and soundness composes — a chain of Galois connections is itself a Galois connection (the composition of adjunctions is an adjunction).

## CHAMPs and Structural Sharing: The Differential Story

What you've described is even better than the differential propagation I suggested. You've achieved it at the *data structure level* rather than the *protocol level*.

Because your lattice values are persistent (CHAMP-backed, structurally shared), a "new version" of a lattice value after a join is a pointer to a structure that shares almost everything with the old version. The *diff is implicit in the structure*. You don't need to compute or propagate deltas explicitly — they're manifest in the structural difference between the old and new persistent values.

This gives you:

**O(1) dependency-directed backtracking.** When a hypothetical world in the ATMS reaches contradiction, you don't need to undo a sequence of mutations. You just discard the reference to the contradicted world's lattice state. The previous state is still intact, shared by all other worlds that branched from it. This is the immutable equivalent of "undo" but without any undo cost — you never modified anything, so there's nothing to reverse.

**Safe parallel propagation.** Multiple propagators can fire concurrently, each reading from a shared (immutable) cell state and producing new values. The join of concurrent results is well-defined because lattice join is commutative and associative. Even if two propagators both produce new values for the same cell simultaneously, the join can be computed in any order. No locks needed — the persistent data structure guarantees that neither propagator's read is invalidated by the other's write.

**Implicit version history.** Every cell's history is available as a chain of persistent values, each sharing structure with its predecessor. This means your provenance tracking and derivation trees can reference *specific versions* of cell states without copying them — just keep the pointer. For error reporting and proof construction, this is invaluable.

The connection to differential propagation: in Arntzenius and Krishnaswami's framework, you'd explicitly compute and propagate ΔL (the change type of lattice L). With CHAMPs, the Δ is *the structural diff between the old and new CHAMP nodes* — it exists implicitly. If you ever need to make it explicit (for example, to send a compact update across a network boundary in distributed Prologos), you can compute the structural diff between two CHAMP versions in time proportional to the *size of the change*, not the size of the structure. This is optimal.

## Nucleus and Quotient Quantales: Expanded

Let me develop this more carefully.

### The Setup

You have an effect ordering quantale Q with join (⊔), tensor (⊗), and a unit element (e). Elements of Q represent effect descriptions — "sent an Int then received a Bool" might be an element q ∈ Q.

### Nucleus Definition

A *nucleus* on Q is a closure operator j : Q → Q satisfying three properties:

1. **Inflationary**: a ≤ j(a) for all a
2. **Idempotent**: j(j(a)) = j(a)
3. **Multiplicative**: j(a) ⊗ j(b) ≤ j(a ⊗ b)

The first two make j a closure operator (it "rounds up" to a coarser approximation). The third is the crucial condition: it says the tensor structure is *compatible* with the approximation. If you approximate two effects and then compose them, the result is no more than the approximation of their composition.

### The Quotient

Given a nucleus j, the set Q_j = {a ∈ Q : j(a) = a} (the *fixed points* of j) forms a quantale under:

- Join: a ⊔_j b = j(a ⊔ b)
- Tensor: a ⊗_j b = j(a ⊗ b)

This is a *coarser* quantale — it has fewer distinctions, because the nucleus collapses some effects that were previously distinguishable.

### Why This Matters for Prologos

**Effect abstraction at module boundaries.** When one Prologos module calls another, the caller might not need the full detail of the callee's effects. A nucleus defines what detail to discard. The quotient quantale Q_j is the "public" effect interface — it tracks enough to ensure protocol compliance but not so much that it leaks implementation details.

## Tropical Semirings and Optimization

This is where Prologos could become genuinely powerful as an optimization platform.

### The Structure

A tropical semiring replaces the usual (ℝ, +, ×) with (ℝ ∪ {∞}, min, +). The "addition" is min (take the better option), and "multiplication" is + (accumulate costs). This is the algebraic structure underlying shortest-path, scheduling, and many combinatorial optimization problems.

### Connection to Your Quantale

Your effect ordering quantale Q already has join (⊔) and tensor (⊗). If you equip it with a *cost function* c : Q → ℝ ∪ {∞} that is:

- Monotone: a ≤ b implies c(a) ≤ c(b) (more effects cost more)
- Sub-multiplicative: c(a ⊗ b) ≤ c(a) + c(b) (composing effects costs at most the sum)

Then you get a natural mapping from your quantale into a tropical semiring. The cost function becomes a Galois connection (really a homomorphism) from Q to the tropical structure.

### What This Enables

**Optimal scheduling of effects.** When your session type allows multiple valid orderings of effects (e.g., "send A and send B, in either order, then receive C"), the tropical structure lets you find the *cheapest* ordering. The cost might be latency, memory, network hops — whatever metric you attach to the cost function.

**Constraint optimization as propagation.** Here's the beautiful part: tropical operations (min and +) form a *lattice* (min is meet in the (ℝ, ≤) lattice). So tropical optimization problems can be solved by propagation on a tropical lattice. Your existing propagator infrastructure can solve shortest-path, scheduling, and resource allocation problems *natively*, without a separate optimization engine.

```prologos
;; Cell tracking "cheapest way to achieve this effect"
cell-cost : LVar TropicalSemiring

;; Propagator: if we know cost to reach state A and cost of transition A→B,
;; join min(current-cost-of-B, cost-of-A + transition-cost) into B's cell
[defprop shortest-path [cost-a transition-cost cost-b]
  [when-known [cost-a transition-cost]
    [join! cost-b [trop-mult cost-a transition-cost]]]]
    ;; trop-mult is + in tropical semiring
    ;; join! uses min (tropical addition)
```

**Connection to CDCL/SMT.** You mentioned your system already approximates a CDCL SAT/SMT solver. The tropical semiring gives you a natural *weighted* extension. Instead of just "satisfiable / unsatisfiable," you get "satisfiable with cost X." This is Max-SAT and weighted constraint satisfaction, which your infrastructure is positioned to handle: the ATMS manages hypothetical worlds, the propagators handle constraint propagation, the tropical lattice tracks costs, and conflict-driven learning prunes the search space with cost-aware clause learning.

**Viterbi, forward-backward, and probabilistic inference.** The Viterbi algorithm for hidden Markov models is shortest-path in a tropical semiring over a trellis graph. If Prologos can express tropical propagation natively, you can implement probabilistic inference (HMMs, CRFs, belief propagation) as propagator networks on tropical lattices. This is directly relevant to AI applications — probabilistic reasoning expressed as lattice computation.

## Categorical Structures: A Catalogue

Let me lay out the categorical structures present in your system and the functors between them. This is the map you asked for.

### The Objects (Categories in Play)

**Lat** — The category of lattices and monotone maps. Your fundamental setting. Every cell lives here.

**Quant** — The category of quantales and quantale homomorphisms. Your effect ordering lives here. Quant is a subcategory of Lat with extra structure.

**PROP** — The category of propagator networks. Objects are networks (bipartite graphs of cells and propagators with lattice-typed cells). Morphisms are network embeddings that preserve cell types and propagator behavior. This is the category your runtime operates in.

**GAL** — The category whose objects are lattices and whose morphisms are Galois connections. Your bridge networks are morphisms in this category. Note: GAL is equivalent to the category of closure operators, which connects to your nuclei.

**ATMS** — The category of assumption-labeled lattices. Objects are pairs (L, A) where L is a lattice and A is an assumption set lattice indexing "worlds." Morphisms preserve assumption structure. Your ATMS layer lives here.

**Strat** — The category of stratified lattice systems. Objects are sequences L₀, L₁, ..., Lₙ of lattices with Galois connections between strata. Your negation and accumulation layer lives here.

**Fib(Lat)** — The fibered category of lattices over a base. Objects over a base lattice B are families of lattices {L_b}_{b ∈ B} indexed by elements of B. This is where your dependent types live — the type lattice is fibered over the value lattice. As the value cell refines, the fiber (the type lattice at that value) changes accordingly.

**SType** — The category of session types and session type simulations. Objects are session types (elements of your quantale), morphisms are subtyping/simulation relations. Duality is a contravariant endofunctor on this category.

### The Functors (Mappings Between Structures)

**Embedding: Lat → PROP.** Every lattice can be realized as a trivial propagator network (one cell, no propagators). This is how you install a new lattice domain into the network.

**Semantics: PROP → Lat.** Every propagator network has a denotational semantics as a lattice (the fixpoint lattice of the network's propagation function). This functor takes a network and returns "the lattice of all reachable fixpoints." This is the semantic foundation for your system.

**Abstraction: Lat → GAL → Lat.** Galois connections between lattices. Each bridge network implements a morphism in GAL. The composition of Galois connections (chain of bridges) is again a Galois connection — this is functorial.

**Stratification: Strat → PROP.** A stratified system compiles to a propagator network with controlled propagation order. The functor takes a stratification scheme and produces the network that implements it.

**Fibration: Fib(Lat) → PROP.** A dependent type system (fibered lattice family) compiles to a propagator network where cell connections change as base values refine. This is the most complex functor and the one most worth formalizing.

**Cost: Quant → Trop.** The cost function from your quantale to the tropical semiring. This is a quantale homomorphism, hence a functor between the corresponding one-object categories (since quantales are one-object enriched categories).

**Nucleus: Quant → Quant.** Each nucleus j on Q gives a functor from Q-computations to Q_j-computations. This is how you implement effect abstraction at module boundaries.

**Forgetful: SType → Quant.** Session types embed into the effect quantale — a session type *is* a specification of permitted quantale elements. The forgetful functor strips the "protocol specification" aspect and gives you just the algebraic structure.

### The Key Adjunctions

**Free ⊣ Forgetful between PROP and Lat.** The free propagator network on a lattice is the trivial one-cell network. The forgetful functor takes a network to its fixpoint lattice. The adjunction captures the fact that building a network from a lattice and extracting a lattice from a network are related by a universal property.

**Abstraction ⊣ Concretization in GAL.** Each Galois connection is an adjunction. Your bridge networks are literally adjunctions deployed at runtime.

**Inclusion ⊣ Nucleus between Q and Q_j.** The inclusion of Q_j into Q (concretization) is right adjoint to the nucleus j (abstraction). This is the adjunction underlying effect abstraction.

### The Natural Transformations

**Propagation as natural transformation.** A single step of propagation in the network is a natural transformation from "current state" to "next state" — it's a family of lattice operations (one per cell) that commutes with all the structural connections. The fixpoint of this natural transformation is the solution.

**ATMS world selection as natural transformation.** Choosing a consistent assumption set is a natural transformation from the ATMS-indexed family of lattices to a single lattice (the selected world). The ATMS manages a *family* of such transformations.

**Stratification ordering as natural transformation.** The ordering "stratum k before stratum k+1" is a natural transformation from the parallel composition of strata to the sequential composition. Stratification recovers the correct sequential semantics from the parallel infrastructure.

## Current State

Right now, facts are represented as **flat positional lists**. The `fact-row` struct in `relations.rkt:69` is simply:

;; terms: a list of values, e.g. '("Alice" "Bob")

## The Triangle You're Describing

You're identifying three vertices:

1. **Positional tuples** — what Prolog uses, what we have now. Compact, fast to unify, but opaque: `("Alice" "Engineering" 95000)` means nothing without the schema nearby.

2. **Keyed maps** — what your Prolog-with-databases workflow uses (`table_name({column_key: Val, ...})`). Self-describing, composable, but heavier and potentially slower for the hot unification path.

3. **Typed records** — what schemas aspire to be. Named fields, type-checked, but with the efficiency of positional access.

The tension: positional is fast but fragile; keyed is robust but heavy; records are ideal but usually require compile-time resolution.

## How to Square It

I think the answer is already implicit in your schema design, but making it explicit reveals the right representation:

### The Schema *is* the Rosetta Stone

A `schema` declaration establishes a **bijection between positions and keys**:

```prologos
schema Employee
  :name   String    ;; position 0
  :dept   Department ;; position 1
  :salary Nat       ;; position 2
```

This means the underlying runtime representation can be **positional vectors** (fast, compact, cache-friendly) while the *surface syntax* supports **both positional and keyed access**. The schema mediates:

- **Storage**: flat vector `#("Alice" Engineering 95000)` — essentially a database row
- **Authoring (positional)**: `|| "Alice" Engineering 95000N` — tabular, concise
- **Authoring (keyed)**: `|| {:name "Alice" :dept Engineering :salary 95000N}` — self-documenting
- **Access (keyed)**: `alice.name` or `:name` in goal patterns
- **Unification**: positional, against the vector, always

The key insight: **keyed syntax is a compile-time convenience that erases to positional access**. The schema provides the mapping; after elaboration, everything is positional. This is exactly what databases do — SQL lets you say `WHERE name = 'Alice'` but the storage engine uses column offsets.

### What About Schema-less Relations?

Bare `defr` without a schema annotation works like traditional Prolog — positional only, untyped. This is fine for quick prototyping and small relations. The moment you want keyed access or database interop, you add a schema.

### The Database Import Use Case

Your Prolog workflow of `table_name({column_key: Val, ...}).` maps beautifully:

```prologos

  :id     Nat
  :name   String
  :dept   String
  :salary Nat

defr employees : Employees

     3N "Carol" "Engineering" 110000N

defr employees : Employees
  :source "data/employees.csv"
```

The schema does double duty: it's the type contract *and* the column mapping for import/export. A CSV header row, a SQL `CREATE TABLE`, a JSON schema, and a Prologos `schema` all describe the same thing — a named sequence of typed fields. The schema is the interop surface.

### PVec vs Flat Vector vs CHAMP Map

For the underlying runtime representation, the options are:

| Representation | Pros | Cons |
|---|---|---|
| **Flat Racket vector** | Fastest access, minimal allocation | Not persistent, no structural sharing |
| **PVec (RRB tree)** | Persistent, shareable, log-time access | Overkill for small fixed-width rows |
| **CHAMP map** | Keyed access, persistent | Heaviest, hash overhead per lookup |
| **Struct** | Fastest, named fields | No dynamism, one struct per schema |

My analysis: for **fact storage**, flat vectors (or even Racket structs generated per-schema) are the right choice. A fact row is small (typically 3-15 fields), fixed-width (schema determines it), and accessed many times during unification. The hot path is "unify position i of fact with goal argument i" — this wants O(1) indexed access, not hash lookup.

PVec and CHAMP are great for *user-facing collections* (lists, maps in the functional language), but using them for the internal fact representation would be paying persistent-data-structure overhead for data that doesn't need persistence — facts are immutable once asserted.

**The representation I'd suggest**: compile-time-generated Racket structs per schema (fastest) with a fallback to flat vectors for schema-less relations. The struct fields correspond to schema columns. The schema provides the name→position mapping for keyed syntax. This is essentially what Ecto changesets do — the schema generates a struct, and the struct is the runtime representation.

### Keyed Pattern Matching in Goals

The really interesting ergonomic question is: can you use keyed patterns in relational goals?

```prologos
;; Positional (current)
(employees id name "Engineering" salary)

;; Keyed (proposed) — only bind what you care about
(employees {:dept "Engineering" :name name :salary salary})

;; Mixed — positional with keyword override

```

The keyed pattern form is a big ergonomic win for wide relations. If your `employees` table has 15 columns, positional patterns require 15 arguments with `_` for every column you don't care about.
