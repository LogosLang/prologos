## The Key Insight: Attribute Grammars ARE Propagator Networks

The deepest connection here isn't hypergraph grammars per se — it's the equivalence between attribute grammars and propagator networks. Context-free hypergraph grammars and attribute grammars generate the same class of term languages. This is the Engelfriet-Heyker theorem, and it's directly relevant to what you want.

An attribute grammar decorates a parse tree with computed values. Synthesized attributes flow upward (child to parent). Inherited attributes flow downward (parent to child). The evaluation of attributes is a fixpoint computation — attributes depend on other attributes, and the evaluator iterates until all values stabilize.

This is literally a propagator network on a tree-shaped graph. Each tree node is a cluster of cells (one per attribute). Each semantic rule is a propagator (reading some cells, writing others). Synthesized attributes are propagators that read from child cells and write to parent cells. Inherited attributes are propagators that read from parent cells and write to child cells. Evaluation to fixpoint is run-to-quiescence.

The equivalence with hypergraph grammars means: anything you can express as a context-free hypergraph grammar, you can express as an attribute grammar, which you can express as a propagator network on a parse tree.

## Parsing as Monotonic Information Accumulation

Here's the vision that maps onto your architecture.

Traditional parsing is a phase: you take a string of tokens, run a parsing algorithm (LR, Earley, PEG, etc.), and produce a parse tree. Then type checking happens on the tree. Then elaboration. Then optimization. Each phase is separate, with a complete data structure handed from one to the next.

On a propagator network, you can collapse these phases. The parse tree is a network of cells. Initially, the cells hold partial information — a token is known, but its syntactic role isn't. As parsing proceeds, information accumulates monotonically:

- The lexer writes token cells (character → token lattice)
- The parser writes syntax cells (token sequence → production lattice)
- The type checker writes type cells (syntax → type lattice)
- The elaborator writes elaborated cells (syntax + type → elaborated form lattice)

All of these are propagators on the same network. There's no "handoff" between phases. Information flows as soon as it's available. If the type checker can determine a type from a partial parse (because enough context is known), it does so immediately — it doesn't wait for parsing to complete.

This is incremental in the strongest sense: not "re-parse the changed region" but "propagate the change through the existing network." A character edit changes a token cell, which triggers the parser propagators, which trigger the type propagators, which trigger the elaboration propagators. Only the affected cells update. Everything else stays at its current value (the no-change guard prevents unnecessary propagation).

### The Lattice Structure for Parsing

The key design question is: what's the lattice for parse state?

**Token lattice**: ⊥ (unknown) → character sequence → token classification → ⊤ (error). Monotonic: once you know a character is part of an identifier, that information only gets refined (not retracted).

**Syntax lattice**: this is trickier because parsing involves ambiguity. A token sequence might be parseable as multiple productions. The lattice needs to represent ambiguity:

- ⊥ = no parse information
- Ambiguous = set of possible parse trees (like a parse forest)
- Unique = single parse tree
- ⊤ = parse error

The ordering: ⊥ ≤ Ambiguous ≤ Unique ≤ ⊤ (more information reduces ambiguity). This is a powerset lattice ordered by reverse inclusion — exactly the same structure as your narrowing choice lattices.

**Attribute lattice**: for each attribute (type, scope, elaborated form), a domain-specific lattice. Your existing type lattice, multiplicity lattice, session lattice all apply here.

### Connecting to Hypergraph Grammars

Context-free hypergraph grammars generalize context-free string grammars to graphs. HRGs have practical applications in graph parsing, information extraction, and learning, bridging expressive graph theory with computational efficiency. The generalization is relevant because your AST isn't really a tree — it's a graph (with variable references creating cross-links, with type annotations creating additional edges, with scope creating hierarchical structure).

A context-free graph grammar generalizes the concept of context-free string grammars to the domain of graphs, with the dominant formalism being hyperedge replacement grammars that enable the generative specification, analysis, and manipulation of graph languages.

For your propagator network, a hyperedge replacement grammar rule becomes a propagator template. When the parser recognizes that a subgraph matches the left-hand side of a production, it installs the propagators corresponding to the production's semantic rules. This is exactly your SRE pattern — register a form, get decomposition/composition/reconstruction propagators for free — applied to parsing.

The context-freeness lemma for hypergraph grammars states that derivations in hyperedge replacement grammars do not interfere with each other as long as they concern different hyperedges. In propagator terms: parsing propagators for independent subexpressions don't interfere. They can fire in parallel. This gives you parallel parsing as a natural consequence of the propagator architecture.

## Graph Rewriting for Compilation

Graph rewrite systems can be used to specify and generate program optimizations, with termination criteria defining exhaustive graph rewrite systems and stratification automatically selecting single normal forms.

This connects directly to your self-hosting compiler story. The key ideas:

**Compiler passes as graph rewrite rules.** Each optimization (constant folding, dead code elimination, inlining) is a graph rewrite rule: match a pattern in the program graph, replace it with an optimized version. The rewrite rules are propagators — they fire when they detect their pattern in the network.

**Stratification for phase ordering.** Different optimizations interact — inlining might enable constant folding, which might enable dead code elimination. The phase ordering problem (which optimization to apply first) maps to your stratification: each optimization phase is a stratum, with monotonic propagation within and controlled transitions between.

**Termination via your lattice structure.** Graph rewrite systems need termination guarantees. Your lattice height bounds + fuel limits provide exactly this. Each optimization moves the program "up" the lattice (toward a more optimized form). The lattice has finite height. Therefore optimization terminates.

**Double-pushout rewriting.** The algebraic approach to graph rewriting is based upon category theory, with the double-pushout approach and single-pushout approach being the most common. DPO rewriting has a clean categorical semantics as two pushout squares in the category of graphs. For your system, DPO rewriting rules would be typed by your polynomial functor/operad type system — a rewrite rule has a typed interface (what it matches, what it produces), and composition of rewrite rules is type-checked.

In this paper, hypergraph rewriting is studied in categorical terms in an attempt to define the notion of events and develop foundations of causality in graph rewriting. The causal structure of rewriting — which rewrites enable which subsequent rewrites — maps directly onto your propagator dependency graph. The causal ordering is your effect ordering quantale. This means your existing infrastructure for tracking effect ordering can also track optimization ordering in the compiler.

## The Unified Network Vision

Here's how all of this comes together for Prologos:

**A single propagator network that spans the entire compilation pipeline.**

```markdown
Source text → Tokens → Parse tree → Typed AST → Elaborated IR → Optimized IR → Target code
```

Each arrow is not a phase boundary but a set of propagators. Information flows left-to-right (parsing to compilation) and right-to-left (type constraints influencing parse disambiguation, optimization constraints influencing elaboration). The network is bidirectional — your Galois connections between subsystems apply to the compilation pipeline just as they apply to type checking.

**What this gives you for self-hosting:**

Source provenance is preserved end-to-end. Every cell in the target code has a derivation tree tracing back through optimization, elaboration, typing, and parsing to the original source text. This derivation tree is the proof object you need for verified compilation — it's the certificate that the compiled code correctly implements the source.

**What this gives you for interactive editing:**

When the user edits a character, the change propagates through the network incrementally. The parser updates the affected region. The type checker updates the affected types. The elaborator updates the affected elaboration. The optimizer updates the affected optimizations. All of this happens via propagation, with the no-change guard ensuring that unchanged parts of the pipeline don't recompute.

This is the dream of language-server architecture: instead of a pipeline of batch processors (parse → typecheck → elaborate → emit diagnostics), you have a live network where every change propagates immediately to all affected analyses.

**What this gives you for formal verification:**

If parsing is on the network, the parse tree is a cell value with provenance. Type checking refines it (monotonically). Elaboration refines it further. Optimization is a graph rewrite (tracked as a cell-tree transformation). The entire compilation is a monotonic lattice ascent from source text (⊥) to compiled artifact (fixpoint). The Layered Recovery Principle handles the non-monotonic aspects (optimization rewriting, negation in type checking, etc.).

The compilation proof object is the network's derivation tree at fixpoint. It includes every parsing decision, every type inference step, every optimization applied, and every elaboration choice — all as first-class data in the network. This is what your proof-carrying code infrastructure would verify. And because it's on the propagator network, it composes with your existing type checking, session checking, and effect tracking.

## Practical Starting Point

For an initial experiment, I'd suggest not trying to put the entire parsing pipeline on propagators immediately. Instead:

**Start with attribute evaluation.** Take your existing parsed AST and attach attribute propagators to it. Type attributes, scope attributes, elaboration attributes — all as propagators on the AST nodes. This replaces your current multi-pass elaboration with a single-network fixpoint computation.

You're essentially already doing this with your type checker's propagator network. The step is to recognize that the AST structure itself — the parent-child relationships, the sibling orderings, the variable references — can be cells and propagators rather than a separate static data structure.

**Then extend downward to parsing.** Once attributes are on the network, extend the network to include parsing state. Token cells, production cells, ambiguity cells. The parser becomes a set of propagators that fire when tokens arrive and produce parse structure. Attribute propagators fire when parse structure stabilizes.

**Then extend upward to optimization.** Graph rewrite rules as propagators that match patterns and produce optimized forms. The e-graph structure we discussed earlier is the natural representation — equivalent forms coexist in the same e-class, and the optimizer picks the best one.

Each step is independently useful and testable. Each step exercises and validates your propagator infrastructure at a new scale. And the end state is a unified network spanning the entire compilation pipeline — which is both the technical foundation for self-hosting and the theoretical validation of propagators as universal substrate.
