* Introduction

Prologos is a functional-logic language unifying dependent types, session types,
linear types (QTT), logic programming, and propagators. This document describes
the /surface syntax/ of the language as written in =.prologos= files.

Prologos has two syntactic modes:

1. *Whitespace-sensitive mode* (=.prologos= files): Indentation-based structure
   with =[]= for grouping and minimal punctuation. This is the primary mode.
2. *S-expression mode*: Parenthesized fallback for use in macros, tests, and
   when embedding in Racket. Every WS-mode form has a canonical sexp
   representation.

The grammar is organized bottom-up: lexical elements, then types, then
expressions, then declarations.

#+begin_quote
*Design Principle*: Prologos is /homoiconic/ --- code and data share the same
representation. All syntactic sugar desugars to s-expressions. Macros operate on
the post-parse representation, making code-as-data a first-class concept.
#+end_quote

* Lexical Grammar

** Comments

Line comments begin with =;= and extend to end of line. There are no block
comments.

#+begin_src prologos
;; This is a comment
def x : Nat zero  ; inline comment
#+end_src

** Identifiers

Identifiers may contain letters, digits, hyphens, question marks, exclamation
marks, and primes. They must not start with a digit.

#+begin_src grammar
identifier  ::= ident-start { ident-char }
ident-start ::= letter | '_' | '+' | '*' | '/' | '!' | '?' | '<' | '>' | '='
ident-char  ::= ident-start | digit | '-' | "'"
#+end_src

Naming conventions:
- Predicates use =?= suffix: =zero?=, =empty?=, =member?=
- Destructive operations use =!= suffix: =persist!=, =tvec-push!=
- Type names are capitalized: =Nat=, =List=, =Option=
- Value names are lowercase with hyphens: =sort-on=, =find-index=

Qualified names use =::= separator:

#+begin_src prologos
list::map       ;; qualified reference to list module's map
nat::add        ;; qualified reference to nat module's add
opt::unwrap-or  ;; qualified reference to option module's unwrap-or
#+end_src

** Numeric Literals

Prologos has four numeric literal forms, plus a decimal variant for approximate literals:

#+begin_src grammar
nat-literal     ::= digit+ 'N'                (* Natural: 0N, 42N, 100N *)
int-literal     ::= ['-'] digit+              (* Integer: 42, -7, 0     *)
rat-literal     ::= ['-'] digit+ '/' digit+   (* Rational: 3/7, -1/3   *)
decimal-literal ::= ['-'] digit+ '.' digit+   (* Decimal: 3.14, 0.5    *)
approx-literal  ::= '~' (int | rat | decimal) (* Posit approx: ~42, ~3/7, ~3.14 *)
#+end_src

#+begin_src prologos
42N          ;; Nat literal (Church-encoded natural — type infrastructure only)
42           ;; Int literal (arbitrary-precision integer)
3/7          ;; Rat literal (exact rational)
3.14         ;; Posit32 literal (approximate — same as ~3.14)
~3/7         ;; approximate Posit literal (from fraction)
~3.14        ;; approximate Posit literal (from decimal, stored as exact rational 157/50)
#+end_src

Bare decimal literals (=3.14=, =0.5=) produce Posit32 values, the same as their tilde-prefixed
equivalents (=~3.14=, =~0.5=). Internally, the decimal is stored as an exact rational (=157/50=)
and encoded to the nearest Posit32 bit pattern.

Nat is intended for type-level infrastructure (indices, lengths, proofs), not general computation.

** String Literals

Double-quoted with standard escape sequences (=\n=, =\t=, =\\=, =\"=).

#+begin_src prologos
"hello world"
"line one\nline two"
#+end_src

** Keyword Literals

Keywords start with =:= and are used as map keys and enum-like values:

#+begin_src prologos
:name
:age
:hello
#+end_src

** Boolean Literals

#+begin_src prologos
true
false
#+end_src

** Special Tokens

| Token     | Meaning                                      |
|-----------+----------------------------------------------|
| =_=       | Wildcard / type hole (inferred)              |
| =_1= =_2= | Numbered placeholders for partial application |
| =??=      | Typed hole (interactive development)          |
| =??name=  | Named typed hole                              |
| =:0=      | Erased multiplicity (use 0 times)            |
| =:1=      | Linear multiplicity (use exactly 1 time)     |
| =:w=      | Unrestricted multiplicity (use any times)    |
| =zero=    | Nat zero constructor                         |
| =nil=     | List empty / Nil value (overloaded)           |
| =unit=    | Unit value                                   |
| =refl=    | Equality reflexivity proof                   |

** Bracket Types

Prologos uses four bracket types, each with distinct semantics:

| Brackets | Purpose                                         |
|----------+-------------------------------------------------|
| =[...]=  | Primary grouping: function application, params  |
| =(...)=  | Special forms: =(fn ...)=, =(match ...)=, types |
| ={...}=  | Implicit type parameters, map literals           |
| =<...>=  | Dependent types, return type annotations          |

Inside any bracket pair, newlines are treated as whitespace (indentation is not
significant).

** Collection Literal Prefixes

| Syntax    | Type             | Example         |
|-----------+------------------+-----------------|
| ='[...]=  | =List=           | ='[1N 2N 3N]=  |
| =@[...]=  | =PVec=           | =@[1 2 3]=     |
| =~[...]=  | =LSeq=           | =~[1 2 3]=     |
| =#{...}=  | =Set=            | =#{1 2 3}=     |
| ={k v}=   | =Map=            | ={:name "Ada"}= |

* Type Expressions

In a dependently-typed language, types and terms share the same expression
syntax. This section highlights type-specific forms.

** Base Types

#+begin_src grammar
base-type ::= 'Type'     (* universe of types, level inferred *)
            | 'Nat'      (* natural numbers                    *)
            | 'Int'      (* arbitrary-precision integers        *)
            | 'Rat'      (* exact rationals                     *)
            | 'Bool'     (* booleans                            *)
            | 'Unit'     (* unit type                           *)
            | 'Symbol'   (* symbols for homoiconicity           *)
            | 'Keyword'  (* keyword type for map keys           *)
            | 'Nil'      (* nil type (safe navigation)           *)
            | 'Datum'    (* code-as-data algebraic type         *)
            | 'PropNetwork'  (* persistent propagator network  *)
            | 'CellId'       (* propagator cell identifier     *)
            | 'PropId'       (* propagator identifier          *)
            | 'UnionFind'    (* persistent disjoint sets       *)
            | 'ATMS'         (* hypothetical reasoning TMS     *)
            | 'AssumptionId' (* ATMS assumption identifier     *)
            | 'TableStore'   (* SLG-style tabling store        *)
            | 'Solver'       (* solver configuration type       *)
            | 'Goal'         (* relational goal type (Prop)     *)
            | 'DerivationTree' (* proof/derivation tree         *)
            | 'Answer'       (* answer with optional provenance *)
#+end_src

#+begin_src prologos
def x : Nat zero
def b : Bool true
def u : Unit unit
#+end_src

** Parameterized Types

Types are applied to arguments via juxtaposition in brackets:

#+begin_src prologos
List Nat            ;; list of natural numbers
Option Bool         ;; optional boolean
Map Keyword Int     ;; keyword-to-int map
Vec Nat 3N          ;; length-3 vector of Nats (dependent!)
Set Int             ;; set of ints
PVec Nat            ;; persistent vector of Nats
#+end_src

** Function Types (Arrows)

Arrows are right-associative infix operators:

#+begin_src grammar
arrow-type ::= type '->' type    (* unrestricted function *)
             | type '-0>' type   (* erased function       *)
             | type '-1>' type   (* linear function       *)
             | type '-w>' type   (* unrestricted, explicit *)
#+end_src

#+begin_src prologos
Nat -> Nat                ;; simple function
Nat -> Nat -> Bool        ;; curried: Nat -> (Nat -> Bool)
[List A] -> Nat           ;; bracketed domain
[A -> B] -> [List A] -> List B  ;; higher-order
#+end_src

** Dependent Types (Angle Brackets)

Angle brackets delimit dependent Pi (universal) and Sigma (existential) types:

#+begin_src grammar
dependent-type ::= '<' '(' ident ':' type ')' '->' type '>'  (* dependent Pi   *)
                 | '<' '(' ident ':' type ')' '*'  type '>'  (* dependent Sigma *)
                 | '<' ident ':' type '->' type '>'           (* shorthand Pi   *)
#+end_src

#+begin_src prologos
;; Dependent function: "for all n : Nat, a vector of length n"
<(n : Nat) -> Vec Nat n>

;; Dependent pair: "a natural n together with a Vec of that length"
<(n : Nat) * Vec Nat n>

;; Shorthand (single binder, no inner parens):
<n : Nat -> Vec Nat n>
#+end_src

** Sigma Types (Product / Pair Types)

Non-dependent pair types use infix =*=:

#+begin_src prologos
;; Non-dependent pair
Nat * Bool

;; In return type of split-at:
[Sigma [_ <List A>] [List A]]

;; Dependent Sigma via angle brackets:
<(x : Nat) * Vec Nat x>
#+end_src

** Equality Type

#+begin_src prologos
;; Eq type : proof that two terms are equal
(Eq Nat zero zero)       ;; 0 = 0 at type Nat
(Eq Nat [add x y] [add y x])  ;; commutativity
#+end_src

** Union Types

Union types use infix =|=:

#+begin_src prologos
Nat | Bool         ;; either a Nat or a Bool
Int | Rat | Nat    ;; right-associative: Int | (Rat | Nat)
#+end_src

** Nil Type and Nilable Types

The =Nil= type represents the absence of a value, used for safe navigation and
nilable map operations. The =nil= value is overloaded: in list context it is the
empty list, in =Nil= context it is the nil value.

#+begin_src prologos
;; Nil as a type
def nothing : Nil nil

;; Nilable type sugar: A? → (A | Nil)
def name : String? "hello"     ;; same as (String | Nil)
def age  : Nat? nil            ;; nil in Nat? context

;; nil? predicate
nil? nil          ;; true
nil? 42           ;; false
nil? "hello"      ;; false
#+end_src

*** =A?= Sugar

Uppercase type names ending with =?= desugar to a union with =Nil=:

#+begin_src grammar
nilable-type ::= UpperName '?'    (* String? → (String | Nil) *)
#+end_src

This only applies to known type names starting with an uppercase letter.
Lowercase identifiers with =?= suffix (=empty?=, =nil?=) are predicates, not
types.

*** Safe Navigation (=#.= and =#:=)

The =#.= and =#:= operators provide nil-safe map access. When the target is
=nil=, the entire chain short-circuits to =nil= instead of raising an error.

#+begin_src prologos
;; Regular access (throws on nil):
user.address.city

;; Nil-safe access (returns nil if user or address is nil):
user#.address#.city

;; Mixed: safe then regular
user#.address.city    ;; nil-safe for address, regular for city

;; Standalone nil-safe key access
#:name                ;; fn (x : _) (nil-safe-get x :name)
#+end_src

In sexp mode, use =nil-safe-get= directly:

#+begin_src prologos
(nil-safe-get user :address)
(nil-safe-get (nil-safe-get user :address) :city)
#+end_src

=nil-safe-get= returns =(V | Nil)= — if the key is missing or the map is =nil=,
it returns =nil= (NOT an error like =map-get=).

*** Postfix Bracket Indexing

Bracket indexing uses adjacency detection: =xs[0]= (no space) is an index,
while =xs [0]= (with space) is function application.

#+begin_src prologos
;; List indexing (accepts Int or Nat keys):
xs[0]             ;; first element
xs[2]             ;; third element

;; Map indexing:
m[:key]           ;; keyword lookup
m[computed-expr]  ;; computed key

;; Chaining with dot-access:
users[0].name     ;; index then field access
m[:config][:port] ;; nested map indexing

;; In sexp mode:
(get xs 0)
(get m :key)
#+end_src

Type-directed dispatch: =get= on =List A= / =PVec A= requires =Nat= or =Int= key
and returns =A=. On =Map K V= requires key type =K= and returns =V=.
Selection and Schema types delegate to =map-get=.

** Universe Levels

#+begin_src prologos
Type           ;; universe, level inferred
(Type 0)       ;; explicit level 0
(Type 1)       ;; explicit level 1 (contains Type 0)
#+end_src

** Type Holes

The wildcard =_= stands for an inferred type:

#+begin_src prologos
def x : _ zero    ;; type inferred as Nat
map _ xs          ;; type argument inferred
#+end_src

** Typed Holes (Interactive Development)

The =??= token is a /typed hole/ --- distinct from =_= (inference hole). Where
=_= means "infer this for me silently," =??= means "I don't know what goes here
--- show me what's possible." The type checker produces a diagnostic report with
the expected type, local context, and suggestions.

#+begin_src prologos
defn reverse
  | [nil]        -> nil
  | [[cons h t]] -> ??
  ;; Report: Hole ?? : [List A]
  ;;   h : A, t : [List A], reverse : [List A] -> [List A]
#+end_src

Named holes label specific incomplete positions:

#+begin_src prologos
defn zip-with [f xs ys]
  match [xs ys]
    | [[cons x xs'] [cons y ys']] -> [cons ??combine ??recurse]
#+end_src

=??= is the syntactic foundation for Editor-Assisted Interactive Development
(Agda/Idris-style hole-driven programming). See the Extended Spec Design
document for the full vision.

* Expressions

** Function Application

Application is juxtaposition, with brackets =[]= for grouping:

#+begin_src prologos
;; Simple application
suc zero                    ;; suc(zero)

;; Multi-argument (uncurried convention)
add x y                     ;; add(x, y)

;; Brackets for grouping
[add x y]                   ;; same as add x y
cons [f a] [map f as]       ;; cons(f(a), map(f, as))

;; Nested application
map [add 1N _] '[1N 2N 3N]  ;; map with partial application
#+end_src

** Lambda Expressions

#+begin_src grammar
lambda ::= 'fn' binder+ body
         | '(' 'fn' '(' ident ':' type ')' body ')'   (* sexp mode *)
#+end_src

#+begin_src prologos
;; WS mode: fn with bracket binder
fn [x : Nat] [suc x]

;; Multiple binder groups
fn [x : Nat] [y : Nat] [add x y]

;; Bare parameter (type inferred from context)
fn x [suc x]

;; Sexp mode: explicit binder
(fn (x : Nat) (suc x))

;; With multiplicity annotation
(fn (x :1 Nat) x)          ;; linear lambda
(fn (x :0 Nat) zero)       ;; erased lambda (can't use x)
#+end_src

** Pattern Matching

=match= is the primary pattern matching form, using =|= for arm separation.
Match arms support the same rich pattern language as =defn= pattern clauses:
nested constructor patterns, numeric literals, head-tail, and wildcards.
Constructor names are disambiguated via the constructor registry.

#+begin_src grammar
match-expr    ::= 'match' scrutinee match-arm+
match-arm     ::= '|' match-pattern guard? '->' body
guard         ::= 'when' expr
match-pattern ::= identifier | wildcard | nat-literal | int-literal
               |  identifier match-pattern+          (* compound: suc zero *)
               |  '[' match-pattern+ ']'             (* bracketed compound *)
               |  '[' match-pattern+ '|' match-pattern ']'  (* head-tail *)
#+end_src

#+begin_src prologos
;; Match on Nat — zero is recognized as constructor
match n
  | zero  -> zero
  | suc k -> add k k

;; Nested constructor pattern: suc zero (zero is ctor, not variable)
match n
  | suc zero    -> true    ;; matches exactly 1N
  | suc [suc k] -> false   ;; matches >= 2N, binds k
  | zero        -> true    ;; matches 0N

;; Nat literal patterns (0N, 1N, ... desugar to zero, suc zero, ...)
match n
  | 0N -> true
  | 1N -> false
  | _  -> false

;; Int literal patterns (bare integers → equality dispatch)
match n
  | 0  -> "zero"
  | 1  -> "one"
  | -1 -> "minus-one"
  | _  -> "other"

;; Guard clauses (when keyword)
match n
  | x when [int-lt x 0] -> [int-neg x]
  | x                   -> x

;; Int literals + guards
defn classify [n : Int] : Int
  | 0                    -> 0
  | n when [int-lt n 0]  -> -1
  | _                    -> 1

;; Match on Bool
match b
  | true  -> suc zero
  | false -> zero

;; Match on List
match xs
  | nil       -> zero
  | cons a as -> suc [length as]

;; Match on Option
match [find pred xs]
  | none   -> default
  | some v -> v

;; Nested match
match xs
  | nil       -> nil
  | cons a as -> match [pred a]
    | true  -> cons a [filter pred as]
    | false -> filter pred as
#+end_src

Available patterns:
- *Variable*: =x=, =n=, =rest= -- binds the matched value
- *Wildcard*: =_= -- matches anything, discards
- *Constructor*: =zero=, =suc k=, =nil=, =cons a as=, =true=, =false=,
  =none=, =some v=, =ok v=, =err e=, =datum-nil=, =datum-cons h t=, etc.
- *Nested constructor*: =suc zero=, =suc [suc k]= -- nested ctor patterns,
  constructors are distinguished from variables via the constructor registry
- *Bracketed*: =[cons x nil]= -- grouped pattern in brackets
- *Head-tail*: =[a b | rest]= -- Prolog/Erlang-style list destructuring,
  desugars to nested =cons= matching: =cons a (cons b rest)=
- *Nat literal*: =0N=, =1N=, =42N= -- desugar to
  =zero=, =suc zero=, =suc (suc ... zero)=
- *Int literal*: =0=, =1=, =-1=, =42= -- bare integers compile to
  equality dispatch via =int-eq=, NOT constructor patterns
- *Guard clause*: =| pattern when guard-expr -> body= -- boolean guard
  checked after pattern matches; if guard fails, falls through to next arm

#+begin_src prologos
;; Head-tail list pattern in match
match xs
  | [a | rest]  -> a       ;; first element
  | nil         -> default

;; Multiple head elements
match xs
  | [a b | rest] -> [add a b]
  | [a | _]      -> a
  | nil           -> zero
#+end_src

** If (Conditional)

=if= is a macro that desugars to =match= on Bool:

#+begin_src prologos
;; Three-argument if
if [zero? n] base-case [step n]

;; Equivalent match:
match [zero? n]
  | true  -> base-case
  | false -> step n
#+end_src

** Cond (Multi-Way Conditional)

=cond= is a macro for multi-way conditional dispatch. Each arm has a boolean
guard and a body separated by =->= . Arms are tried top-to-bottom; the first
true guard fires. Use =true= as a catch-all default.

#+begin_src grammar
cond-expr ::= 'cond' cond-arm+
cond-arm  ::= '|' guard-expr '->' body
#+end_src

#+begin_src prologos
;; Classify an integer
defn classify [n : Int] : Int
  cond
    | [int-eq n 0]  -> 0
    | [int-lt n 0]  -> -1
    | true          -> 1

;; Absolute value
defn abs [n : Int] : Int
  cond
    | [int-lt n 0] -> [int-neg n]
    | true         -> n
#+end_src

** Let (Local Binding)

=let= is a macro that desugars to application:

#+begin_src prologos
;; Single binding
let x = [add 1N 2N]
  suc x

;; With type annotation
let x : Nat = [add 1N 2N]
  suc x
#+end_src

** Pipe Operator (|>)

The pipe operator threads a value left-to-right through functions:

#+begin_src prologos
;; Binary pipe
zero |> suc |> suc |> suc    ;; = suc(suc(suc(zero))) = 3

;; Block pipe with indented steps
|> nums
  map suc-fn
  filter is-positive
  reduce sum-rf zero

;; Block pipes support automatic loop fusion:
;; consecutive map/filter steps fuse into a single-pass
;; transducer when followed by a terminal (reduce, sum, etc.)
#+end_src

** Compose Operator (>>)

Left-to-right function composition:

#+begin_src prologos
;; Compose two functions
[suc >> suc] zero           ;; = suc(suc(zero)) = 2

;; Pipe into a composed function
zero |> [suc >> double]     ;; = double(suc(zero))
#+end_src

** Type Annotations (the)

Explicit type annotation on an expression:

#+begin_src prologos
;; Annotate with explicit type
(the Nat zero)
(the [List Nat] nil)
#+end_src

** Pairs

#+begin_src prologos
;; Construct a pair
pair 1N true

;; Project from a pair
first [pair 1N true]        ;; = 1N
second [pair 1N true]       ;; = true

;; Aliases: fst, snd
fst [pair x y]
snd [pair x y]
#+end_src

** Quote and Quasiquote

Quote ='= converts code to =Datum= (code-as-data):

#+begin_src prologos
'foo          ;; -> datum-sym
'42           ;; -> datum-nat
':hello       ;; -> datum-kw
'true         ;; -> datum-bool
'()           ;; -> datum-nil
'(add 1 2)   ;; -> datum-cons chain

;; Quasiquote with unquote
def val : Datum [datum-nat 10]
`,val          ;; splice val into template
`(add 1 ,val)  ;; template with hole
#+end_src

** Collection Literals

#+begin_src prologos
;; List literal (linked list)
'[1N 2N 3N]

;; List with tail
'[1N 2N | rest-list]

;; PVec (persistent vector)
@[1 2 3 4 5]

;; Set
#{1 2 3}

;; Lazy sequence
~[1 2 3]

;; Map (key-value pairs)
{:name "Ada" :age 36}

;; Mixed-type map: value type auto-inferred as union
;; {:name "Alice" :age zero} : Map Keyword (Nat | String)
;; map-assoc widens: (map-assoc nat-map :k "s") : Map Keyword (Nat | String)
{:name "Alice" :age zero}
#+end_src

** Partial Application

Numbered placeholders =_1=, =_2= enable positional reordering:

#+begin_src prologos
;; Wildcard _ fills rightmost position
[add 1N _]           ;; fn x -> add 1N x

;; Numbered holes for reordering
[div _2 _1]          ;; fn x y -> div y x
#+end_src

** Varargs

The =...= marker in specs indicates variadic arguments. In =defn=, =...name=
collects excess arguments into a =List=:

#+begin_src prologos
spec sum-all Nat ... -> Nat
defn sum-all [...xs]
  sum xs

sum-all 1 2 3          ;; = 6
sum-all 1 2 3 4 5      ;; = 15

;; Mixed fixed + varargs
spec first-plus-rest Nat Nat ... -> Nat
defn first-plus-rest [first ...rest]
  add first [sum rest]

first-plus-rest 100 1 2 3  ;; = 106
#+end_src

* Declarations

** Namespace (ns)

Must be the first form in a file. Controls module identity and prelude loading:

#+begin_src prologos
;; Standard namespace: auto-imports prelude
ns my-project.utils

;; Bare namespace: no prelude
ns prologos.data.list :no-prelude
#+end_src

The prelude automatically provides: =Nat=, =Bool=, =List=, =Option=, =Result=,
=Pair= operations, =Eq=/=Ord=/=Add=/=Sub=/=Mul=/=Neg=/=Abs=/=FromInt=/=Num=/
=Fractional= traits and instances.

** Imports

Import modules with named or aliased references:

#+begin_src prologos
;; Named imports
imports [prologos.data.list :refer [List nil cons map length]]

;; Aliased module
imports [prologos.data.nat :as nat :refer [add mult]]

;; Multiple imports
imports [prologos.data.list   :refer [List nil cons map]]
        [prologos.data.nat    :refer [add mult]]
        [prologos.data.option :refer [Option none some]]
#+end_src

** Value Definition (def)

#+begin_src prologos
;; With type annotation
def x : Nat zero

;; Shorthand with :=
def x := zero

;; Multi-line body (indented)
def nums : [List Nat]
  cons 1N [cons 2N [cons 3N [nil Nat]]]

;; Private (not auto-exported)
def- internal-state : Nat zero
#+end_src

** Type Signature (spec)

=spec= declares a type signature separately from the definition:

#+begin_src prologos
;; Simple function type
spec factorial Nat -> Nat

;; Bare type variables — auto-introduced
;; (kind-`Type` vars and higher-kinded vars constrained by `:where`)
spec map [A -> B] [List A] -> List B

;; With trait constraints — the implicit binder is also auto-introduced
spec elem [Eq A] A [List A] -> Bool

;; Higher-kinded — `C` is inferred to have kind `Type -> Type`
;; from the `Seqable C` / `Buildable C` constraints.
spec gmap [A -> B] -> [C A] -> [C B]
  :where (Seqable C) (Buildable C)

;; With varargs
spec sum-all Nat ... -> Nat
#+end_src

*** Auto-Introduced Implicit Binders

A capitalized identifier that appears free in a =spec= signature (and is not a
known type or trait name) is auto-introduced as an implicit binder.  Two
directions of inference cooperate:

- *Direction 1.* Free type variables are auto-introduced with kind =Type=.
  ={A : Type}= almost never needs to be written.
- *Direction 2.* Higher-kinded variables (e.g. =C : Type -> Type=) are
  inferred from =:where= clauses and from inline trait constraints.  When
  =Seqable= is declared over ={C : Type -> Type}=, a free =C= used at
  =[Seqable C]= or =:where (Seqable C)= is auto-introduced with the trait's
  declared kind.

#+begin_src prologos
;; Equivalent — the bare form is the canonical style:
spec gmap {A B : Type} {C : Type -> Type} [A -> B] -> [C A] -> [C B]
  :where (Seqable C) (Buildable C)

spec gmap [A -> B] -> [C A] -> [C B]
  :where (Seqable C) (Buildable C)
#+end_src

Auto-detected binders are placed after explicit binders and sorted by kind
(=Type= first, then higher-kinded), matching the convention users have
written by hand.

Explicit binders remain useful for:

- *Pedagogic clarity* — when documenting a teaching example.
- *Disambiguation* — when no constraining position pins the kind, e.g.
  =spec empty {A : Type} [List A]=.
- *Order-sensitive call sites* — when a body explicitly passes type
  arguments in a particular order, declaring the binders explicitly fixes
  the order against the source-walk-order auto-introduction would produce.

#+begin_src prologos
;; Multiple implicits and constraints (still works):
spec sort-on {A B : Type} [B -> B -> Bool] [A -> B] [List A] -> List A
#+end_src

*** Extended Spec with Metadata

=spec= optionally accepts a trailing metadata map using the implicit map
syntax. Keyword-headed children after the type signature are collected into a
metadata hash. Recognized keys include =:implicits=, =:where=, =:doc=,
=:examples=, =:properties=, =:see-also=, =:pre=, =:post=, =:refines=.
Unrecognized keys are stored but not acted upon (forward-compatible).

The =:implicits= key lifts implicit binders out of the type signature into
metadata, freeing the signature to express pure function shape. Inline
={A : Type}= and =:implicits= coexist; both are merged.

#+begin_src prologos
;; With :implicits — clean signature:
spec sort [List A] -> [List A]
  :implicits {A : Type}
  :where (Ord A)
  :doc "Sorts a list in ascending order"
  :examples
    - [sort '[3N 1N 2N]] => '[1N 2N 3N]
    - [sort '[]] => '[]
  :properties (sortable-laws A)
  :see-also [reverse filter]

;; HKT case — :implicits dramatically improves readability:
spec gmap [A -> B] -> [C A] -> [C B]
  :implicits {A B : Type} {C : Type -> Type}
  :where (Seqable C) (Buildable C)

;; With functor — Pi types hidden entirely:
spec xf-compose [Xf A B] -> [Xf B C] -> [Xf A C]
  :implicits {A B C : Type}
  :properties (transducer-fusion-laws A B C)
#+end_src

The metadata is entirely optional. A simple =spec add Nat Nat -> Nat= is
unchanged. Metadata keys follow the principle of progressive disclosure: add
=:doc= and =:examples= for public APIs, =:properties= for libraries,
=:pre=/=:post= for safety-critical code, =:refines= for verified code.

Each =:properties= entry is a map with three keys:

| Key | Meaning |
|-----------+---------------------------------------------|
| =:name= | Human-readable label for the property |
| =:forall= | Universally quantified variables (binders) |
| =:holds= | Boolean expression that must be true |

See the Extended Spec Design document
(=docs/tracking/2026-02-22_EXTENDED_SPEC_DESIGN.org=) for the full research
survey and design rationale.

** Function Definition (defn)

#+begin_src prologos
;; Basic function
spec add Nat -> Nat -> Nat
defn add [x y]
  match x
    | zero  -> y
    | suc n -> suc [add n y]

;; With inline type
defn suc-all [xs : List Nat] : List Nat
  map [add _ 1N] xs

;; With return type annotation
impl Eq Nat
  defn eq? [x y] <Bool>
    nat-eq? x y

;; Pattern dispatch (Haskell/Erlang-style)
;; Clauses with [patterns...] -> body are grouped by arity
;; and compiled to match/reduce at macro expansion time.
defn fact
  | [zero]  -> suc zero
  | [suc n] -> mult [suc n] [fact n]

;; Multi-arg patterns: bare tokens = atomic, inner [] = compound
defn add
  | [zero n]    -> n
  | [[suc m] n] -> suc [add m n]

;; Params+patterns: named params declare arity, bare positional patterns
;; Arity 1: all forms before -> grouped into one pattern
defn pred [n]
  | suc zero    -> zero
  | suc [suc k] -> suc k
  | zero        -> zero

;; Arity 2: each bare form is one pattern argument
;; Use brackets for compound patterns in multi-arg positions
defn add2 [m n]
  | zero n      -> n
  | [suc m'] n' -> suc [add2 m' n']

;; Typed params + return type with patterns
defn safe-head [xs : List Nat] : Option Nat
  | [x | rest] -> some x
  | nil        -> none

;; Head-tail list patterns: [head1 head2 | rest]
;; Desugars to nested cons matching.
defn list-head
  | [[a | rest]] -> some a
  | [nil]        -> none

;; Mixed: arity clauses + pattern clauses (different arities)
defn f
  | [zero] -> true        ;; arity 1, pattern clause
  | [n]    -> false       ;; arity 1, pattern clause
  | [x y]  <Nat> [add x y]  ;; arity 2, arity clause

;; Private
defn- helper [x]
  suc x
#+end_src

** Abstract Interpretation Traits

Prologos provides trait infrastructure for abstract interpretation --- modular
abstract domains connected by soundness-preserving Galois connections, with
widening/narrowing for infinite-height lattices. These traits build on the
existing =Lattice= and =HasTop= traits.

*** Widenable: Widening and Narrowing

The =Widenable= trait adds widening (∇) and narrowing (Δ) operators to a lattice.
Widening forces convergence on infinite ascending chains; narrowing recovers
precision after widening stabilizes. Required for domains like =Interval= that
have infinite height.

#+begin_src prologos
;; Trait definition (in prologos::core::widenable-trait)
trait Widenable {A}
  widen  : A -> A -> A    ;; ∇ — over-approximate for convergence
  narrow : A -> A -> A    ;; Δ — recover precision after widening

;; Instance for Interval (Cousot-style widening)
impl Widenable Interval
  defn widen [old <Interval> new <Interval>] <Interval>
    ...  ;; if bounds tighten without stabilizing, jump to bot (unconstrained)
  defn narrow [old <Interval> new <Interval>] <Interval>
    ...  ;; intersect old and new to recover precision

;; Creating a widenable cell (generic)
spec new-widenable-cell {A} PropNetwork -> [PropNetwork * CellId]
  where (Lattice A) (Widenable A)
defn new-widenable-cell [net] where (Lattice A) (Widenable A)
  net-new-cell-widen net bot [fn x y [join x y]] [fn x y [widen x y]] [fn x y [narrow x y]]
#+end_src

*** GaloisConnection: Abstraction and Concretization

A =GaloisConnection= pairs a concrete domain =C= with an abstract domain =A=
via an abstraction function (α: C → A) and a concretization function (γ: A → C).
The adjunction law α(c) ≤ a ⟺ c ≤ γ(a) ensures soundness: anything provable in
the abstract domain reflects a real property of the concrete domain.

#+begin_src prologos
;; Trait definition (in prologos::core::galois-trait)
trait GaloisConnection {C A}
  alpha : C -> A    ;; abstraction (concrete → abstract)
  gamma : A -> C    ;; concretization (abstract → concrete)

;; Instance: Interval → Bool ("is constrained?" abstraction)
impl GaloisConnection Interval Bool
  defn alpha [iv <Interval>] <Bool>
    match iv
      | interval-bot -> false
      | interval-top -> true
      | mk-interval lo hi -> true
  defn gamma [b <Bool>] <Interval>
    if b interval-top interval-bot
#+end_src

The multi-type-parameter trait generates dict names following the pattern
=ConcreteType-AbstractType--GaloisConnection--dict=, e.g.,
=Interval-Bool--GaloisConnection--dict=. Accessors follow the standard pattern:
=GaloisConnection-alpha dict val= and =GaloisConnection-gamma dict val=.

*** Cross-Domain Propagation

Cross-domain propagation creates bidirectional links between concrete and abstract
cells using α/γ functions from a =GaloisConnection=. Changes to either cell
automatically propagate through the Galois connection:

- Concrete cell changes → α(c-val) written to abstract cell
- Abstract cell changes → γ(a-val) written to concrete cell

Termination is guaranteed by the no-change guard in =net-cell-write= (assumes
monotone α/γ over finite-height lattices).

*** Abstract Domain Library Modules

Prologos ships with ready-made abstract domains as library modules (not
prelude-exported --- use explicit =imports=):

- =prologos::data::sign= --- =data Sign = sign-bot | sign-neg | sign-zero | sign-pos | sign-top=
- =prologos::core::sign-lattice= --- =impl Lattice Sign=, =impl HasTop Sign=
- =prologos::data::parity= --- =data Parity = parity-bot | parity-even | parity-odd | parity-top=
- =prologos::core::parity-lattice= --- =impl Lattice Parity=, =impl HasTop Parity=

#+begin_src prologos
;; Example: using the Sign domain
imports [prologos::data::sign :refer [Sign sign-neg sign-pos sign-top]]
imports [prologos::core::sign-lattice :refer []]
imports [prologos::core::lattice-trait :refer [Lattice Lattice-join]]

;; => sign-top
#+end_src

** Property Declarations

A =property= declares a named, composable group of propositions --- analogous
to =bundle= for traits. Properties compose via =:includes= (conjunction) and
attach to specs via =:properties= or to traits via =:laws=.

#+begin_src prologos
;; Basic property declaration
property sortable-laws {A : Type}
  :where (Ord A)
  - :name "idempotent"
    :forall {xs : [List A]}
    :holds [eq? [sort [sort xs]] [sort xs]]
  - :name "length-preserving"
    :forall {xs : [List A]}
    :holds [eq? [length [sort xs]] [length xs]]

;; Composition via :includes (like bundle for traits)
property monoid-laws {A : Type}
  :where (Add A) (AdditiveIdentity A)
  :includes (semigroup-laws A)
  - :name "left-identity"
    :forall {x : A}
    :holds [eq? [add additive-identity x] x]
  - :name "right-identity"
    :forall {x : A}
    :holds [eq? [add x additive-identity] x]

;; Hierarchical: monad includes applicative includes functor
property monad-laws {M : Type -> Type}
  :where (Monad M)
  :includes (applicative-laws M)
  - :name "left-identity"
    :forall {a : A} {f : [A -> [M B]]}
    :holds [eq? [bind [pure a] f] [f a]]

;; Usage in spec
spec sort [List A] -> [List A]
  :implicits {A : Type}
  :where (Ord A)
  :properties (sortable-laws A)

;; Higher-order: parameterized over functions
property sorting-properties {A : Type} {f : [List A] -> [List A]}
  :where (Ord A)
  - :name "idempotent"
    :forall {xs : [List A]}
    :holds [eq? [f [f xs]] [f xs]]

spec sort [List A] -> [List A]
  :where (Ord A)
  :properties (sorting-properties A sort)
#+end_src

Property clause names are scoped to their declaring block. When =:includes=
flattens a hierarchy, names are qualified with =/=:
=functor-laws/identity=, =monad-laws/left-identity=, etc.

** Functor Declarations (Named Type Abstractions)

A =functor= declares a named, parameterized type abstraction with optional
category-theoretic metadata. Transparent by default --- the type checker unfolds
to the =:unfolds= form during elaboration. Metadata keys map CT concepts to
approachable language.

#+begin_src prologos
;; Simple type synonym
functor FilePath
  :unfolds String

;; Parameterized synonym with documentation
functor Result {A}
  :unfolds [Either String A]
  :doc "A computation that may fail with a string error"

;; Full category-theoretic treatment
functor Xf {A B : Type}
  :doc "A transducer: transforms A-reductions into B-reductions"
  :compose xf-compose
  :identity id-xf
  :laws (transducer-fusion-laws A B)
  :unfolds <(S :0 Type) -> [S -> B -> S] -> S -> A -> S>

;; Optics
functor Lens {S T A B : Type}
  :doc "A bidirectional accessor: view and update a part of a structure"
  :compose lens-compose
  :identity lens-id
  :laws (lens-laws S T A B)
  :see-also [Prism Traversal Iso]
  :unfolds <{F : Type -> Type} -> (Functor F) -> [A -> [F B]] -> S -> [F T]>

;; Usage in specs — the functor name replaces raw Pi types:
spec xf-compose [Xf A B] -> [Xf B C] -> [Xf A C]
  :implicits {A B C : Type}
spec into-list [Xf A B] -> [List A] -> [List B]
  :implicits {A B : Type}
#+end_src

| CT concept | Key | Meaning in plain English |
|---------------------+---------------+------------------------------------------------|
| Object mapping | =:unfolds= | "What this expands to under the hood" |
| Morphism composition | =:compose= | "How to chain two of these together" |
| Identity morphism | =:identity= | "The do-nothing version" |
| Laws | =:laws= | "What rules these always follow" |

Only =:unfolds= is required. The rest are progressive.

** User-Defined Macros

#+begin_src prologos
;; Simple macro: double-it x --> add x x
defmacro double-it [$x] [add $x $x]

;; Conditional macro
defmacro when-nonzero [$n $body]
  [if [not [zero? $n]] $body zero]

(expand-full (when true [suc zero]))  ;; all steps with labels
#+end_src

* Subtype Declarations

The =subtype= declaration registers a subtype relationship between two types,
enabling automatic widening in the type checker and implicit runtime coercion
in the reducer. This extends the built-in numeric subtype lattice
(=Nat <: Int <: Rat=, =Posit8 <: Posit16 <: Posit32 <: Posit64=) to
library-defined refined types.

** Syntax

#+begin_src prologos
;; Auto-inferred coercion: single-field wrapper types unwrap automatically
subtype PosInt Int
subtype NegInt Int
subtype PosRat Rat
subtype NegRat Rat

;; Explicit coercion via a named function (required for nullary constructors)
subtype Zero Int via zero-to-int
#+end_src

** Semantics

- The type checker's =subtype?= predicate recognizes the declared relationship,
  so a =PosInt= value can be used wherever =Int= is expected.
- The reducer's coercion helpers (=try-coerce-to-int=, =try-coerce-to-rat=)
  unwrap refined values at runtime when they appear as operands to numeric
  operations (=int+=, =rat*=, etc.).
- Transitive closure is computed automatically: declaring =PosInt <: Int= when
  =Int <: Rat= is already known yields =PosInt <: Rat= with composed coercion.
- Auto-inference works for single-constructor, single-field wrapper types: the
  coercion simply unwraps the constructor. Nullary constructors (like =mk-zero=)
  require an explicit =via= clause naming a coercion function.

** Subtype Lattice (Numeric)

#+begin_example
           Rat
          / | \
    NegRat  |  PosRat
         \  |  /
          Int
        / | \
  NegInt Zero PosInt
          |
         Nat
#+end_example

* Schema, Selection, and Path Algebra

Prologos provides /schemas/ for declaring structured data shapes, /selections/ for
declaring field-level access policies, and a /path algebra/ for navigating and
transforming nested data.

** Schema Definitions

Schemas declare named map types with typed fields and optional properties:

#+begin_src prologos
;; Basic schema
(schema User :name String :age Nat :address Address)

;; Schema with properties
(schema Config
  :host String :default "localhost"
  :port Nat    :default 8080N :check [pos?]
  :debug Bool  :default false)

;; Closed schema (rejects extra keys)
(schema Point :x Nat :y Nat :closed)
#+end_src

Field properties: =:default val= (auto-fill on construction), =:check [pred]= (runtime
assertion), =:closed= (reject unknown fields). Schema-typed values are constructed as
map literals: ~(def u : User := {:name "Alice" :age 3N :address {...}})~.

** Selection Definitions

Selections declare which fields are accessible through a given view of a schema:

#+begin_src prologos
;; Allow access to name and address only
(selection NameAddr from User :requires [:name :address])

;; Deep paths: only address.zip
(selection AddrZip from User :requires [:address.zip])

;; Composition via :includes
(selection Extended from User :includes [NameAddr] :requires [:age])
#+end_src

A function typed with a selection type can only access the declared fields:
~(spec get-name NameAddr -> String)~ can access ~:name~ but not ~:age~.

** Path Algebra

The path algebra provides a rich syntax for navigating nested data:

| Syntax                    | Meaning                                  |
|---------------------------+------------------------------------------|
| =:address.zip=            | Navigate two levels deep                 |
| =:address.*=              | All fields of address (wildcard)         |
| =:address.**=             | All descendants (globstar)               |
| =:address.{zip city}=     | Branch: expand to two paths              |
| =:a.{b.c d.e}=           | Per-branch sub-paths                     |
| =:a.{b.{c d} e}=         | Nested braces (recursive)                |
| =:a.{b c}.**=             | Post-brace suffix applied to all         |

Brace items are full sub-paths, enabling mixed-depth branching:
~:user.{name address.{zip city} age}~ expands to five independent paths.

** Path Expressions: =get-in= and =update-in=

Two expression forms apply the path algebra to values:

#+begin_src prologos
;; Navigate: get value at path
(get-in user :address.zip)           ;; → Nat value

;; Navigate + branch: project subset
(get-in user :address.{zip city})    ;; → {:zip 10001N :city "NYC"}

;; Navigate + transform: apply fn at leaf, rebuild structure
(update-in user :address.zip (fn [n] 0N))  ;; → user with zip set to 0N
#+end_src

*Semantics*:
- =get-in= with single path desugars to chained =map-get=: ~(map-get (map-get user :address) :zip)~
- =get-in= with branched paths builds a map literal with projected fields
- =update-in= desugars to nested =map-get= / =map-assoc= with function application at leaf

Both forms reuse the existing =map-get= and =map-assoc= type checking infrastructure.

* Relational Language (Logic Programming)

Prologos makes logic programming a first-class paradigm via =defr=, the
relational analogue of =defn=. Relations use round brackets for goals,
mode-annotated parameters, and dual sigils for facts (=||=) and rules (=&>=).

** Relation Definition (defr)

#+begin_src prologos
;; Single-arity: fact block
defr parent [?x ?y]
  || "alice" "bob"
     "bob" "carol"

;; Single-arity: rule clause
defr ancestor [?x ?y]
  &> (parent x y)

;; Multi-arity with | dispatch
defr ancestor
  | [?x ?y] &> (parent x y)
  | [?x ?z] &> (parent x y) (ancestor y z)

;; Mode annotations: + = input, - = output, ? = free
defr lookup [+key -val]
  &> (table key val)
#+end_src

** Anonymous Relations (rel)

#+begin_src prologos
;; Inline relation
(eval (rel [?x] &> (parent x "bob")))
#+end_src

** Solve and Explain

#+begin_src prologos
;; Solve: returns Seq (Map Keyword Value)
(eval (solve (parent x y)))

;; Solve-one: returns first answer only
(eval (solve-one (ancestor "alice" y)))

;; Solve-with: named solver
(eval (solve-with my-solver (ancestor x y)))

;; Explain: returns Seq (Answer Value) with derivation trees
(eval (explain (ancestor "alice" "carol")))

;; Explain-with: named solver + overrides
(eval (explain-with debug-solver (ancestor x y)))
#+end_src

** Solver Configuration

#+begin_src prologos
;; Solver definitions (pre-expanded by macros)
(solver my-solver :execution :parallel :timeout 5000)
(solver debug-solver :provenance :full :strategy :depth-first)
#+end_src

| Type | Kind | Description |
|------+------+-------------|
| =Solver= | =Type 0= | Solver configuration |

| =DerivationTree= | =Type 0= | Proof / derivation tree |

* Multiplicity (QTT)

Prologos uses Quantitative Type Theory for resource tracking. Each variable
binding has a /multiplicity/ annotation:

| Annotation | Meaning                      | Syntax       |
|------------+------------------------------+--------------|
| =:0=       | Erased (0 uses at runtime)   | =(x :0 Nat)= |
| =:1=       | Linear (exactly 1 use)       | =(x :1 Nat)= |
| =:w=       | Unrestricted (any uses)      | =(x :w Nat)= |

Arrow types carry multiplicities:

#+begin_src prologos
Nat -> Nat            ;; unrestricted (default :w)
Nat -0> Nat           ;; erased: argument not used at runtime
Nat -1> Nat           ;; linear: argument used exactly once
Nat -w> Nat           ;; unrestricted: explicit
#+end_src

#+begin_src prologos
;; Erased argument (type-level only)
(fn (A :0 Type) (fn (x :w A) x))     ;; polymorphic identity

;; Linear function
(fn (x :1 Nat) x)                    ;; must use x exactly once
#+end_src

* Session Types (Process Calculus)

Session types describe structured communication protocols between concurrent
processes. Each protocol step specifies the direction (send/receive) and the
type of data being transmitted.

** Session Declarations

#+begin_src prologos
;; A simple request-response protocol
session Greeter
  ? String       ;; receive a name
  ! String       ;; send back a greeting
  end
#+end_src

** Session Step Operators

| WS Operator      | Sexp Form         | Meaning                              |
|-------------------+-------------------+--------------------------------------|
| =! Type=          | =(Send T S)=      | Send a value of Type                 |
| =? Type=          | =(Recv T S)=      | Receive a value of Type              |
| =!: name Type=    | =(DSend (n:T) S)= | Dependent send (binds name in cont)  |
| =?: name Type=    | =(DRecv (n:T) S)= | Dependent recv (binds name in cont)  |
| =!! Type=         | =(AsyncSend T S)= | Asynchronous (buffered) send         |
| =?? Type=         | =(AsyncRecv T S)= | Asynchronous (buffered) recv         |
| =+>=              | =(Choice ...)=    | Internal choice (select a branch)    |
| =&>=              | =(Offer ...)=     | External choice (offer branches)     |
| =rec= /[label]/   | =(Mu S)=          | Recursive reference point            |
| =end=             | =End=             | Session termination                  |

** Choice and Offer

#+begin_src prologos
session FileRead
  rec
    +>                            ;; internal choice (client decides)
      | :read-all  -> ? String -> end
      | :read-line -> ? String -> rec
      | :close     -> end
#+end_src

** Dependent Sessions

Dependent send (=!:=) and dependent recv (=?:=) bind a name so
that the continuation type can reference the transmitted value.

#+begin_src prologos
;; Length-indexed vector protocol:
;; first send the length, then send exactly that many elements.
session VecProto
  !: n Nat                        ;; send n, bind it in continuation
  ! [Vec String n]                ;; send exactly n strings
  end
#+end_src

** Process Definitions

#+begin_src prologos
;; defproc defines a process that implements a session protocol
defproc greeter : Greeter
  name := self ?                  ;; receive from self channel
  self ! [concat "Hello, " name]  ;; send greeting
  stop                            ;; terminate

;; with-open: sugar for file IO with auto-close
defproc reader
  with-open "data.txt" : FileRead
    select ch :read-all
    data := ch ?
  stop
#+end_src

** Natural Number Elimination (natrec)

The low-level eliminator for natural numbers:

#+begin_src prologos
;; natrec motive base step target
;; motive : Nat -> Type
;; base   : motive zero
;; step   : (n : Nat) -> motive n -> motive (suc n)
;; target : Nat
natrec motive base step target
#+end_src

In practice, =match= is preferred:

#+begin_src prologos
;; Idiomatic: match instead of natrec
defn double [n]
  match n
    | zero  -> zero
    | suc k -> suc [suc [double k]]
#+end_src

#+begin_src prologos

;; Given proof : Eq A left right
;; J computes: motive(left, right, proof) via base(left)
J motive base left right refl
#+end_src

#+begin_src prologos
ns examples.demo

imports [prologos.data.list   :refer [List nil cons map reduce sum length]]
        [prologos.data.nat    :refer [add mult]]
        [prologos.data.option :refer [Option none some]]

;; Type signature with implicit params
spec factorial Nat -> Nat
defn factorial [n]
  match n
    | zero  -> suc zero
    | suc k -> mult n [factorial k]

;; Higher-order function with trait constraint
spec sum-mapped {A : Type} [A -> Nat] [List A] -> Nat
defn sum-mapped [f xs]
  sum [map f xs]

;; Data type definition
data Tree {A}
  leaf
  node : A -> Tree A -> Tree A

;; Pattern matching on user-defined types
spec tree-size [Tree A] -> Nat
defn tree-size [t]
  match t
    | leaf       -> zero
    | node _ l r -> suc [add [tree-size l] [tree-size r]]

;; Using pipe operator
def result : Nat
  |> '[1N 2N 3N 4N 5N]
    map [add 1N _]
    reduce add 0N

;; Macro definition
defmacro unless [$cond $body] [if $cond zero $body]

;; Using the macro
unless [zero? 5] [factorial 5]
#+end_src

* Appendix: S-Expression Mode

Every WS-mode form has a canonical s-expression representation. In sexp mode,
all grouping uses parentheses and whitespace is not significant:

#+begin_src prologos
;; WS mode:
defn add [x y]
  match x
    | zero  -> y
    | suc n -> suc [add n y]

;; Equivalent sexp mode:
(defn add (x y)
  (match x
    (zero  -> y)
    (suc n -> (suc (add n y)))))

;; WS mode:
spec map {A B : Type} [A -> B] [List A] -> List B

;; Equivalent sexp mode:
(spec map {A B : Type} (-> (-> A B) (-> (List A) (List B))))
#+end_src

* Appendix: Whitespace Reader Rules

The WS reader converts indentation to explicit structure:

1. *Same indentation* as previous line: NEWLINE token (sibling separator)
2. *Deeper indentation*: INDENT token (child block begins)
3. *Shallower indentation*: DEDENT token(s) (child blocks end)
4. *Inside brackets* =[]=, =()=, ={}=, =<>=: newlines are whitespace
5. *Blank/comment lines*: ignored for indentation purposes
6. *Column 0*: always starts a new top-level form

* Appendix: Reader Desugaring Table

| Surface syntax  | Reader output            | Meaning                      |
|-----------------+--------------------------+------------------------------|
| ='expr=         | =($quote expr)=          | Quote to Datum               |
| =`expr=         | =($quasiquote expr)=     | Quasiquote template          |
| =,expr=         | =($unquote expr)=        | Unquote (splice into QQ)     |
| =42N=           | =($nat-literal 42)=      | Natural number literal       |
| =~42=           | =($approx-literal 42)=   | Approximate (Posit) literal  |
| =3.14=          | =($decimal-literal 157/50)= | Bare decimal → Posit32    |
| =:name=         | =keyword token=          | Keyword literal              |
| ='[1 2 3]=      | =($list-literal 1 2 3)=  | List literal                 |
| =@[1 2 3]=      | =($vec-literal 1 2 3)=   | PVec literal                 |
| =~[1 2 3]=      | =($lseq-literal 1 2 3)=  | Lazy sequence literal        |
| =#{1 2 3}=      | =($set-literal 1 2 3)=   | Set literal                  |
| =...name=       | =($rest-param name)=     | Rest/vararg parameter        |
| =x \vert> f=    | =($pipe-gt x f)=         | Pipe left-to-right           |
| =f >> g=        | =($compose f g)=         | Compose left-to-right        |
| =x.name=        | =($dot-access name)=     | Map field access              |
| =x#.name=       | =($nil-dot-access name)= | Nil-safe field access         |
| =x#:name=       | =($nil-dot-key :name)=   | Nil-safe keyword access       |
| =xs[0]=          | =($postfix-index 0)=     | Postfix bracket index         |
| =m[:key]=        | =($postfix-index :key)=  | Postfix keyword index         |
