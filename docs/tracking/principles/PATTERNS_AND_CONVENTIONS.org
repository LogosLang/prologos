* Naming Conventions

** Functions and Values
- Lowercase with hyphens: ~find-index~, ~sort-on~, ~map-get~
- Predicates end in ~?~: ~zero?~, ~empty?~, ~member?~, ~even?~
- Destructive / side-effecting operations end in ~!~: ~persist!~, ~tvec-push!~
- Transducers end in ~-xf~: ~map-xf~, ~filter-xf~, ~take-xf~

** Types and Constructors
- Capitalized: ~Nat~, ~List~, ~Option~, ~Bool~, ~Map~
- Multi-word types: PascalCase: ~QueryResult~, ~AdditiveIdentity~
- Constructors: lowercase of their type: ~some~, ~none~, ~cons~, ~nil~

** Modules and Namespaces
- Separated by ~::~: ~prologos::data::list~, ~prologos::core::map-ops~
- Module aliases with ~:as~: ~imports [prologos::core::map-ops :as m]~
- Selective import: ~:refer [merge fold-entries keys vals]~
- Side-effect import: ~:refer []~ (triggers instance registration)

** Library Functions Should Not Repeat the Library Name
By convention, users import libraries with an alias. This means library
functions should use short, unqualified names:

#+begin_src prologos
;; GOOD: short names, users alias the module
;; In prologos::core::map-ops:
spec keys : {K V : Type} [Map K V] -> [List K]
spec vals : {K V : Type} [Map K V] -> [List V]
spec merge : {K V : Type} [Map K V] [Map K V] -> [Map K V]

;; Usage: qualified access prevents collision
imports [prologos::core::map-ops :as m :refer [merge]]
m::keys my-map
m::vals my-map
merge m1 m2

;; BAD: redundant prefix
spec map-keys-list ...   ;; "map" is already in the module name
spec map-vals-list ...   ;; user would write m::map-vals-list (double "map")
#+end_src

When a library function name collides with a parser keyword or another
prelude name, use the module alias for disambiguation rather than
lengthening the function name.

** Trait Methods
Short, action-oriented names: ~eq?~, ~from~, ~add~, ~sub~, ~compare~.
The trait name provides context: ~(Eq A)~ implies ~eq?~ compares for
equality within type ~A~.

* Nat vs Int: When to Use Each

~Nat~ (Peano naturals: ~zero~, ~suc~) is /type-level machinery/ --- it
exists for dependent type indexing, structural induction, and proof
terms. It is NOT for general-purpose arithmetic.

~Int~ (arbitrary-precision integers) is the default numeric type for
computation. ~Rat~ for exact rationals. ~Posit32~ for approximate.

| Use Case | Type | Example |
|-------------------------------+-----------+-------------------------------------|
| General arithmetic | ~Int~ | ~.{3 + 4}~ => ~7 : Int~ |
| Exact fractions | ~Rat~ | ~3/7 : Rat~ |
| Approximate computation | ~Posit32~ | ~~3.14 : Posit32~ |
| Type-level vector length | ~Nat~ | ~<(n : Nat) -> Vec A n>~ |
| Structural induction / proofs | ~Nat~ | ~defn by-induction \| [zero] ...~ |
| List length (structural) | ~Nat~ | ~spec length : [List A] -> Nat~ |
| List length (computational) | ~Int~ | ~spec length-int : [List A] -> Int~ |
| Indexing (structural) | ~Nat~ | ~spec nth : Nat [List A] -> A?~ |
| Indexing (computational) | ~Int~ | ~spec nth-int : Int [List A] -> A?~ |
| Narrowing / logic programming | ~Nat~ | ~add ?x ?y = 5N~ |
| Generic numeric code | ~Num A~ | ~spec double : (Num A) A -> A~ |

#+begin_src prologos
;; GOOD: Int for computation
spec factorial : Int -> Int
defn factorial [n]
  if [<= n 1] 1 [* n [factorial [- n 1]]]

;; GOOD: Nat for type-level indexing
spec safe-head : {A : Type} (xs : [List A]) {pf : [NonEmpty xs]} -> A

;; GOOD: Int-indexed list operations for computational contexts
[nth-int 2 my-list]     ;; Int index, returns Option A
[take-int 5 my-list]    ;; Int count, returns List A
[length-int my-list]    ;; returns Int (not Nat)

;; BAD: Nat for general computation
;; spec factorial : Nat -> Nat   -- Nat is for types, not computation
#+end_src

** Index Type Strategy

Collection operations have two families of index types:

- *Nat-indexed* (~nth~, ~take~, ~drop~, ~length~): Use for structural
  recursion, dependent types, and proof terms. ~length~ returns ~Nat~
  because it mirrors the structural depth of the list.

- *Int-indexed* (~nth-int~, ~take-int~, ~drop-int~, ~length-int~): Use
  for computational contexts where you work with Int values. These
  handle negative indices safely (returning ~none~ / ~nil~) without
  requiring manual Nat conversion.

#+begin_src prologos
;; Computational code: use Int-indexed ops
let n := [length-int xs]       ;; Int, works with int arithmetic
let elem := [nth-int 2 xs]     ;; no from-nat needed

;; Type-level / proof code: use Nat-indexed ops
spec safe-index : {A : Type} (n : Nat) (xs : [List A])
  {pf : [lt? n [length xs]]} -> A
#+end_src

*Recommendation*: Default to Int-indexed operations in new code.
Use Nat-indexed only when Nat is structurally required (dependent
types, inductive proofs, narrowing).

* Function Definition Patterns

** Prefer ~spec~ / ~defn~ Over ~def~ With Lambda

#+begin_src prologos
;; GOOD: spec + defn
spec factorial : Int -> Int
defn factorial [n]
  if [<= n 1] 1
    [* n [factorial [- n 1]]]

;; ACCEPTABLE but less idiomatic:
def factorial : [Int -> Int]
  fn [n] (if (<= n 1) 1 (* n (factorial (- n 1))))
#+end_src

** Prefer Multi-Arity Over Helper Functions
Use the ~|~ separator for pattern-matching arities rather than defining
separate helper functions. Note that Nat multi-arity is appropriate for
structural/type-level functions (like ~length~), while computational
functions use ~Int~:

#+begin_src prologos
;; GOOD: multi-arity for structural recursion (Nat is appropriate here)
spec length : {A : Type} [List A] -> Nat
defn length
  | [nil]        -> 0N
  | [[cons _ t]] -> [suc [length t]]

;; GOOD: Int for computational functions
spec sum-list : [List Int] -> Int
defn sum-list
  | [nil]        -> 0
  | [[cons h t]] -> [+ h [sum-list t]]
#+end_src

Multi-arity definitions keep related logic together, are more readable,
and let the compiler see the full pattern space for exhaustiveness checking.

** Uncurried Arguments, Curried Results
Functions take arguments as a flat group, not curried:

#+begin_src prologos
;; GOOD: uncurried
spec fold : {A B : Type} [B -> A -> B] B [List A] -> B
defn fold [f z xs] ...

;; AVOID: fully curried (Haskell style)
spec fold : {A B : Type} [B -> A -> B] -> B -> [List A] -> B
#+end_src

The uncurried convention matches the ~[f x y z]~ call syntax and avoids
partial application confusion. When currying is desired, use explicit
lambda: ~[fn [x] [fold f x]]~.

* Bracket Usage Rules

| Context | Use | Example |
|--------------------+---------+------------------------------|
| Function call | ~[...]~ | ~[map inc xs]~ |
| Logic clause | ~(...)~ | ~(edge ?from ?to)~ |
| Type grouping | ~<...>~ | ~<(n : Nat) -> Vec A n>~ |
| Parser keywords | ~(...)~ | ~(match ...)~, ~(fn ...)~ |
| Implicit binders | ~{...}~ | ~{A B : Type}~ |
| Map literal | ~{...}~ | ~{:name "Ada"}~ |
| PVec literal | ~@[..]~ | ~@[1 2 3]~ |
| List literal | ~'[..]~ | ~'[1 2 3]~ |
| Set literal | ~#{..}~ | ~#{:a :b}~ |
| Lazy seq literal | ~~[..]~ | ~~[1 2 3]~ |

Key rule: *outer trees are implicit* --- top-level forms don't need
wrapping brackets. Write ~defn foo [x] body~, not ~[defn foo [x] body]~.

* Collection Conventions

** Seq as Universal Hub
All collection operations go through ~LSeq~ (lazy sequence) as the
universal intermediate representation:

#+begin_example
Any Collection → to-seq → transform (map/filter/fold) → to-seq → Any Collection
#+end_example

Use ~Seqable~, ~Buildable~, and ~Foldable~ traits for generic code.
Use collection-specific modules (~pvec-ops~, ~set-ops~, ~map-ops~) for
type-specific operations.


** Map as Two-Param Type
~Map K V~ has two type parameters, which means it cannot implement the
standard one-param traits (~Seqable~, ~Foldable~, ~Buildable~). Instead,
use standalone functions from ~map-ops~:

#+begin_src prologos
;; These are standalone functions, not trait methods
[m::fold-entries f z my-map]   ;; fold over (K, V) pairs
[m::map-vals f my-map]         ;; map over values
[m::keys my-map]               ;; extract key list
#+end_src

* Import and Module Patterns

** Standard Import Forms

#+begin_src prologos
;; Import with alias (preferred for libraries)
imports [prologos::core::map-ops :as m :refer [merge]]

;; Import specific names
imports [prologos::data::list :refer [List nil cons map filter]]

;; Side-effect only (registers trait instances)
imports [prologos::core::eq-nat :refer []]

;; Full prelude
ns my-module

;; Bare mode (no prelude)
ns my-module :no-prelude
#+end_src

** Library Modules Skip Prelude
Library modules under ~prologos::data::*~ and ~prologos::core::*~ must
use ~:no-prelude~ (or the equivalent Racket-side mechanism) to avoid
circular dependencies. They import only what they need explicitly.

** Prelude Exports
The prelude (~ns foo~) provides:
- Core types: ~Nat~, ~Bool~, ~List~, ~Option~, ~Result~, ~Pair~
- Collections: ~PVec~, ~Map~, ~Set~, ~LSeq~
- All traits and instances: ~Eq~, ~Ord~, ~Add~, ~Sub~, etc.
- Collection operations and conversions
- Generic numeric functions: ~sum~, ~product~, ~int-range~
- Generic wrappers: ~plus~, ~minus~, ~times~, ~divide~, ~negate-fn~, ~abs-fn~
- Int-indexed list ops: ~nth-int~, ~take-int~, ~drop-int~, ~length-int~
- Nat ops behind ~nat::~ alias (e.g., ~nat::add~, ~nat::mult~); ~zero?~ unqualified

Own-definition priority: a user's ~def map~ shadows the prelude's
~list::map~.

* Type Annotation Patterns

** Spec for Public Functions
Every exported function should have a ~spec~:

#+begin_src prologos
spec sort : {A : Type} where (Ord A) [List A] -> [List A]
defn sort [xs] ...
#+end_src

** Inline Annotation for Local Bindings
Use ~(the Type expr)~ or ~(def x : Type val)~ for local type ascription:

#+begin_src prologos
def xs : [List Int] '[1 2 3]
(the [List Int] '[1 2 3])
#+end_src

** Implicit vs. Explicit Type Parameters
- ~{A : Type}~ in ~spec~ --- implicit, inferred at call site
- ~(A : Type)~ in ~spec~ --- explicit, must be provided
- Convention: type parameters are implicit unless the caller needs to
  specify them (e.g., ~map-empty K V~ where both must be explicit)

* Error Handling Conventions

** Use ~A?~ (Nilable) for Simple Absence

#+begin_src prologos
spec find : {A : Type} [A -> Bool] [List A] -> A?
#+end_src

** Use ~Result E A~ for Recoverable Errors

#+begin_src prologos
spec parse-int : String -> [Result ParseError Int]
#+end_src

** Use Dependent Types for Preconditions

#+begin_src prologos
spec head : {A : Type} (xs : [List A]) -> {pf : [NonEmpty xs]} -> A
#+end_src

* Pipe and Composition

** ~|>~ for Value Threading (Last Argument Position)

#+begin_src prologos
|> @[1 2 3 4 5]
  pvec-filter even?
  pvec-map inc
  pvec-fold add 0
#+end_src

Default: value goes to last argument position. Use ~_~ for explicit
placement: ~|> x [f _ y]~ threads ~x~ into the ~_~ position.

** ~>>~ for Function Composition (Left-to-Right)

#+begin_src prologos
def process := >> parse validate transform serialize
#+end_src

** Block-Form ~|>~ Enables Loop Fusion
Consecutive ~map~/~filter~ steps in a block-form pipe fuse into a
single pass when followed by a terminal (~fold~, ~sum~, ~reduce~).

* Propagator Network Conventions

** Termination Argument Convention
Every new propagator or lattice added to the network must include a
termination argument in its definition comment. The argument identifies
which level of the termination guarantee hierarchy applies:

- *Level 1 (Tarski)*: "Finite lattice height /h/; monotone propagator."
- *Level 2 (Well-founded)*: "Measure /m/ decreases on each step; /m/ is well-founded."
- *Level 3 (Widening)*: "Concrete domain widened to finite abstract lattice."
- *Level 4 (Productive)*: "Each observation terminates; overall computation is coinductive."
- *Level 5 (Fuel)*: "Bounded by fuel; structural argument not available because [reason]."

Example:
#+begin_src racket
;; Readiness propagator for trait constraint.
;; Termination: Level 1 (Tarski). Each dependency cell transitions
;; at most once (⊥ → solved). This propagator fires at most once per
;; dependency. Finite dependencies = finite firings.
(define (add-readiness-propagator! dep-cell-id meta-id tc-info)
  ...)
#+end_src

See [[file:GÖDEL_COMPLETENESS.org][GÖDEL_COMPLETENESS.org]] for the full hierarchy and design discipline.

** Layer Tagging Convention
Propagators must be tagged with their stratum layer at creation time:
- ~'retraction~ (S(-1)): assumption cleanup propagators
- ~'propagation~ (S0): type unification, lattice merge propagators
- ~'readiness~ (L1): constraint readiness detection
- ~'resolution~ (L2): resolution commitment propagators

The layered scheduler processes propagators in layer order. Untagged
propagators default to ~'propagation~ (S0) for backward compatibility.
