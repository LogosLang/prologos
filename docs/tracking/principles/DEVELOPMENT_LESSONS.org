#+TITLE: Prologos Development Lessons
#+AUTHOR: Prologos Project
#+DATE: 2026-02-22

* Why We Track and Document Efforts

Language implementation is a long journey with compounding decisions.
Every design choice constrains future choices. Every bug reveals an
assumption. Every workaround reveals an abstraction gap. By tracking
lessons systematically, we:

1. *Avoid repeating mistakes* --- the same meta-variable pitfall doesn't
   bite twice
2. *Preserve institutional knowledge* --- when context windows reset, the
   lessons survive
3. *Inform future design* --- patterns that recur across subsystems
   suggest deeper abstractions
4. *Measure progress* --- timing data, test counts, and phase completions
   give concrete signals

* Workflow Lessons

For the full design-to-implementation lifecycle, see
[[file:DESIGN_METHODOLOGY.org][DESIGN_METHODOLOGY.org]] --- the five-phase process
(Research → Refinement → Design Iteration → Implementation → Composition)
that governs how we approach significant features. For how lessons are
extracted from completed work via Post-Implementation Reviews, see
[[file:POST_IMPLEMENTATION_REVIEW.org][POST_IMPLEMENTATION_REVIEW.org]].

** Phase-Gated Implementation
Break large features into lettered sub-phases (a, b, c...) with explicit
"done" vs. "remaining" tracking. This:
- Prevents scope creep within a single session
- Creates natural commit points
- Makes it easy to resume after interruption

** Tracking Documents Before Code
Create ~docs/tracking/YYYY-MM-DD_HHMM_TOPIC.md~ /before/ implementation.
Write down what you plan to build, what the phases are, and what success
looks like. Update after completion with lessons learned.

** Deferred Work Is Real Work
When something is deferred, immediately add it to ~docs/tracking/DEFERRED.md~.
Deferred work that isn't tracked is forgotten work. The DEFERRED file is
the single source of truth for all postponed tasks.

** Targeted Testing During Development
Run only affected tests (~racket tools/run-affected-tests.rkt~), not the
full suite. The full suite is for end-of-day regression. This keeps the
feedback loop tight (seconds, not minutes).

** Whale File Splitting
Test files exceeding 30s wall time bottleneck the parallel runner. Split
into ~-01~, ~-02~ parts with ~20 test-cases each. Check with
~racket tools/benchmark-tests.rkt --slowest 10~ after full runs.

* Type System Lessons

** Two-Phase Zonking
Intermediate zonking must NOT default unsolved metas --- it would commit
to wrong answers before all information is available. Only final zonking
(after type-checking completes) defaults metas to ~lzero~ / ~mw~.

This was a painful lesson: defaulting too early caused "spooky action at
a distance" where solving one meta would silently lock in defaults for
unrelated metas.

** Speculative Type-Checking
~save-meta-state~ / ~restore-meta-state!~ enables try-and-backtrack:
attempt a type check, and if it fails, restore the meta-variable store
to its prior state. Critical for:
- Church fold type inference (try fold interpretation, fall back to list)
- Union type widening (try homogeneous, widen if mismatched)
- Trait resolution (try one instance, backtrack to another)

** Dict Params Use ~:w~, Not ~:0~
Trait dictionary parameters must have unrestricted (~:w~) multiplicity.
If they're erased (~:0~), the body can't call the dict's methods at
runtime. This caused cryptic QTT violations until the invariant was
established.

** Meta Variables in ~infer-level~
Every AST node that appears in type position must have an ~infer-level~
case. When we added mixed-type maps, unsolved metas appeared in
~expr-map-empty~ type annotations, and ~infer-level~ had no case for
~expr-meta~. The fix: unsolved metas default to ~just-level(lzero)~,
matching zonk-final's behavior.

** Pattern Unification Boundaries
Miller's pattern fragment (the decidable subset of higher-order
unification) handles most practical cases, but higher-rank types push
beyond it. The workaround for higher-rank polymorphism: use explicit
~def ... : Type body~ instead of ~spec~/~defn~ when the type is too
complex for inference.

* Parser and Syntax Lessons

** Parser Keywords Are Sacred
Names like ~map-vals~, ~set-member?~ are parser keywords --- the parser
recognizes them at read time before name resolution. A library function
with the same name will be shadowed in unqualified usage. Solutions:
- Use qualified access: ~m::map-vals~
- Choose a non-conflicting name for the library function

** WS Reader ~$~ Is Quote
The ~$~ prefix in whitespace mode is the quote operator: ~$Foo~ desugars
to ~($quote Foo)~. It is NOT a sigil for variable names. This tripped up
dict parameter naming when traits were first implemented.

** Nested Bracket Arrow Syntax
~[B -> [K -> [V -> B]]]~ fails to parse (inner brackets create separate
argument groups). The flattened form ~[B -> K -> V -> B]~ works correctly.
This is a consequence of Prologos's uncurried convention.

** Consumer-Side Normalization
When special characters (~...~, ~>>~, ~|~) have readtable conflicts,
normalize on the consumer side (parser/elaborator) rather than hacking
the readtable. This is safer and keeps the reader simple.

* Trait and Instance Lessons

** Single-Method Trait = The Function
When a trait has exactly one method, the dict IS the function. No wrapper
struct, no field access. ~(Add Nat)~ resolves directly to the Nat addition
function. This gives zero-overhead dispatch for the common case.

** Instance Registration Is a Side Effect
~(imports [prologos::core::eq-nat :refer []])~ loads the module for its
side effect (registering the Eq instance for Nat). The empty ~:refer []~
signals "I don't need any names, just register the instances."

** Own-Definition Priority
When the user defines ~def map ...~ in their module, it shadows the
prelude's ~list::map~. This is intentional: user definitions always win.
Qualified access (~list::map~) remains available.

** Prelude Errors Are Silently Swallowed
~process-ns-declaration~ wraps module loading in ~with-handlers~ that
silently catches errors. When a prelude module fails to load, the only
signal is that its names are missing. This made debugging prelude changes
extremely difficult. Future improvement: log prelude loading errors.

* Collection and Library Lessons

** Map Can't Use Standard Traits
~Map K V~ has two type parameters, but ~Seqable~, ~Foldable~, and
~Buildable~ expect ~{C : Type -> Type}~ (one parameter). This is a
fundamental HKT limitation. The workaround: standalone functions in
~map-ops~ that operate directly on ~Map K V~.

** Type Constructors Aren't Terms
~(infer Add)~ doesn't work because ~Add~ is a type constructor (deftype),
not a term. To test type constructors, use a typed binding:
~(def x : (Add Nat) val)~.

** ~inferQ~ Returns Result Type
For AST keyword nodes (like ~expr-map-get~, ~expr-set-insert~),
~inferQ~ must return the RESULT type, not the input type. Getting this
wrong causes the QTT checker to track multiplicities against the wrong
type, leading to phantom violations.

** Library Function Names Don't Repeat Module Names
When ~prologos::core::map-ops~ exports ~map-keys-list~, the user writes
~m::map-keys-list~ --- "map" appears twice. Better: export ~keys~, user
writes ~m::keys~. This lesson drove the map-ops rename.

* Numerics Lessons

** ~expr->impl-key-str~ Must Handle All Types
The trait instance dispatch function must have cases for every numeric
type (~Int~, ~Rat~, ~Posit8~, ~Posit16~, ~Posit32~, ~Posit64~,
~Keyword~). A missing case causes silent dispatch failure with an
unhelpful error.

** Posit Widening Through Exact
Converting between posit sizes goes through exact rational:
~p16-from-rat (p8-to-rat x)~. This preserves as much precision as
the target format allows.

** Into Blanket Impl Pattern
~impl Into A B where (From A B)~ provides the reverse direction
automatically using parametric constraint resolution. This halves the
number of conversion instances needed.

* Homoiconicity Lessons

** Both Modes Must Produce Identical ASTs
The whitespace reader and the s-expression reader must produce
bit-identical ASTs. This is tested extensively and is a hard invariant.
Any divergence means macros behave differently depending on input mode.

** Quote Roundtrips Must Preserve Structure
~'[fn [x] [add x 1N]]~ must produce a ~Datum~ value that, when
reconstructed, yields the original expression. This is the foundation
for code-as-data metaprogramming.

** Quasiquote Comma Conflict
Racket's reader consumes commas before our readtable sees them. The
solution: a nested readtable (~prologos-qq-readtable~) that handles
unquoting within quasiquoted expressions.

* Architecture Lessons

** 14-File AST Pipeline
Adding a new AST node requires touching 14 files (syntax, surface-syntax,
parser, elaborator, typing-core, qtt, reduction, substitution, zonk,
pretty-print, and sometimes unify, macros, foreign). This is the cost
of a uniform AST representation --- but the benefit is that every
subsystem handles every node consistently.

** Callback Pattern for Circular Dependencies
When module A needs to call module B, but B already requires A, use a
callback parameter: B exposes a function that takes a callback, and A
provides the implementation at the call site. This avoids Racket's
circular require limitation.

** Sentinel AST Nodes
The reader emits sentinel nodes (~$dot-access~, ~$dot-key~, ~$pipe-gt~)
that are rewritten by preparse into standard AST forms. This keeps the
reader simple (it doesn't need to understand semantics) and the preparse
predictable (it transforms known sentinels).

* Optimization Lessons

** Prefer ~where~ Constraints for Specialization
When a function needs a trait constraint, use ~where~ syntax rather than
explicit positional dict parameters:

#+begin_src prologos
;; Good — eligible for call-site specialization:
spec my-fn {A} SomeArg -> Result where (Trait A)
defn my-fn [arg] where (Trait A)
  ...method-name arg...

;; Unavoidable but less optimal — invisible to rewriter:
spec my-fn [Trait A] A -> A
defn my-fn [dict x]
  Trait-method dict x
#+end_src

The ~where~ style threads the trait dictionary automatically via
~maybe-inject-where~ in ~macros.rkt~, and the ~rewrite-specializations~
pass in ~driver.rkt~ can replace calls with zero-overhead specialized
versions when the type argument is statically known. The explicit
positional dict style works, but the rewriter can't detect it.

** Call-Site Specialization: When to Add ~specialize~ Blocks
Rule of thumb:
- 3+ concrete types regularly used → add specializations for each
- Hot loop (propagator merge, inner folds) → specialize even for 1 type
- One-off generic usage → dict-passing overhead is negligible, don't bother

Each specialization is small (~4 lines: spec + specialize + defn + body)
and eliminates all dict-passing overhead for that type, collapsing to a
direct function call. The generic dict-passing version remains as the
correct fallback for types without a specialization.

** Two Dict-Passing Styles Coexist
The codebase has two styles for polymorphic functions that need trait
constraints:

| Style               | Example           | Rewriter sees it? | Ergonomics     |
|---------------------+-------------------+-------------------+----------------|
| ~where (Trait A)~   | ~new-lattice-cell~ | Yes               | Auto-injected  |
| ~[Trait A]~ positional | ~plus~, ~sum~   | No                | Manual at call |

Migration path for the positional style:
1. Change spec to use ~where~ clause
2. Change defn to use bare method names
3. Add ~specialize~ blocks for common types
4. Existing call sites continue to work (the desugared form is identical)

This migration is optional --- the positional style works correctly, it
just doesn't benefit from automatic specialization. Prioritize migration
for functions called in performance-sensitive paths.

* Meta-Lessons

** Completeness Over Deferral
When you have the clarity, the vision, and the full context --- finish
the work now. Half-built pieces that get deferred are half-built pieces
that get forgotten. The cost of re-acquiring context later almost always
exceeds the cost of doing the work while the understanding is fresh.

This doesn't mean never defer --- sometimes layering and phasing is the
only sensible approach. But the default should be: complete the solution.
Defer only when there is a genuine dependency on unbuilt infrastructure
or when the design is genuinely uncertain. "We'll come back to it" is a
red flag; if the work is clear enough to describe, it's clear enough to
do.

Corollary: when deferral /is/ necessary, it must be immediately tracked
in ~DEFERRED.md~. Deferred work that isn't tracked is abandoned work.

** Callbacks Are a Propagator-First Anti-Pattern

When a subsystem injects behavior across module boundaries via callback
parameters, it's a sign that the behavior belongs in the propagator
network, not outside it. Callback parameters are:
- Not correct-by-construction (they can be ~#f~; every call site must guard)
- Not data-oriented (the callback is an opaque function, not inspectable data)
- Not propagator-first (the callback fires imperatively, not through cell state)

The alternative: make the triggering condition a cell, and the response a
propagator that fires when the cell reaches the ready state. The module
boundary problem disappears because the propagator captures its closure at
registration time (in the module that has the resolution logic) and fires
at the appropriate stratum (in the network's scheduler).

Discovered during Track 6 Phase 8d: three callback parameters existed to
break circular dependencies between ~metavar-store.rkt~ and ~driver.rkt~.
Deep audit revealed them vestigial --- the stratified resolution loop was
the load-bearing path. The pattern: when you find callbacks in a propagator-
based system, ask whether the callback's trigger should be a cell and its
behavior a propagator.

** Let Pain Drive Design
Don't add features until they're needed. Every deferred item in
~DEFERRED.md~ was deferred because the pain wasn't yet sufficient to
justify the complexity. When the pain arrives, the solution is better
informed by real usage.

** Tests Are Documentation
The test suite (3000+ tests) is the most accurate documentation of what
the language actually does. When docs and tests disagree, tests win.
New features aren't done until they have tests.

** Grammar Is Living
~docs/spec/grammar.ebnf~ and ~docs/spec/grammar.org~ are canonical and
must be updated whenever syntax changes. Stale grammar docs cause more
confusion than no grammar docs.

** State Termination Arguments Explicitly
When adding a propagator or lattice to the network, state why it
terminates. "This propagator terminates because the lattice has height 3"
or "because type depth decreases on each cycle." If you can't state the
argument, the computation may not terminate. Fuel bounds catch bugs in
termination arguments; they should not BE the argument. See
[[file:GÖDEL_COMPLETENESS.org][GÖDEL_COMPLETENESS.org]] for the hierarchy of termination guarantees
(Levels 1--5) and the discipline for choosing the strongest available.

Discovered during Track 6/7 design: the stratified propagator network's
layered quiescence raised the question "what guarantees termination when
L2 feeds back to S0?" The answer required identifying a well-founded
measure (type depth × unsolved metas). Without stating this explicitly,
the design would have relied on fuel alone --- missing the structural
insight that makes the architecture provably decidable.

** Non-Equality Relations Break Equality Assumptions

When extending the SRE from equality-only to support additional structural
relations (subtyping, duality, coercion), every decomposition function
must be audited for equality-specific assumptions. Three classes of
assumptions were discovered in SRE Track 1:

1. /Lattice ordering assumption/: Equality uses a flat lattice where
   ~merge(Nat, Int) = top~ (incompatible). Subtyping needs a
   partially-ordered lattice where ~merge(Nat, Int) = Int~ (the
   supertype). Two orderings on the same carrier require separate merge
   functions --- a single merge cannot serve both relations.

2. /Symmetric decomposition assumption/: Equality decomposes with one
   descriptor (both sides have the same constructor tag). Duality
   decomposes with two descriptors (Send on one side, Recv on the other).
   ~sre-decompose-generic~ assumed symmetric decomposition; duality
   required a dedicated ~sre-duality-decompose-dual-pair~.

3. /Value copying assumption/: Equality safely copies the "unified" value
   to both sides (they converge to the same value). Duality and subtyping
   cannot --- copying source-side values into target-side sub-cells
   produces un-dualized or un-subtyped values. For non-equality relations,
   always initialize target-side sub-cells with bot and let propagators
   push the correct values.

When adding a new relation, analyze it along four axes before
implementation: (a) symmetry (symmetric, directional, involutive),
(b) merge semantics (merges cells, checks cells, swaps constructors),
(c) constructor mapping (same tag, dual tag pairs, coercion mapping),
(d) binder requirements (needs fresh metas, or operates on ground types).

Discovered in SRE Track 1 + 1B: 7 bugs total, 3 from equality-specific
assumptions. See [[file:../2026-03-23_SRE_TRACK1_PIR.md][SRE Track 1 PIR §4, §13, §14]].

** Retraction Is Aggregation's Dual
Both retraction (removing entries from a cell) and aggregation (collecting
the complete set of derivations) are non-monotonic operations. Both are
addressed by stratification: compute to fixpoint in a lower stratum
before the non-monotone operation in a higher stratum observes the result.
This insight from well-founded logic programming (NAF-LE) directly
informed the S(-1) retraction stratum design in Track 7. See
[[file:GÖDEL_COMPLETENESS.org][GÖDEL_COMPLETENESS.org]] § "Application to the Stratified Propagator Network."

** Data at Rest, Closures Derived on Demand

When persistent state contains closures (Racket procedures), serialization
fails. The fix is always the same: store the DATA from which the closure
can be deterministically derived, and derive the closure on demand (at
load time, at call time, or lazily).

This pattern appeared 6 times in PM Track 10's =.pnet= serialization:
1. /Foreign function procedures/ → store =(module-path . binding-name)=,
   re-link via =dynamic-require= at deserialize time.
2. /Preparse expander closures/ → store expander symbol name, look up
   in expander table at call time.
3. /Coercion functions/ → store =(coerce-via Sub Super via-fn-name)=,
   derive the closure from the coercion spec at deserialize time.
4. /Marshal-in / marshal-out closures/ → derive from foreign function's
   type signature (already stored as data in =source-module= +
   =racket-name= fields).
5. /Capability-wrapped foreign functions/ → store unwrapped reference,
   re-wrap with capability token pass-through at deserialize time.
6. /Gensym identifiers/ → store as tagged interned symbols (=symbol$$N=),
   reconstruct fresh gensyms from per-module table at deserialize time.

The general principle: registries and persistent state should hold DATA
REFERENCES, not closures. The closure is a CACHE — an optimization of the
data reference. If the data reference is lost (not serialized), the cache
cannot be reconstructed. If the data reference IS serialized, the cache
can always be re-derived.

This is Data Orientation applied to persistence: the serialized form IS
the NTT-correct representation. The live form (closures) is an
optimization cache derived from the data. When closures block
serialization, the data model is incomplete — the closure's provenance
(where it came from, what data generates it) is not recorded.

Discovered in PM Track 10: 22/40 prelude modules failed serialization
due to foreign function closures. The comprehensive fix (Option A:
embed provenance in struct) solved all 22 in one pattern. The
whack-a-mole phase (5 incremental fixes before the comprehensive audit)
demonstrated the anti-pattern: fixing ONE closure category at a time
reveals the next. The audit ("how many closure categories ARE there?")
broke the cycle.

See [[file:../2026-03-25_PM_TRACK10_PIR.md][PM Track 10 PIR]] and [[file:../research/2026-03-24_SELF_DESCRIBING_SERIALIZATION.md][Self-Describing Serialization Research]].

** Validated Is Not Deployed — Flip the Switch or Delete the Parameter

When new infrastructure is built alongside old infrastructure (hybrid
paths, dual pipelines, feature-flagged switchovers), a common pattern
emerges: the new path is validated (all tests pass) and the track is
declared complete — but the parameter controlling the switch defaults to
the OLD path. The new code runs in test configurations but never in
production. This creates four compounding costs:

1. *Invisible technical debt*: the old path can't be deleted because
   it's still the production default
2. *False confidence*: the track's PIR says "switchover complete" but
   the switchover is parameterized, not actual
3. *Deferred cleanup accumulates*: each subsequent track inherits
   the dual paths and must work around both
4. *Context loss*: when the switch is finally attempted (tracks later),
   the implementation context is gone and edge cases resurface

*The rule*: when a validation phase proves the new path works (full
suite GREEN), the NEXT phase MUST either (a) flip the default to the
new path and re-validate, or (b) delete the parameter entirely if the
old path is no longer needed. A parameter that defaults to the old
path is not a "safety net" — it's an admission that the switchover
isn't complete. Name it honestly: "Phase N: validated (not deployed)."

*Discovered in*: PPN Track 2 Phase 6g. ~use-tree-parser?~ validated
at ~#t~ (383/383 GREEN) but defaulted to ~#f~. The tree parser — the
entire point of the track — never ran in ~process-file~ or the default
~process-string-ws~ path. Three follow-on commits were needed to
clean up what should have been one phase: flip the switch, validate,
delete the parameter.

*Red-flag phrases*: "defaults to #f for safety", "validated but keeping
the old path as fallback", "can be enabled via environment variable",
"opt-in for now." These all mean: not actually deployed.

*Anti-pattern*: ~use-new-reader?~, ~use-tree-parser?~, ~use-propagator-X?~
parameters that default to ~#f~. Each one represents a completed
architecture that isn't running. The parameter itself is the debt.

** "Pragmatic" Is a Rationalization for Incomplete

When the word "pragmatic" appears in a design document or commit message
to justify keeping both old and new code paths alive, it is almost always
a signal that the Completeness principle is being violated. Audit the
specific claims:

- "The merge is a *pragmatic* bridge" → the new path can't handle all
  cases. What cases are missing? Can they be addressed in the current
  track? If not, is the gap explicitly tracked with a concrete plan?
- "A *pragmatic* approach is to keep preparse for X" → X is a
  responsibility not yet migrated. Is migration blocked by a genuine
  dependency, or by effort avoidance? Genuine dependency = acceptable
  deferral. Effort avoidance = Completeness violation.
- "The *pragmatic* path is ..." → rephrase as "the INCOMPLETE path is
  ..." and see if the sentence still feels acceptable.

*The test*: replace "pragmatic" with "incomplete" in every design
decision. If the rephrased sentence is alarming, the original decision
needs revisiting. If it's genuinely acceptable (e.g., "the incomplete
path is appropriate because X is blocked by unbuilt Track 4
infrastructure"), then the deferral is principled and should say
"deferred to Track 4 because [specific dependency]" not "pragmatic."

*Discovered in*: PPN Track 2 PIR analysis. The merge strategy was
described as "pragmatic" 4 times in the design doc. Rephrased: "the
INCOMPLETE path maintains two code paths because the tree parser doesn't
handle data/trait/impl." This reframing immediately clarifies that the
gap is the tree parser's form coverage, and the fix is extending it —
not accepting dual paths as an architecture.

*The discipline*: the word "pragmatic" is banned from design documents
and commit messages when used to justify keeping old code paths. Use
"incomplete (deferred to Track N because [reason])" instead. This forces
the deferral to be specific, tracked, and justified — rather than
rationalized.

** CALM Requires Fixed Topology — Dynamic Topology Requires Stratification

The CALM theorem (Consistency As Logical Monotonicity) guarantees that
monotone computations over a *fixed* lattice converge to the same
fixpoint regardless of evaluation order. This is the theoretical
foundation for BSP scheduling, parallelism, and order-independent
propagation. The key word is FIXED.

If a propagator's fire function creates new cells or propagators
(dynamic topology), the lattice itself changes during computation.
CALM says nothing about convergence over a changing lattice. In
practice, BSP scheduling silently drops structural changes (new
cells, propagators) because =fire-and-collect-writes= only diffs cell
VALUES. The structural changes exist in the returned network but are
discarded during bulk merge. This produces wrong results — not errors,
not crashes, just silently wrong answers.

*Discovered in*: BSP scheduler vs SRE subtype decomposition.
=sre-decompose-generic= creates sub-cells and sub-propagators inside
fire functions. Under DFS, this works because the full returned
network is used. Under BSP, the structural changes are lost:
=PVec Int <: PVec Nat= returned =true= (should be =false=) because
the decomposition that would detect the violation never materialized.

*The invariant*: Within a stratum, the network topology is FIXED.
Propagator fire functions may only write cell values
(=net-cell-write=). Creating cells (=net-new-cell=) or propagators
(=net-add-propagator=) requires a stratum boundary. A dynamic guard
(=current-bsp-fire-round?=) enforces this — violation produces an
immediate error rather than a silent wrong result.

*The architectural pattern*: Separate two fixpoints.
1. *Topology fixpoint*: what cells and propagators exist. The set of
   cells/propagators grows monotonically (a lattice under subset
   ordering). Topology changes are processed in a dedicated stratum.
2. *Value fixpoint*: what values cells hold. BSP-safe, parallelizable,
   order-independent. Runs within a stratum over the fixed topology.

The cycle: value stratum (BSP) → topology stratum (sequential:
process decomposition requests, create cells/propagators) → value
stratum (BSP over expanded topology) → repeat until neither changes.

*Implications*:
- *SRE decomposition*: Must emit decomposition REQUESTS as cell
  values, not directly create cells/propagators. A topology stratum
  reads requests and builds structure.
- *Hypergraph rewriting (PPN, PRN)*: Rewrite rules that add nodes or
  edges are topology changes. They must be stratified — rewrite
  application in one stratum, value propagation in the next.
- *Tabling/memoization*: If tabling creates new cells for cached
  results, that's a topology change requiring stratification.
- *Any future construct* that dynamically extends the network: the
  CALM guard will catch it at the API level, preventing silent
  incorrectness.

This is the Correct-by-Construction principle applied to scheduling:
the guard makes the invariant structural (error on violation) rather
than maintained by discipline (hoping fire functions don't add cells).

See commit =af35f5e= (CALM topology guard implementation).

** Racket Parallelism Has a Hard ~8μs Floor — Measure Before Parallelizing

Racket's cross-thread communication has a measured ~8μs floor (OS
thread wakeup cost). Sync-pool crossover against sequential execution
for ~0.5μs per-propagator work is N≈256. No Racket-level mechanism
beats this floor:

- Spin-wait: workers sleep during merge phase — spin window too short
- Sync events: 15–20μs dispatch per call
- Async-channel: ~30μs per call
- Spin-poll done-semaphores: 2400μs catastrophic busy-spin

*The arithmetic*: parallelism only helps when per-propagator work
exceeds a multiple of the wakeup cost. Below ~16× the floor
(~125μs per propagator), thread overhead dominates — parallelism is
a regression, not a speedup.

*Methodology — measure before parallelizing*:
- Measure actual per-propagator work (μs) at typical workloads
- Benchmark sync-pool vs sequential at multiple N values
- Don't trust design-time crossover estimates — Phase 0c estimated
  thread crossover at N≈128 with 1.9× speedup; actual is N≈256 and
  payoff only at N≥512
- Run 4+ alternative approaches before concluding (spin/sync/async/poll)

*Memory cost is a separate axis, measure it too*: parallel
infrastructure allocates even when unused (worker pool threads,
channels, per-worker state). Racket's GC pauses also affect
cross-thread coordination — a GC in one worker stalls others at
the barrier. Measure allocation rates AND retention, not just
wall-clock:
- Worker pool baseline: memory for K threads + channels (regardless of use)
- Per-dispatch allocation: channel write + thread wake + CHAMP fork
- GC amplification: parallel rounds have K× allocation pressure
  vs sequential for the same work

*Implications*:
- Parallelism in Phase 0 (Racket) pays off only for specific workloads
  (N≥256, per-propagator work ≥ ~5μs)
- True parallelism gains are a compiler/runtime problem, not a
  library problem — self-hosted Prologos with lighter threads shifts
  this boundary
- Parallel infrastructure should be threshold-gated: install the
  infrastructure, but don't use it below measured crossover
- Architecture design should NOT assume free parallelism even if the
  theoretical structure is parallelizable

*Discovered in*: BSP-LE Track 2B Phase 2d (streaming BSP
investigation — 4 approaches benchmarked exhaustively, all worse
than sync pool). Full decomposition in Track 2B PIR §7 (Where We Got
Lucky), §8.1 (surprise), §12.5 (what we learned), §16.6. Strategic
decision: wait for self-hosting. Research note:
=docs/research/2026-03-26_PARALLEL_PROPAGATOR_SCHEDULING.md=.

names, present **Wide vs Narrow** as a decision point to the user.

- *Wide* = comprehensive cleanup atomically (retire all
  audit-discovered surfaces in one sub-phase)
- *Narrow* = retire only design-named surfaces; defer
  audit-discovered surfaces to follow-up sub-phase or future phase

   "stick to the design; breaking into multiple subphases is fine."

2. *S2.e-v Wide retirement*: design named 2 functions; audit
   revealed 6. User-directed Wide ("Wide + Migrate, I agree

- The cost difference is usually small — audit-discovered surfaces

- Audit-discovered surfaces require fundamentally different

- Capture-gap (above) names the failure mode of Narrow without

- Wide is the principled response that prevents drift

** Sed-Deletion of Parameterize Bindings Is Error-Prone for Last-Binding Lines

When sed-deleting parameterize bindings, MUST use 2-pass pattern:
substitution preserves the closing paren for last-binding-in-block;
deletion removes plain bindings. Single-pass deletion breaks the
parameterize blocks where the deleted line had the closing paren.

*The bug*:

#+begin_src bash
# WRONG — matches both `(make-hasheq)]` (regular) AND
# `(make-hasheq)])` (last-binding-in-parameterize-block).
# For the latter, deletes the closing `)` → unbalanced parameterize.
sed -i '' '/\[X (make-hasheq)\]/d' file.rkt
#+end_src

*The fix (2-pass pattern)*:

#+begin_src bash
sed -i '' \
  -e 's/^\([[:space:]]*\)\[X[[:space:]][[:space:]]*(make-hasheq)\])/\1)/' \
  -e '/\[X[[:space:]][[:space:]]*(make-hasheq)\]/d' \
  file.rkt
#+end_src

The first line (substitution) handles `(make-hasheq)])` — replaces
the entire binding (preserving leading whitespace via captured `\1`)
with just the closing `)`. The second line (deletion) handles
`(make-hasheq)]` (no trailing paren).

*Verification before batch*: test on 1 file via copy to /tmp, run
sed, then ~tools/check-parens.sh~ to verify balance. Don't apply
batch sed to many files without single-file verification.

*Pattern observed in PPN 4C S2.e-iv-c (2026-04-25)*: initial
single-pass sed broke 94 parameterize blocks. Detection only via
FULL SUITE (test-session-runtime-03.rkt read-syntax error;
targeted tests on the metavar-store/universe domains all PASSED
because the broken parameterize blocks were in OTHER test files).
Recovery via ~git checkout HEAD --~ revert + smart 2-pass sed.

*Why this matters*: large-scale test fixture surgery is high-leverage
when pattern uniformity is verified (audit-first), but high-risk
when sed patterns have edge cases. The 2-pass pattern removes the
edge case. Test-mode isolation (1 file via copy + check-parens)
catches the bug before batch.

*Operational rule*:
1. Audit-first to verify 100% pattern uniformity across target files
2. Test sed on 1 file copy in /tmp before batch
3. Use 2-pass pattern (substitution + deletion) for parameterize
   bindings
4. Run check-parens after batch to verify balance
5. Run FULL SUITE as regression gate (NOT just targeted tests —
   the bug may manifest in unrelated test files via cascading
   read-syntax errors)

*Reinforces existing rule*: ~workflow.md~ "full suite as regression
gate when touching code is RULE." This sed-mistake is a strong
data point for that rule — targeted tests on the directly-affected
files all PASSED while the regression was real.

*Discovered in*: PPN 4C Phase 9+10+11 addendum Step 2 S2.e-iv-c
(2026-04-25). 1 high-confidence data point — codifying as
operational rule because the failure mode is structural (parameterize
bindings universally have this shape) and the fix is mechanical
(2-pass pattern).
