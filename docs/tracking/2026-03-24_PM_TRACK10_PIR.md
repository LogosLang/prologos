## 1. Stated Objectives

From the design document:
- Eliminate the ~20s cold-start prelude loading tax via `.pnet` serialized module cache
- Achieve <200s test suite wall time (down from ~240s)
- Provide three-layer fork model for test isolation (prelude → test-file → test-case)
- Absorb PM 8F deferred items (CHAMP fallback, defaults at solve-time)
- Establish foundation for Track 10B (zonk elimination, test-granular scheduling)

**Objectives evolved during implementation**: The #lang prologos drop (Phase 5) was not in the original design — it emerged from the Phase 4 finding that expander.rkt was the sole CHAMP fallback consumer.

## 5. What Went Well

1. **Pre-0 benchmarking changed the design fundamentally.** D.1 assumed parameterize overhead was the bottleneck (based on vision). Pre-0 showed: 3.4μs per parameterize (negligible). Re-elaboration = 20s (the real cost). This pivoted the entire approach from "eliminate parameterize" to "serialize and cache modules." Without Pre-0, we'd have spent days optimizing the wrong thing.

2. **The `struct->vector` + tag dispatch mechanism was zero-blast-radius.** No struct definition changes. No pipeline exhaustiveness burden. 40/40 modules round-tripped on first validation. The D.3 critique's concern about extraction overhead was measured at 3ns (negligible). This was the right serialization choice.

3. **The comprehensive registry audit (`e25061a`) broke the whack-a-mole cycle.** Five incremental fixes each revealed the next missing registry. The audit question "how many registries ARE there?" (not "which one is missing NOW?") found 28, identified 17 non-zero, serialized all of them in ONE commit. This is the Completeness principle applied to process.

4. **The diagnostic protocol proved its value twice.** Phase 2 (registry whack-a-mole → comprehensive audit) and Phase 3d (process-string scoping → box mutation identified in one narrowing cycle). Now codified in workflow.md.

5. **The `#lang prologos` drop was the right pivot.** 1315 lines deleted. No impact on active workflows. Removed the sole blocker for CHAMP fallback removal. This is the Completeness principle applied to architecture: remove dead compatibility layers rather than upgrading them.

6. **Foreign function provenance (Option A) was the polyglot-correct choice.** `(module-path . binding-name)` pairs are host-agnostic. dynamic-require is the standard Racket re-linking pattern. This generalizes to future polyglot hosts without redesign.

## 6. What Went Wrong

1. **5 wrong assumptions in the D.1 design.** (a) Parameterize is the bottleneck (it's 3.4μs). (b) ~225 zonk sites (actually 31). (c) fasl available in Racket 9.0 (it's not). (d) All modules serialize cleanly (22/40 have foreign procs). (e) 7 registries is all load-module manages (actually 17+). Each was corrected by data, not discovered by analysis. The lesson: the combined Stage 2/3 format let VISION contaminate the AUDIT.

2. **The registry whack-a-mole consumed ~4 hours.** Each fix (preparse closures, coercion refs, trait registry, capability wrappers, marshal stubs) passed 380/382 then a NEW test failed. The comprehensive audit approach should have been applied FIRST, not after 5 iterations. This anti-pattern is now named and codified.

3. **The `process-string` box mutation leak was a deep bug.** 252 test files used bare `parameterize` (no prop-net-box) → `process-string` MUTATED the caller's box → elab-network leaked between test cases. The fix was 3 lines (parameterize in process-string itself), but finding it required the full diagnostic protocol. This bug existed since Phase 1a but was invisible until fork-integrated tests exposed it.

4. **Stale process detection was a recurring time sink.** Multiple test runs were killed or produced stale results because previous Racket processes lingered. We needed to manually `kill` processes and verify CPU activity. A process cleanup step should be automatic before test runs.

## 7. Where We Got Lucky

1. **The gensym round-trip issue was caught by the 40/40 validation.** If we'd only tested 18/40 clean modules and assumed the rest were fine, the 4 polymorphic modules (with gensym metas) would have failed in production. The comprehensive validation (40/40) caught it preemptively.

2. **The prop-network sentinel in pnet-serialize.rkt was added "as defense-in-depth" and turned out to be essential.** The process-string scoping bug caused elab-network wrappers to appear in serialized data. The sentinel caught them silently. Without it, the deserialized data would have contained corrupt network references.

## 8. What Surprised Us

1. **Module state had 28 parameter-based registries.** The design assumed 7 (the ones load-module explicitly captures). The audit found 17 non-zero in the prelude. The other 11 are managed by Racket's parameter scoping (inherited by modules during loading, not explicitly captured). This distinction between "managed" and "inherited" registries was not in any design document.

2. **The CHAMP fallback was needed for ONE consumer: expander.rkt.** The entire CHAMP fallback infrastructure (6 code sites, dual-read logic, worldview-aware filtering) existed for a 261-line compile-time expander that runs in Racket's phase 1. Removing that one consumer eliminated the need for the entire fallback.

3. **Serialization IS dynamic linking.** Every closure in serialized data needs a reconstruction strategy. The strategy is always: store DATA from which the closure derives, derive on load. This pattern recurred 6 times (foreign procs, preparse expanders, coercion functions, marshallers, capability wrappers, preparse-macro templates).

4. **The `process-string` box mutation bug had been present since Phase 1a but was invisible.** Without fork-integrated tests, the mutation leaked between test cases but produced CORRECT results (each test's elaboration happened to work with the leaked state). The fork exposed it because the leaked state was from a DIFFERENT test's elaboration.

## 9. Architecture Assessment

The `.pnet` serialization architecture is clean:
- One serialization mechanism (struct->vector + tag dispatch) for all struct types
- Named-reference re-linking for all closures (foreign, preparse, coercion, marshal)
- Gensym preservation via symbol$$N tagging
- Feature flag (`current-use-pnet-cache?`) for clean rollback
- Staleness detection via source hash

The fork model is sound but incomplete:
- `fork-prop-network` provides O(1) structural sharing
- `with-forked-network` integrates into test-support
- BUT: 306 `with-fresh-meta-env` sites still use the old parameterize pattern
- Full migration to fork-based isolation is Track 10B scope

The `#lang prologos` removal was architecturally significant:
- Single compilation pipeline (process-file only)
- No phase-1/phase-0 boundary to bridge
- Simplifies self-hosting target (one pipeline to self-host, not two)

## 10. What This Enables

1. **Track 10B**: CHAMP fallback removal (sole consumer deleted), zonk elimination (expander's unconverted sites deleted), id-map elimination, test-granular scheduling.

2. **LSP instant startup**: `.pnet` cache means prelude loads in 108ms (memory cache) instead of ~20s. Restarting the language server is nearly instant.

3. **Self-hosting trajectory**: Single compilation pipeline means the self-hosting target is `process-file` only. No Racket phase-1 expander to replicate.

4. **NTT serialize/deserialize forms**: Track 10 revealed that serialization needs first-class typing in NTT. Proposed `serialize`/`deserialize` forms added to NTT syntax design.

5. **Polyglot foreign function framework**: The `(module-path . binding-name)` provenance pattern generalizes to any host language. Future Python/JS/C foreign functions use the same `dynamic-require` re-linking model.

## 12. What Would We Do Differently

1. **Start with the comprehensive audit, not the incremental approach.** The registry whack-a-mole would have been avoided entirely if we'd asked "how many registries exist?" before "which registry is this test missing?" The diagnostic protocol codifies this, but we learned it the hard way.

2. **Separate Stage 2 audit from Stage 3 design.** The combined format let vision contaminate the audit (225 zonk sites from vision, 31 from reality). A separate audit document forces data-grounded scoping. This lesson is now codified in DESIGN_METHODOLOGY.org.

3. **Measure serialization round-trip time in Pre-0.** The D.4 critique correctly noted the <100ms target was unmeasured. We hit 108ms (close) but didn't know until post-implementation. Pre-0.5 benchmark should have measured it.

## 14. What We Learned About the Problem

**Module loading is dynamic linking, not compilation.** The traditional view: "compile module from source, cache the compiled form." The propagator view: "elaborate module on the network, serialize the network state, deserialize and re-link on cold start." The serialization format IS the module's interface — it carries types, registries, definition locations, foreign function references. The re-linking on deserialize IS the import mechanism.

**Closures in data structures are the universal serialization blocker.** Every time serialization fails, the root cause is a closure stored where data should be. The fix is always the same: store the DATA from which the closure derives, derive on load. This pattern is so fundamental it should be a design principle: "data at rest, closures derived on demand."

## 15. Are We Solving the Right Problem?

Yes. The performance regression analysis (March 20: 185s→240s from First-Class Paths) identified the root cause as per-test prelude re-elaboration through a heavier pipeline. Track 10's `.pnet` cache directly addresses this: modules are elaborated ONCE and cached. The 44% improvement (240→134s) validates the approach.

The `#lang prologos` drop reframes the problem further: instead of "how do we make two compilation pipelines work with the propagator network?", the question becomes "why do we have two pipelines?" The answer: we don't need two. One pipeline, one serialization format, one compilation model.

### Lessons for Codification

1. **"Data at rest, closures derived on demand"** → DEVELOPMENT_LESSONS.org. 6 instances in one track. When storing state (registries, env-snapshots, module state), store the data reference, not the closure. Derive closures at load time from named references.

2. **"Remove dead compatibility layers"** → watching. 1 instance. If a subsystem exists solely for compatibility with a workflow you've replaced, removing the subsystem is simpler and more principled than upgrading it.
