* Purpose

This document is the single reference for ALL toplevel forms in Prologos
that carry keyword metadata (the "config language"). Each form's available
keywords, their types, defaults, and implementation status are documented
here.

Consult this document to answer: "What keywords does =spec= accept?",
"What metadata can I put on a =trait=?", "Is =:laws= implemented?"

Forms are organized by domain. Implementation status uses:
- *Implemented* (I): Parsed, elaborated, type-checked in WS-mode
- *Parsed only* (P): Sexp parsing exists, WS-mode or elaboration incomplete
- *Designed* (D): In a design document, not yet in the parser
- *Research* (R): Proposed in research/NTT, no grammar entry yet

* Functional Core

** =spec= — Function Type Signature

Declares the type signature for a function. The primary carrier of
metadata in the language.

#+begin_example
spec sort {A : Type} [List A] -> [List A]
  :where [Ord A]
  :doc "Sorts a list in ascending order"
  :examples
    - [sort '[3 1 2]] => '[1 2 3]
  :properties [sortable-laws A]
#+end_example

| Keyword | Type | Default | Status | Description |
|---------+------+---------+--------+-------------|
| =:where= | =[Constraint ...]= | none | I | Trait constraints on type parameters |
| =:doc= | =String= | none | I | Documentation string |
| =:examples= | =[(input => output) ...]= | none | I | Input/output pairs for auto-test generation |
| =:properties= | =[PropertyRef ...]= | none | P | References to =property= groups that hold for this function |
| =:see-also= | =[Symbol ...]= | none | P | Related function names |
| =:pre= | =Expr= | none | D | Precondition expression (Extended Spec Phase 3) |
| =:post= | =Expr= | none | D | Postcondition expression (Extended Spec Phase 3) |
| =:refines= | =Symbol= | none | D | Reference to a verified/refined implementation |
| =:instrument= | =Bool= | =false= | D | Enable contract enforcement at runtime |
| =:implicits= | ={A B : Type}= | inferred | I | Explicit implicit binder specification |

*Source*: =macros.rkt= ~line 3086 (=parse-spec-metadata=), struct =spec-entry= ~line 439.
*Grammar*: =grammar.org= lines 806-877. *Design*: Extended Spec Design (2026-02-22).

** =defn= — Function Implementation

Implements a function declared by =spec=. Carries no metadata of its own
(metadata lives on =spec=). Multi-arity dispatch via =|= patterns.

#+begin_example
defn sort
  | '[]        -> '[]
  | [cons h t] -> [insert h [sort t]]
#+end_example

*Status*: I. *Source*: =macros.rkt=.

** =def= — Value Binding

Top-level value binding. Minimal metadata (type annotation only).

#+begin_example
def greeting : String := "hello"
def answer := 42                    ;; type inferred
#+end_example

| Keyword | Type | Default | Status | Description |
|---------+------+---------+--------+-------------|
| =:= | =Type= | inferred | I | Type annotation |
| =:=​== | =Expr= | required | I | Binding operator (value) |

*Status*: I. *Source*: =macros.rkt=.

* Trait System

** =trait= — Type Class Declaration

Declares a type class with method signatures and optional laws.

#+begin_example
trait Functor {F : Type -> Type}
  :laws [functor-laws F]
  spec fmap {A B : Type} [A -> B] -> [F A] -> [F B]
#+end_example

| Keyword | Type | Default | Status | Description |
|---------+------+---------+--------+-------------|
| =:laws= | =[PropertyRef ...]= | none | P | References to =property= groups defining algebraic laws |
| =:where= | =[Constraint ...]= | none | D | Constraints on type parameters (superclass constraints) |
| =:extends= | =[Trait ...]= | none | D | Superclass hierarchy (alternative to =:where=) |
| =:doc= | =String= | none | P | Documentation |

*Note*: =:laws= is parsed and stored but not yet verified against
implementations. When =property= elaboration is complete, =impl= blocks
will be checked against trait laws automatically.

*Status*: I (core); =:laws= is P; =:where=, =:extends= are D.
*Source*: =macros.rkt=. *Design*: First-Class Traits (2026-03-09).

** =impl= — Trait Implementation

Implements a trait for a specific type. Method bodies are pattern-matched
by name against the trait's =spec= signatures.

#+begin_example
impl Eq Color
  eq?
    | x x -> true
    | _ _ -> false
#+end_example

| Keyword | Type | Default | Status | Description |
|---------+------+---------+--------+-------------|
| (none currently) | — | — | — | Method names matched positionally |

*Note*: The implicit-=defn= syntax (method name directly, no =defn=
keyword) is the target ergonomic form. Currently requires =defn= in sexp
mode.

*Status*: I. *Source*: =macros.rkt= struct =impl-entry= ~line 6080.

* Property System

** =property= — Named Proposition Group

Declares a group of named propositions (laws) that can be referenced by
=spec :properties= and =trait :laws=.

#+begin_example
property sortable-laws {A : Type}
  :where [Ord A]
  :includes [permutation-laws A]
  - :name "idempotent"
    :forall {xs : [List A]}
    :holds [eq? [sort [sort xs]] [sort xs]]
  - :name "length-preserving"
    :forall {xs : [List A]}
    :holds [eq? [length [sort xs]] [length xs]]
#+end_example

| Keyword | Type | Default | Status | Description |
|---------+------+---------+--------+-------------|
| =:where= | =[Constraint ...]= | none | P | Constraints on type parameters |
| =:includes= | =[PropertyRef ...]= | none | P | Compose other property groups (set union) |

*Clause-level keywords*:

| Keyword | Type | Default | Status | Description |
|---------+------+---------+--------+-------------|
| =:name= | =String= | required | P | Human-readable label for this proposition |
| =:forall= | ={bindings}= | none | P | Universally quantified variables |
| =:holds= | =Expr= | required | P | Boolean expression that must hold |

*Status*: P (sexp parsing complete, WS-mode preparse not wired, elaboration
hooks incomplete). *Source*: =macros.rkt= struct =property-entry= ~line
7343. *Design*: Extended Spec Design (2026-02-22) Part IV.

* Type Abstraction

** =functor= — Named Type Abstraction

Declares a named, composable type abstraction with category-theoretic
metadata. Used for optics (Lens, Prism), transducers (Xf), and other
compositional patterns.

#+begin_example
functor Xf {A B : Type}
  :doc "A transducer: transforms A-reductions into B-reductions"
  :unfolds <(S :0 Type) -> [S -> B -> S] -> S -> A -> S>
  :compose xf-compose
  :identity id-xf
  :laws [transducer-fusion-laws A B]
  :see-also [map-xf filter-xf take-xf]
#+end_example

| Keyword | Type | Default | Status | Description |
|---------+------+---------+--------+-------------|
| =:unfolds= | =Type= | required | P | Transparent expansion type |
| =:compose= | =Symbol= | none | P | Composition function (=functor . functor -> functor=) |
| =:identity= | =Symbol= | none | P | Identity element |
| =:laws= | =[PropertyRef ...]= | none | D | Property groups (composition laws) |
| =:doc= | =String= | none | P | Documentation |
| =:see-also= | =[Symbol ...]= | none | P | Related functors |
| =:transforms= | =Functor= | none | D | Natural transformation target |
| =:adjoint= | =Functor= | none | R | Adjoint functor (far future) |

*Status*: P (sexp parsing complete, WS-mode preparse not wired).
*Source*: =macros.rkt= struct =functor-entry= ~line 7615.
*Design*: Extended Spec Design (2026-02-22) Part VII.

* Data Schema

** =schema= — Typed Record Shape

Declares a typed record schema with per-field metadata.

#+begin_example
schema Config
  :host String :default "localhost"
  :port Nat    :default 8080N :check [pos?]
  :debug Bool  :default false
  :closed
#+end_example

| Keyword (schema-level) | Type | Default | Status | Description |
|-------------------------+------+---------+--------+-------------|
| =:closed= | flag | open | P | Reject unknown fields |

| Keyword (field-level) | Type | Default | Status | Description |
|-----------------------+------+---------+--------+-------------|
| =:default= | =Expr= | none | P | Default value for field |
| =:check= | =Fn= | none | D | Validation predicate |

*Status*: P (sexp parsing complete, WS via implicit-map works partially).
*Source*: =macros.rkt= struct =schema-field= ~line 728.
*Design*: Schema/Selection Design (2026-03-02).

** =selection= — Field-Level Access Policy

Declares required/provided fields for a schema, enabling row-polymorphic
access control.

#+begin_example
selection user-login :from User
  :requires [:id :email :password]
  :provides [:auth-token]
  :includes [base-auth-selection]
#+end_example

| Keyword | Type | Default | Status | Description |
|---------+------+---------+--------+-------------|
| =:from= | =Schema= | required | D | Parent schema |
| =:requires= | =[Keyword ...]= | none | D | Required field paths |
| =:provides= | =[Keyword ...]= | none | D | Guaranteed field paths |
| =:includes= | =[Selection ...]= | none | D | Compose other selections |
| =:properties= | =[PropertyRef ...]= | none | D | Schema-level invariants |

*Status*: D (design complete, not in grammar).
*Design*: Schema/Selection Design (2026-03-02) §3.2-4.12.

* Session Types

** =session= — Protocol Type Declaration

Declares a session type (communication protocol) with WS operators.

#+begin_example
session login-protocol
  :doc "Two-phase login with error handling"
  ! UserCredentials
  ? Bool
  | true  -> ! AuthToken . end
  | false -> ! ErrorMsg  . end
#+end_example

| Keyword | Type | Default | Status | Description |
|---------+------+---------+--------+-------------|
| =:doc= | =String= | none | P | Documentation |
| =:deprecated= | =String= | none | D | Deprecation notice |
| =:throws= | =[Type ...]= | none | D | Exception types |
| =:linear= | flag | linear | D | Linearity constraint (default: linear) |

*WS Operators* (within session body):

| Operator | Meaning | Status |
|----------+---------+--------|
| =!= | Send | P |
| =?= | Receive | P |
| =\vert= | Branch/choice | P |
| =.= | Continuation (single-line) | P |
| =end= | Termination | P |
| =&= | Selection (far future) | D |

*Status*: P (sexp IR complete, WS operators tokenized, declaration form
incomplete). *Design*: Session Type Design (2026-03-03).

* Logic Engine

** =defr= — Named Relation

Defines a relation (logic program) with facts and rules.

#+begin_example
defr parent [?x ?parent]
  || facts ||
    ('alice' 'bob')
    ('bob' 'charlie')
  &> rules &>
    parent ?x ?z &> parent ?z ?y &> (! [parent ?x ?y])
#+end_example

*WS Operators*:

| Operator | Meaning | Status |
|----------+---------+--------|
| =\vert\vert= | Fact block sentinel | P |
| =&>= | Rule clause separator | P |
| =\vert= | Goal conjunction | P |
| =!= | Assert (cut) | P |

*Status*: P (sexp parsing complete, WS operators tokenized, elaboration
incomplete). *Design*: Logic Engine Design (2026-02-24), Relational
Surface (2026-02-25).

** =solver= — Solver Configuration

Declares a named solver with scheduling and resource configuration.

#+begin_example

#+end_example

| Keyword | Type | Default | Status | Description |
|---------+------+---------+--------+-------------|
| =:scheduler= | =Symbol= | =:auto= | P | Scheduling algorithm (=bsp=, =gauss-seidel=, =:auto=) |
| =:fuel= | =Nat= | =1000= | P | Maximum reduction steps |
| =:strategy= | =Symbol= | =:dfs= | P | Search strategy (=dfs=, =bfs=, =round-robin=) |

*Status*: P (sexp parsing exists, WS not wired, dispatch not complete).
*Design*: Logic Engine Design (2026-02-24) §7.6.2.

** =solve= / =solve-one= / =solve-with= — Query Forms

Executes a logic query against the current relation database.

#+begin_example

#+end_example

*Status*: P (sexp parsing complete, WS not designed).

** =explain= / =explain-one= — Provenance Queries

* Design Pattern: The Config Language

All metadata-carrying forms follow the same structural pattern:

#+begin_example
<form-keyword> <name> [<type-params>] [<positional-args>]
  :<keyword1> <value1>
  :<keyword2> <value2>
           <continuation-value>
  <body>
#+end_example

*Parsing infrastructure*:
- =keyword-like-symbol?= (macros.rkt ~line 330): Classifies =:foo= tokens
- =flatten-ws-kv-pairs= (macros.rkt ~line 2272): Normalizes WS-reader
  sub-lists for keyword-value forms
- =rewrite-implicit-map= (macros.rkt ~line 2452): Detects keyword-headed
  children, wraps as =$brace-params=
- =parse-spec-metadata= (macros.rkt ~line 3086): Extracts =:key value=
  pairs into hasheq
- =split-session-metadata= (macros.rkt ~line 1258): Extracts leading
  metadata from body items

*Design principles*:
- *Minimal syntax*: The 80% use case requires no annotation. Safe defaults
  are implicit; exceptions are explicit.
- *Block grouping*: Multiple values under one keyword use indentation, not
  repeated keyword prefixes (e.g., =:where= with indented constraints).
- *Forward compatibility*: Unrecognized keywords are stored but not acted
  upon, allowing future extensions without breaking existing code.
- *Progressive disclosure*: Simple forms look simple. Metadata reveals
  complexity gradually. Layer 1 users never see =:laws= or =:where=.
