### Research: Recovering Order-Dependent Effects on Propagator Substrate (commit `548cf5d`)

Created `docs/tracking/2026-03-06_EFFECTFUL_PROPAGATORS_RESEARCH.md` (~891 lines).

Responds to PIR §4a challenge: if propagators can't order side effects, can a
multi-layered architecture recover this capability — as the logic engine
recovered choice-point semantics and non-monotonic negation on the same substrate?

**External research surveyed**:
- CALM Theorem (Hellerstein): IO effects are non-monotonic → coordination required
- LVars (Kuper & Newton): Threshold reads + freeze = quasi-determinism; effect-level indexing
- Timely Dataflow (McSherry): Logical timestamps on a partially ordered set
- CRDTs: Separate data plane (monotone) from ordering plane (causal context)
- Radul & Sussman: Propagator designers explicitly excluded non-commutative effects
- BloomL: Monotone functions + coordination at points of order
- Algebraic Effects: Computation/handler separation
- Flix: Lattice-based fixed points for practical languages

**Three candidate architectures**:
- **A (Stratified Barriers)**: Current approach formalized. Verify → execute → verify.
- **B (Timestamped Effect Cells)**: Effect descriptors accumulate monotonically with causal metadata; execute in timestamp order at barriers.
- **C (Reactive Effect Streams)**: Topological scheduling of effect DAGs with freeze semantics.

**Key principle**: "The propagator substrate handles convergent reasoning about effects. A control layer handles sequential execution of effects."

### Research: Session Types as Causal Timelines for Effect Ordering (commit `5fbe496`)

Created `docs/tracking/2026-03-06_SESSION_TYPES_AS_EFFECT_ORDERING.org` (~851 lines).

Deep investigation of §9c from the effectful propagators research. The central
thesis: session types are not merely protocol specifications — they are **causal
clocks**. Each position in a session type's continuation chain is a
causally-ordered event.

**Core formal contribution — Galois connection (α, γ)**:

```
α : Session → EffectPosition      (extract causal position from session state)
γ : EffectPosition → Session       (reconstruct remaining protocol from position)
```

Both monotone. Adjunction guarantees soundness: session fidelity ⇒ effect ordering
correctness. At deeper level: both lattices are effect quantales; the Galois
connection is a quantale morphism.

**Grounding in existing code**:
- Session lattice (`session-lattice.rkt`): flat at top level, chain structure within
- Session advancement (`session-runtime.rkt`): chain of propagator cells mirrors session positions
- `compile-live-process` walk order = session type order = effect order (already true today, but implicit)

**Architecture D (Session-Derived Effect Ordering)**:
- Ordering source is *intrinsic* (session types), not external (timestamps/barriers/DAGs)
- Soundness follows from session fidelity theorem, not verification
- Session advancement = Lamport clock per channel
- Multi-channel = vector clock (partial order for concurrent effects)
- Truly concurrent effects (different channels) don't need ordering

**Algebraic structure**:
- Parameterised monad view: session types ARE pre/post state indices (Atkey, Orchard-Yoshida)
- Sequential composition = bind matching postcondition to precondition
- Effect quantale structure shared by session lattice and effect position lattice
- Category-graded monad unification (Katsumata et al.)

**Formalization roadmap (D1-D5)**:
1. Effect position lattice (eff-bot/eff-pos/eff-top)
2. Session-effect bridge propagator
3. Effect descriptor cells
4. Effect handler (execute in position order)
5. Vector clock for multi-channel

**Open questions**:
- Recursive sessions → infinite timelines (modular positions?)
- Cross-channel data deps (process structure provides, not session types alone)
- Unsessioned effects (SysCap top-level eval → fallback to Architecture A/B)
- Session-effect duality: dual endpoints have same positions, complementary effects

### Design Discussion: Resolving Open Questions + Architecture Decision

Discussion session exploring the open questions from the session types research.
Key findings captured in updated research doc and new principles document.

**Cross-channel data dependencies resolved (§9c)**:
- Transitive closure of ordering edges (session edges + data-flow edges) is a **monotone fixed point**
- Propagator can incrementally compute the complete partial order — this is propagator-native
- Cross-channel data flow creates ordering edges; transitive closure propagates them

**ATMS for branching effect orders**:
- `proc-case` branches create different data-flow patterns → different orderings
- ATMS maintains per-branch ordering hypotheses (same pattern as logic engine `amb`)
- Resolved when branch is chosen; unchosen worldviews become nogoods

**`rel`/`proc` structural parallel**:
- Both engines: propagator network (monotone) + ATMS (hypothetical) + control layer (non-monotone)
- `rel`: variables → equality constraints → constraint closure → stratification (negation)
- `proc`: sessions → ordering constraints → ordering closure → effect handler (IO)
- The **Layered Recovery Principle** is the common theory

**Architecture decision: A + D** (B subsumed):
- D for session-typed IO (ordering from session fidelity — a theorem)
- A for unsessioned IO (walk-based ordering — simple fallback)
- B adds nothing that A + D don't provide
- `main` is a degenerate case of D (implicit single-channel session)

**Partial Order Planning connection**: Data-flow edges = causal links; session ordering = initial constraints; `proc-case` = plan alternatives; ATMS = threat postponement; least commitment principle.

**Critique**: Data-flow analysis complexity bounded by restricted `proc` language; cycle detection = static deadlock detection; CALM optimality — architecture achieves minimum coordination required.

### Formalization: Stratified Effect Barriers — Phase 1 (commit `bc34e44`)

Formalized the current `compile-live-process` + `rt-execute-process` pipeline as
"Stratified Effect Barriers" — Architecture A from the effectful propagators research.

**Three strata documented**:
1. **Pre-Execution (Monotone)**: Type checking, capability inference, session type verification, QTT — elaboration + constraint solving
2. **Effect Execution (Non-Monotone)**: Sequential AST walk with inline IO — `compile-live-process`. Walk order = session type order = effect order (soundness chain)
3. **Post-Execution (Monotone)**: Propagator network quiescence — `rt-run-to-quiescence`. Session advancement, protocol completion, contradiction detection

**Key observations formalized**:
- Interleaving of effects + propagator installation within Stratum 2 is safe (propagators don't fire until quiescence)
- Correctness breaks down for multi-channel concurrent processes → need Architecture D
- Architecture A is a degenerate case of Architecture D (single implicit session)

Updated Phase 2 in the research doc to target Architecture D (not B), per A+D decision.
Added deferred work entries to DEFERRED.md (Phases 2-3).
