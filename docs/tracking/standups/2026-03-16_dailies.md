## Lessons Learned

- Session/effect/IO aren't separate networks — they're sub-graphs in one `prop-network` with different lattice merge functions, connected by cross-domain bridges (Galois connections). The module network follows the same pattern.
- CHAMP structural sharing means "separate network per module" and "shared pointers" are the same thing at the data structure level — logically separate, physically sharing.
- Definition cells are the stable part of a module network; per-command state (metas, constraints) is ephemeral. This separation was already implicit; Track 5 makes it explicit.
- Shadow-cell + callback is the right cross-network pattern — don't try to make propagator schedulers span networks. Module deps form a DAG; LSP walks the DAG; within each network, propagation is local.
- `prop:dict` wrappers for backwards compatibility are more trouble than they're worth — dual materialized-hash + live-network during transition, then cut over, is cleaner and proven (Track 3 pattern).
- Batch-worker's `parameterize` save/restore of `current-module-registry` automatically covers persistent module networks (they live inside `module-info` which lives inside the registry).
