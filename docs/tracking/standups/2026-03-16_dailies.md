## Lessons Learned

- Session/effect/IO aren't separate networks — they're sub-graphs in one `prop-network` with different lattice merge functions, connected by cross-domain bridges (Galois connections). The module network follows the same pattern.
- CHAMP structural sharing means "separate network per module" and "shared pointers" are the same thing at the data structure level — logically separate, physically sharing.
- Definition cells are the stable part of a module network; per-command state (metas, constraints) is ephemeral. This separation was already implicit; Track 5 makes it explicit.
- Shadow-cell + callback is the right cross-network pattern — don't try to make propagator schedulers span networks. Module deps form a DAG; LSP walks the DAG; within each network, propagation is local.
- `prop:dict` wrappers for backwards compatibility are more trouble than they're worth — dual materialized-hash + live-network during transition, then cut over, is cleaner and proven (Track 3 pattern).
- Batch-worker's `parameterize` save/restore of `current-module-registry` automatically covers persistent module networks (they live inside `module-info` which lives inside the registry).
- TMS tree structure has a nested-read bug: `tms-read` with stack `(inner outer)` falls back to base when `inner` branch is absent, instead of checking `outer` branch. This means depth-0 speculation works correctly but nested speculation misses outer writes. The fix (for Phase 4) is `(tms-read cell-val (cdr stack))` instead of `(tms-cell-value-base cell-val)` on branch miss.
- The `current-prop-net-box` holds an `elab-network`, not a bare `prop-network`. Any code that calls `prop-network-cells` on the box contents must unwrap via `elab-network-prop-net` first.
- `tms-commit` for nested TMS must **flatten** sub-trees, not blindly promote sub-tree base. When inner speculation succeeds, its branches (which may reference outer hypotheses) must be lifted into the outer cell. Promoting `tms-bot` base from fresh sub-trees was the root cause of `match: no matching clause for 'tms-bot` errors.
- Two bugs in the TMS tree model were tightly coupled: the `tms-read` fallback bug (falls to base instead of outer hypothesis) and the `tms-commit` flatten bug (promotes `tms-bot` instead of merging). Both needed fixing simultaneously for nested speculation to work. The Phase 2+3 depth-0-only workaround was the right de-risking approach.
