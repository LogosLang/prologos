### Phase 8b: Switchover — Replace metavar-store.rkt Internals

The core of today's work. Rewrote `metavar-store.rkt` internally to delegate to the propagator network.

**Key changes to `metavar-store.rkt` (~242 lines changed):**
- Added `champ.rkt` import (no cycle — pure data structure)
- Added callback parameters for network operations (5 parameters):
  - `current-prop-make-network`, `current-prop-fresh-meta`, `current-prop-cell-write`, `current-prop-cell-read`, `current-prop-add-unify-constraint`
- Inlined `type-bot?`/`type-top?` as simple symbol predicates (`prop-type-bot?`/`prop-type-top?`) to avoid importing `type-lattice.rkt`
- Added `install-prop-network-callbacks!` — called by `driver.rkt` at module load to inject real network functions
- Added internal state: `current-prop-net-box` (boxed immutable network), `current-prop-id-map-box` (CHAMP: meta-id -> cell-id)
- Rewrote core API functions: `fresh-meta`, `solve-meta!`, `meta-solved?`, `meta-solution`, `save-meta-state`, `restore-meta-state!`, `reset-meta-store!`
- Added `extract-shallow-meta-ids` for propagator constraint creation in `add-constraint!`

**Key changes to `driver.rkt` (~48 lines changed):**
- Added `(require "elaborator-network.rkt")` for callback installation
- Calls `install-prop-network-callbacks!` at module load with real network functions
- Updated `load-module` parameterize to include `current-prop-net-box` and `current-prop-id-map-box`

**Critical obstacle: Circular dependency.**
Adding `(require "elaborator-network.rkt" "type-lattice.rkt")` to `metavar-store.rkt` created a cycle:
```
metavar-store → elaborator-network → type-lattice → reduction → metavar-store
```
**Solution**: Callback parameter pattern (same approach as `current-retry-unify`). Network operations are injected via parameters at startup. The trivial lattice predicates (`type-bot?`, `type-top?`) are inlined as symbol checks since they're just `(eq? v 'type-bot)`.

**Result**: Clean compile, all 4308 tests pass.

### Key Decisions

1. **Callback parameter pattern over direct requires** — breaks circular dependency without touching consumer files. Same proven pattern as `current-retry-unify`.

2. **Inlined type-lattice predicates** — `type-bot` and `type-top` are just symbols (`'type-bot`, `'type-top`), so `prop-type-bot?`/`prop-type-top?` are trivial `eq?` checks. No need to import the full lattice module.

3. **Kept hash-table backward compatibility** — `current-meta-store` still exists and is updated alongside the network. This means test files that directly manipulate the hash (e.g., for isolated unit tests) continue to work. Can be removed in a future cleanup phase.

4. **Deferred ATMS** — the design doc (§5.4) describes an Assumption-based Truth Maintenance System for dependency-directed error messages. Not needed for the switchover; the current `with-speculative-rollback` works better now (O(1) forking). ATMS is a potential Phase 9.

### Architectural Patterns Identified

2. **Boxed CHAMP (immutable-core, mutable-shell)** — O(1) snapshots for speculation
3. **Dual-path migration** — write to both old/new, read from new
4. **Phased validation** — shadow -> adapter -> switchover -> cleanup
5. **Trivial predicate inlining** — simpler than callbacks for stable, pure functions

## Phases A–D: Post-Switchover Optimization Arc

### Phase A: Eliminate Hash Backward Compatibility ✅

Added `current-prop-meta-info-box` — a CHAMP storing meta-id → meta-info as the primary metadata store. Eliminated dual writes (production no longer writes to the hash). `save-meta-state`/`restore-meta-state!` are now fully O(1) when CHAMP stores are active (removed O(N) hash snapshot/restore). Legacy hash path preserved for test compatibility.

**Key changes:**
- `metavar-store.rkt`: CHAMP meta-info store, rewrite of 8 core functions (fresh-meta, solve-meta!, meta-lookup, reset-meta-store!, save-meta-state, restore-meta-state!, all-unsolved-metas)
- `driver.rkt`: champ.rkt import, recover-name-map dual-path, load-module parameterize
- `test-speculation-bridge.rkt`: Fixed make-hash → make-hasheq
- `test-error-messages.rkt`: Added reset-meta-store! for CHAMP isolation

### Phase B: Level/Mult/Session Meta Migration to CHAMP ✅

Added three CHAMP box parameters for auxiliary meta stores. All reads/writes use dual-path (CHAMP primary, hash fallback).

**Critical bug fix:** `save-meta-state`/`restore-meta-state!` now include all six CHAMPs (network, id-map, meta-info, level, mult, session). Previously, level/mult/session meta mutations leaked through `with-speculative-rollback` — a latent speculation bug.

**Key changes:**
- `metavar-store.rkt`: 3 new CHAMP box parameters, rewrote 12 API functions
- `driver.rkt`: Added 3 CHAMP boxes to load-module parameterize

### Phase C: Incremental Trait Resolution ✅

Added incremental trait resolution via wakeup-based callbacks (same proven pattern as Sprint 5 constraint retries). When `solve-meta!` resolves a type-arg meta, `retry-trait-for-meta!` checks a reverse wakeup index and attempts to resolve any trait constraints that reference it.

**Simpler than full propagator integration:**
- Reuses the proven wakeup pattern
- No modifications to the elaboration network
- Catches cascading resolutions that the single-pass sweep misses
- `resolve-trait-constraints!` kept as fallback at 9 existing call sites

**Key changes:**
- `metavar-store.rkt`: Added `current-trait-wakeup-map`, `current-retry-trait-resolve`, `install-trait-resolve-callback!`, `retry-trait-for-meta!`, `extract-shallow-meta-ids-from-list`
- `driver.rkt`: Added unify.rkt import, installed trait resolution callback

### Phase D1: ATMS Error Integration (Foundation) ✅

Threaded ATMS through the speculation bridge for dependency-directed error tracking. Each speculative branch now creates an ATMS hypothesis, and failures record nogoods. The ATMS is initialized per-command.

**Opt-in infrastructure** — no changes to error formatting yet. Enables future D2-D4 phases to add derivation chains to error messages.

**Key changes:**
- `elab-speculation-bridge.rkt`: Added `current-command-atms`, ATMS hypothesis creation, nogood recording, extended `speculation-failure` with `hypothesis-id`
- `driver.rkt`: Added atms.rkt import, ATMS initialization per-command
