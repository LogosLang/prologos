### Cell & Propagator Allocation Audit (commit `f7bd03d`)

14-section audit document: `docs/tracking/2026-03-20_CELL_PROPAGATOR_ALLOCATION_AUDIT.md`

**Core finding**: `struct-copy prop-network` (13-field copy) is the dominant per-operation allocation cost, with 25 call sites in propagator.rkt alone. A representative command (~15 metas, 50 worklist iterations, 30 cell writes) generates ~125 prop-network struct copies = ~14KB of struct allocation per command.

**Six optimization opportunities identified** (all preserve pure data-in → data-out contract):
1. **Batch cell registration** — use existing transient CHAMP builder for `register-global-env-cells!` (saves ~49 struct-copies per 50-cell batch)
2. **Field-group struct splitting** — partition 13 fields into hot (worklist, fuel), warm (cells, contradiction), cold (9 others). Worklist loop copies drop from 13→2 fields
3. **Mutable worklist/fuel in quiescence loop** — worklist+fuel only used within `run-to-quiescence`; mutable box inside pure boundary eliminates per-iteration struct-copy
4. **Propagator input batching** — transient CHAMP for cells map during `net-add-propagator` input registration
5. **eq?-first fast path in cell write** — skip expensive structural `equal?` when merge returns identical pointer (80% case)
6. **Identity-preserving short-circuits** — verify all wrappers propagate eq? identity through full call chain

**Incremental GC section**: Threshold propagators fire at most once (monotone lattice guarantee). But deallocation in a persistent network = producing a new network without the propagator — same allocation cost as creation. The network IS the provenance/replayability trail. Four exploration patterns proposed: provenance lifetime analysis, generation-based retention, lazy dependent pruning, snapshot-and-compact. Recommendation: optimize allocation first (§9), measure whether dead-propagator retention is measurable, then decide.

**Key insight from standup discussion**: GC for propagator networks is itself a propagator computation — liveness propagates "dead" upward through the dependency graph monotonically. This connects to existing ATMS infrastructure (justifications as decoupled provenance records). An incremental propagator GC would produce ATMS-compatible provenance records as a byproduct.
