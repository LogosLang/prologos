- `#=` chosen as narrowing operator (distinct from `=` unification)

#### PUnify Adversarial Benchmarks (commit `76eb5d2`)
Three-tier benchmark suite for PUnify A/B comparison:

1. **bench-solver-unify.rkt** (19 micro-benchmarks): unify-terms decomposition (deep 50, wide 100, binary trees 511 nodes), walk/walk* traversal (chains 100-500, wide subst), normalize-ast-to-solver-term, combined patterns (growing subst, transitive chains, partial overlap).
   - Key finding: deep unification 37ms, late-failure same cost (38ms — no early termination). Walk chain-500: 9.5ms. Tree (511 nodes): 4ms.
2. **bench-solve-pipeline.rkt** (24 micro-benchmarks): full solve pipeline with controlled stores — fact-only (10-1000), clause chains (5-50 depth), backtracking (first/last/fail), multi-hop joins (2-5 hops), conjunction (10-200 goals), diamond joins (5-50×), inline unify.
   - Key finding: 5-hop×5 join: 236ms (exponential DFS cost). Diamond 50×50: 27ms. Conjunction 200: 27ms.
3. **solve-adversarial.prologos** (92 commands): comparative benchmark for bench-ab.rkt — exercises all solver paths through full pipeline. Baseline: **14.3s median, 0.3% CV**.

Also exported `walk`, `unify-terms`, `normalize-term-deep`, `solve-goals` from relations.rkt and `normalize-ast-to-solver-term` from reduction.rkt for benchmark access.

#### Observations
- DFS solver doesn't handle recursive relations (`ancestor`) at API level — diverges even with ground queries. This is a known limitation; the defr/solve through full pipeline uses stratification which avoids the issue.
- `defr` with `|` clause form doesn't register relations (only `||` facts and `&>` conjunction forms work)
- Constructor unification through `=` in solve returns nil for some prelude constructor terms — PUnify should fix this
