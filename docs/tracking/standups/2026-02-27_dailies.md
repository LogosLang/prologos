# Claude Daily Summary: 2026-02-27

## Dict-Passing + Call-Site Specialization — Complete

### Overview

Completed the full plan from `buzzing-launching-pascal.md`: (1) `new-lattice-cell` generic wrapper using dict-passing across closures, (2) call-site specialization rewriting that transparently replaces generic calls with zero-overhead specialized versions.

### Key Decisions

- **`where (Lattice A)` syntax over inline `[Lattice A]`**: The inline constraint syntax `{A} [Lattice A] PropNetwork -> ...` wraps in `$angle-type` which `extract-inline-constraints` can't detect. Using `where` is the correct path — it's processed by `extract-where-clause` which handles trait constraints properly.

- **De Bruijn shifts must be explicit in typing rules**: The `arrow` helper (`syntax.rkt:1019`) is just `(expr-Pi mw a b)` — it does NOT shift for de Bruijn. When building multi-arg Pi types like `A -> A -> A` where `A` contains bvars, each Pi binder requires explicit `(shift n 0 ...)`. This is a subtle invariant that isn't documented anywhere.

- **`checkQ` over `inferQ` for standalone lambdas**: QTT's `inferQ` has no standalone lambda case — lambdas only type-check via `checkQ` against a known Pi type. This was a pre-existing bug affecting ALL `net-new-cell` usage in `def`/`defn` contexts, not just the polymorphic version.

- **Specialization rewriting placement**: After `zonk-final` but before QTT and reduction. This gives us ground types for specialization matching while keeping the optimization invisible to the type checker and multiplicity checker.

- **Fast path for empty registry**: `(hash-empty? (current-specialization-registry))` returns immediately. Zero overhead when no specializations are registered.

- **Library modules need explicit type imports**: In `:no-prelude` modules (like `propagator.prologos`), data types defined via `data` in other modules are registered as preparse macros during loading but NOT imported into the local namespace unless explicitly `refer`'d. The `Interval` type required `[prologos::core::lattice-instances :refer [Interval interval-bot interval-join]]`.

### Key Findings

1. **The "meta-resolution limitation" deferred blocker was a false alarm.** Phase D of trait resolution (`elaborator.rkt:729-736`) already propagates `current-where-context` into lambda bodies. The MapLattice instance (`lattice-instances.prologos:148-154`) proves this works: `[fn x y [join x y]]` inside `where (Lattice V)` compiles correctly.

2. **Four independent compiler edge cases** were the real blockers — none fundamental, all fixable:
   - `*` operator in `param-type->angle-type` (macros.rkt)
   - Propagator types missing from `known-type-name?` (macros.rkt)
   - De Bruijn shifts in `net-new-cell` typing (typing-core.rkt)
   - QTT `inferQ` vs `checkQ` for lambdas (qtt.rkt)

3. **Closure capture of trait dictionaries works end-to-end.** The `new-lattice-cell` body `[fn x y [join x y]]` captures `$Lattice-A` from the enclosing scope. The `join` resolves to `(Lattice-join A $Lattice-A)` via `resolve-method-from-where`. The closure carries the dict through `net-new-cell` into the merge function.

4. **`raco test` vs `racket` for test execution**: `raco test` respects module boundaries (only exported names visible), while `racket` as a script exposes all module-level bindings. The `rewrite-specializations` function needed explicit `provide` export.

### Lessons Learned

- **Deferred work accumulates false assumptions.** The `new-lattice-cell` was deferred months ago with the note "blocked on meta-resolution limitation." Revisiting with fresh eyes revealed the infrastructure already existed — the deferral was based on a misunderstanding. Lesson: challenge deferral reasons periodically.

- **Testing in parallel runners surfaces isolation bugs.** The specialization test passed standalone but failed in the parallel test runner because `rewrite-specializations` wasn't exported. Always verify with `raco test`.

- **Dict-passing IS the semantics; specialization IS the optimization.** These compose cleanly because specialization is purely syntactic sugar — it rewrites the call site but doesn't change the type system. If a specialization is missing, dict-passing works. If a specialization exists, it produces the same result with less overhead.

## Phase 6: Galois Connections & Abstract Interpretation — Complete

### Overview

Completed all five sub-phases of Phase 6 (Galois Connections & Abstract Interpretation) from the plan at `buzzing-launching-pascal.md`. This adds modular abstract domain support to the logic engine: widening/narrowing for infinite-height lattices, Galois connections for sound domain bridging, cross-domain propagation, and two ready-made abstract domain libraries (Sign, Parity).

### What Was Built

| Phase | Deliverable | Key Files |

| 6b | `GaloisConnection {C A}` 2-type-param trait + `impl GaloisConnection Interval Bool` | galois-trait.prologos, galois-instances.prologos |
| 6c | `net-add-cross-domain-propagator` bidirectional α/γ combinator | propagator.rkt |
| 6d | Sign (5-element) + Parity (4-element) data types with Lattice/HasTop instances | sign.prologos, parity.prologos, sign-lattice.prologos, parity-lattice.prologos |
| 6e | `specialize new-widenable-cell for Interval` + grammar docs + integration tests | propagator.prologos, grammar.org |

### Key Lessons Learned

1. **WS-mode continuation line nesting**: Deeper-indented continuation lines get grouped as sub-lists by the WS reader. Multi-argument function calls like `net-new-cell-widen` with 5 args must have ALL args on the same line (or use explicit bracketing). Debugging required using `read-all-forms-string` from the reader module to inspect actual sexp output.

2. **Match wildcard type mismatch**: The `_` wildcard pattern in ADT match fails when the return type differs from the scrutinee type (e.g., matching on `Interval` but returning `Bool`). Must enumerate all constructors explicitly.

3. **Trait names are not term-level values**: `def x : Type Widenable` fails — trait names exist only in the type/ctor registry. Test trait existence via accessors (`Widenable-widen`, `Widenable-narrow`).

4. **Angle-bracket annotations required in impl bodies**: `defn widen [old <Interval> new <Interval>] <Interval>` not `defn widen [old : Interval new : Interval] Interval`.

5. **Stale struct compilation**: Adding `widen-fns` field to `prop-network` (9→10 fields) caused "mismatch; expected struct-type value with 9 fields" in modules compiled against the old struct. Fix: `raco make` the affected .rkt files explicitly.

6. **Cross-domain propagation as pure combinator**: No new struct fields needed — `net-add-cross-domain-propagator` creates two `net-add-propagator` calls (C→A via α, A→C via γ). Bidirectional termination guaranteed by the no-change guard.

- `connect-domains` Prologos-level wrapper (needs AST keyword or FFI approach)

## Numeric Literal Cleanup + Refined Numeric Types — Complete

### Overview

Fixed two bugs preventing ergonomic use of negative numeric literals, and added refined numeric types (PosInt/NegInt/Zero, PosRat/NegRat) as library-level data types with Eq/Ord trait instances.

### What Was Built

| Phase | Deliverable | Key Files |
|-------|------------|-----------|
| A | Sexp-mode negative integer fix (`exact-nonnegative-integer?` → `exact-integer?`) | parser.rkt |
| B | WS-mode negative number literal tokenization (`-42`, `-3/7`, `-3.14`) | reader.rkt |
| B.2 | Tilde handler extension for `~-N` approximate literals | reader.rkt |
| C | 25 tests: sexp, WS tokenizer, roundtrip, approximate, non-regression, e2e | test-negative-literals.rkt |
| D.1 | `PosInt`, `NegInt`, `Zero` data types + smart/unsafe constructors + extractors | refined-int.prologos |
| D.2 | `PosRat`, `NegRat` data types + smart/unsafe constructors + extractors | refined-rat.prologos |
| D.3 | `impl Eq/Ord` for PosInt, NegInt, Zero, PosRat, NegRat (delegate to underlying Int/Rat) | refined-int-instances.prologos, refined-rat-instances.prologos |
| D.4 | 26 + 18 = 44 tests for refined types | test-refined-int.rkt, test-refined-rat.rkt |

### Key Lessons Learned

1. **WS reader `-` dispatch order is critical**: `-0>`, `-1>`, `-w>` (multiplied arrows) → `->` (arrow) → `-3` (negative literal, new) → `-foo` (identifier). The peekahead `(char-numeric? c2)` after `-` differentiates negative numbers from identifiers.

2. **Trait method names vs accessor names**: In `:no-prelude` modules, `eq?` is the trait body method name but the export is `Eq-eq?`. Calling `(Eq-eq? PosInt PosInt--Eq--dict x y)` with explicit dict-passing is the correct pattern. The short name `eq?` only works through the prelude's trait dispatch mechanism.

3. **`:no-prelude` for library modules, prelude-aware for tests**: Library `.prologos` files use `:no-prelude` to avoid circularity. Tests can either use the prelude (for standard types) or explicit dict-passing (for custom types not in the prelude).

4. **Negative Nat guard**: `-3N` is correctly rejected with an error at the tokenizer level — natural numbers cannot be negative by definition. This applies to both bare `-3N` and approximate `~-3N`.
