### Overview

Completed the full plan from `buzzing-launching-pascal.md`: (1) `new-lattice-cell` generic wrapper using dict-passing across closures, (2) call-site specialization rewriting that transparently replaces generic calls with zero-overhead specialized versions.

### Key Decisions

- **`where (Lattice A)` syntax over inline `[Lattice A]`**: The inline constraint syntax `{A} [Lattice A] PropNetwork -> ...` wraps in `$angle-type` which `extract-inline-constraints` can't detect. Using `where` is the correct path â€” it's processed by `extract-where-clause` which handles trait constraints properly.

- **De Bruijn shifts must be explicit in typing rules**: The `arrow` helper (`syntax.rkt:1019`) is just `(expr-Pi mw a b)` â€” it does NOT shift for de Bruijn. When building multi-arg Pi types like `A -> A -> A` where `A` contains bvars, each Pi binder requires explicit `(shift n 0 ...)`. This is a subtle invariant that isn't documented anywhere.

- **`checkQ` over `inferQ` for standalone lambdas**: QTT's `inferQ` has no standalone lambda case â€” lambdas only type-check via `checkQ` against a known Pi type. This was a pre-existing bug affecting ALL `net-new-cell` usage in `def`/`defn` contexts, not just the polymorphic version.

- **Specialization rewriting placement**: After `zonk-final` but before QTT and reduction. This gives us ground types for specialization matching while keeping the optimization invisible to the type checker and multiplicity checker.

- **Fast path for empty registry**: `(hash-empty? (current-specialization-registry))` returns immediately. Zero overhead when no specializations are registered.

- **Library modules need explicit type imports**: In `:no-prelude` modules (like `propagator.prologos`), data types defined via `data` in other modules are registered as preparse macros during loading but NOT imported into the local namespace unless explicitly `refer`'d. The `Interval` type required `[prologos::core::lattice-instances :refer [Interval interval-bot interval-join]]`.

### Key Findings

1. **The "meta-resolution limitation" deferred blocker was a false alarm.** Phase D of trait resolution (`elaborator.rkt:729-736`) already propagates `current-where-context` into lambda bodies. ### Key Decisions

- **Registry-based over hardcoded**: Rather than adding more cases to the existing hardcoded `subtype?` match, introduced `current-subtype-registry` and `current-coercion-registry` parameters. This makes the system extensible to any library-defined type, not just numerics.

- **`hash` not `hasheq` for registries**: Keys are `(cons sub-key super-key)` pairs. Racket's `eq?` compares pair identity, not structure â€” two `(cons 'PosInt 'Int)` created separately are NOT `eq?`. Must use `equal?`-based `hash`. This was the most subtle bug.

- **Dual-key registration (FQN + short name)**: `type-key` extracts FQN from `(expr-fvar name)` (e.g., `prologos::data::refined-int::PosInt`), but `ctor-meta-type-name` stores short names (e.g., `PosInt`). Rather than normalizing everywhere, we register under both keys. Clean separation of concerns.

- **Auto-inference for single-field wrappers**: `subtype PosInt Int` (no `via`) auto-infers coercion by unwrapping the single constructor. Nullary constructors like `mk-zero` require explicit `via zero-to-int`. This covers the common case ergonomically while handling edge cases explicitly.

### Key Findings

1. **`expand-top-level` is a gatekeeper**: Every new top-level `surf-*` struct needs a pass-through clause in `expand-top-level` (macros.rkt), or it falls to the `[else` case and gets wrapped in `surf-eval`, causing "Cannot elaborate" errors. This is not obvious â€” the error message doesn't point to `expand-top-level` at all.

2. **`qualify-name` exists in namespace.rkt**: Don't hand-roll qualification with `(format "~a::~a" ...)` â€” use `(qualify-name short-name (ns-context-current-ns ctx))`. The hand-rolled version also had a bug: it formatted the entire `ns-context` struct instead of extracting `current-ns`.

3. **`ctor-short-name` pattern in reduction.rkt**: The reducer already has `(or (lookup-ctor name) (lookup-ctor (ctor-short-name name)))` at lines 580-581 for handling FQNâ†’short-name ctor lookups. New reducer code should follow this pattern.

4. **Transitive closure must query BOTH short and FQN supertypes**: Built-in pairs like `Int <: Rat` are registered under short names (`'Int`, `'Rat`), while user-defined pairs use FQNs. The transitive closure computation must query `(all-supertypes super-fqn)` AND `(all-supertypes super-short)` to find all existing relationships.

### Lessons Learned

- **Two hash table types in Racket are a footgun**: `hasheq` (identity comparison) vs `hash` (structural comparison) â€” the wrong choice silently drops all lookups when keys are compound values like pairs. Always think about key equality semantics.

- **The "14-file AST pipeline" has a hidden 15th file**: `macros.rkt` contains `expand-top-level` which acts as a top-level form router. It's not in the linear pipeline but every top-level form passes through it. When adding new `surf-*` structs, this is the file most likely to be forgotten.

- **Dual-key registration is simpler than normalization**: Rather than adding normalization logic to every lookup site (type-key, ctor-meta, subtype?, coercion lookup), registering the same data under multiple keys is O(1) extra space and eliminates an entire class of FQN/short-name mismatch bugs.

| 6 | Tests + grammar + docs | 38 tests, grammar.ebnf + grammar.org, DEFERRED.md, tracking doc |

### Key Design Decisions

1. **Overloaded `nil`**: Single keyword, type inference disambiguates. `infer(nil)`
   defaults to `List ?A` for backward compat; `check(nil, Nil)` and `check(nil, List A)`
   both succeed. Full test suite passed after Phase 1b â€” zero regressions.

2. **`#.`/`#:` syntax (NOT `?.`)**: `?` suffix conflicts with predicate naming convention
   (`empty?`, `nil?` are lowercase). `#` prefix is already used for `#{` set literals,
   extending naturally. `#.field` for nil-safe dot-access, `#:key` for nil-safe key-access.

3. **`nil-safe-get` vs `map-get`**: Missing key returns `nil` (NOT error). Nil input
   short-circuits to `nil`. Return type is always `(V | Nil)`.

4. **`A?` sugar is parser-only**: No reader change needed since `?` is `ident-continue?`.
   Only applies to uppercase names that are `known-type-name?`.

### Key Lessons Learned

1. **Backward compat for keyword hijacking**: Changing `nil` from `surf-var 'nil` to
   `surf-nil` keyword was the highest-risk change. The overloading semantics (infer defaults
   to `List ?A`) preserved ALL existing code. Key pattern: when hijacking an existing name,
   ensure the infer case returns what it always returned.

2. **List helper cascade in reduction.rkt**: `racket-list->prologos-list`,
   `prologos-list->racket-list`, `try-as-list` (pretty-print), and `try-builtin-reduce`
   ALL had `(expr-fvar 'nil)` checks that needed updating to `(expr-nil?)`. Missing any
   one breaks list display or pattern matching. Lesson: search for ALL occurrences of the
   thing you're replacing, not just the obvious ones.

3. **Sentinel-based desugaring scales well**: The `($nil-dot-access field)` /
   `($nil-dot-key :keyword)` pattern (parallel to `($dot-access field)`) keeps the reader
   thin and the preparse rewriter composable. Mixed access like `user#.address.city`
   naturally produces `(map-get (nil-safe-get user :address) :city)` through two
   independent rewrite passes.

4. **Union type extraction for nil-safe-get**: The typing rule for `nil-safe-get` when
   the map argument has a union type (e.g., `(Map K V | Nil)`) needs to extract Map
   components, check the key against each, and union the result Vs with Nil. This reuses
   the existing `union-members` infrastructure.

## Spec / Functor / Trait / Property Configuration Language Audit â€” Complete

### Overview

Comprehensive audit of the `spec`, `functor`, `trait`, `property`, and `bundle`
configuration languages: their metadata keys, type-theoretic and category-theoretic
mappings, ergonomic gaps, and opportunities. Follows Phase 1â€“2 of design methodology
(research + gap analysis). Produced a design document at

### Key Findings

1. **All Phase 1 implementations are structurally complete**: `spec` (8-field struct,
   10 metadata keys), `property` (6-field struct, clause parsing, `:includes` flattening),
   `functor` (4-field struct, transparent deftype expansion), `trait` (`:laws` parsing),
   `??` typed holes (full pipeline), `:deprecated` warnings.

2. **Two structural gaps found**:
   - **G5**: `trait-meta` struct has NO `metadata` field â€” `:doc`, `:deprecated`,
     `:see-also` are parsed but silently discarded (only `:laws` is extracted)
   - **G6**: `bundle-entry` struct has NO `metadata` field â€” same issue

3. **Four ambiguity/conflict risks identified**:
   - G1: `:pre`+`:post`+`:invariant` overlap (no resolution designed)
   - G2: `:implicits` kind disagreement between inline and metadata (silent override)
   - G3: Property `:where` vs spec `:where` subset mismatch (unverifiable properties)
   - G4: Functor/data name collision (silent clobbering via deftype)

4. **Pi/Sigma surface leaks are minimal**: The design covers all common cases. `functor`
   hides transducer/optics types. `:implicits` hides most Pi binders. `:refines` (Phase 3)
   will hide Sigma returns. Only truly dependent return types (e.g., `Vec A n` where `n`
   is runtime) still need angle brackets â€” proposed `:depends` key for these.

5. **Category theory coverage**: `functor` captures the core category structure (objects,
   composition, identity, laws). Proposed extensions: `:variance`, `:fold`/`:unfold`
   (recursion schemes), `:free-over`. Concepts that don't fit `functor` (monads, arrows,
   profunctors) correctly belong on `trait`/`bundle`.

6. **`schema` is a genuine future opportunity**: Closed-world product types with validation
   metadata. Distinct from `data` (algebraic types) and `trait` (open-world behaviors).
   Preliminary design sketched but deferred as orthogonal.

### Key Lessons Learned

1. **Metadata field consistency is an invariant**: All configuration keywords should have
   a `metadata` hash field. `trait-meta` and `bundle-entry` broke this invariant. The
   generic `parse-spec-metadata` function can parse metadata for any keyword, but the
   storage struct must have a field to receive it.

2. **The Postel principle (store unrecognized keys) works for forward compatibility**:
   Users can write `:pre`, `:post` today even though they're Phase 2 features. The keys
   are stored silently. This is correct behavior â€” but explicit recognition with shape
   validation would catch typos (`:propreties` vs `:properties`).

3. **The graduation path from testing to proving is the core design insight**: The same
   metadata syntax upgrades from QuickCheck (Phase 2) to proof obligations (Phase 3)
   without any syntax changes. This is only possible because the type-theoretic
   interpretations were designed upfront.

### Critique Cycle

Independent critique received and processed. Key improvements adopted:
- **G1**: `:pre`+`:post`+`:invariant` upgraded from warning to error (genuinely different
  TT semantics: caller vs implementer proof obligations)
- **`:refines` erasure**: Default erased Sigma (multiplicity 0); future `:refines :relevant`
- **CT precision**: Fixed imprecise claim that functor declaration "IS a category" â†’ correctly
  "describes the data of a category." Retained naming rationale per Extended Spec Design.
- **`:depends` deferred**: Angle brackets are sufficient and explicit for dependent returns
- **Linearity**: `:pre`/`:post` predicates operate on multiplicity-0 copies (observation
  without consumption of linear resources)
- **Roadmap additions**: G8 (error message audit), G9 (gap test cases), O11 (`:refines :relevant`)
Rejected: struct inheritance (14-file match disruption), `:w` on `:implicits` (violates
progressive disclosure), `profunctor` keyword (trait covers it), named args outside angle
brackets (separate design concern).

## Literate Book System â€” Phase 1 Proof of Concept â€” Complete

### Overview

Built the tangler infrastructure and first chapter of the standard library book.
The system uses `.prologos` itself as the literate format â€” comments are prose,
`module` directives mark compilation unit boundaries, and the tangler extracts
machine-loadable modules from human-readable chapters.

### What Was Built

| Artifact | Description |
|----------|-------------|
| `lib/prologos/book/OUTLINE` | Reading order manifest (named chapters, not numbered) |
| `lib/prologos/book/equality.prologos` | First chapter: Eq trait + 6 instance modules + propositional equality |
| `tools/tangle-stdlib.rkt` | Text-processing tangler (~175 LOC) â€” splits chapters at `module` directives |
| `.tangled/` | Output directory (gitignored) with 7 tangled module files |
| `.gitignore` | Updated with `.tangled/` entry |
| `docs/tracking/2026-02-27_STDLIB_DESIGN.md` | Updated with Part VIII (literate book evolution) |

### Key Design Decisions

- **Format is `.prologos` itself**: No org-mode, no markdown wrappers. Prologos's metadata
  system (`spec` + `:doc` + `:examples` + `:laws` + `:see-also`) IS a documentation language
- **`module` directive**: New preparse-level form â€” `module prologos::core::eq-trait` means
  "subsequent definitions belong to this compilation unit"
- **Chapters named, not numbered**: Ordinals derived from OUTLINE position. Insert = add a line
- **Instance placement**: Instances with the motivating concept (trait's chapter), not the type
- **Tangler is text processing**: No Prologos parser invocation â€” splits on `module` lines,
  wraps each section with `ns`, writes to `.tangled/`
- **Module paths stable**: Tangled output preserves all existing module paths identically

### Verification

- Code-only diff: all 7 tangled modules have identical code to their originals
- 471+ tests pass across 20 test files â€” zero regressions
- Comment differences are intentional: richer prose in chapters vs simpler headers in originals
