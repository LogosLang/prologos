## Dict-Passing + Call-Site Specialization â€” Complete

### Overview

Completed the full plan from `buzzing-launching-pascal.md`: (1) `new-lattice-cell` generic wrapper using dict-passing across closures, (2) call-site specialization rewriting that transparently replaces generic calls with zero-overhead specialized versions.

### Key Decisions

- **`where (Lattice A)` syntax over inline `[Lattice A]`**: The inline constraint syntax `{A} [Lattice A] PropNetwork -> ...` wraps in `$angle-type` which `extract-inline-constraints` can't detect. Using `where` is the correct path â€” it's processed by `extract-where-clause` which handles trait constraints properly.

- **De Bruijn shifts must be explicit in typing rules**: The `arrow` helper (`syntax.rkt:1019`) is just `(expr-Pi mw a b)` â€” it does NOT shift for de Bruijn. When building multi-arg Pi types like `A -> A -> A` where `A` contains bvars, each Pi binder requires explicit `(shift n 0 ...)`. This is a subtle invariant that isn't documented anywhere.

- **`checkQ` over `inferQ` for standalone lambdas**: QTT's `inferQ` has no standalone lambda case â€” lambdas only type-check via `checkQ` against a known Pi type. This was a pre-existing bug affecting ALL `net-new-cell` usage in `def`/`defn` contexts, not just the polymorphic version.

- **Specialization rewriting placement**: After `zonk-final` but before QTT and reduction. This gives us ground types for specialization matching while keeping the optimization invisible to the type checker and multiplicity checker.

- **Fast path for empty registry**: `(hash-empty? (current-specialization-registry))` returns immediately. Zero overhead when no specializations are registered.

- **Library modules need explicit type imports**: In `:no-prelude` modules (like `propagator.prologos`), data types defined via `data` in other modules are registered as preparse macros during loading but NOT imported into the local namespace unless explicitly `refer`'d. The `Interval` type required `[prologos::core::lattice-instances :refer [Interval interval-bot interval-join]]`.

### Key Findings

1. **The "meta-resolution limitation" deferred blocker was a false alarm.** Phase D of trait resolution (`elaborator.rkt:729-736`) already propagates `current-where-context` into lambda bodies. The MapLattice instance (`lattice-instances.prologos:148-154`) proves this works: `[fn x y [join x y]]` inside `where (Lattice V)` compiles correctly.

2. **Four independent compiler edge cases** were the real blockers â€” none fundamental, all fixable:
   - `*` operator in `param-type->angle-type` (macros.rkt)
   - Propagator types missing from `known-type-name?` (macros.rkt)
   - De Bruijn shifts in `net-new-cell` typing (typing-core.rkt)
   - QTT `inferQ` vs `checkQ` for lambdas (qtt.rkt)

3. **Closure capture of trait dictionaries works end-to-end.** The `new-lattice-cell` body `[fn x y [join x y]]` captures `$Lattice-A` from the enclosing scope. The `join` resolves to `(Lattice-join A $Lattice-A)` via `resolve-method-from-where`. The closure carries the dict through `net-new-cell` into the merge function.

4. **`raco test` vs `racket` for test execution**: `raco test` respects module boundaries (only exported names visible), while `racket` as a script exposes all module-level bindings. The `rewrite-specializations` function needed explicit `provide` export.

### Lessons Learned

- **Deferred work accumulates false assumptions.** The `new-lattice-cell` was deferred months ago with the note "blocked on meta-resolution limitation." Revisiting with fresh eyes revealed the infrastructure already existed â€” the deferral was based on a misunderstanding. Lesson: challenge deferral reasons periodically.

- **Testing in parallel runners surfaces isolation bugs.** The specialization test passed standalone but failed in the parallel test runner because `rewrite-specializations` wasn't exported. Always verify with `raco test`.

- **Dict-passing IS the semantics; specialization IS the optimization.** These compose cleanly because specialization is purely syntactic sugar â€” it rewrites the call site but doesn't change the type system. If a specialization is missing, dict-passing works. If a specialization exists, it produces the same result with less overhead.

### Key Lessons Learned

1. **WS reader `-` dispatch order is critical**: `-0>`, `-1>`, `-w>` (multiplied arrows) â†’ `->` (arrow) â†’ `-3` (negative literal, new) â†’ `-foo` (identifier). The peekahead `(char-numeric? c2)` after `-` differentiates negative numbers from identifiers.

2. **Trait method names vs accessor names**: In `:no-prelude` modules, `eq?` is the trait body method name but the export is `Eq-eq?`. Calling `(Eq-eq? PosInt PosInt--Eq--dict x y)` with explicit dict-passing is the correct pattern. The short name `eq?` only works through the prelude's trait dispatch mechanism.

3. **`:no-prelude` for library modules, prelude-aware for tests**: Library `.prologos` files use `:no-prelude` to avoid circularity. Tests can either use the prelude (for standard types) or explicit dict-passing (for custom types not in the prelude).

4. **Negative Nat guard**: `-3N` is correctly rejected with an error at the tokenizer level â€” natural numbers cannot be negative by definition. This applies to both bare `-3N` and approximate `~-3N`.

## Refined Numeric Subtyping (Phase E) â€” Complete

### Overview

### Key Decisions

- **Registry-based over hardcoded**: Rather than adding more cases to the existing hardcoded `subtype?` match, introduced `current-subtype-registry` and `current-coercion-registry` parameters. This makes the system extensible to any library-defined type, not just numerics.

- **`hash` not `hasheq` for registries**: Keys are `(cons sub-key super-key)` pairs. Racket's `eq?` compares pair identity, not structure â€” two `(cons 'PosInt 'Int)` created separately are NOT `eq?`. Must use `equal?`-based `hash`. This was the most subtle bug.

- **Dual-key registration (FQN + short name)**: `type-key` extracts FQN from `(expr-fvar name)` (e.g., `prologos::data::refined-int::PosInt`), but `ctor-meta-type-name` stores short names (e.g., `PosInt`). Rather than normalizing everywhere, we register under both keys. Clean separation of concerns.

- **Auto-inference for single-field wrappers**: `subtype PosInt Int` (no `via`) auto-infers coercion by unwrapping the single constructor. Nullary constructors like `mk-zero` require explicit `via zero-to-int`. This covers the common case ergonomically while handling edge cases explicitly.

### Key Findings

1. **`expand-top-level` is a gatekeeper**: Every new top-level `surf-*` struct needs a pass-through clause in `expand-top-level` (macros.rkt), or it falls to the `[else` case and gets wrapped in `surf-eval`, causing "Cannot elaborate" errors. This is not obvious â€” the error message doesn't point to `expand-top-level` at all.

2. **`qualify-name` exists in namespace.rkt**: Don't hand-roll qualification with `(format "~a::~a" ...)` â€” use `(qualify-name short-name (ns-context-current-ns ctx))`. The hand-rolled version also had a bug: it formatted the entire `ns-context` struct instead of extracting `current-ns`.

3. **`ctor-short-name` pattern in reduction.rkt**: The reducer already has `(or (lookup-ctor name) (lookup-ctor (ctor-short-name name)))` at lines 580-581 for handling FQNâ†’short-name ctor lookups. New reducer code should follow this pattern.

4. **Transitive closure must query BOTH short and FQN supertypes**: Built-in pairs like `Int <: Rat` are registered under short names (`'Int`, `'Rat`), while user-defined pairs use FQNs. The transitive closure computation must query `(all-supertypes super-fqn)` AND `(all-supertypes super-short)` to find all existing relationships.

### Lessons Learned

- **Two hash table types in Racket are a footgun**: `hasheq` (identity comparison) vs `hash` (structural comparison) â€” the wrong choice silently drops all lookups when keys are compound values like pairs. Always think about key equality semantics.

- **The "14-file AST pipeline" has a hidden 15th file**: `macros.rkt` contains `expand-top-level` which acts as a top-level form router. It's not in the linear pipeline but every top-level form passes through it. When adding new `surf-*` structs, this is the file most likely to be forgotten.

- **Dual-key registration is simpler than normalization**: Rather than adding normalization logic to every lookup site (type-key, ctor-meta, subtype?, coercion lookup), registering the same data under multiple keys is O(1) extra space and eliminates an entire class of FQN/short-name mismatch bugs.
