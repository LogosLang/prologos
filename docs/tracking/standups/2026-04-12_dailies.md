### Phase 2 NAF Design Evolution

Three approaches tried, each teaching something:

1. **Worldview-bitmask isolation (D.10)**: inner goal on same network, NAF bitmask for isolation. Failed: construction-time vs BSP-time confusion, ground goal detection impossible (no writes for ground args), variable bindings not visible (cell promotion issues). Root cause: conflating monotone (S0) and non-monotone (NAF) on the same stratum.

2. **Probe variables**: fresh vars + unification to ensure fact matches produce bindings. Failed: unification setup writes to probe scope cells indistinguishable from inner goal match writes.

3. **NAF-success cell**: dedicated cell written by discrimination when viable alternatives found. Worked for basic cases but not ground inner goals (discrimination writes viability, not scope cells).

4. **Stratified evaluation (D.11, final)**: NAF registered at S0, evaluated at S1 (S0 fixpoint). Discrimination provability check reads S0-resolved args directly. Variable bindings visible because S0 has completed. SIMPLEST approach (net negative lines: 103 added, 125 removed).

### Key Lesson: NAF is Non-Monotone → Stratification Required

The user identified this: CALM guarantees confluence only for monotone operations. NAF inverts provability (not monotone). Every issue in the bitmask approach was a symptom of this fundamental mismatch. Stratification is the principled solution — well-understood in logic programming (stratified evaluation, well-founded semantics). The BSP architecture already has strata; adding S1 for NAF fits naturally.

### Kan Extension Connection

The S1 evaluation is a right Kan extension from positive provability to the negated domain. For N independent NAFs: broadcast evaluation (O(1)). For dependent NAFs: S0↔S1 fixpoint iteration (well-founded semantics).

- **Design mantra as live challenge**: The mantra caught architectural drift that tests missed. Multiple tracks have drifted from "on-network" — the mantra makes this visible at every decision point.

#### Phase 2d: Streaming BSP Investigation (`1c691f86`)

Exhaustive investigation of 4 approaches to lower the parallel crossover:
- Hybrid spin-wait: ineffective (merge phase > spin window)
- Streaming via sync events: 15-20us/call overhead
- Streaming via async-channel: 30us/call
- Spin-poll done-semaphores: 2400us catastrophically slow

**Root cause**: Racket inter-thread communication cost (8-300us) exceeds per-propagator work (~0.5us). Same-thread semaphore 0.012us, cross-thread ~8us — 667x overhead from OS thread wakeup. Reverted to sync pool as best Phase 0 path.

**Lesson**: Racket's concurrency primitives have a ~8us floor per cross-thread interaction. The parallel infrastructure is architecturally correct and will pay off when: per-propagator work becomes heavier (self-hosted compiler), lighter-weight runtime available (Prologos concurrency), or larger workloads (N≥256).

**Strategic discussion**: Considered FFI to C for lightweight threads vs waiting for self-hosting. Conclusion: design concurrency model now, implement at self-hosting. Racket FFI would be throwaway work. Self-hosted compiler gets heavier work per propagator → lower crossover naturally. SH Series placeholder in Master Roadmap; research note spawned on LLVM path.

#### Phase 3: Guard as Worldview Assumption (`83276b0d`)

Guards follow the NAF pattern: allocate assumption h_guard, tag subsequent goals, fire-once evaluates condition at S0, nogood if falsy. `install-conjunction` pre-scans for BOTH NAF and guard goals (gating kinds). Guard bit position detected via worldview cache XOR.

Key discovery: with scope sharing + all-at-once conjunction installation, guards have no gating effect on subsequent goals (they're all installed independently). The worldview assumption pattern provides gating via bitmask tagging — same as NAF.

#### Phase 5a: Propagator Flags + Tier 1 Flush + Self-Clearing (`333a5667`→`2181fac0`)

Three-component optimization:
1. **Propagator flags**: `PROP-FIRE-ONCE` (bit 0), `PROP-EMPTY-INPUTS` (bit 1) on propagator struct.
2. **Tier 1 flush**: worldview==0 AND all worklist propagators have FIRE-ONCE|EMPTY-INPUTS AND no NAF pending → single-pass flush, skip BSP entirely.
3. **Self-clearing**: fired-set (hasheq per BSP session). After successful fire: remove from input cells' dependents. No future enqueuing.

**Belt-and-suspenders incident**: Initially added BOTH closure wrapper AND scheduler flags. User correctly flagged "belt-and-suspenders" as a workflow red flag. Investigation revealed: the Tier 1 check had a bitwise bug (`flags & (A|B)` passes when EITHER flag set, should require BOTH). Fix: `flags & (A|B) == (A|B)`. With correct Tier 1 check, scheduler-level mechanism is sufficient. Closure wrapper removed. Fire-once is purely a scheduler concept.

**Lesson**: Belt-and-suspenders masks bugs in the new mechanism. The old wrapper hid the Tier 1 detection bug. Removing it forced proper diagnosis.

#### Phase 5b/5c: Network Template + Tier 1 Direct Return + Boundary Normalization (`d998b06c`→`01de93f5`→`b4001cd7`)

**PPN boundary normalization insight**: normalize at the domain boundary (data entry), not at each comparison site. `normalize-solver-value` extracts raw values from AST nodes. Applied at: `build-discrimination-data`, discrimination propagators, S1 NAF handler. Removed all inline `normalize-for-compare` calls.

**Tier 1 direct fact return**: single-variant, fact-only relations bypass ALL solver infrastructure. Direct fact matching: normalize + compare + return. No cells, no propagators, no BSP.

**Performance results**:
- ATMS single-fact: **0.49us** (was 30.6us) — **62x speedup**
- ATMS is now **2.3x FASTER than DFS** for Tier 1 queries (0.49us vs 1.13us)
- Tier 1 lifted to universal dispatch (`stratified-solve-goal`) — benefits ALL strategies

**Network template** (Phase 5c): pre-built network forked O(1) per query. Scope-sensitive fire-once: direct writes when bitmask=0 (no tagging needed).

**Lesson**: The biggest optimization wasn't "faster BSP" — it was "skip BSP entirely." Tier 1 direct return eliminated the entire propagator infrastructure for trivial queries. The PPN boundary normalization insight (normalize at source, not at consumer) was key to making it work across AST/raw value representations.

#### Phase 6: Adaptive :auto Dispatch (`7d77d52a`)

`:auto` strategy now adaptive:

- NAF or guard in relation → ATMS (mandatory, needs worldview)
- Total alternatives ≥ threshold (configurable, default 256) → ATMS (parallel benefit)

## Lessons Distillation

### New lessons from this session

1. **Design mantra as live challenge**: "All-at-once, all in parallel, structurally emergent information flow ON-NETWORK" caught 6 architectural violations that tests missed. Codified in 4 rules files.

2. **Belt-and-suspenders IS a bug indicator**: when two mechanisms do the same thing, the overlap masks bugs in either one. The workflow rule was right.

3. **Normalize at the boundary (PPN insight)**: normalize values at data entry, not at each comparison site. Single normalization function, multiple consumers. Eliminated an entire class of AST/raw comparison bugs.

4. **Module Theory for scope decomposition**: R = C₁ ⊕ ... ⊕ Cₙ realized as bitmask layers on shared carrier cell, not separate cells with morphisms. Eliminated bridges and the tag-collapse bug class.

5. **Skip the mechanism, not optimize it**: Tier 1 direct return (skip BSP entirely) delivered 62x speedup. All BSP optimizations combined delivered negligible improvement. The biggest win was NOT making BSP faster — it was recognizing when BSP isn't needed.

6. **Racket parallelism ceiling**: OS thread wakeup ~8us dominates all cross-thread communication. No Racket-level mechanism can beat it for ~0.5us work granularity. Infrastructure is correct; payoff comes at self-hosting with heavier work.

7. **Measure before optimizing, measure after**: Phase 0b data guided the optimization targets. Phase 2d exhaustive benchmarking revealed the Racket ceiling. Phase 5 benchmarks showed the regression from Phase R. Each measurement changed the strategy.

8. **N+1 Principle**: When considering N options, ask "what's the N+1?" Applied to NAF stratification (threshold → fork-based S1), bridge composition (bridges → scope sharing), parallelism (faster BSP → skip BSP).

### Principles confirmed

- **Completeness over deferral**: fixing all 6 mantra violations NOW (Phase R) prevented cascading issues and produced a cleaner architecture than incremental fixes would have.
- **Prior art drives design**: PPN Track 1 (PU cells), Track 4 (typing PU), SRE Track 2D (PU micro-stratum), NTT interface/stratification — all informed the design. Designing without this context produced ad hoc solutions.
- **Data beats theory**: Phase 0b estimates were optimistic (N≈8-16 crossover). Actual measurements (N≈256) changed the strategy. Every approach was measured before adoption.

#### Watching

- **Phase T parity**: ~~2 wf-comparison test failures from adaptive dispatch routing NAF to ATMS.~~ RESOLVED in Phase T-a (`1eb8b8cc`, `35a39865`). See session log below.
- **Self-hosting performance path**: per-propagator work at ~5-50us in self-hosted compiler would drop BSP pool crossover to N≈25. The parallel infrastructure is ready.
- **Concurrency runtime**: SH Series design scope. Process language (`defproc`, `session`) integrated with type system. Lightweight processes as self-hosting deliverable, not Phase 0 FFI.
- **Dissolution entry pre-merge**: fact-row propagators write x and y in separate cell writes → separate tagged entries at same bitmask. Pre-merge step fixes product-worldview computation. Self-hosted compiler should use single-write scope-cell updates to avoid this.

#### Key Findings

1. **CWA semantics are per-`semantics` configuration**: "correct" depends on the configured semantics. The propagator solver must honor the same contract as DFS for each semantics mode.

2. **Discrimination is necessary but not sufficient for provability**: Static discrimination checks structural compatibility (are there matching alternatives?) but not runtime success (did the clause body actually succeed after NAF evaluation?). The DFS solver handles nested NAF chains correctly through recursive evaluation.

3. **Gating-only clause bodies are a gap in the propagator model**: When a clause body consists entirely of NAF/guard goals, no positive goals write to scope cells. Dissolution has nothing to read. Fix: explicit success markers under the gating bitmask.

4. **Per-variable writes need pre-merging**: Fact-row propagators write x and y in separate `net-cell-write` calls → separate tagged entries at the same bitmask. Product-worldview dissolution treats them as independent groups → cross-product creates compound worldviews (3|5=7) that combine independent fact rows. Pre-merge step consolidates same-bitmask entries before grouping.

5. **Compilation-sensitive Heisenbug led to root cause**: Adding debug code to the test file changed Racket's memory layout → hash iteration order → different tagged-cell-value entry ordering → exposed the duplicate product-worldview bug that was latent but hidden by lucky iteration order in the clean build.

6. **`raco make driver.rkt` does NOT recompile test files**: Tests aren't in driver.rkt's dependency graph. The suite runner's `precompile-modules!` handles both, but `--no-precompile` bypasses it. Stale .zo detection extended to warn about this.

#### BSP-LE Track 2B Addendum A1: Topology → General Stratum Unification
