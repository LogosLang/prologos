## Considerations

- **Context loss as architectural risk**: Server outages during Phases 7b-c led to implementation that diverged from Track 5's established patterns. The sync-back approach was the opposite of what the propagator-first architecture called for. Lesson: when resuming after context loss, re-read the prior track's design and PIR before implementing.
- **Dual-write is persistence + propagation, not redundancy**: The macros/warnings register functions write to both parameter (persistence across commands) and cell (propagation within command). This matches Track 5's `global-env-add` pattern exactly. Phase 7b-c's error was mischaracterizing this as "redundant dual-write."
- **Phase 7d was the real Phase 7 work**: The global-env lookup migration to module-network-ref was the Track 5 → Track 6 cutover that the Track 5 design doc explicitly planned. It was erroneously deferred as "not a dual-write."
- **Callbacks are a propagator-first anti-pattern**: When resolution logic lives outside the network and is injected via callbacks, it's a sign the logic should be a propagator. The stratified resolution loop already IS the correct control boundary; the callbacks are vestigial indirection from pre-Track-2 architecture.
- **Stratified prop-net architecture**: The most significant architectural insight of this track. Each resolution stratum as a propagator layer, with inter-stratum Galois connections. Correct-by-construction readiness (cell-based, not scan-based), efficient (O(changed) not O(total)), data-oriented (readiness and resolution are cell values). This pattern should be applied retroactively to existing infrastructure.
- **Lazy initialization > hard errors for correct-by-construction**: The ATMS bug existed because initialization was mandatory at a specific call site. Lazy init makes the system work regardless of entry path — correct by construction.
