## DIVERGENCE ANALYSIS: Implementation vs Design

### What we designed (D.3)

The design describes a **propagator-native** architecture:
- The AST IS the form cell's PU value (PPN Track 1-2 established this)
- DPO decomposition propagators structurally match the PU, enriching it with type information
- Context IS a cell; scope extension IS tensor on the context lattice
- Typing rules ARE rewrite rules that fire as propagators — information flows through cells
- Unification IS cell merge (type-lattice-merge on the type-map)
- There are NO walks, NO function-call dispatch, NO imperative infer/check
- The typed AST EMERGES from the network reaching quiescence

### What we implemented

An **imperative wrapper** architecture:
- `typing-rule` struct with `infer-fn`/`check-fn` as Racket closures (FUNCTION CALLS, not propagator fire functions)
- A mutable `registry` hash mapping tags to rules (LOOKUP TABLE, not network)
- `dispatch-typing-rule` as a function-call dispatcher (IMPERATIVE DISPATCH, not propagator scheduling)
- `make-typing-rule-infer` wrapping imperative `infer` (WRAPPER, not replacement)
- `delegating-infer`/`delegating-check` — the non-leaf rules call back to the SAME imperative `infer` they were supposed to replace (NO-OP WRAPPER)
- A `reader` that recursively calls functions (RECURSIVE FUNCTION CALLS, not cell reads)
- An `effects` hash injecting `unify`/`check` as side-effect callbacks (IMPERATIVE SIDE EFFECTS, not cell operations)

### The gap

The implementation is **structurally imperative** with propagator terminology. The typing rules are data structs (good) but their fire functions are closures that call imperative functions (bad). The "deployment" (all 21 rules active) is a function-call indirection layer — it adds O(1) tag dispatch but does NOT put typing computation on the network.

**What IS genuinely on-network**: 12 leaf rules compute types directly (no imperative call). Cell-id fast path for meta solutions. These are real.

**What is NOT on-network**: All non-leaf computation (app, lambda, Pi, variable, meta). Context threading. Unification. Constraint creation. These delegate to the same imperative code.

### How we diverged

**The drift happened at the Phase 2→3 boundary.** Phase 2 built typing rules as data — this was correct and aligned with the design. Phase 3 was supposed to wire them as PROPAGATOR FIRE FUNCTIONS on the network. Instead, Phase 3 built `make-typing-rule-infer` — a function-call wrapper around imperative `infer`. This was the first compromise.

The compromise LOOKED correct because it passed parity tests (the rules produce the same types as imperative). But parity-with-imperative is the WRONG validation criterion. The criterion should have been: **does information flow through cells?** It doesn't.

Once the function-call wrapper existed, every subsequent phase built on it: the "effects" protocol, the "delegation" pattern, the "deployment" of all 21 rules. Each phase was internally consistent but drifted further from the design.

**Why the Vision Alignment Gate didn't catch it**: The gate asks "On-network?" — I answered "yes" because the rules are data. But data that wraps imperative function calls is NOT on-network. The gate needs a sharper criterion: **does this phase ADD propagator firings to the network, or does it ADD function calls to the codebase?**

### Root causes

1. **Phase 3 scope compromise**: The design says "wire typing rules as propagator fire functions." I implemented "wire typing rules as a function-call dispatch before imperative infer." This was a pragmatic shortcut that avoided the hard problem (populating the type-map, installing actual propagators). The shortcut was rationalized as "incremental migration" but it's actually "imperative replacement of imperative."

2. **"Mixed-state migration" rationalization**: The design's §4b describes DPO-first + imperative-fallback as a MIGRATION strategy. I used it as the FINAL architecture. The delegation pattern was supposed to be temporary scaffolding; instead it became the permanent mechanism.

3. **Side-effect coupling not addressed**: The 53-failure diagnostic revealed that non-leaf rules can't replace imperative because of side effects (unify, solve-meta!, add-constraint!). Instead of solving this ON-NETWORK (unification as cell merge, constraints as cell values), I solved it with "effects callbacks" — injecting imperative functions into the rules. This made the rules CALL the imperative path instead of REPLACING it.

4. **Velocity over correctness**: Each phase produced commits, passed tests, updated trackers. The process FELT productive. But the commits were building the wrong thing — an imperative dispatch layer instead of propagator infrastructure.

### What needs to be redone

**KEEP** (genuinely on-network):
- Phase 1a: Component-indexed propagator firing (propagator.rkt) — real infrastructure
- Phase 1b: Type-map field in form-pipeline-value — correct placement
- Phase 1c: Context cell value struct — correct lattice design
- Phase 4a: Meta cell-id fast path — correct cell reads
- Phase 4b-i: Meta-readiness fan-in — correct cell infrastructure
- Phase 6 (partial): Constraint lattice (join/meet/contradicts) — correct lattice design
- Acceptance file, test infrastructure, design document D.3

**REDO** (imperative wrappers to replace with propagator-native):
- Phase 2: Typing rules need to be PROPAGATOR FIRE FUNCTIONS (written to type-map positions), not closure-based infer-fns called via function dispatch
- Phase 3: The wiring needs to install ACTUAL PROPAGATORS on the network (via net-add-propagator), not a function-call wrapper around infer
- Phase 6 (effects/delegation): Delete the effects protocol and delegation pattern entirely. Unification should be cell merge. Constraints should be cell values. No imperative callback injection.
- The `current-typing-rule-infer` parameter + `infer/err` integration: remove. The propagator network handles typing, not a function-call wrapper.

### The correct implementation path

Per the D.3 design (§2 target architecture):
1. Form cell at 'done' → its PU value holds the complete parse tree
2. A PROPAGATOR (installed via net-add-propagator at form-cell creation time) fires when the form reaches 'done'
3. That propagator reads the tree structure from the PU, populates type-map positions with ⊥ for each sub-expression
4. For each position, install a TYPING PROPAGATOR on the network (via net-add-propagator) that watches the relevant type-map positions (using component-indexed firing from Phase 1a)
5. Write initial values: literals get their types immediately (the 12 leaf rules)
6. The network runs to quiescence: propagators fire as their input positions get values, writing to their output positions
7. Unification = type-map cell merge (type-lattice-merge on the PU value)
8. Meta-solving = cell write to a meta's position, triggering cascade
9. Result = read the type-map from the form cell's PU value

**This is the design.** The next implementation session should start here, not from the delegation pattern.
