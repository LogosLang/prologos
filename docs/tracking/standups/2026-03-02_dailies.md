## Capabilities Phase 7a-7d: Dependent Capabilities

Parameterized capability types indexed by values. `(capability FileCap (p : Path))` declares
a type family where `[FileCap "/data"]` is path-specific file access.

### Key Architectural Insight
The elaborator does NOT import `unify.rkt`. Dependent index matching happens in the type
checker, not the elaborator. The elaborator resolves capabilities optimistically by functor
name; the type checker verifies indices via existing `expr-app` decomposition. Two-phase:
elaborate optimistically → type-check precisely.

### Phase 2c: Selection Type Checking (commit `b5ba86d`)
- **Field-gating**: Selection case in `expr-map-get` gates access to only `:requires`/`:provides` fields
- **QTT support**: `inferQ` for `map-get` on schema/selection fvar types; `checkQ` for map constructors
- **3 bug fixes discovered**:
  1. `known-type-name?` didn't check schema/selection registries → uppercase names like `NameOnly`
     auto-detected as type variables `{NameOnly : Type}`. Fixed by adding `lookup-schema`/`lookup-selection`.
  2. Selections not pre-registered in Pass 0 → forward references fail during spec processing.
     Fixed by adding selection pre-registration alongside schema.
  3. QTT `inferQ` for `map-get` only handled `(expr-Map K V)`, not schema/selection fvar types.
     Fixed with fallback delegating to `infer` from typing-core.
- 12 typing tests

### Lessons Learned

1. **Preparse ordering matters for type name detection**: Schema/selection names must be registered
   in preparse Pass 0 (before Pass 1 where specs are processed) so `known-type-name?` recognizes
   them and prevents auto-implicit binder detection.

2. **Sexp reader representation of Prologos keywords**: In sexp mode, `:foo` is a symbol `':foo`,
   `[:x :y]` becomes a list `'(:x :y)`. Use `syntax->datum` (deep) not `stx->datum` (shallow).

3. **`unless` is truthy in Racket**: `(unless cond body)` returns `(void)` when condition is true,
   and `(void)` is truthy. Never use `unless` in positions where the return value matters (e.g.,
   `for/or`, `cond` branches). Use explicit `if` with `#f` instead.

4. **QTT must parallel typing-core for new type patterns**: Every new type pattern added to
   `infer`/`check` in typing-core.rkt needs a corresponding `inferQ`/`checkQ` clause in qtt.rkt,
   or QTT will reject valid programs with multiplicity errors.

5. **Test in sexp mode: no `:` in spec**: Write `(spec f Point -> Nat)` not `(spec f : Point -> Nat)`.
   The colon is WS syntactic sugar that becomes a literal token in sexp mode.

### Phase 3a: Structured Path Parsing (commit `5d21089`)
- Parser converts `:address.zip` → `((#:address #:zip))`, `:address.*` → `((#:address *))`,
  `:address.{zip city}` → `((#:address #:zip) (#:address #:city))`
- Elaborator validates first segment of each path against parent schema
- Existing tests updated for new structured path format (paths now lists of segments, not flat keywords)
- **Bug fix: Schema field type qualification** — `schema-field-type->expr` returned unqualified
  `(expr-fvar 'Address)` while the type system used qualified `test::Address`. Fixed by qualifying
  user-defined type names in schema field datums at registration time (`qualify-type-datum` in macros.rkt).
- 21 tests in test-selection-paths.rkt

### Phase 3b: Deep Path Validation (commit `a409ce9`)
- Elaborator recursively validates all segments of deep paths against nested schemas
- `:address.zip` → validates `:address` in User, then `:zip` in Address
- `:name.foo` → errors because `name` is `String` (not a schema, can't traverse deeper)
- Brace-expanded paths also validated: `:address.{zip bogus}` catches `:bogus` not in Address
- 9 new tests (30 total in test-selection-paths.rkt)

### Phase 3c: Nested Field-Gating — DEFERRED
- When a selection has deep path `:address.zip`, accessing `.address` returns the full
  `Address` schema type (all fields accessible). Ideally it should return a sub-selection
  gating only `:zip`. Deferred — requires type-level sub-selection design.
- Added to DEFERRED.md

### Lessons Learned (Phase 3)

6. **Schema field types must be namespace-qualified**: Schema fields store type datums as raw
   symbols (e.g., `'Address`). When types are registered under qualified names (`test::Address`),
   field type lookups fail. Fix: qualify user-defined type names at schema registration time,
   not at lookup time.

7. **Test with spec+defn, not value construction, for nested schemas**: Constructing values of
   nested schema types (e.g., `(the User ($brace-params :address (the Address ...)))`) hits
   type-checking complexity. Use spec+defn patterns (`(spec f AddrZip -> Address)` / `(defn f [u] (map-get u :address))`) to test field-gating without value construction.

### Phase 4: Selection Composition (commit `fa288eb`)
- `resolve-includes` function resolves `:includes [A B]` by looking up named selections in registry
- `path-union` computes set-union with join semantics: wildcard `*` subsumes specific fields
- Cross-schema includes and unknown selection references produce clear errors
- 12 tests in test-selection-compose.rkt

### Phase 5: Schema Properties (commit `e27a3f8`)
- `:closed` schema-level property: parsed after schema name, stored in `schema-entry-closed?`
- `:default val` field-level property: parsed after field type, stored in `schema-field-default-val`
- `:check pred` field-level property: parsed after field type, stored in `schema-field-check-pred`
- Extended `schema-field` struct from 2 to 4 fields (keyword, type-datum, default-val, check-pred)
- `parse-field-properties` helper consumes `:default`/`:check` in any order after each field type
- Closed schemas reject unknown fields at construction time in typing-core.rkt `check`
- 16 tests in test-schema-properties.rkt

### Lessons Learned (Phase 4-5)

8. **Schema registry uses bare names**: Schema registry keys are unqualified (`'ClosedConfig`),
   not namespace-qualified (`'test::ClosedConfig`). `lookup-schema-by-name` in typing-core.rkt
   handles this by trying both full and short names. Direct `hash-ref` on the registry needs
   bare names.

9. **Stale compiled caches after struct changes**: Changing a struct definition (adding fields)
   requires recompiling ALL dependent files, not just the defining file. `raco make driver.rkt`
   triggers the full dependency chain.
