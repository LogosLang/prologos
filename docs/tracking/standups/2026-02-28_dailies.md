## Design Discussion: Eliminating Intermediate LSeq in Generic Collection Operations

### The Problem

Generic ops like `map`, `filter`, `reduce` from `collection-fns` currently route through LSeq as a universal hub:

```
Collection A → (Seqable to-seq) → LSeq A → transform → LSeq B → (Buildable from-seq) → Collection B
```

For PVec and Set, `to-seq` goes through **two** hops: `PVec → List → LSeq`. So a generic `map` on PVec does:

```
PVec → List → LSeq → LSeq → List → PVec   (5 intermediate allocations)
```

Meanwhile, native primitives that bypass this entirely already exist (`pvec-map`, `pvec-fold`, `set-fold`, `map-fold-entries`, etc.) — they just aren't wired into the generic dispatch path.

### Three-Layer Design

**Layer 1: Fold+Build (eliminates LSeq intermediary)**

Replace the `Seqable + Buildable` path with `Reducible + Buildable.conj`:

- New `Reducible` trait — left fold, natural for accumulator-style building
- Extend `Buildable` with `conj : C A -> A -> C A` — single-element append
- Keep `Foldable` unchanged — right fold for structural recursion
- Rewrite generic ops: `map f xs = reduce (λ acc x → conj acc (f x)) empty-coll xs`
- Source folds natively (via Reducible), target builds natively (via conj). No LSeq.

**Layer 2: Trait-Level Specialization (native same-type ops)**

When the compiler knows source type = target type AND a native op exists, bypass even fold+build. E.g., `map f (pvec : PVec A)` → `pvec-map f pvec` directly.

Two potential mechanisms discussed:

1. **Trait method overrides** — richer trait (e.g., `Mappable`) with default implementation via fold+build, overridable with native implementations. User-extensible: anyone can implement `Mappable` for their own types.

2. **Compiler rewrite rules** — recognize fold+build patterns on known types and emit native calls. Only benefits built-in types unless we expose a user-facing rule-definition mechanism (like GHC RULES).

**Decision**: Trait overrides are the primary mechanism (user-facing, extensible). Compiler rewrite rules are a future optimization layer.

**Layer 3: Transducer Fusion (chained pipelines)**

For `pvec |> map f |> filter g |> take 5`, compose transformations into a single reducer applied in one fold pass. Builds on Layer 1's fold+build foundation.

### Key Design Discussion: Clause-Style Constraint Matching

The Layer 2 dispatch question surfaced a deeper language design issue: how should Prologos handle **prioritized resolution over disjoint trait constraints**?

The problem: Layer 2 wants "if this trait constraint is satisfiable, dispatch here; otherwise fall back to that constraint." This isn't union types (value-level), isn't bundles (conjunction of constraints), and isn't trait aliasing — it's **pattern matching at the constraint level**.

Three design shapes were considered:

1. **Default methods in traits** — Haskell/Rust approach. The trait provides a default body using its supertraits. Every type with the supertraits gets the trait for free; types that want to specialize override.

2. **Instance priority** — Scala 3-style. Instances have explicit priority numbers; resolver picks the most specific. Essentially overlapping instances with a tie-breaker.

3. **Clause-style constraint matching** — most natural for Prologos given its logic programming roots. The function itself declares alternative constraint signatures:

```prologos
spec map {A B} {C}
  | (Mappable C) -> (A -> B) -> C A -> C B
  | (Reducible C) -> (Buildable C) -> (A -> B) -> C A -> C B

defn map
  | [$mappable f xs] [map-native $mappable f xs]
  | [$red $build f xs] [reduce $red [fn [acc x] [conj acc [f x]]] [empty-coll $build] xs]
```

The trait resolver tries clauses top-to-bottom — first satisfiable match wins. This is Prolog-style clause ordering applied to constraint resolution. It reuses the `|` clause syntax users already know, it's compositional, and it falls naturally out of the logic programming substrate.

**Implementation cost**: Make trait resolution fallible (try/fail) and add clause-level constraint alternatives to the resolver. Currently, unsatisfiable constraints are hard errors — they would need to become backtrackable choice points.

**Decision**: Shape 3 (clause-style constraint matching) is the design direction. It aligns with Prologos's identity as a functional-logic language. Layer 2 + constraint matching tracked as a separate design doc; Layer 1 proceeds first as it delivers the main performance win without needing any new dispatch mechanism.

### Decisions Made

1. **Extend Buildable** with `conj` — no backward compat concerns (zero users, experimental phase)
2. **Add `Reducible` trait** (left fold) alongside existing `Foldable` (right fold)
3. **Layer 1 first** — Reducible + Buildable.conj + rewrite generic ops. This is the 80% win.
4. **Layer 2 tracked separately** — clause-style constraint matching is interesting language design that deserves its own tracking and design doc
5. **MapEntry already complete** — Clojure-style approach implemented in Phase 3c, in prelude. Partially resolves the Map HKT tension (Map can participate in sequence world via `map-seq`/`map-from-seq`) but doesn't let Map satisfy `Collection` bundle constraints.
