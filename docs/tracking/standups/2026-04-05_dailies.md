### Failure triage: 17 files → Phase 3 acceptance criteria

Systematic triage of all 17 failing test files (59 failing tests). Read all failure logs. Result: **uniform root cause** — imperative trait resolution fires before type metas are solved.

Three categories, one disease:
- **A (10 files, 30 tests)**: Unsolved dict-meta `?metaNNNN` in output. Trait resolution never completed.
- **B (4 files, 23 tests)**: `no instance found for (Eq _)` — trait resolution fires but type arg is still `_`.
- **C (3 files, 6 tests)**: Compound — primarily trait resolution + secondary symptoms (boolrec arity, WS parse diagnostics, punify contradiction).

All 17 tagged as Phase 3 acceptance criteria in §7b of the design doc. Phase 3 is not complete until these 59 tests pass.

**Key insight**: the failures VALIDATE the Track 4B problem statement. The imperative pipeline's fragile ordering is the disease. S0→S1 stratification (type inference before trait resolution) is the cure. The failures are the acceptance test.

**Lesson**: "pre-existing" failures that share a root cause with your current track aren't pre-existing — they're your acceptance tests. Triaging them early converts noise into signal.

### Phase 1: Attribute Record PU (commit `90e4979e`)

Extended type-map from flat `(position → value)` to nested `(position → (hasheq facet → value))`. Core deliverables:

- `attribute-map-merge-fn`: two-level pointwise merge, facet-dispatched. Each facet has its own merge function and bot value. `:type` uses `type-lattice-merge`, `:context` uses `context-cell-merge`.
- `that-read` / `that-write`: the attribute record API from §14 of the design. Internal API, designed for future user-facing exposure.
- `pu-value-diff`: extended to produce compound `(position . facet)` changed-paths for nested records. Enables precise component-indexed firing — a typing propagator watching `(pos . :type)` doesn't fire when `(pos . :context)` changes.
- Component-paths: all updated to compound keys. Context-watching propagators use `(cons ctx-pos ':context)`, type-watching propagators use `(cons position ':type)`.
- Context propagators migrated to `:context` facet: `make-context-extension-fire-fn` writes to `:context`, `make-bvar-fire-fn/ctx-pos` reads from `:context`. No more context-cell-value special-casing in the merge function.
- Zero changes to ~150 SRE domain rules or fire function bodies (they call `type-map-read/write` which are thin wrappers).

**Design conversation before implementation**: agreed on nested hasheq over compound flat keys, facet-dispatched merge, and backward-compatible wrappers. Implementation matched design exactly — no divergences.

**Network Reality Check**: same propagator count, same information flow. Cell values richer, topology unchanged. Correct-by-construction refactoring confirmed.

Full suite: 7574 tests, 17 failures (same pre-existing set). Zero new regressions.

### Phase 3: Trait Resolution Propagators — S1 (commit `74f79506`)

**The acceptance test phase.** All 17 previously-failing files (59 tests) now pass individually.

Three discoveries drove the design:

1. **Type-narrowing cycle**: The type-narrows-constraints bridge can't fire because type-arg metas start at ⊥ in the ephemeral PU. **Solution**: meta-feedback in the APP propagator — when domain is a meta and arg has concrete type, write arg type to meta's position. This IS §3.5 "Unification = Merge" applied to meta-solving.

2. **Kind vs solution conflict**: The APP propagator's downward write puts `Type(0)` (the KIND of `?A`) at `?A`'s `:type` facet. The meta-feedback puts `Nat` (the SOLUTION). `merge(Type(0), Nat) = type-top` (contradiction). **Discovery**: `:type` conflates two semantic levels for metas. **Solution (Option C)**: skip downward write for meta positions — kind info is implicit from elaboration, `:type` reserved for solutions.

3. **Identity mismatch**: type-arg-exprs from trait-constraint-info vs Pi domain after substitution — confirmed these ARE the same expr-meta by eq? once the kind/solution conflict was resolved.

**Information flow chain** (the whole Phase 2+3 pipeline working end-to-end):
```
literal(0N) → Nat at arg-pos
  → APP fires, domain is ?A, arg is Nat → FEEDBACK: Nat at ?A-pos
    → type-narrows-constraints: Nat → narrow Eq to {Nat--Eq--dict}
      → constraint-one → S1 resolution → dict at ?D-pos
        → bridge: solve-meta! → main network sees resolved dict
```

**Suite results**: 17→12 failures. 5 files fully fixed in suite. 12 pass individually but fail in batch — batch-worker isolation issue (pre-existing, not regression).

**Lessons**:
- The `:type` facet conflation of kind vs solution was invisible until the feedback mechanism tried to use it. The design conversation caught it: "Does this hurt self-hosting?" led to the feedback design, which revealed the conflict, which led to Option C.
- Option C (skip downward write) is 1 line of code. The architectural insight behind it took 3 rounds of design discussion. This is why we do design conversations before implementation.

### Phase 6: On-Main-Network Typing + Meta-Bridge Output Cell (commit `a8b597de`)

**The biggest architectural shift in Track 4B.** Typing evaluation now runs on the MAIN elab-network's prop-net, not an ephemeral PU. No separate network, no boundary, no scan.

**Design conversation highlights** (4 rounds of principled pushback):
1. "scan... iterate... loop... is it justified?" → recognized the imperative scan as step-think
2. "What are our principled options?" → identified §3.3 scoped-on-main as the target
3. "Could they not be in a single PU on network?" → confirmed single persistent cell vision (§9)
4. "We're not seeing efficiency until we retire the old" → honest about the cost/payoff timing

**What was removed**: ephemeral PU creation (`make-prop-network`), constraint-metas collection, `current-constraint-meta-positions` parameter, `install-typing-network/with-constraints` wrapper, post-quiescence scan bridge.

**What was added**: meta-bridge propagators (per-meta, watches :type, writes to output cell), meta-solution output cell (monotone list, append-merge), elab-network unbox→modify→rebox pattern.

**What remains as scaffolding**: `solve-meta!` calls from ONE cell read (output cell) to bridge to the imperative CHAMP meta-store. Phase 9 retires these.

**Suite**: 8 failures (down from 10). 1 new batch timeout (accumulation on main network — passes individually). Suite time 164.8s (20% above baseline — at investigation threshold).

**Observation**: The 20% suite time increase is the cost of running on the main network. Old commands' inert propagators exist in the dependents CHAMP. Component-indexed firing prevents them from firing, but `filter-dependents-by-paths` still folds over them. For large batch runs, this accumulates. Mitigation: propagator cleanup after each command (future optimization).

### Phase 6b: P1 Initial Writes + P3 Cleanup (commit `6df6e7cc`)

Three optimizations recovering the performance:
- **P1**: Literals, type constructors, fvar write types directly during installation (no propagator). Eliminates ~25-30% of propagators per command. Net -46 lines.
- **P3**: `net-clear-dependents` clears propagators from cells after quiescence while retaining values. Zero scheduling overhead for next command.
- **Global cell (§9) attempted but reverted**: 178 failures from cross-command position contamination. Per-command cells with cleanup are correct. Global cell needs position namespacing — positions are eq?-identity on expr nodes, fresh per command, so collisions SHOULDN'T happen. Needs investigation.

Suite: 7578 tests, 7 failures, 142.5s (recovered from 164.8s).

**New propagator patterns identified** (for design docs / PTF):
- **P1 (initial writes)**: Constants write directly, no propagator. Good for any value known at installation time.
- **P2 (self-cleaning)**: Fire-once propagators emit cleanup topology request. Not yet implemented; noted for future.
- **P3 (per-command cleanup)**: `net-clear-dependents` clears cell dependents bulk. Retains values. Good for any cell with transient computation.
