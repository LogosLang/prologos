### WFLE Phase 4a: Well-Founded Engine Core (commit `333130a`)
New `wf-engine.rkt` (~350 lines) — the orchestration layer that wraps the existing DFS solver with bilattice cells to compute three-valued well-founded semantics.

**Architecture: Hybrid DFS + Bilattice**
- Reuses the existing DFS solver (`solve-goal`) as-is — no fork or rewrite
- Adds a bilattice layer on top: one bilattice-var per predicate name (Phase 4a uses predicate-level granularity)
- NAF oracle (`current-naf-oracle` parameter in relations.rkt) intercepts negation-as-failure: returns `'succeed` (definitely false), `'fail` (definitely true), or `'defer` (unknown → skip clause conservatively)
- Iterative fixpoint: probe predicates → update bilattice bounds → check stability → repeat until convergence. Max iterations = max(10, 2n) for n negation-participating predicates.

**Key design decisions:**
- `current-naf-oracle` is a parameter, not a field: clean separation — oracle is optional, standard NAF path completely unchanged when oracle is `#f`
- Defer tracking via `current-naf-deferred?` parameter: distinguishes conclusive "no proofs" (upper := #f) from inconclusive "NAF deferred" (leave as unknown). Without this, stratifiable negation like `a :- not b. b :- not c. c.` would incorrectly mark `a` as definitely false in early iterations when NAF for `b` defers.
- `wf-relation-info->dep-info` inlined to break circular dependency (wf-engine.rkt ↔ stratified-eval.rkt). Alternative was restructuring module hierarchy, but inlining is simpler for a small function.
- Solver dispatch via `solver-config-semantics` key: `'stratified` (default) or `'well-founded`. Added to `solver.rkt` valid keys/defaults/accessor.

**Components:**
- `wf-solve-goal`: top-level entry — transitive pred closure, negation identification, bilattice creation, iterative fixpoint, answer annotation
- `wf-iterate`: fixpoint loop with stability check
- `make-wf-naf-oracle`: creates oracle from bilattice state, tracks defer via side-effect parameter
- `probe-single-predicate`: facts → lower := #t, no clauses → upper := #f, clauses → DFS with oracle
- `annotate-answers`: tags DFS answers with 'definite or 'unknown based on final bilattice state
- `wf-answers->standard`: converts to standard hasheq format ('strict filters unknowns, 'all adds certainty tag)
- `wf-explain-goal`: explanation with undeterminacy cycle identification via Tarjan SCC

**Solver dispatch wired in `stratified-eval.rkt`:**
- `stratified-solve-goal` checks `solver-config-semantics` — 'well-founded dispatches to `wf-solve-goal` + `wf-answers->standard`, else original stratified path
- `stratified-explain-goal` similarly dispatches

**Tests (24):** NAF oracle (4), bilattice stability (2), transitive closure (2), preds-with-negation (1), answer conversion (2), solver config (2), WF solve basic queries (3), WF-specific (odd cycle, self-reference, mixed definite+unknown) (3), solver dispatch (3), edge cases (2)
- E2E tests through `:with solver-config` syntax deferred to Phase 6 (WS syntax not wired yet)
- Full suite: 7019 tests, 364 files, 204.4s, all pass

**Lessons:**
- Circular dependency is the #1 gotcha when adding a new module that both provides to and requires from existing modules. Plan the dependency direction early.
- The `replace_all` edit mode can double-prefix names: renaming `relation-info->dep-info` to `wf-relation-info->dep-info` also renamed already-prefixed occurrences to `wf-wf-relation-info->dep-info`. Use targeted edits instead.
- Upper bound conservatism matters: without defer tracking, the fixpoint converges to the wrong answer for programs where negation is stratifiable but the iteration hasn't stabilized yet.

### WFLE Phase 5: Three-Valued Tabling Extension (commit `9bc99fc`)
Extended `tabling.rkt` with three-valued table entries for the well-founded engine.

**What was added (~80 lines new code):**
- `wf-table-entry`: extends `table-entry` with a `certainty` field (`'definite` | `'unknown`)
- `wf-all-mode-merge`: merge function that deduplicates by answer value, preferring `'definite` over `'unknown` when the same answer appears with different certainties — monotonic in the certainty ordering
- `wf-table-register`: creates a table with `wf-all-mode-merge` and initial certainty `'unknown`
- `wf-table-add`: adds `(answer . certainty)` pair to a three-valued table
- `wf-table-answers`: reads `(listof (cons answer certainty))`
- `wf-table-complete`: marks table as complete with a final certainty
- `wf-table-certainty`: reads the table-level certainty tag

**Design choices:**
- Subclassing via `struct ... table-entry`: `wf-table-entry` extends `table-entry`, so all existing `table-entry` accessors (name, cell-id, answer-mode, status) work on WF entries. Existing code that uses `table-entry?` also matches WF entries — no breakage.
- Cell stores `(cons answer certainty)` pairs, not bare answers: the merge function keys on `car` (answer) for dedup, and resolves certainty via subsumption (`'definite` > `'unknown`). This is cleaner than a separate certainty registry.
- WF tables coexist with regular tables in the same `table-store` — no parallel infrastructure needed.

### WFLE Phase 4b: Engine + Tabling Integration (commit `7281c1a`)
Connected the WF engine's bilattice fixpoint with three-valued tabling — after convergence, answers are stored in WF tables with per-predicate certainty.

**What was added:**
- `wf-solve-goal-tabled`: wraps `wf-solve-goal` to populate WF tables after fixpoint. For each negation-participating predicate, registers a WF table, stores answers with certainty derived from the bilattice (`'definite` if bilattice reads true/false, `'unknown` if bilattice reads unknown), then marks the table complete.
- `current-wf-table-store` parameter: holds the WF table store during tabled evaluation
- `stratified-solve-goal` now parameterizes `current-wf-table-store` and uses `wf-solve-goal-tabled` for the `'well-founded` path

**Design choices:**
- Per-predicate certainty (not per-answer): the bilattice operates at predicate-level granularity, so all answers for a predicate inherit the same certainty. Per-variant NAF refinement would require changing the oracle interface to accept ground arguments, which is a future enhancement.
- No-negation fast path also stores in WF tables: even when the fast path is taken (no negation-participating predicates), answers are stored with `'definite` certainty for API consistency.
- Table store is a parameter, not a return value: allows nested calls and composability without changing the `wf-solve-goal` return signature.

This allows unstratifiable predicates (odd cycles defined for WF use) to coexist with stratifiable `solve` queries in the same program. Previously, defining `p :- not q. q :- not p.` anywhere in the program poisoned ALL subsequent `solve` calls, even for unrelated relations.

2. **Cycle-scoped bilattice tracking** (`wf-engine.rkt`): `preds-with-negation` rewritten to use Tarjan SCC + negative edge detection. Now returns only predicates in actual negative SCCs — stratifiable negation targets (like `blocked` in `reachable :- edge, not blocked`) no longer get bilattice vars. This was causing the WF engine to use predicate-level granularity for `blocked` (has facts → oracle says "true" → `not blocked` fails for ALL values), making it disagree with the stratified engine on stratifiable programs.

3. **Closed-world oracle for fast path** (`wf-engine.rkt` + `relations.rkt`): When the WF engine takes the no-cycle fast path, it now installs a `make-closed-world-oracle` that returns `'succeed` for undefined predicates and `'standard` for defined ones. The NAF handler in `relations.rkt` gains a new `'standard` oracle result that falls through to standard DFS NAF. This handles the edge case where `a :- not b` and `b` is undefined — the oracle correctly applies closed-world assumption.

## Lessons Learned

- **WS reader sub-list wrapping**: The WS reader wraps each indented continuation line as a sub-list. For keyword-value forms like `solver name :key val`, this produces `(solver name (:key val))` instead of flat `(solver name :key val)`. Need `flatten-ws-kv-pairs` to selectively flatten only `:keyword`-headed sub-lists (initial version was too aggressive and broke compound types like `(List Nat)` in schema tests).
- **`$brace-params` is wrong for solver config**: `{:key val}` in WS → `$brace-params` → `parse-map-literal` → treats values as expressions (variable references). Solver config values are symbols, not expressions. Need a dedicated sentinel (`$solver-config`) that routes to elaborator code that builds `expr-solver-config` with symbol values directly.
- **Syntax-aware predicates**: Parser args are syntax objects, not raw datums. `brace-params?` checks `(pair? d)` which is `#f` for syntax objects — must use `brace-params-stx?` which unwraps first.
- **Circular dependency via callback**: `relations.rkt` can't import `whnf`/`nf` from `reduction.rkt` (circular). Solution: `current-is-eval-fn` parameter set via `parameterize` in reduction.rkt.
- **Parser context sensitivity for `is` expression**: `parse-datum` in relational context produces `surf-goal-app` for `(add ?x ?y)` — must `parameterize ([current-parsing-relational-goal? #f])` when parsing `is` expression arg.
- **`#f` goal propagation**: When `expr-guard` has `#f` goal (1-arg form), ALL pipeline stages that traverse it (zonk×3, substitution×2, reduction, pretty-print) must guard against it. Easy to miss one.
- **Descending cells need no net-cell-write changes**: The existing `merge-fn(old, new)` + `merged = old` check is direction-agnostic. For ascending cells merge-fn = join (raises); for descending cells merge-fn = meet (lowers). Only the constructor and direction registry are new.
- **Stale .zo cache on struct field addition**: Adding `cell-dirs` to `prop-network` requires recompiling ALL transitive dependents, not just `driver.rkt`. Modules like `session-propagators.rkt` and `trace-serialize.rkt` that import `prop-network` by struct linklet fail with "expected 12 fields" if not recompiled.
- **`andmap` + `member` returns truthy, not `#t`**: Racket's `andmap` returns the last truthy value, and `member` returns the tail of the list. Use `(and (andmap ...) #t)` to coerce to proper boolean for `check-true`.

### WF Probe Fix + Expanded Acceptance Coverage (commit `3ea9ae9`)

**Bug**: `probe-single-predicate` in `wf-engine.rkt` used `expr-logic-var` structs as goal-args for probing parameterized predicates. The DFS solver's `unify-terms` only treats plain symbols as logic variables — `expr-logic-var` structs are opaque values that can't unify with ground terms. This caused the first positive goal (e.g., `(quaker x)`) to silently fail during the probe, preventing the NAF oracle from ever being consulted. The probe concluded "definitely false" when the result was actually inconclusive.

**Fix**: One line — `(define goal-args query-vars)` instead of wrapping each in `(expr-logic-var qv 'free)`.

**Impact**: The Nixon diamond (C6), win/lose game (C7), and any WF program with parameterized predicates was giving wrong results (definite answers instead of unknown).

**Lesson**: The DFS solver's `unify-terms` treats only Racket symbols as logic variables. Any other Racket value (structs, lists, strings) is treated as ground. When constructing probe goals, use the same representation the solver expects — plain symbols, not AST nodes.
