### WFLE Phase 2: Bilattice Module (commit `edb3fad`)
New `bilattice.rkt` module — the abstraction layer that pairs ascending and descending cells into bilattice variables with three-valued reading.
- `lattice-desc` struct: bot, top, join, meet, leq — parameterizes the base lattice
- `bool-lattice`: Boolean lattice {false < true} for standard logic programming
- `make-set-lattice`: powerset lattice over a universe (for non-ground extensions)
- `bilattice-var`: pairs lower (ascending, starts bot) + upper (descending, starts top) cells
- `bilattice-new-var`: creates both cells, optional `#:consistency-check?` propagator
- `bilattice-read`: returns `(list 'exact val)` | `(list 'approx lo hi)` | `'contradiction`
- `bilattice-read-bool`: convenience — returns `'true` | `'false` | `'unknown` | `'contradiction`
- `bilattice-lower-write` / `bilattice-upper-write`: monotone writes via underlying cell merge-fns
- `bilattice-add-consistency-propagator`: enforces lower <= upper, sets contradiction if violated
- Design choice: consistency propagator is optional (`#f` by default for Boolean lattice where the invariant is maintained by construction; `#t` for non-Boolean or testing)

## Lessons Learned

- **WS reader sub-list wrapping**: The WS reader wraps each indented continuation line as a sub-list. For keyword-value forms like `solver name :key val`, this produces `(solver name (:key val))` instead of flat `(solver name :key val)`. Need `flatten-ws-kv-pairs` to selectively flatten only `:keyword`-headed sub-lists (initial version was too aggressive and broke compound types like `(List Nat)` in schema tests).
- **`$brace-params` is wrong for solver config**: `{:key val}` in WS → `$brace-params` → `parse-map-literal` → treats values as expressions (variable references). Solver config values are symbols, not expressions. Need a dedicated sentinel (`$solver-config`) that routes to elaborator code that builds `expr-solver-config` with symbol values directly.
- **Syntax-aware predicates**: Parser args are syntax objects, not raw datums. `brace-params?` checks `(pair? d)` which is `#f` for syntax objects — must use `brace-params-stx?` which unwraps first.
- **Circular dependency via callback**: `relations.rkt` can't import `whnf`/`nf` from `reduction.rkt` (circular). Solution: `current-is-eval-fn` parameter set via `parameterize` in reduction.rkt.
- **Parser context sensitivity for `is` expression**: `parse-datum` in relational context produces `surf-goal-app` for `(add ?x ?y)` — must `parameterize ([current-parsing-relational-goal? #f])` when parsing `is` expression arg.
- **`#f` goal propagation**: When `expr-guard` has `#f` goal (1-arg form), ALL pipeline stages that traverse it (zonk×3, substitution×2, reduction, pretty-print) must guard against it. Easy to miss one.
- **Descending cells need no net-cell-write changes**: The existing `merge-fn(old, new)` + `merged = old` check is direction-agnostic. For ascending cells merge-fn = join (raises); for descending cells merge-fn = meet (lowers). Only the constructor and direction registry are new.
- **Stale .zo cache on struct field addition**: Adding `cell-dirs` to `prop-network` requires recompiling ALL transitive dependents, not just `driver.rkt`. Modules like `session-propagators.rkt` and `trace-serialize.rkt` that import `prop-network` by struct linklet fail with "expected 12 fields" if not recompiled.
- **`andmap` + `member` returns truthy, not `#t`**: Racket's `andmap` returns the last truthy value, and `member` returns the tail of the list. Use `(and (andmap ...) #t)` to coerce to proper boolean for `check-true`.
