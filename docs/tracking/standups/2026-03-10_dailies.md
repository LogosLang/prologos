## Key Lessons & Debugging Insights

### Audit Methodology
- Many "CRASHes" turned out to be wrong audit syntax (GADT field-list syntax vs positional, wrong arg order for stdlib functions). Reclassification eliminated ~15 phantom issues — always verify against existing working examples before assuming a bug.
- Bottom-up fix ordering (reader → preparse → parser → typing → reduction) is essential. Reader fixes change the datum shapes preparse sees; preparse fixes change what the parser/elaborator receives.

### Preparse (macros.rkt)
- `expand-def-assign` is the central desugaring for `def x := val` and `def x : T := val`. Private `def-` needed its own entry point — the pattern-matching in macros.rkt is keyword-literal-sensitive.
- Multi-token RHS after `:=` requires auto-wrapping in brackets. Without this, `def x := add 1N 2N` parses as three separate top-level forms.
- Pass ordering matters: `ns`/`imports` must be processed in Pass -1 (before any other preparse) so that imported names are available for subsequent passes.
- Flat `$pipe` grouping: multi-clause `defn` uses `|` which the reader produces as `$pipe`. The preparse must flatten nested `$pipe` sequences before pattern-group compilation.

### Narrowing Engine
- **Shared variable constraint**: `narrow-match` result-side logic var handler was overwriting bindings without checking consistency. Target-side already had the check — made both sides consistent. This was the root cause of `my-double ?x = 6N` over-generating solutions.
- **Constructor inversion**: When `func-name` is `#f` (no function to narrow through), the engine can still structurally decompose constructor expressions against concrete targets. `narrow-constructor-match` in reduction.rkt handles `[suc ?n] = 3N` by peeling the constructor and recursing.
- **Boolrec narrowing**: `if` desugars to `expr-boolrec` but `narrow-subst-bvars` didn't traverse it, and `narrow-match` had no case for it. Both needed adding. When the scrutinee is a logic var, enumerate both Bool values. Attempted to use `whnf` from reduction.rkt but that creates a circular dependency — handled inline instead.
- **Circular dependency avoidance**: `narrowing.rkt` imports from `reduction.rkt` and vice versa. Adding new imports between them requires care — sometimes inline handling is better than cross-module calls.

### WS Reader (reader.rkt)
- Commas serve dual roles (separator vs unquote). The `current-qq-depth` parameter pattern cleanly separates concerns — backtick increments, unquote decrements, all comma-skip sites check `(zero? depth)`.
- The fix touches 8 separate parser functions but the change at each site is identical: `(eq? tt 'comma)` → `(and (eq? tt 'comma) (zero? (current-qq-depth)))`. Mechanical and safe.
- Unquote handler must decrement depth so commas in unquoted subexpressions (`` `(f ,(g a, b)) ``) revert to separator behavior.

### General
- Test expectations need updating when semantics change: Phase 5b constructor inversion made `?x = 5N` return `[{:x 5N}]` instead of `nil` — two test files expected the old behavior.
- The `pp-expr` export name (not `pretty-print-expr`) is a recurring gotcha when importing from `pretty-print.rkt`.

### Generic Numerics Audit (commit `d61669f`)

Systematic audit of Nat misuse across the codebase, cataloging every instance where Nat is used for computation instead of Int or generic `Num`. Produced a 350-line report covering:

- **16 NAT-API findings**: Collection indexing functions (`length`, `nth`, `take`, `drop`, `find-index`, `count`, `glength`) all hardcoded to Nat
- **15 NAT-PRELUDE findings**: Nat-specific `add`, `mult`, `sub`, `pow`, `double`, `pred`, comparisons imported unqualified
- **4 SHADOW findings**: `min`/`max` from `data/nat` shadow generic `ord-min`/`ord-max`; `sum`/`product` from `data/list` shadow generic algebra versions
- **5 root cause clusters** (D1-D5): prelude overexposure, collection API hardcoding, Nat-first examples, missing Int convenience, narrowing Nat dependency
- **22 repair tasks** across 5 priority levels, organized into sprint phases

Key design question identified: collection indices should stay Nat for type-level (Vec, dependent types) but provide Int alternatives for the common computational case.
