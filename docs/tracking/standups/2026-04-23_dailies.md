### Design refinement — broadcast at install layer (mid-audit dialogue)

User asked: "Is this also 'parallel ready' according to the 'all-at-once, all in parallel' part of our propagator mantra? We have some prior art on this." That prior art is broadcast (BSP-LE Track 2 Phase 1B: 2.3× faster at N=3, 75.6× at N=100 vs N-propagator model). The set-latch as initially codified used N fire-once propagators per fan-in site — the N-propagator model that broadcast outperforms.

After analysis: set-latch and broadcast are COMPLEMENTARY, not substitutes:
- Set-latch is the STRUCTURAL shape (latch + threshold + per-input watcher)
- Broadcast is the REALIZATION strategy for processing N independent items in 1 propagator with parallel decomposition profile

The architecturally-aligned fan-in uses BOTH: set-latch's structural shape + broadcast's realization at the watcher layer for the universe sub-set; per-cell legacy sub-sets fall back to fire-once until they migrate. For mixed-domain inputs, partition.

Multi-layer mantra alignment articulated:
- all-at-once: N watchers installed in one helper call
- all-in-parallel: 1 broadcast fire processes N items via broadcast-profile metadata; OR N independent fire-once propagators in parallel BSP rounds
- structurally emergent: latch state IS the readiness signal
- information flow: input change → watcher → latch → threshold → ready-queue
- on-network: every step is `net-cell-{read,write}`
