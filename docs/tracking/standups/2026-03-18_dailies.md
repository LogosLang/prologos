- **Tests discovered a real bug**: `run-retraction-stratum!` used merge-based `elab-cell-write` to write cleaned values. For monotone merge (`merge-hasheq-union`), `union(old, cleaned) = old` — the write was a no-op. S(−1) retraction was silently broken; masked by `restore-meta-state!` box snapshot.
- Fix: `net-cell-replace` in `propagator.rkt` — bypasses merge, replaces cell value directly. `elab-cell-replace` wrapper. `current-prop-cell-replace` callback. `run-retraction-stratum!` rewired.
- Without the PIR's honest self-assessment, this bug would have been discovered during Track 8 when trying to retire `restore-meta-state!` — much harder to diagnose and fix at that point.

### Considerations

- **PIR → tests → bug → fix chain validates the methodology**: The PIR's self-critique (§8.1: "no unit tests") led to writing tests that found a real bug (merge-based write in S(−1)). The bug was silently masked in production. This is exactly the kind of latent issue that PIRs are designed to surface. Without the methodology, this would have been a Track 8 debugging session.
- **`net-cell-replace` is a new primitive**: The propagator network now has three write operations: `net-cell-write` (monotone merge), `net-cell-write-widen` (widening merge), `net-cell-replace` (raw replacement). The replace operation is non-monotone — it's only safe within the S(−1) stratum (stratification contains the non-monotonicity). This should be documented in the propagator taxonomy.
