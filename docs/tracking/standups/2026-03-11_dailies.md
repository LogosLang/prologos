### Propagator-First Migration: Phase 3a (commit `dae48b7`)

**Two-layer architecture for global-env:**
- Layer 1: `current-definition-cells-content` — per-file definitions, checked first in lookups
- Layer 2: `current-global-env` — prelude/module definitions, frozen fallback

**Key design decisions:**
- Structural freeze: module loading parameterizes `current-global-env-prop-net-box #f`, so `global-env-add` falls back to legacy hasheq. Per-file processing activates cell writes via `process-command` parameterize.
- Scoped activation: `current-global-env-prop-net-box` set in `process-command`'s parameterize (auto-reverts when command finishes), preventing state leakage between test cases.
- Module env capture: `global-env-snapshot` merges both layers, fixing a subtle issue where `reset-meta-store!` creates a prop-net even during module loading (overriding the parameterized `#f`).

**Files changed (16):**
- `global-env.rkt`: Two-layer lookups, cell-write helper, register fn, snapshot, names
- `driver.rkt`: Callback installation, 3 register sites, scoped prop-net-box, module isolation, 6 error recovery sites, capability inference uses snapshot
- `capability-inference.rkt`, `cap-type-bridge.rkt`: `global-env-snapshot` instead of `(current-global-env)` for inference entry points
- `metavar-store.rkt`: `current-definition-cells-content` in `with-fresh-meta-env`
- `test-support.rkt`: All 6 parameterize blocks updated
- `batch-worker.rkt`: Per-file isolation for new parameters
- 9 test files: `global-env-snapshot` for state inspection

**Learnings:**
1. `reset-meta-store!` creates a prop-net even when `current-prop-net-box` is parameterized to `#f` — it mutates the parameter. This means the "structural freeze" design from the plan needed adjustment: can't rely on `(current-prop-net-box)` being `#f` inside `process-command` during module loading.
2. Functions with `[env (current-global-env)]` default args (capability-inference, cap-type-bridge) needed migration to `[env (global-env-snapshot)]` — these are "read" sites that the plan didn't enumerate.
3. Batch worker needs per-file isolation for `current-definition-cells-content`, `current-definition-cell-ids`, and `current-global-env-prop-net-box` — without this, 12 tests failed due to cross-file leakage.
4. Test files using `run-capturing` to inspect `(current-global-env)` directly needed `(global-env-snapshot)` — architectural change affects 83 files but only 9 actually inspect the captured env.

- Added §9.8 "Reduction Cache Cells" to LSP Stage 2 roadmap (`2026-03-11_LSP_VSCODE_STAGE2_REFINEMENT.md`)
- Key insight: this is purely an LSP concern — batch mode has no invalidation problem (caches are populated once and discarded). The question only arises when a user edits a definition and the LSP needs incremental re-elaboration.
- Two approaches documented: fine-grained (per-reduction-entry cells, precise but potentially expensive) vs coarse-grained (bulk invalidation per dependency set, zero batch overhead)
- Recommendation: start with coarse-grained — dependency graph from Phase 3b already identifies which definitions to re-elaborate; just clear caches for those. Fine-grained is a potential optimization gated behind `current-track-reduction-deps?` parameter.
- 5 explicit benchmarking requirements defined: baseline measurement, per-call overhead, cache hit rate, re-reduction cost, 5% regression gate

### Whole-System Propagator Migration Design Document

**Design document** (commit `78bf556`): `docs/research/2026-03-11_WHOLE_SYSTEM_PROPAGATOR_MIGRATION.md`

Articulates the architectural thesis for completing the dual-write → cell-primary migration. Key sections:
- §2: Information-theoretic argument (parameters are black holes; cells carry provenance, dependencies, nogoods)

- §4: Three-stage migration architecture — (1) query interface design, (2) read migration in 4 tiers, (3) self-hosting bridge
- §5: Interaction with deferred work (P-Unify, session runtime, effectful computation, extended specs, homoiconicity Phase IV)
- §6: Probabilistic extensions — confidence-weighted assumptions, lattice-valued propositions, Bayesian networks as propagator networks, dependent type `Belief A p`

Motivated by K-Framework/KEVM formal verification experience: three separate systems, fifteen-plus people, hand-bridged semantic gaps. Prologos thesis: eliminate gaps by construction — paper model, operational semantics, implementation, and verification share one substrate.
