### IO-F: Linear Handle Type â€” Complete (2 sub-phases) (commit `d249098`)

**IO-F1: Linear Handle Type**
- Created `lib/prologos/core/fio.prologos` with `Handle` ADT (`data Handle  mk-handle : Nat`)
- Integer port table in `io-ffi.rkt`: `fio-port-table` maps Nat indices to Racket ports (avoids `Opaque:*` naming issues in WS reader)
- Linear functions use sexp `(def name : [Pi [h :1 <Handle>] ...] body)` â€” `spec`/`defn` cannot express `:1` multiplicity
- Data-flow FFI trick: `fio-write-ret` returns handle-id (not void) so the FFI call is in the data flow and always evaluated by the lazy reducer
- Read-and-cache pattern: `fio-read-ret` reads eagerly + caches data + returns handle-id; `fio-read-cached` retrieves cached data â€” forces read before close in bracket pattern

**IO-F2: Bracket Pattern**
- `fio-with-file` opens file, runs body, closes handle, returns result
- Match-on-unit forces close: `(match (fio-close (first p)) (unit -> (second p)))` â€” evaluating the scrutinee forces the close side effect
- QTT enforcement: 4 tests verify that `:1` Handle must be used exactly once (positive: type-checks; negative: unused handle â†’ multiplicity error; negative: double-use â†’ multiplicity error)

**Type Checker Fix: Let + Match (beta-redex check propagation)**
- Discovered general type checker bug: `infer` has no case for `expr-reduce` (match expressions), falls to `[_ (expr-error)]`
- When `check` encounters `let` expressions (desugared as `(expr-app (expr-lam ...) arg)`), no special case existed, falling to conversion fallback which calls `infer` â€” fails on match bodies
- Fix: Added `check` case for the let pattern (beta-redex) that propagates expected type into body with `shift 1 0` to account for the new binder context
- This is a general fix benefiting all let+match combinations, not IO-F specific

**Key Discoveries**:
- Global env stores elaborated/zonked expressions (NOT reduced values). Every reference re-evaluates the expression, causing FFI side effects to re-execute. Multi-`def` chains with side effects produce different port IDs on each evaluation. Solution: use single-expression bracket pattern for test lifecycle operations.
- `spec`/`defn` CANNOT express `:1` linear multiplicity â€” must use sexp `(def name : Type body)` form for linear functions
- Data-flow trick essential for lazy evaluation: embed FFI calls in constructor arguments that MUST be evaluated (e.g., `(mk-handle (fio-write-ret idx data))`)
