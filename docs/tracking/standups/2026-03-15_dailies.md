## Lessons Learned

- **Direct parameter mutation leaks outside elaboration scope**: `register-macros-cells!` sets `current-macros-prop-net-box` and cell-id parameters via direct `(current-X val)` mutation (not `parameterize`). These persist beyond the elaboration scope, causing test assertion failures where cell reads see stale/cumulative data. Solution: `current-macros-in-elaboration?` boolean set via `parameterize` in `process-command` provides proper scoping.
- **Elaboration-context guard is essential for all phases**: The `macros-cell-read-safe` + `current-macros-in-elaboration?` pattern is the structural boundary between elaboration reads (cell-primary) and non-elaboration reads (parameter fallback). Must be reused for all subsequent phases.
- **Phase 2 was purely mechanical**: Once the guard pattern was established in Phase 1, Phase 2 applied identically to 7 more registries with zero surprises. This validates the design doc's "mechanical pattern" characterization for Phases 1-4.
- **Narrowing requires its own elaboration guard**: Even though `current-macros-in-elaboration?` already exists, narrowing cells live in `global-constraints.rkt` (no macros.rkt import). Added `current-narrow-in-elaboration?` to avoid circular deps. Test failure from missing guard: all-different constraint returning extra solutions because cell read returned stale `'()` instead of parameterized constraints.
- **Warnings DON'T need elaboration guard**: Warnings only read inside process-command (driver.rkt), so there's no code path where they'd read cells outside elaboration. This saved a parameter.
