## Reducible / Layer 1 Collection Generics

### Design Discussion (commit `cdd6e99`)

Three-layer architecture for eliminating LSeq intermediary:
- **Layer 1**: `Reducible` trait (left-fold) + `Buildable.conj` — value-reducing ops fold natively
- **Layer 2**: Trait-level specialization via clause-style constraint matching
- **Layer 3**: Transducer fusion for O(n) chained pipelines

Key design decision: **clause-style constraint matching** (Prolog-style clause ordering over constraint alternatives) chosen as Layer 2 dispatch mechanism. Aligns with Prologos's logic programming identity.

### Implementation (commits `c6372c6` through `6cc204f`)

- **Traits**: `Reducible` (single-method, dict IS the reduce function) + `Buildable.conj` extension
  - Buildable dict: `(pair from-seq empty-coll)` → `(pair from-seq (pair empty-coll conj))`
- **Instances**: 4 Reducible (List, PVec, Set, LSeq wrapping native folds) + 4 Buildable `conj` updates
- **Generic ops rewrite**: `reduce`, `length`, `any?`, `all?`, `find` switched from Seqable→LSeq to Reducible
- **Collection bundle**: expanded to 4 traits (Seqable, Buildable, Foldable, Reducible)
- **Infrastructure**: PRELUDE 87→92 entries, namespace.rkt regenerated, dep-graph.rkt updated
- **Tests**: `test-reducible.rkt` — 26 tests, full regression 4664 tests pass

### Considerations
- List `conj = cons` reversal: `map`/`filter` can't use Reducible without Layer 2
- Map HKT: 2-type-param can't implement 1-type-param traits — needs partial application
- `lseq-fold` type arg order swapped vs Reducible convention; instance wraps to compensate

## Design Discussion: Capability Security → I/O Design Reordering

User critique: the I/O library design jumped ahead of prerequisite design work. The correct dependency chain:

1. **Capability Security model** (principles-level) — how authority is expressed in the type system
2. **Session Types design** (language-level) — prerequisite for protocol-governed I/O
3. **I/O library** (application of 1 + 2)

### Key Insight: World Token Was Wrong Starting Point
Mercury/Clean needed the World token because they lacked richer machinery. Prologos has QTT + dependent types + session types — we can express fine-grained capabilities, not just "does I/O."

### Decision: Capabilities as Implicit Parameters
- `{Fs : ReadCap}` in spec — like trait constraints, resolved implicitly
- Traits are ambient (globally registered); capabilities are NOT (lexically scoped)
- Resolution from call chain, not global registry — a separate mechanism from trait resolution
- `:0` for authority checking (zero-cost, erased), `:1` for authority transfer (linear)

### QTT Multiplicity ↔ Capability Mode Mapping
| QTT | Capability Mode | seL4 Analogue |
|-----|----------------|---------------|
| `:0` | Authority proof (compile-time only) | Capability slot type-checked |
| `:1` | Transferable authority | Capability transfer/delegation |
| `:w` | Ambient authority (use sparingly) | Global objects |

User critique identified two errors in `CAPABILITY_SECURITY.md`:

## Capabilities as Types Design Document (commit `2f0eec6`)

Design doc: `docs/tracking/2026-03-01_1500_CAPABILITIES_AS_TYPES_DESIGN.md`

### Phase 1 Research (3 parallel research tracks)
- **seL4**: CNodes, CDT (capability derivation tree), 6 operations (Copy/Mint/Move/Delete/Revoke/Retype), formal verification, badge model → maps to QTT multiplicities and dependent type indices
- **Mark Miller / E / Agoric**: 5 capability discipline rules, confused deputy, eventual sends, membrane pattern, powerbox pattern, SES/Compartments, CapTP → validates ocap model, membranes map to linear scope exit
- **Fuchsia/Typed Caps**: Zircon handles/rights, FIDL `resource` keyword, Pony reference capabilities (6-cap lattice), WASI pre-opened directories, Crary-Morrisett capability calculus, Austral (linear+caps), Granule (graded modal types) → Crary-Morrisett validates `:0` as capability proof

### 5 Research Themes
1. Capabilities must be unforgeable and non-ambient
2. Attenuation is monotonic (rights only decrease) → subtyping
3. Authority root is special → `main` as powerbox
4. Linear types are the ideal enforcement mechanism → QTT
5. Static verification strictly superior to runtime checking → type system

### Gap Analysis (5 infrastructure gaps)
1. **Capability kind marker** — compiler can't distinguish capability types from trait types
2. **Lexical capability resolution** — separate from global trait instance resolution
3. **Capability inference through call graphs** — backward propagation of requirements
4. **Multiplicity defaulting** — capabilities default to `:0`, warn on `:w`
5. **Foreign function capability gating** — `foreign` imports need capability annotations

### Design Decisions
- `capability` keyword (desugars to `trait ... :capability`)
- Capability registry separate from trait instance registry
- Resolution: search function parameters → let-bindings → module grants → propagate to caller
- Inference: post-type-checking pass computes transitive capability closures
- Standard hierarchy: SysCap > FsCap (ReadCap | WriteCap), NetCap (HttpCap | WsCap), DbCap, SpawnCap, ClockCap
- **Supersedes I/O design's World token** — fine-grained erased capabilities (`{fs :0 ReadCap}`) replace threaded `World :1`

### 8-Phase Implementation Roadmap
1. Capability declaration + kind marker
2. Multiplicity defaulting + `:w` warning
3. Standard capability hierarchy
4. Lexical capability resolution (most significant phase)
5. Capability inference through call graph
6. Foreign function capability gating
7. Dependent capabilities (future)
8. Session types for capability protocols (future)

### Open Questions (6 for design discussion)
- `capability` as keyword vs trait annotation
- Module-level capability declarations
- Capability parametricity (`{C : Capability}`)
- Implicit vs explicit delegation
- Macro interaction
- Pattern matching on capabilities
