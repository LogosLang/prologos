## Reducible / Layer 1 Collection Generics

### Design Discussion (commit `cdd6e99`)

Three-layer architecture for eliminating LSeq intermediary:
- **Layer 1**: `Reducible` trait (left-fold) + `Buildable.conj` ŌĆö value-reducing ops fold natively
- **Layer 2**: Trait-level specialization via clause-style constraint matching
- **Layer 3**: Transducer fusion for O(n) chained pipelines

Key design decision: **clause-style constraint matching** (Prolog-style clause ordering over constraint alternatives) chosen as Layer 2 dispatch mechanism. Aligns with Prologos's logic programming identity.

### Implementation (commits `c6372c6` through `6cc204f`)

- **Traits**: `Reducible` (single-method, dict IS the reduce function) + `Buildable.conj` extension
  - Buildable dict: `(pair from-seq empty-coll)` ŌåÆ `(pair from-seq (pair empty-coll conj))`
- **Instances**: 4 Reducible (List, PVec, Set, LSeq wrapping native folds) + 4 Buildable `conj` updates
- **Generic ops rewrite**: `reduce`, `length`, `any?`, `all?`, `find` switched from SeqableŌåÆLSeq to Reducible
- **Collection bundle**: expanded to 4 traits (Seqable, Buildable, Foldable, Reducible)
- **Infrastructure**: PRELUDE 87ŌåÆ92 entries, namespace.rkt regenerated, dep-graph.rkt updated
- **Tests**: `test-reducible.rkt` ŌĆö 26 tests, full regression 4664 tests pass

### Considerations
- List `conj = cons` reversal: `map`/`filter` can't use Reducible without Layer 2
- Map HKT: 2-type-param can't implement 1-type-param traits ŌĆö needs partial application
- `lseq-fold` type arg order swapped vs Reducible convention; instance wraps to compensate

## Design Discussion: Capability Security ŌåÆ I/O Design Reordering

User critique: the I/O library design jumped ahead of prerequisite design work. The correct dependency chain:

1. **Capability Security model** (principles-level) ŌĆö how authority is expressed in the type system
2. **Session Types design** (language-level) ŌĆö prerequisite for protocol-governed I/O
3. **I/O library** (application of 1 + 2)

### Key Insight: World Token Was Wrong Starting Point
Mercury/Clean needed the World token because they lacked richer machinery. Prologos has QTT + dependent types + session types ŌĆö we can express fine-grained capabilities, not just "does I/O."

### Decision: Capabilities as Implicit Parameters
- `{Fs : ReadCap}` in spec ŌĆö like trait constraints, resolved implicitly
- Traits are ambient (globally registered); capabilities are NOT (lexically scoped)
- Resolution from call chain, not global registry ŌĆö a separate mechanism from trait resolution
- `:0` for authority checking (zero-cost, erased), `:1` for authority transfer (linear)

### QTT Multiplicity Ōåö Capability Mode Mapping
| QTT | Capability Mode | seL4 Analogue |
|-----|----------------|---------------|
| `:0` | Authority proof (compile-time only) | Capability slot type-checked |
| `:1` | Transferable authority | Capability transfer/delegation |
| `:w` | Ambient authority (use sparingly) | Global objects |

### 5 Research Themes
1. Capabilities must be unforgeable and non-ambient
2. Attenuation is monotonic (rights only decrease) ŌåÆ subtyping
3. Authority root is special ŌåÆ `main` as powerbox
4. Linear types are the ideal enforcement mechanism ŌåÆ QTT
5. Static verification strictly superior to runtime checking ŌåÆ type system

### Gap Analysis (5 infrastructure gaps)
1. **Capability kind marker** ŌĆö compiler can't distinguish capability types from trait types

5. **Foreign function capability gating** ŌĆö `foreign` imports need capability annotations

- `capability` keyword (desugars to `trait ... :capability`)
- Capability registry separate from trait instance registry
- Resolution: search function parameters ŌåÆ let-bindings ŌåÆ module grants ŌåÆ propagate to caller
- Inference: post-type-checking pass computes transitive capability closures
- Standard hierarchy: SysCap > FsCap (ReadCap | WriteCap), NetCap (HttpCap | WsCap), DbCap, SpawnCap, ClockCap
