### Memory as first-class characterization (4 distinct measurement points)

Per DESIGN_METHODOLOGY: "Memory cost is a separate axis from wall-clock... different axes catch different problems."

Tropical fuel substrate specifically has FOUR memory concerns:
1. **Per-operation allocation** — cell-write allocates tagged-cell-value entry per worldview vs counter's struct-copy
2. **Sustained retention** — long-running computations accumulate fuel-cell history under speculation; vs counter's bounded state
3. **GC pressure under load** — high-frequency decrement scenarios generate GC pressure proportional to allocation rate
4. **Per-consumer overhead** — multi-consumer scenarios test whether per-consumer fuel cells scale memory linearly

Every M/A/E/R test gets dual-axis (wall + memory) measurement via existing `bench-mem` macro (bench-ppn-track4c.rkt:73). R-series dedicated to memory-as-primary-signal.

## Parallel session continuation — language exploration + syntax convention codification

### prologos-syntax.md update — multi-line clause body convention

Codified the canonical layout for multi-line clause bodies in `.claude/rules/prologos-syntax.md` § "Pattern matching and dispatch". The form

```
defn nth [n xs]
  | n nil -> none
  | n [cons h t] ->
    match [eq n 0]
      | true  -> [some h]
      | false -> [nth [- n 1] t]
```

works (verified empirically) — body indented past the `|` becomes a continuation. Body at the SAME indent as `|` is a layout violation that currently produces a hard parser error (now tracked as Case 6 in #27).

The mechanism reuses the existing parse-reader indent-grouping rule that PR #11 (multi-line let body) builds on. Same architectural pattern: continuation lines indented past parent get wrapped into the parent's body. The convention is consistent across `defn` body, `def := body`, `let` body, AND clause `-> body` — once users learn the rule once, it generalizes.

## Late-day — Solver-output regression diagnosed and fixed

User flagged: `solve (color "apple" c)` returning `'[{:c unknown, :item_ "apple"}]` instead of `'[{:c "red", :item_ "apple"}]`. WF solver works correctly; default solver path broken.

### Root cause

`ground->prologos-expr` in `racket/prologos/reduction.rkt:261` is the formatter that converts solver-produced binding values to displayable AST. Its `cond` had cases for keywords, symbols, pairs, AST-wrapped values (including `expr-string?`), and an `else` clause that returns `(expr-fvar (if (symbol? v) v 'unknown))`.

**No case for raw Racket strings.** Raw strings fell through to the `else` clause and rendered as `unknown`.

### When it regressed

Commit `32d62c67` ("PUnify Phase 0a: wire structural unification through solve pipeline") rewrote `ground->prologos-expr` and added cases for keywords / symbols / pairs. Per `relations.rkt:188`: "resolved-args normalizes AST→raw (PPN boundary), but DFS expects AST." PUnify Phase 0a introduced this AST→raw normalization at the PPN boundary, but didn't update the formatter to handle the new raw types.

Pre-PUnify: solver output was AST-wrapped → `expr-string?` predicate caught everything.
Post-PUnify: raw strings/integers/booleans flow through → miss every case → render as `unknown`.

### Fix

Three lines added to `ground->prologos-expr`:
```racket
[(string? v) (expr-string v)]
[(boolean? v) (if v (expr-true) (expr-false))]
[(exact-integer? v) (expr-int v)]
```

- Bug repro fixed: `solve (color "apple" c)` → `[{:c "red", :item_ "apple"}]`
- 146 targeted tests pass: test-relations-runtime, test-solver-context, test-parser-relational, test-solver-parity, test-relational-types
- Full suite GREEN

Acceptance files (`examples/*.prologos`) document expected behavior with `;; =>` comments but are NOT part of the test suite. The bug's discovery path — user manually inspecting an acceptance file — is exactly what should be automated:

## Late-evening — Second solver bug: parser keyword `path` vs user `defr path`

User flagged: `solve (path "a" dest)` returning `(solve [?path "a" ?dest])` (unevaluated symbolic form) instead of running the solver. WF solver path was OK; this was the default-solve dispatch path again, but a different bug from the morning's `unknown` rendering issue.

### Root cause

`path` is a parser keyword in `racket/prologos/parser.rkt:1984`, with dual semantics:
- `(path :address.zip)` — first-class path literal (selection paths via keyword args)
- `(path "hello")` — falls through to "normal application" (line 1999) — i.e., `parse-application` producing `surf-app`, **bypassing relational-goal-context dispatch**

Inside `solve (path "a" dest)`, the parser was in relational-goal context (`current-parsing-relational-goal? = #t`). For non-keyword names this dispatches to `parse-goal-application` producing `surf-goal-app`. But the `path` keyword's else-branch went directly to `parse-application` — losing the relational dispatch.

Result: `(path "a" dest)` parsed as `surf-app` instead of `surf-goal-app`. Then the elaborator treated `path` via the relational-fallback rule (bare names → logic vars), producing `?path` as a logic var argument rather than a relation predicate name. The whole thing then printed as the unevaluated `(solve [?path "a" ?dest])`.

### Fix

One-block change at `parser.rkt:1999` — the else-branch now respects `current-parsing-relational-goal?`:

```racket
[else
 (if (current-parsing-relational-goal?)
     (parse-goal-application head args loc)
     (parse-application head-stx args loc))]
```

### Verification

- `solve (path "a" dest)` returns 4 answers (b, c, d, e — all reachable from a) ✓
- `solve (path src "e")` returns 4 answers (a, b, c, d — all that reach e) ✓
- 146 targeted tests pass: same set as the morning's fix
- Full suite: GREEN

### Why this kept working for `myroute`, `trans`, `color`

- `myroute`, `trans` — not parser keywords; default symbol-head dispatch correctly used `current-parsing-relational-goal?` to pick `parse-goal-application`
- `color` — same: not a parser keyword, default dispatch
- `path` — parser keyword, `else` branch had its own dispatch that ignored context

Other parser keywords with similar fall-through patterns (e.g., the generic-arithmetic keywords `+`, `-`, `*`, `/`, `lt`, `gt`, `eq`, `mod`) are unlikely to conflict with relation names, but the same pattern fix applies if they ever did. Surgical fix here; broader audit deferred unless another conflict surfaces.

### Lessons / observations

- **Parser-keyword fall-throughs need to respect relational context**: any `[else ...]` branch in a keyword case that calls `parse-application` should check `current-parsing-relational-goal?`. The codified pattern is "context-sensitive dispatch, not fixed dispatch."
- **The bug was extra-confusing because of the silent fall-through**: `(solve [?path "a" ?dest])` looks like a "result" but is actually an unevaluated form. Same class as the morning's `(expr-fvar 'unknown)` sentinel. Two regressions today, both with silent-wrong-answer presentation. Compounds the case for issue #41 (.prologos coverage gap) — neither bug had test coverage, both were caught by user inspection of acceptance file.
- **Two solver bugs in one day**: morning's `unknown` rendering, evening's `path` parser keyword. Both surfaced from the SAME acceptance file (`2026-03-14-wfle-acceptance.prologos`) and would have been caught by Layer 1 acceptance-file CI from #41. Concrete demonstration of the gap; the issue's priority rises.

## Late-evening — Parser-keyword fall-through audit

User direction: audit ALL parser keywords with the same fall-through pattern that hit `path`. After the morning's `unknown` rendering fix and the evening's `path` keyword fix, the user wanted a thorough pass: any other parser keywords with `[else (parse-application head-stx args loc)]` branches that ignore relational context.

### Audit findings — 22 fall-through sites, all migrated to context-sensitive helper

Found via `grep "parse-application head-stx args loc" parser.rkt`:

- **Generic arithmetic** (10 sites, lines 1746-1822): `+ - * / lt le gt ge eq mod`
- **Unary arithmetic** (2 sites, 1830, 1835): `negate abs`
- **Relational keywords** (4 sites, 2863-2909): `is not guard cut`
- **Constraint keywords** (4 sites, 2921-2957): `all-different element cumulative minimize`
- **Default symbol-head dispatch** (1 site, 3160): existing context-sensitive path; kept the redundancy because `parse-app-or-goal` correctly no-ops in non-relational context
- **Catch-all for non-symbol head** (1 site, 3164): KEPT as `parse-application` — compound expressions can't be relation names; goal-app branch would fail on non-symbol head

Plus the original `path` fix (line 2003), now simplified to just call the helper.

### The helper

`parse-app-or-goal head head-stx arg-stxs loc` defined near `parse-application` and `parse-goal-application`. Single point of policy:

```racket
(define (parse-app-or-goal head head-stx arg-stxs loc)
  (if (current-parsing-relational-goal?)
      (parse-goal-application head arg-stxs loc)
      (parse-application head-stx arg-stxs loc)))
```

Every keyword fall-through now calls this helper instead of `parse-application` directly. Centralizes the dispatch decision; future keywords only need to call the helper too.

### Verification

- `solve (path "a" dest)` works correctly (regression test from earlier today's fix)
- `solve (color "apple" c)` works correctly (morning's regression test)
- 254 targeted tests across solver/relations/parser-relational/relational-types + test-parse-reader
- Full suite: 8131/8132 tests pass; only the same pre-existing 2 failures in test-parse-reader (unbalanced parens in `2026-03-15-track3-acceptance.prologos`) remain — unrelated, tracked under #41

### Lessons / observations

- **Helper-as-policy-locus**: collapsing 22 sites to call a single helper means future keyword additions automatically get the right behavior IF authors use the helper. Worth a doc note in `propagator-design.md` or wherever parser keyword conventions live: "in fall-through branches that may receive a relational-goal, use `parse-app-or-goal`, not `parse-application`."
- **The audit was small in code volume but architecturally meaningful**: 54 lines added, 31 removed. The change is mostly mechanical (replace `parse-application` with `parse-app-or-goal`) but the architectural property — parser context-sensitivity at all keyword fall-throughs — is meaningful.
- **Some keyword names still cause deeper crashes when used as relation names** (e.g., `defr minimize` produces a zonk-time error). The audit fix addresses parser-level dispatch only; deeper elaboration/zonking issues with reusing keyword names are a separate concern. For names users actually reach for (`path` is the realistic case), the fix is complete.
