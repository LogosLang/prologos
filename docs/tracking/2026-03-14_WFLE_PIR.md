## 1. What Was Built

The WFLE is an alternate solver backend for Prologos's logic engine that computes the **well-founded semantics** of logic programs with negation. Where the existing stratified engine errors on programs with cyclic negation (e.g., `p :- not q. q :- not p.`), the WFLE assigns the three-valued answer `unknown` — the semantically correct result per Van Gelder (1991).

### Architecture: Hybrid DFS + Bilattice

The WFLE wraps the existing DFS solver — no fork, no rewrite — adding a bilattice layer on top:

- Each negation-participating predicate gets a **bilattice variable**: a pair of propagator cells (lower ascending / upper descending) whose gap encodes truth status
- A **NAF oracle** (`current-naf-oracle` parameter) intercepts negation-as-failure during DFS solving, answering `'succeed` / `'fail` / `'defer` based on current bilattice state
- An **iterative fixpoint** probes predicates, updates bilattice bounds, and checks stability until convergence

Users select the WF backend via solver configuration:

```
solver wf-solver
  :semantics well-founded
  :tabling by-default

solve-with wf-solver (paradox)   ;; → nil (unknown, no definite answers)
explain-with wf-solver (paradox) ;; → [{:__certainty unknown, :__cycle '["paradox"]}]
```

## 4. Bugs Found and Fixed During Validation

These are the correctness issues that only surfaced during Level 3 WS-mode validation — they would have been missed by unit tests alone.

### 4.1 Global Stratification Poisoning (commit `2e8f9f6`)

**Problem**: `stratified-solve-goal` stratified ALL predicates in the store, not just those reachable from the query. Defining `p :- not q. q :- not p.` (for WF use) poisoned all subsequent `solve` calls for unrelated relations.

**Fix**: Scoped stratification to only predicates reachable from the query via `transitive-pred-closure`.

### 4.2 Over-Broad Bilattice Tracking (commit `2e8f9f6`)

**Problem**: `preds-with-negation` returned ALL predicates that appeared in any negation, including stratifiable negation targets like `blocked` in `reachable :- edge, not blocked`. These got bilattice vars with predicate-level granularity, causing the NAF oracle to return wrong results (a predicate with any facts reads as "true" → `not` fails for ALL ground instances).

**Fix**: Rewrote `preds-with-negation` to use Tarjan SCC + negative edge detection. Only predicates in actual negative SCCs get bilattice vars; non-cyclic negation uses standard DFS NAF.

### 4.3 Probe Goal-Args Representation Mismatch (commit `3ea9ae9`)

**Problem**: `probe-single-predicate` wrapped query vars in `(expr-logic-var qv 'free)` structs. The DFS solver's `unify-terms` only treats plain Racket symbols as logic variables — `expr-logic-var` structs are opaque values that never unify with ground terms. Probes for parameterized predicates (Nixon diamond, win/lose) silently failed.

**Fix**: One line — `(define goal-args query-vars)` instead of wrapping in `expr-logic-var`.

### 4.4 Empty Explain for Unknown Atoms (commit `9c16fb3`)

**Problem**: `wf-explain-goal` returned empty list for unknown atoms like `paradox`. The bilattice had no definite answers to annotate, and `annotate-answers` iterates over an empty list producing `'()`.

**Fix**: Added a `cond` branch that checks bilattice state when `annotated` is null — generates `wf-explained-answer` with undeterminacy cycle info.

### 4.5 Explain Routing Bypass (commit `9c16fb3`)

**Problem**: `run-explain-goal` in `reduction.rkt` called `explain-goal` directly, bypassing the stratified/WF dispatch. `explain-with wf-solver (paradox)` crashed because standard `explain-goal` doesn't handle cyclic programs.

**Fix**: Route through `stratified-explain-goal`, handle both `answer-record` and `wf-explained-answer` return types.

## 5. Design Decisions and Rationale

### 5.1 Cut Intentionally Omitted

Prologos does not implement Prolog's `cut` operator. Cut is a procedural control-flow hack for managing chronological backtracking — it breaks declarative semantics, makes program behavior order-dependent, and is incompatible with WF/tabled evaluation. Prologos covers cut's legitimate use cases through:
- `solve-one` — committed choice (first answer only)
- `guard` — conditional filtering within clauses
- `|` (multi-arity dispatch) — deterministic pattern matching
- WF semantics — non-monotonic reasoning without procedural tricks

### 5.2 NAF Oracle as Parameter, Not Struct Field

`current-naf-oracle` is a Racket parameter (dynamic variable), not a field on `relation-info` or `solver-config`. This keeps the standard NAF path completely unchanged when the oracle is `#f` — zero overhead for non-WF evaluation, and the existing solver code doesn't need to know about WF semantics.

### 5.3 Defer Tracking for Correct Upper Bounds

The `current-naf-deferred?` parameter distinguishes "no proofs found" (conclusive → upper := `#f`) from "NAF deferred during probe" (inconclusive → leave as unknown). Without this, stratifiable negation like `a :- not b. b :- not c. c.` would incorrectly mark `a` as definitely false in early iterations.

### 5.4 Predicate-Level Granularity (Intentional Simplification)

The bilattice operates at predicate level (one bilattice-var per predicate name), not per ground instance. This is conservative — `win(a)` and `win(b)` share the same bilattice var, so if any instance is unknown, all are marked unknown. Per-variant refinement requires changing the oracle interface to accept ground arguments, deferred as a future enhancement.

## 6. Lessons Learned

### 6.1 Research-Heavy Design Pays for Itself

The 7-hour research+design phase (Stages 0–1) produced a 1532-line implementation guide with concrete Racket code. The 7-phase implementation took 73 minutes. The ratio — 6:1 design-to-implementation time — is extreme but correct for this kind of work. Every architectural decision was settled before touching production code. There were zero "oh wait, this doesn't work" moments during implementation.

### 6.2 Level 3 Validation Is Non-Negotiable

All five bugs in §4 were invisible to unit tests (Level 1) and WS string tests (Level 2). They only appeared when running a `.prologos` file through `process-file` — because that path exercises top-level scoping, file-level preparse, multi-form interaction, and cross-relation state that the other levels don't touch. The acceptance file caught issues that could have shipped as silent correctness bugs.

### 6.3 Acceptance File as Design Document

Writing the acceptance `.prologos` file BEFORE implementation served three purposes:
1. **Gap detection**: Revealed 5 pre-existing WS pipeline gaps (solver form, `solve-one`, `is`-goals, `guard`, `rel`+`solve`) that would have blocked WFLE validation
2. **Scope definition**: 9 sections with explicit expected outputs defined "done" precisely
3. **Incremental validation**: Sections could be uncommented one at a time as phases completed

### 6.4 Circular Dependencies Are the #1 Integration Gotcha

Adding `wf-engine.rkt` required both providing to and requiring from existing modules (`stratified-eval.rkt`, `reduction.rkt`, `relations.rkt`). The callback parameter pattern (`current-is-eval-fn`, `current-naf-oracle`) consistently solved these: the low-level module defines the parameter, the high-level module sets it.

### 6.5 Self-Critique Before Implementation Finds Real Issues

The self-critique pass (commit `063ca8c`) found 11 issues in the Stage 3 design. The most significant: the original Phase 4 designed a standalone solver that would have duplicated the DFS engine. The critique identified this as unnecessary — the hybrid DFS+bilattice architecture wraps the existing solver instead. This saved an estimated 200+ lines of duplicated solver logic.

### 6.6 Representation Mismatches Between Layers

The probe bug (§4.3) is a general lesson: the DFS solver treats plain Racket symbols as logic variables, but the AST layer uses `expr-logic-var` structs. When constructing inputs to the solver, use the solver's representation, not the AST's. This applies to any interface between layers with different data representations.

### 6.7 Predicate-Level Granularity Interacts Badly with Non-Cyclic Negation

A predicate that has any facts reads as "true" at predicate level → NAF fails for ALL ground instances, even those not in the fact set. The fix (§4.2) was architectural: only track predicates in actual negative cycles in the bilattice; non-cyclic negation uses standard DFS NAF which correctly evaluates per-ground-instance. This is a fundamental tension in any predicate-level approximation.

### 6.8 Stale `.zo` Cache on Struct Field Addition

Adding `cell-dirs` to `prop-network` (Phase 1) required recompiling ALL transitive dependents. Modules like `session-propagators.rkt` that import `prop-network` by struct linklet fail with "expected 12 fields" if not recompiled. Always run `raco make driver.rkt` after struct changes.

## 8. What's Next

### Immediate (enabled by WFLE)

- **Per-variant bilattice refinement**: Change oracle interface to accept ground arguments, enabling per-instance truth values instead of per-predicate approximation
- **Per-relation `:tabled` annotation**: Currently tabling is all-or-nothing via solver config; per-relation control requires `parse-defr` keyword handling
- **Typed-relational integration**: `def x := (solve ...)` fails with QTT multiplicity error — the functional-relational bridge needs multiplicity-aware binding

### Medium-term (using WFLE infrastructure)

- **Coinductive session types**: μ/ν recursion using bilattice pairs — descending cells compute greatest fixpoints for session type recursive unfolding
- **Runtime monitoring**: Bilattice-based three-valued monitors for temporal property evaluation at runtime
- **ATMS composition**: Branching (ATMS) + incomplete information (bilattice) compose orthogonally — ATMS branches could each carry bilattice state

### Long-term (research-stage)

- **Model checking on propagator networks**: μ-calculus formulas as propagator wiring patterns
- **Shield synthesis from session types**: Reactive controllers that enforce session type compliance
