## 1. What Was Built

Track 7 delivers three workstreams that eliminate transitional debt from Tracks 1-6 and advance the propagator-first architecture:

**WS-C (Persistent Registry Cells)**: A separate persistent `prop-network` holds 29 registry cells (macros, warnings, narrowing) with stable cell-ids that survive across commands. Cell reads/writes go directly through `net-cell-read`/`net-cell-write` — no callback indirection. Eliminates per-command cell recreation (~29 cells × thousands of commands).

**WS-B (Stratified Retraction + Readiness Propagators + Pure Resolution)**: Assumption-tagged scoped cells enable S(-1) retraction of infrastructure cell entries. Readiness propagators (threshold-cell composition, fan-in + readiness per constraint) replace 6 O(total) scanning functions with O(changed) propagator-driven readiness detection. The resolution chain is purified (`enet → enet*`) with `solve-meta!` as the sole box-writing entry point. A new `resolution.rkt` module replaces 3 callback parameters with a unified dispatcher.

**WS-A (QTT Mult Cells)**: Infrastructure confirmed complete from Track 4. Bridge wiring blocked by prop-net/elab-net boundary — documented as Track 8 scope.

## 4. Bugs Found and Fixed

**Cross-file leakage in batch-worker** (Phase 6g): The persistent registry network box was shared across test files via `parameterize`. Mutations from one file leaked to the next via `set-box!`. Fix: snapshot box CONTENTS (immutable CHAMP value), create fresh box per file. Root cause: confusing box identity restoration with box contents restoration — a CHAMP value is immutable but the box holding it is mutable.

**Paren errors from sed replacement** (Phase 7a): Bulk sed replacement of `[current-retry-unify #f]` left orphaned closing parens in test files where the original had trailing comments. Fix: manual cleanup. Root cause: mechanical text replacement without paren-awareness.

**5 acceptance file bugs** (Phase 0a): User-defined `data` constructors unbound at file level, `{A}` + multi-arity defn parser crash, defn+match body crash, let-in-match reader failure, `->` in identifier parsing. All preparse/parser level — predating Track 7.

## 5. Design Decisions and Rationale

**Two-network architecture** (Phases 1-3): Separate persistent `prop-network` for registries vs per-command `elab-network` for inference. Rationale: lifecycle decomplection — mixing persistent registries with ephemeral metas in one network requires selective reset (`reset-elab-network-command-state`), which grows fragile with each new field. The two-network approach eliminates this class of bugs structurally.

**Raw `prop-network` for registries** (Phase 1): Registries are simple monotone accumulators — no TMS, no metas, no structural decomposition needed. Using `elab-network` would be overengineering.

**Threshold-cell composition over lifecycle state machine** (Phase 8a): One-shot readiness via lattice monotonicity (⊥ → ⊤, never back) rather than runtime `status != queued` checks. Structural guarantee — correctness is a property of the data, not the code. From external critique D.2, Concern 2.

**`with-enet-reads` bridge for read functions** (Phase 7b): Rather than duplicating 500+ lines of zonk/ground-expr?/normalize-for-resolution, temporarily `parameterize` the net-box within the resolution chain. Tight local scope, data-in → data-out contract. Track 8 eliminates this bridge when metas become network citizens.

**`restore-meta-state!` retained** (Phase 6a): Cannot retire because `meta-info` and `id-map` are elab-network fields, not TMS cells. Structural rollback still needs the box restore. Track 8 scope.

## 6. Lessons Learned

**Track 6's base-network pattern was never activated.** `current-persistent-base-network` and `reset-elab-network-command-state` were designed in Track 6 Phase 6 but never set to non-`#f`. The parameters were always the real persistence layer. Track 7's persistent registry network was the first working cell-based persistence. Lesson: audit whether "designed" infrastructure is actually exercised before planning migration from it.

**Module-load-time registrations block parameter elimination.** Built-in Nat/Bool/Unit ctor registrations happen at `require` time before any persistent network exists. Parameters are needed as seed values for `init-macros-cells!`. Full parameter removal requires either a registration queue or moving built-in registrations to a separate init function. This dependency was not anticipated in the design doc's Phase 3. Lesson: module-load-time side effects are a hidden dependency — audit them before planning parameter removal.

**The narrowing system is extremely effective.** The adversarial benchmark (Phase 0c) revealed that generic arithmetic (`[+ 1 2]`) never creates trait constraints — the narrowing fast path resolves everything eagerly. The stratified resolution loop is driven by meta allocation + unification from implicit resolution, not by trait dispatch. Lesson: benchmark BEFORE designing optimizations — the bottleneck may not be where you expect.

**`eq?` identity comparison for progress detection is fragile.** In `run-stratified-resolution-pure`, progress is detected by `(eq? enet-s2 enet-s0)`. This works because CHAMP structural sharing means unchanged networks are `eq?`. But any unnecessary copy (e.g., a struct-copy that doesn't change values) would break progress detection. This is correct-by-construction with CHAMP but invisible to readers. Lesson: document structural invariants that correctness depends on.
