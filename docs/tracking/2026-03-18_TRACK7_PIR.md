## 1. What Was Built

Track 7 delivers three workstreams that eliminate transitional debt from Tracks 1–6 and advance the propagator-first architecture:

**WS-C (Persistent Registry Cells)**: A separate persistent `prop-network` holds 29 registry cells (24 macros, 3 warning, 2 narrowing) with stable cell-ids that survive across commands. Cell reads/writes go directly through `net-cell-read`/`net-cell-write` — no callback indirection. Eliminates per-command cell recreation (~29 cells × thousands of commands ≈ 200K allocations per full suite).

**WS-B (Stratified Retraction + Readiness Propagators + Pure Resolution)**: Assumption-tagged scoped cells (14 cells) enable S(−1) retraction of infrastructure cell entries on speculation failure. Readiness propagators (threshold-cell composition — fan-in → boolean cell → readiness propagator, each firing exactly once by lattice monotonicity) replace 6 O(total) scanning functions with O(changed) propagator-driven readiness detection. The resolution chain is purified (`enet → enet*`) with `solve-meta!` as the sole box-writing entry point. A new `resolution.rkt` module replaces 3 callback parameters with a unified dispatcher.

**WS-A (QTT Mult Cells)**: Infrastructure confirmed complete from Track 4 (`elab-fresh-mult-cell`, `elab-add-type-mult-bridge`). Bridge wiring blocked by prop-net/elab-net boundary in `decompose-pi` — documented as Track 8 scope.

## 4. Bugs Found and Fixed

### 4.1 Cross-file leakage in batch-worker (Phase 6g)

**Problem**: The persistent registry network box was shared across test files via `parameterize`. Mutations from one file leaked to the next via `set-box!` — the parameterize restored the box identity, but the box contents had been mutated by the previous file.

**Why the wrong path seemed right**: `parameterize` restores parameter values on scope exit, so saving/restoring the box should restore the state. But `set-box!` mutates the box in place — the restored box IS the same box, with its already-mutated contents.

**Fix**: Snapshot box CONTENTS (immutable CHAMP value), create fresh box per file wrapping the snapshot. Root cause: confusing box identity restoration (what `parameterize` does) with box contents restoration (what we need).

**Cross-reference**: This is the same class of bug as Track 6's Phase 7b-c sync-back divergence — both involve misunderstanding the interaction between mutable containers and parameterize. The Track 6 PIR §4.1 documented this: "context loss causes architectural divergence." Here the context was the distinction between box identity and box contents.

### 4.2 Orphaned parens from sed replacement (Phase 7a)

**Problem**: Bulk sed replacement of `[current-retry-unify #f]` left orphaned closing parens in test files where the original had trailing comments on the same line.

**Fix**: Manual cleanup. Minor but illustrative — mechanical text replacement without syntax awareness is fragile.

### 4.3 Five acceptance file bugs (Phase 0a)

| Bug | Severity | Root Cause |
|-----|----------|------------|
| User-defined `data` constructors unbound at file level | High | Preparse registers constructors but L3 file-level scope doesn't see them |
| `{A}` implicit + multi-arity `defn` crashes parser (car #f) | High | `parse-defn-with-implicits` / `parse-reduce` interaction |
| `defn name [x] (match x ...)` crashes parser | Medium | `parse-defn-new` doesn't handle match body |
| `let` inside `match` arm fails reader | Medium | Reader bracket nesting in indented match continuation |
| `int->str` in pattern arm parsed as arrow | Low | Reader tokenizer treats `->` as delimiter inside identifiers |

All preparse/parser level — predating Track 7. The first two are the most significant: user-defined ADTs and polymorphic multi-arity functions are broken at L3 (file level) despite passing L1-L2. This is the exact class of gap the acceptance file methodology is designed to catch (DESIGN_METHODOLOGY.org § "WS-Mode Validation Protocol").

## 5. Design Decisions and Rationale

### 5.1 Two-network architecture (Phases 1-3)

Separate persistent `prop-network` for registries vs per-command `elab-network` for inference. Lifecycle decomplection — mixing persistent registries with ephemeral metas in one network requires selective reset, which grows fragile with each new field. The persistent network is small (29 cells) and static; the elab-network is large and dynamic.

**Principle**: Decomplection (DESIGN_PRINCIPLES.org)

### 5.2 Raw `prop-network` for registries (Phase 1)

Registries are simple monotone accumulators — no TMS, no metas, no structural decomposition. Using `elab-network` would import unused machinery and create lifecycle coupling. The raw `prop-network` is the minimal correct structure.

**Principle**: Simplicity of Foundation

### 5.3 Threshold-cell composition over lifecycle state machine (Phase 8a)

External critique (D.2 Concern 2) identified that the countdown-latch design couldn't prevent duplicate firings. The threshold-cell composition resolves this structurally: fan-in propagator writes to a boolean cell (⊥ → ⊤, never back), readiness propagator watches that cell. The one-shot guarantee is a property of the lattice, not a runtime check. No per-firing `status != queued` guard needed.

**Principle**: Correct by Construction — correctness is a property of the data, not the code.

### 5.4 Option A: full purity for resolution chain (Phase 7b)

D.3 self-critique identified the tension: resolution functions called imperative helpers (`zonk`, `meta-solution`, `ground-expr?`) that read from the network via box. Option A (full purity — `enet → enet*` threading) was chosen over Option B (defer to Track 8) because: (1) the write chain purification is permanent architecture that survives into Track 8, (2) imperative patterns compound — deferring means building more code on the wrong foundation, (3) the `with-enet-reads` bridge is a clean transitional pattern.

**Principle**: Completeness over Deferral, Propagator-First Infrastructure

### 5.5 `with-enet-reads` bridge (Phase 7b)

Rather than duplicating 500+ lines of zonk/ground-expr?/normalize-for-resolution as pure variants, temporarily `parameterize` the net-box within the resolution chain. Tight local scope, data-in → data-out at the contract boundary. Track 8 eliminates this bridge when meta-info and id-map become network-level citizens.

### 5.6 `restore-meta-state!` retained (Phase 6a)

Cannot retire because `meta-info` (CHAMP: meta-id → meta-info struct), `id-map` (CHAMP: meta-id → cell-id), and `next-meta-id` (counter) are plain elab-network fields — not TMS-aware cells. Structural state created during speculation persists even after TMS retraction. Track 8 scope.

**Principle**: Belt-and-Suspenders — don't remove the safety net until the replacement is proven.

## 6. What Went Well

1. **Design investment paid off in implementation speed**: 8 hours of design (D.1 through scanning audit) + 6 hours of implementation with zero wrong turns. Compare Track 6's 4h design + 14h implementation with 2 reverts. The D.2/D.3 critique cycles caught the threshold-cell and resolution purity issues before any code was written.

2. **External critique was genuinely useful**: The reviewer's Concern 2 (countdown latch under feedback) led directly to the threshold-cell composition — a better design than the countdown-latch we'd originally proposed. Concern 6 (loop audit) led to the scanning function audit that discovered the dual-scan asymmetry (constraints use mutual-exclusive paths; traits/hasmethods use BOTH cell-driven AND wakeup-driven). Both improved the final architecture.

3. **Resolution purification was cleaner than expected**: The `enet → enet*` threading through 6 write functions + 4 read functions required no algorithmic changes — just signature changes and `for/fold` instead of `for-each`. The `with-enet-reads` bridge encapsulated all imperative reads in a single 3-line macro. The entire Phase 7b was ~2 hours.

4. **Performance improved 12% (implementation) → 23% (post-fix)**: 235.2s → 207.4s from persistent registry cells (−12%). Post-PIR discussion of the CHAMP `eq?` invariant (§6, §8.3) led to identity-preservation in `elab-cell-write`, `elab-cell-replace`, and `rewrap` — avoiding unnecessary struct allocations on no-op writes. Result: 207.4s → 181.2s (additional −13%, commit `5dd5d68`). Total: 235.2s → 181.2s (−23%), fastest suite time in project history. The `eq?` concern was a correctness investigation that accidentally found a performance win — every no-op cell write was allocating a new 5-field struct, compounding across thousands of writes per command.

5. **12 phases, zero regressions**: Every phase passed 7154 tests. The belt-and-suspenders pattern (Phases 5-6 for retraction, Phase 8b for readiness) caught zero divergences — the new infrastructure matched the old exactly.

6. **Acceptance file discovered 5 real bugs**: All at the parser/preparse level, not Track 7 regressions — but the session type and relational logic sections exercise code paths no prior acceptance file touched. This is the first acceptance file to exercise the process language.

## 7. What Went Wrong

1. **No new unit tests**: Track 7 added zero new test cases. The 7154 existing tests provide integration coverage, but there are no isolated tests for S(−1) retraction, threshold-cell readiness, or ready-queue consumption. If any of these break in Track 8, diagnosing the regression will require bisecting through integration failures rather than checking a targeted unit test. Compare Track 4, which added 28 TMS cell unit tests.

2. **Phase 6 scope grew significantly**: The original Phase 6 was "belt-and-suspenders retirement." It expanded to 7 sub-phases (6a-g) covering dead code removal, test fixture updates, read simplification, parameter assessment, and batch-worker fixes. The expansion was necessary (each sub-phase addressed a real concern), but the Phase 6 entry in the progress tracker understates the actual work.

3. **Module-load-time registration dependency not anticipated**: The design doc's Phase 3 assumed parameter writes could be fully eliminated. Implementation discovered that Nat/Bool/Unit constructor registrations happen at `require` time — before any persistent network exists. This forced Phase 3b (full parameter removal) to be deferred to Phase 6e, where it was resolved by making reads cell-primary rather than eliminating parameter writes.

4. **PIR initially sparse**: The first version of this PIR was missing What Went Well, What Went Wrong, Where We Got Lucky, What Surprised Us, Assumptions That Were Wrong, and Cross-References — exactly the evaluative and meta-learning sections that the PIR methodology (POST_IMPLEMENTATION_REVIEW.org §4-8, 12-16) identifies as the most valuable. A PIR written without consulting the methodology will systematically omit these sections.

5. **S(−1) retraction was silently broken** (discovered post-PIR by tests written to address §7.1): `run-retraction-stratum!` used merge-based `elab-cell-write` to write cleaned cell values. For monotone merge functions (`merge-hasheq-union`), `union(old, cleaned) = old` — the write was a no-op. Retracted entries were silently restored. The bug was masked in production by `restore-meta-state!` (box snapshot handles rollback). Fixed by adding `net-cell-replace` to `propagator.rkt` — bypasses merge, replaces cell value directly (`commit 2b6b576`). **This is the strongest validation of the PIR methodology to date**: PIR §7.1 self-identified the testing gap → tests written → tests found a real bug → bug fixed. Without the PIR's honest self-assessment, this would have surfaced during Track 8 when trying to retire `restore-meta-state!` — much harder to diagnose and fix with all of Track 8's other changes in flight.

## 8. Where We Got Lucky

1. **Track 6's base-network was dead code**: `current-persistent-base-network` and `reset-elab-network-command-state` were designed in Track 6 Phase 6 but never set to non-`#f` values. If they had been active, Track 7's two-network architecture would have required migrating FROM an active base-network pattern rather than replacing dead code. The "retirement" was actually dead code deletion — much simpler. If someone had activated the base-network pattern between Track 6 and Track 7, we'd have had a real migration challenge.

2. **The narrowing fast path masked a benchmark design flaw**: The adversarial benchmark (Phase 0c) was designed to stress the stratified resolution loop, but narrowing resolves generic arithmetic eagerly — the loop was driven by implicit resolution, not trait dispatch. The benchmark still provided a valid baseline, but if we'd relied on it to measure scanning vs. readiness-propagator performance specifically, we'd have been measuring the wrong thing. The full test suite comparison caught the real performance story.

3. **CHAMP structural sharing makes `eq?` progress detection work**: Phase 7b's pure resolution loop detects progress via `(eq? enet-s2 enet-s0)`. This works because CHAMP returns the identical object when no changes are made. If CHAMP returned structurally-equal-but-not-`eq?` objects (as many immutable data structures do), the progress detection would silently fail, causing infinite loops. We never explicitly tested this invariant — it just happens to hold.

4. **No test exercises the threshold-cell under concurrent readiness**: All 7154 tests pass with both old scanners and new readiness propagators, but no test has a constraint whose dependencies become ready in a single BSP round where the threshold-cell fires at the boundary. The existing tests exercise sequential meta-solving patterns. If the threshold-cell had an off-by-one in the ground-count comparison, it might not have been caught.

## 9. What Surprised Us

1. **The 12% performance improvement**: Track 7 was expected to be performance-neutral (eliminate cell recreations but add bridge propagators and assumption tagging). Instead, 235.2s → 207.4s — faster than the pre-migration baseline of 208.6s. The improvement comes primarily from eliminating per-command `register-*-cells!` calls (29 cell allocations + network insertions × thousands of commands), which had more overhead than anticipated. The assumption tagging + S(−1) retraction adds negligible overhead at depth 0 (fast path confirmed — same lesson as Track 4's TMS depth-0 fast path).

2. **The scanning functions were not the bottleneck**: The design doc emphasized replacing 6 O(total) scanning functions with O(changed) readiness propagators as a major performance win. But the adversarial benchmark showed that most constraint resolution is driven by meta allocation + unification from implicits, not by trait dispatch. The scanning functions execute quickly because most constraints have few dependencies. The real performance win was the persistent cell infrastructure (WS-C), not the readiness propagators (WS-B Phase 8).

3. **`decompose-pi` as the natural mult bridge call site is inaccessible**: Phase 9 intended to wire `elab-add-type-mult-bridge` when Pi types are decomposed. But `decompose-pi` in `elaborator-network.rkt` operates on the raw `prop-network` inside a propagator fire function — it cannot access the elab-network's `id-map` to look up mult-meta cell-ids. This boundary between prop-network and elab-network wasn't visible until implementation. The bridge function exists and works (tested), but the natural call site is one layer too deep.

4. **The external critique's most valuable contribution was indirect**: The reviewer's Concern 1 (channel cell complexity) was largely a non-issue in our architecture (Gauss-Seidel prevents the scenario they described). But the *process of reasoning about why* it wasn't an issue clarified the inter-stratum feedback semantics more precisely than the original design had. The response improved the design document even though the concern itself was rejected.

## 10. Architecture Assessment

### How the architecture held up

The two-network architecture (Phase 1-3) integrated cleanly. The persistent network is genuinely simple — 29 cells, no TMS, no metas, raw `prop-network` with direct read/write. The elab-network remained unchanged. No cross-network routing needed — writes go to the appropriate network based on the data's lifecycle.

The assumption-tagged scoped cells (Phase 4) required touching 14 write sites and 11 read sites — more pervasive than anticipated, but each change was mechanical (`tagged-entry` struct wrapping + unwrap helpers).

The resolution purification (Phase 7b) was the most significant architectural improvement. The `enet → enet*` threading gives us a pure functional core surrounded by a thin imperative shell (`solve-meta!` box read/write). This is the right layering for Track 8, where resolution actions will be wired as propagator fire functions — pure functions compose naturally with propagator infrastructure.

### Extension points that were sufficient
- **`prop-network` as standalone**: The raw prop-network was sufficient for persistent registries — no need for elab-network's meta/TMS machinery
- **`tagged-entry` struct**: Clean extension point for assumption tagging without modifying cell merge functions
- **Propagator registration**: `net-add-propagator!` worked unchanged for threshold-cell and readiness propagators
- **Action descriptor pattern** (Track 2): The `action-retry-constraint` / `action-resolve-trait` / `action-resolve-hasmethod` structs threaded through the new readiness propagator pipeline without modification

### Friction points
- **prop-net/elab-net boundary**: Propagator fire functions operate on `prop-network` but resolution logic needs `elab-network` fields (meta-info, id-map). The `with-enet-reads` bridge is a workaround, not a solution. Track 8 must address this by making these fields accessible from the prop-network layer.
- **Module-load-time registrations**: Built-in Nat/Bool/Unit registrations happen at `require` time, before any persistent network exists. Parameters are needed as seed values. This is a deep Racket-level issue — module-load-time side effects create initialization ordering constraints that the propagator architecture can't easily model.

## 11. Assumptions That Were Wrong

1. **"Phase 3 will fully eliminate parameter writes."** Module-load-time registrations create a seed dependency — parameters are needed to initialize persistent cells. The correct approach was making reads cell-primary (Phase 6e) rather than eliminating writes.

2. **"The scanning functions are the main performance bottleneck."** The adversarial benchmark showed narrowing resolves most generic arithmetic eagerly. The stratified resolution loop is driven by meta allocation + unification from implicits. Scanning overhead is small relative to total resolution time.

3. **"Track 6's base-network pattern needs migration."** It was dead code. We designed a retirement strategy for infrastructure that was never activated.

4. **"Phase 6 is a single retirement step."** It expanded to 7 sub-phases. The design doc should have anticipated the scope based on Track 6 Phase 8's similar expansion.

5. **"The resolution chain can be purified incrementally."** D.3's analysis showed the write chain and read bridge must be implemented together — a write-only purification without read bridging would leave `zonk` calling imperative box reads from within a pure chain, violating the contract. Option A (full purity) was an all-or-nothing change.

## 12. What Did We Learn About the Problem Itself?

The fundamental insight from Track 7 is that **cell persistence and cell lifecycle are orthogonal concerns**. Prior tracks conflated them — cells were ephemeral because they lived in the elab-network, and persistence required parameters. Track 7's two-network architecture separates these axes:

| | Ephemeral | Persistent |
|---|---|---|
| **Value cells** (metas) | Elab-network | — |
| **Infrastructure cells** (constraints) | Elab-network + assumption tags | — |
| **Registry cells** (types, traits) | — | Persistent registry network |
| **Definition cells** (per-name) | — | `current-definition-cells-content` (Track 5 pattern) |

Each quadrant has a different lifecycle, different merge semantics, and different retraction behavior. The two-network design makes this explicit. A single-network design would require each cell to carry its lifecycle metadata — possible but more complex.

The secondary insight is that **readiness is cheap**. The scanning functions we replaced were not expensive. The value of readiness propagators isn't performance — it's structural: adding a new constraint type means adding a readiness propagator, not modifying a scanning function. The architectural benefit (extensibility) outweighs the performance benefit (negligible for current constraint counts).

## 13. Is This Part of a Pattern?

### Recurring: Belt-and-suspenders as validation methodology

Track 7 is the 5th consecutive track to use belt-and-suspenders validation. The pattern has never found a divergence — every new path matched the old exactly. This raises the question: is the validation actually testing anything, or are we just running the same code twice in different ways?

**Assessment**: The validation IS valuable, but its value is in the confidence it provides for the retirement step, not in catching bugs. Every retirement (Phase 6a's retaining of restore, Phase 8c's scanner removal) was preceded by zero-divergence validation. Without it, we'd be removing code on faith rather than evidence.

**Cross-reference**: Track 4 PIR §5.2, Track 5 PIR §7, Track 6 PIR §6.6

### Recurring: Phase scope expansion

Track 6 Phase 8 expanded from 2 sub-phases to 4 (8a-d). Track 7 Phase 6 expanded from 1 step to 7 (6a-g). Both expansions were driven by discovering dependencies during implementation that the design didn't anticipate. The pattern: retirement phases are consistently underestimated because they require coordinating multiple systems that the design analyzed independently.

**Action**: Future retirement phases should be budgeted at 2-3× the design estimate. Explicitly list all consumers of the retired infrastructure in the design doc.

### Recurring: Depth-0 fast path

Track 4 (TMS cells at depth 0: single `null?` check), Track 7 (S(−1) at depth 0: single `set-empty?` check). The pattern: new infrastructure that handles the general case (arbitrary speculation depth) must have zero overhead for the common case (depth 0). Track 7 validated this — assumption tagging + S(−1) retraction added no measurable overhead to the 95% of commands that don't involve speculation.

**Cross-reference**: Track 4 PIR §5.1, Track 4 PIR §6.5

### New: Design critique as design improvement

Track 7 is the first track where the external critique (D.2) directly improved the design (threshold-cell composition replacing countdown-latch). Prior tracks treated critique as validation — this track treated it as collaboration. The difference was concrete questions with code-grounded responses rather than abstract observations.

### New: PIR → tests → bug → fix chain (strongest PIR validation to date)

PIR §7.1 self-identified: "Track 7 added zero new test cases." Tests written to address the gap discovered a real bug: S(−1) retraction was silently broken (merge-based write restored retracted entries). Bug fixed by adding `net-cell-replace` (commit `2b6b576`). Without the PIR's honest self-assessment, this would have been a Track 8 debugging session — much harder to diagnose with all of Track 8's other changes in flight. **Lesson**: PIR testing gap analysis is not bureaucratic box-checking — it finds real bugs. The "What Went Wrong" section should always include an honest assessment of testing coverage. **Action**: Add to workflow rules: new infrastructure features MUST have unit tests written during the implementation phase, not deferred to post-PIR. The Track 4 precedent (28 TMS cell unit tests) was the right approach; Track 7's deferral was the wrong one.

## 14. What Would We Do Differently If Starting Over?

1. **Write unit tests for S(−1) and readiness propagators**: The belt-and-suspenders validation provides integration coverage, but isolated unit tests would make Track 8 regressions easier to diagnose. Should have written 10-15 tests during Phases 5 and 8a.

2. **Anticipate Phase 6 scope**: The design doc listed Phase 6 as one step. It should have been designed as 6a-g from the start, based on Track 6 Phase 8's precedent of retirement-phase expansion.

3. **Design the adversarial benchmark to target the scanning vs. readiness comparison directly**: The current benchmark validates the scheduler overall, but the narrowing fast path means it doesn't stress the exact code path that Phase 8 changed. A benchmark with explicit trait constraints (not generic arithmetic) would have measured the readiness propagator improvement more precisely.

4. **Move `init-macros-cells!` seeding to a separate init function**: Rather than retaining parameter writes for module-load-time seeding, factor the built-in registrations into an explicit init function called before the persistent network is created. This would have simplified Phase 3 and removed a dependency chain.
