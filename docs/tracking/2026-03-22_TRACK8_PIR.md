## 1. What Was Built

Track 8 completed the Propagator Migration Series' infrastructure goal: **bring all elaboration onto the propagator network, dissolving the boundary between type inference and trait resolution.**

Three Parts delivered a progression from data to operations to control flow:
- **Part A** (5 phases): All structural state (meta-info, id-map, mult, level, session) assumption-tagged for ATMS worldview management. `restore-meta-state!` partially retired. Progress box removed.
- **Part B** (8 phases): Elaboration operations moved onto the network. Worldview-aware reads make retracted speculation entries invisible without synchronous restore — `restore-meta-state!` fully retired. 14/23 callback parameters replaced with direct cell-ops. HKT `impl` registration and resolution proven (4 parser fixes). New modules: `cell-ops.rkt`, `elab-network-types.rkt`.
- **Part C** (8 phases): Resolution control flow moved onto the network. Trait resolution, hasmethod resolution, and constraint retry fire as bridge propagators during S0 quiescence — the S1→S2 readiness→action path becomes a safety-net fallback. Gauss-Seidel vs BSP scheduling comparison: inconclusive (0.998×).

The elaborator is now a thin AST walker that emits constraints into the propagator network. Type inference and trait resolution are propagation events in the same S0 pass.

## 4. Bugs Found and Fixed

### B1: Ghost Meta Problem (3 attempts to solve)

**Bug**: Metas created during failed speculation persist in id-map/meta-info after retraction. "Unknown metavariable" and "already solved" errors.

**Root cause**: S(-1) retraction is deferred (runs at loop start, not at failure). The elaborator needs synchronous structural cleanup between speculation branches, but deferred retraction provides only value cleanup.

**Why it seemed right**: A4 assumed that tagging + S(-1) retraction was sufficient for all state. The distinction between value state (lattice-monotone, retractable) and structural state (existence — either exists or doesn't) was not recognized until the bug surfaced.

**Resolution**: B1's worldview-aware reads — the third attempt — succeeded because visibility is a property of the read path (structural, correct-by-construction), not the cleanup timing (imperative, fragile). Retracted entries exist in the CHAMP but are invisible to reads filtered by the speculation stack.

**Lesson**: The same bug required three architecturally distinct approaches across A4, A4b, and B1. Each attempt deepened understanding: A4 showed values aren't enough, A4b showed id-map tagging isn't enough, B1 showed the read path is the correct intervention point. The D.4 design revision (worldview-aware reads) came from this understanding.

### B2: Circular Dependency (B0 analysis was wrong)

**Bug**: `cell-ops.rkt → elaborator-network.rkt → type-lattice.rkt → reduction.rkt → metavar-store.rkt → cell-ops.rkt`.

**Root cause**: B0 analysis said "no circular deps at file level — callbacks are historical artifact." This was wrong. The cycle is real and structural: cell operations depend on lattice merge functions which depend on reduction which depends on meta state.

**Resolution**: `elab-network-types.rkt` extraction — move struct definitions + cell operations (with no type-lattice deps) to a separate module that both sides can import.

**Lesson**: Module-level dependency analysis from reading imports is insufficient when `require` chains are transitive. The cycle was invisible until `raco make` failed.

### B3: HKT Preparse Infinite Loop

**Bug**: `{C : Type -> Type}` in trait definition causes infinite preparse expansion.

**Root cause**: `pattern-var?` treated `$brace-params` (a reader sentinel) as a defmacro pattern variable. `datum-subst` errored "Unbound pattern variable: $brace-params", triggering re-expansion.

**Resolution**: Four parser fixes: (1) `$brace-params` opaque in `preparse-expand-form`, (2) `$brace-params` skipped in `preparse-expand-subforms`, (3) all 6 reader sentinels excluded from `pattern-var?`, (4) `extract-implicit-binders` for method spec Pi wrapping.

**Lesson**: The parser's macro system doesn't distinguish "reader sentinel" from "user-defined macro." Any new reader sentinel is a potential `pattern-var?` collision. This class of bug is structural — it will recur for the next sentinel unless `pattern-var?` is made sentinel-aware (which fix (3) now does for all known sentinels).

### B2b: Unconditional Network Creation

**Bug**: `make-elaboration-network` in `reset-meta-store!` broke test contexts that don't have propagator infrastructure.

**Root cause**: B2b replaced a guarded callback with an unconditional direct call. The callback was `#f` in test contexts (no network); the direct call always creates a network, leaving infrastructure cells uninitialized.

**Resolution**: Gated network creation on `current-prop-new-infra-cell` availability.

### D.4: The Critical Design Revision

The D.1–D.3 design failed to apply Propagator-First, Data Orientation, and Correct-by-Construction principles. "Partial retirement" of `restore-meta-state!` was a rationalization — the imperative save/restore pattern IS a callback-like mechanism for state that should be on the network. The D.3 principles alignment check *listed* these principles but failed to *apply* them.

D.4 applied the principles: worldview-aware reads make retraction a property of the read path (structural, correct-by-construction), not the cleanup timing (imperative, fragile).

**This is the most important lesson of Track 8**: a principles alignment checklist is cosmetic unless it challenges workarounds. The question isn't "does this principle apply?" but "does this design *violate* this principle, and is the violation necessary or a failure of imagination?"

## 6. Lessons Learned

### 6.1: Worldview-Aware Reads as Universal Mechanism

The ATMS worldview (speculation stack) filtering all CHAMP reads is the single most impactful architectural change in the Propagator Migration Series. It solves the ghost-meta problem, retires restore-meta-state!, and makes speculation cleanup automatic. The mechanism is O(depth) per read (depth rarely > 3) — negligible cost for structural correctness.

**Apply to**: Any future state that needs speculation awareness. The pattern is: tag entries with assumption ID at creation, filter by `worldview-visible?` at read time. S(-1) becomes GC.

### 6.2: Bridge Fire Functions Violate Propagator-First, Data Orientation, and Completeness

The C0 prototype used pure α/γ functions — the correct pattern (`net-add-cross-domain-propagator`). The production implementation (C1-C3) abandoned α/γ in favor of `elab-add-propagator` with `enet-box`-syncing fire functions. The fire functions capture a mutable box (`current-prop-net-box`), `unbox` it, call pure resolution, then `set-box!` the result back. This is a callback pattern wearing propagator clothes.

**Principles violations**:
- **Propagator-First**: The impl registry, constraint store, and meta-info CHAMPs are struct fields on `elab-network`, not cells on the propagator network. The bridge propagators reach *outside* the network through `enet-box` to access them.
- **Data Orientation**: `set-box!` is an inline mutation, not an effect description. The fire function braids pure computation with side effects.
- **Correct-by-Construction**: The `enet-box` sync pattern (unbox → rewrap → resolve → set-box!) requires discipline — if a bridge forgets to write back, state is lost silently. No structural guarantee prevents this.
- **Completeness**: Part C was titled "The Phase Boundary Dissolves" — but the boundary didn't dissolve. Bridge propagators fire in S0 (good), but they do so by side-effecting the elaboration network (not dissolved). The phase boundary moved from "S2 resolves after quiescence" to "S0 resolves during quiescence via mutable box" — an improvement in timing, but not in architecture. A complete solution would make the impl registry and constraint store cells on the network, enabling pure α/γ bridges with no `enet-box`.

**What pure α/γ would require**: impl registry, constraint store, and meta-info must become cells on the propagator network. Then the bridge fire function is `(lambda (net) (net-cell-write net result-cell (alpha-fn (net-cell-read net dep-cell))))` — pure `net → net`, no box, no side effects.

**This is not Track 9 debt — it is Track 8 incompleteness.** The enet-box pattern compounds: every future bridge or propagator that needs resolution state will replicate the box-sync dance. The foundation must be corrected before building on it.

### 6.3: Principles Alignment Must Challenge, Not Catalogue

The D.1–D.3 principles check listed 10 aligned principles and 2 tensions. It did NOT challenge the "partial retirement" workaround against Propagator-First. The D.4 revision applied the principles and produced a fundamentally different (and correct) design.

**Apply to**: Every future design critique round. Add an explicit step: "For each principle, identify whether any component of this design is a *workaround* that the principle would eliminate. If so, justify the workaround or redesign."

### 6.4: Incomplete Instrumentation Produces Misleading Conclusions

The first C5a benchmark overrode only the stratified loop's quiescence calls. Result: BSP 0.8% slower — interpreted as "hypothesis confirmed." The second C5a benchmark overrode ALL quiescence calls (propagator.rkt level). Result: BSP 0.2% faster — opposite sign.

**Apply to**: Any A/B comparison. Instrument at the lowest possible level. Partial instrumentation captures a subset of behavior and may flip the conclusion.

### 6.5: The Design Doc Is the Implementation Plan

Track 8's design doc (860+ lines, D.4 revision) served as the implementation plan with zero deviation. Every phase had explicit files, rationale, and acceptance criteria. Implementation was mechanical — read the phase, implement, commit. No plan mode, no additional planning.

**Apply to**: Invest in design documents proportionally to implementation complexity. Track 8's 6h design → 12h implementation with 0 wrong approaches (excepting the circular-dep discovery, which the design noted as a risk).

### 6.6: Belt-and-Suspenders Has Legitimate and Illegitimate Uses

**Legitimate**: C1-C3 installed bridge propagators alongside the existing readiness propagators. The readiness path still fires; the S2 executor finds everything already resolved. Dual mechanisms with `meta-solved?` guards are zero-cost defense-in-depth during migration.

**Illegitimate**: The enet-box-sync pattern itself is belt-and-suspenders rationalized as necessity — "we need the box because resolution state isn't on the network." The correct response is not "accept the box" but "put the state on the network." The distinction: legitimate belt-and-suspenders runs two *correct* mechanisms in parallel; illegitimate belt-and-suspenders keeps an *incorrect* mechanism because the correct one hasn't been built yet, then calls the workaround "architecturally correct."

**Apply to**: When encountering belt-and-suspenders during implementation, ask: "Are both paths architecturally sound, or is one path a workaround for incomplete infrastructure?" If the latter, the infrastructure must be completed — not the workaround accepted.

## 11. Architecture Assessment

### What Held Up

- **Two-network architecture** (Track 7): persistent-registry + per-command elab-network. Part A/B/C all operated within this split without modification.
- **Stratified resolution loop** (Track 2): S0→S1→S2 absorbed bridge propagators gracefully. Bridges fire in S0; S2 is safety net. No loop restructuring needed.
- **Tagged-entry pattern** (Track 7): Per-entry assumption tagging scaled to all 5 CHAMPs (meta-info, id-map, mult, level, session) without pattern modification.
- **Pure resolution functions** (Track 7 Phase 7b): `resolve-trait-constraint-pure` and friends reused directly as bridge fire function bodies. The `enet → enet*` contract was the right abstraction.

### What Needed Modification

- **Module structure**: The elaborator-network → type-lattice → reduction → metavar-store cycle required extraction. This was a genuine structural issue, not an oversight — the cycle exists because cell operations depend on merge functions which depend on type normalization which depends on meta state.
- **Parser**: `pattern-var?` didn't account for reader sentinels. This is a class bug — any reader sentinel added in the future will collide unless the sentinel set is explicit.
- **`restore-meta-state!`**: Required three architecturally distinct attempts (A4: tagging, A4b: id-map tagging, B1: worldview-aware reads) before the correct mechanism was found. The difficulty was fundamental — synchronous restore and deferred retraction are genuinely different mechanisms.

### Architectural Debt: Accepted

1. **Module-load-time dual writes**: `macros.rkt` still writes to both cells and parameters for registry functions. Module loading context has no network — can't remove parameter writes until module loading is redesigned.

### Architectural Debt: NOT Accepted — Requires Immediate Follow-Up

All items below are scoped into **PM Track 8D**. They trace to a single root cause: the elaboration network is accessed as a mutable box (`current-prop-net-box`), not as a propagator-native value. See §12 for the full audit.

1. **Enet-box sync in bridge fire functions (Completeness violation)**: The C1-C3 bridge propagators use `enet-box` to reach outside the propagator network for resolution state. Part C was titled "The Phase Boundary Dissolves" but the boundary was relocated, not dissolved.
2. **`current-prop-net-box` as central architectural seam**: 126 `set-box!/unbox` in metavar-store.rkt, 197 across 12 files. All state operations orbit this mutable box.
3. **Impl/trait registries as parameters, not cells**: `lookup-impl`, `lookup-trait`, `lookup-param-impls` read from Racket parameters invisible to the propagator network. New impl registrations can't trigger re-resolution via propagation.
4. **`with-enet-reads` fiction of purity**: "Pure" resolution functions create temporary mutable boxes so `zonk`/`meta-solved?`/`ground-expr?` can read state. Read functions are coupled to the box, not their arguments.
5. **9 remaining callbacks — 3 legitimate, 6 symptomatic**: `fresh-level-cell`, `fresh-sess-cell`, `structural-meta-lookup`/`mult-bridge` are genuine domain injection. `run-quiescence`, `unwrap-net`/`rewrap-net`, and the 3 bridge-fn callbacks are symptoms of the enet-box architecture.

## 12. Post-Completion Principles Audit

Systematic review of Track 8 implementation against all 10 load-bearing principles (Principles-First Design Gate, codified `c7cda29`). This audit was conducted after the PIR was drafted and immediately revealed violations the original PIR had rationalized.

### Finding 1: `current-prop-net-box` — The Pervasive Mutable Box (ROOT CAUSE)

**Scope**: 126 `set-box!/unbox` in `metavar-store.rkt`. 197 across 12 files.

**Violations**: Propagator-First, Data Orientation, Correct-by-Construction, Completeness.

`current-prop-net-box` is the central architectural seam. It holds the entire `elab-network` as a mutable box. Virtually all state operations — constraint registration, meta solving, trait lookup, bridge resolution — go through `(unbox net-box) → modify → (set-box! net-box modified)`. This is an imperative core that the propagator network wraps around, rather than being the propagator network.

**Why it matters**: Every future feature that touches elaboration state will replicate this pattern. The box is not a containable local issue — it's the access pattern for all of `metavar-store.rkt`, all resolution functions, all bridge propagators, and the speculation bridge.

**Correct-by-Construction violation**: If any code path reads the box, modifies state, but fails to write back, the modification is silently lost. No structural guarantee prevents this — it requires discipline.

### Finding 2: `with-enet-reads` — Temporary Box Creation in "Pure" Functions

**Where**: `resolution.rkt` line 172.

```racket
(define-syntax-rule (with-enet-reads enet body ...)
  (parameterize ([current-prop-net-box (box enet)])
    body ...))
```

**Violations**: Data Orientation, Decomplection.

The "pure" resolution functions (`resolve-trait-constraint-pure`, `resolve-hasmethod-constraint-pure`, `retry-unify-constraint-pure`) have contracts `enet → enet*`. But internally they create a temporary mutable box so that `zonk`, `ground-expr?`, `normalize-for-resolution`, and `meta-solved?` can read state through the `current-prop-net-box` parameter. The purity is a contract-level fiction.

**Why it matters**: The read functions (`zonk`, `meta-solved?`, `ground-expr?`) are coupled to the mutable box, not to their logical input. This prevents them from being called in a pure propagator fire function — they *require* a box in scope. This is the technical reason the C1-C3 bridges couldn't use pure α/γ: the α function needs to call `zonk`, which needs a box, which means the fire function must set up a box, which means side effects.

**Root dependency**: This finding is downstream of Finding 1. If read functions accepted a network argument instead of reading a parameter, `with-enet-reads` would be unnecessary, and α/γ bridges would "just work."

### Finding 3: Impl/Trait/Param-Impl Registries — Parameters, Not Cells

**Where**: `macros.rkt` lines 6050, 6113, 6252.

- `lookup-impl`: reads `(current-impl-registry)` — a Racket parameter (hash).
- `lookup-trait`: reads `(read-trait-registry)` — parameter with cell fallback.
- `lookup-param-impls`: reads `(read-param-impl-registry)` — parameter with cell fallback.

**Violations**: Propagator-First, Composition, Completeness.

The registries are invisible to the propagator network. A new `impl` registration can't trigger re-resolution of pending constraints via propagation — there's no cell to watch. This is why the S2 scan loop must still exist as more than a safety net: it re-scans because propagation-time resolution can't be notified of registry changes.

**What cells would enable**: A bridge propagator watching the impl-registry cell could fire when a new impl is registered, re-resolving any pending trait constraints that match the new impl. Registration-triggered resolution would fall out of the network topology instead of requiring scan-loop discovery.

**Dual-write complication**: `read-trait-registry` already has a cell fallback (`macros-cell-read-safe`), but the parameter is still written alongside the cell (B2e dual-write). The cell isn't authoritative. Track 8D should make the cell authoritative and the parameter the fallback (inverted hierarchy), then eventually eliminate the parameter path.

### Finding 4: Speculation Bridge — Imperative Commit/Retract Access Pattern

**Where**: `elab-speculation-bridge.rkt` lines 236-257. 21 `set-box!/unbox` calls.

**Violations**: Data Orientation (access pattern, not operations).

`net-commit-assumption` and `net-retract-assumption` are network-native operations — correct. But the access pattern extracts the prop-network from a box, applies the operation, and writes back. In a value-threaded architecture, the speculation bridge would receive and return the network, not extract it from and inject it into a mutable container.

**Severity**: Lower than Findings 1-3. The operations themselves are structurally correct (TMS commit/retract are propagator-native). Only the access pattern is imperative. Fixing Finding 1 fixes this automatically.

### Finding 6: `solve-meta!` — Imperative Side-Effect Chain

**Where**: `metavar-store.rkt`.

**Violations**: Data Orientation.

`solve-meta!` writes to meta-info CHAMP via box, writes to the cell via box, triggers the stratified resolution chain, which may trigger more `solve-meta!` calls re-entrantly. This is deeply nested imperative effects with re-entrant callbacks — the antithesis of data orientation's "effects as descriptions at boundaries."

The pure variant (`solve-meta-core-pure` in resolution.rkt) is better but still relies on `with-enet-reads` (Finding 2). The chain is: bridge fire fn → `with-enet-reads` creates box → `resolve-trait-constraint-pure` → `solve-meta-core-pure` → reads/writes through box.

**What data orientation would look like**: `solve-meta-core-pure` returns a description `(SolveMeta meta-id value)`. The quiescence loop interprets descriptions, applying them to the network value. No re-entrancy, no box, inspectable worklist.

### Root Cause Summary

All 6 findings trace to one architectural decision: **the elaboration network is held in a mutable box and accessed imperatively, rather than threaded as a value through pure functions and propagator fire functions.**

```
Finding 1 (box)
  ├── Finding 2 (with-enet-reads: fake purity over box)
  ├── Finding 3 (registries as parameters: not on network)
  ├── Finding 4 (speculation: imperative access to correct operations)
  ├── Finding 5 (callbacks: 6/10 exist because of box architecture)
  └── Finding 6 (solve-meta!: re-entrant side effects through box)
```

**Track 8D scope**: Address the root cause (Finding 1), which makes Findings 2, 4, 5 fall out. Findings 3 and 6 require additional targeted work (registry cells, action descriptors).
