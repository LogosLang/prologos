## 1. What Was Built

Track 8 completed the Propagator Migration Series' infrastructure goal: **bring all elaboration onto the propagator network, dissolving the boundary between type inference and trait resolution.**

Three Parts delivered a progression from data to operations to control flow:
- **Part A** (5 phases): All structural state (meta-info, id-map, mult, level, session) assumption-tagged for ATMS worldview management. `restore-meta-state!` partially retired. Progress box removed.
- **Part B** (8 phases): Elaboration operations moved onto the network. Worldview-aware reads make retracted speculation entries invisible without synchronous restore — `restore-meta-state!` fully retired. 14/23 callback parameters replaced with direct cell-ops. HKT `impl` registration and resolution proven (4 parser fixes). New modules: `cell-ops.rkt`, `elab-network-types.rkt`.
- **Part C** (8 phases): Resolution control flow moved onto the network. Trait resolution, hasmethod resolution, and constraint retry fire as bridge propagators during S0 quiescence — the S1→S2 readiness→action path becomes a safety-net fallback. Gauss-Seidel vs BSP scheduling comparison: inconclusive (0.998×).

The elaborator is now a thin AST walker that emits constraints into the propagator network. Type inference and trait resolution are propagation events in the same S0 pass.

## 4. Bugs Found and Fixed

### B1: Ghost Meta Problem (3 attempts to solve)

**Bug**: Metas created during failed speculation persist in id-map/meta-info after retraction. "Unknown metavariable" and "already solved" errors.

**Root cause**: S(-1) retraction is deferred (runs at loop start, not at failure). The elaborator needs synchronous structural cleanup between speculation branches, but deferred retraction provides only value cleanup.

**Why it seemed right**: A4 assumed that tagging + S(-1) retraction was sufficient for all state. The distinction between value state (lattice-monotone, retractable) and structural state (existence — either exists or doesn't) was not recognized until the bug surfaced.

**Resolution**: B1's worldview-aware reads — the third attempt — succeeded because visibility is a property of the read path (structural, correct-by-construction), not the cleanup timing (imperative, fragile). Retracted entries exist in the CHAMP but are invisible to reads filtered by the speculation stack.

**Lesson**: The same bug required three architecturally distinct approaches across A4, A4b, and B1. Each attempt deepened understanding: A4 showed values aren't enough, A4b showed id-map tagging isn't enough, B1 showed the read path is the correct intervention point. The D.4 design revision (worldview-aware reads) came from this understanding.

### B2: Circular Dependency (B0 analysis was wrong)

**Bug**: `cell-ops.rkt → elaborator-network.rkt → type-lattice.rkt → reduction.rkt → metavar-store.rkt → cell-ops.rkt`.

**Root cause**: B0 analysis said "no circular deps at file level — callbacks are historical artifact." This was wrong. The cycle is real and structural: cell operations depend on lattice merge functions which depend on reduction which depends on meta state.

**Resolution**: `elab-network-types.rkt` extraction — move struct definitions + cell operations (with no type-lattice deps) to a separate module that both sides can import.

**Lesson**: Module-level dependency analysis from reading imports is insufficient when `require` chains are transitive. The cycle was invisible until `raco make` failed.

### B3: HKT Preparse Infinite Loop

**Bug**: `{C : Type -> Type}` in trait definition causes infinite preparse expansion.

**Root cause**: `pattern-var?` treated `$brace-params` (a reader sentinel) as a defmacro pattern variable. `datum-subst` errored "Unbound pattern variable: $brace-params", triggering re-expansion.

**Resolution**: Four parser fixes: (1) `$brace-params` opaque in `preparse-expand-form`, (2) `$brace-params` skipped in `preparse-expand-subforms`, (3) all 6 reader sentinels excluded from `pattern-var?`, (4) `extract-implicit-binders` for method spec Pi wrapping.

**Lesson**: The parser's macro system doesn't distinguish "reader sentinel" from "user-defined macro." Any new reader sentinel is a potential `pattern-var?` collision. This class of bug is structural — it will recur for the next sentinel unless `pattern-var?` is made sentinel-aware (which fix (3) now does for all known sentinels).

### B2b: Unconditional Network Creation

**Bug**: `make-elaboration-network` in `reset-meta-store!` broke test contexts that don't have propagator infrastructure.

**Root cause**: B2b replaced a guarded callback with an unconditional direct call. The callback was `#f` in test contexts (no network); the direct call always creates a network, leaving infrastructure cells uninitialized.

**Resolution**: Gated network creation on `current-prop-new-infra-cell` availability.

## 5. Design Decisions and Rationale

| # | Decision | Rationale |
|---|----------|-----------|
| 1 | Per-entry assumption tagging (not TMS cell for entire CHAMP) | Granular retraction; aligns with Track 7's tagged-entry pattern. O(1) per lookup. |
| 2 | Worldview-aware reads (B1) as architectural centerpiece | Read path filtering is structural (correct-by-construction), not timing-dependent (imperative). Third attempt at restore-meta-state! retirement succeeded because the mechanism is fundamentally different. |
| 3 | `elab-network-types.rkt` extraction (not module merge) | Minimal change to break cycle. Structs + cell ops with no type-lattice deps. ~150 lines moved. |
| 4 | Bridge propagators use `elab-add-propagator`, not `net-add-cross-domain-propagator` | Resolution needs meta-info access (zonk, ground check). Pure α/γ functions can't read metavar state. Enet-box sync in fire function bridges the prop-network/elab-network gap. |
| 5 | Readiness propagators retained alongside bridges (belt-and-suspenders) | Defense-in-depth during migration. If bridge resolves first, S2 action is no-op (meta-solved? guard). Zero cost; removes a class of regression risk. |
| 6 | Domain-injection callbacks for bridge fire functions | metavar-store.rkt can't import trait-resolution.rkt (cycle). 1 callback per resolution type is architecturally correct — it's domain injection, not structural indirection. |
| 7 | B5 transferred to CIU Track 3 | Track 8 = infrastructure (HKT resolution works, bridges work). CIU = content (convert manual dicts to `impl`, wire `surf-get`). Clean decomposition. |
| 8 | `current-use-bsp-scheduler?` at propagator.rkt level | Global override for proper A/B comparison. First attempt overrode only stratified loop (partial, misleading). User correctly identified methodology gap. |

### D.4: The Critical Design Revision

The D.1–D.3 design failed to apply Propagator-First, Data Orientation, and Correct-by-Construction principles. "Partial retirement" of `restore-meta-state!` was a rationalization — the imperative save/restore pattern IS a callback-like mechanism for state that should be on the network. The D.3 principles alignment check *listed* these principles but failed to *apply* them.

D.4 applied the principles: worldview-aware reads make retraction a property of the read path (structural, correct-by-construction), not the cleanup timing (imperative, fragile).

**This is the most important lesson of Track 8**: a principles alignment checklist is cosmetic unless it challenges workarounds. The question isn't "does this principle apply?" but "does this design *violate* this principle, and is the violation necessary or a failure of imagination?"

## 6. Lessons Learned

### 6.1: Worldview-Aware Reads as Universal Mechanism

The ATMS worldview (speculation stack) filtering all CHAMP reads is the single most impactful architectural change in the Propagator Migration Series. It solves the ghost-meta problem, retires restore-meta-state!, and makes speculation cleanup automatic. The mechanism is O(depth) per read (depth rarely > 3) — negligible cost for structural correctness.

**Apply to**: Any future state that needs speculation awareness. The pattern is: tag entries with assumption ID at creation, filter by `worldview-visible?` at read time. S(-1) becomes GC.

### 6.2: Bridge Propagators Are Not Cross-Domain Propagators

The C0 prototype used pure α/γ functions (C0 pattern = C prototype pattern). The production implementation (C1-C3) uses `elab-add-propagator` with enet-box-syncing fire functions. The pure `net-add-cross-domain-propagator` can't access meta-info for zonk/normalize/ground checks.

**Apply to**: Future bridge patterns that need elaboration state access. The enet-box-sync pattern (read box → rewrap with live pnet → call pure function → write box → return pnet) is the bridge between propagation domain and elaboration domain until Track 9 merges them.

### 6.3: Principles Alignment Must Challenge, Not Catalogue

The D.1–D.3 principles check listed 10 aligned principles and 2 tensions. It did NOT challenge the "partial retirement" workaround against Propagator-First. The D.4 revision applied the principles and produced a fundamentally different (and correct) design.

**Apply to**: Every future design critique round. Add an explicit step: "For each principle, identify whether any component of this design is a *workaround* that the principle would eliminate. If so, justify the workaround or redesign."

### 6.4: Incomplete Instrumentation Produces Misleading Conclusions

The first C5a benchmark overrode only the stratified loop's quiescence calls. Result: BSP 0.8% slower — interpreted as "hypothesis confirmed." The second C5a benchmark overrode ALL quiescence calls (propagator.rkt level). Result: BSP 0.2% faster — opposite sign.

**Apply to**: Any A/B comparison. Instrument at the lowest possible level. Partial instrumentation captures a subset of behavior and may flip the conclusion.

### 6.5: The Design Doc Is the Implementation Plan

Track 8's design doc (860+ lines, D.4 revision) served as the implementation plan with zero deviation. Every phase had explicit files, rationale, and acceptance criteria. Implementation was mechanical — read the phase, implement, commit. No plan mode, no additional planning.

**Apply to**: Invest in design documents proportionally to implementation complexity. Track 8's 6h design → 12h implementation with 0 wrong approaches (excepting the circular-dep discovery, which the design noted as a risk).

### 6.6: Belt-and-Suspenders Is a Migration Strategy, Not Technical Debt

C1-C3 installed bridge propagators alongside the existing readiness propagators. The readiness path still fires and populates the ready-queue; the S2 executor finds everything already resolved. This is not waste — it's defense-in-depth during a phased migration. The readiness propagators can be removed later (after the bridges are battle-tested), or retained permanently as a verification layer.

**Apply to**: Any migration that replaces one mechanism with another. Keep both running until confidence is established. The `meta-solved?` guard pattern (check if already resolved before executing action) makes dual mechanisms zero-cost.

## 11. Architecture Assessment

### What Held Up

- **Two-network architecture** (Track 7): persistent-registry + per-command elab-network. Part A/B/C all operated within this split without modification.
- **Stratified resolution loop** (Track 2): S0→S1→S2 absorbed bridge propagators gracefully. Bridges fire in S0; S2 is safety net. No loop restructuring needed.
- **Tagged-entry pattern** (Track 7): Per-entry assumption tagging scaled to all 5 CHAMPs (meta-info, id-map, mult, level, session) without pattern modification.
- **Pure resolution functions** (Track 7 Phase 7b): `resolve-trait-constraint-pure` and friends reused directly as bridge fire function bodies. The `enet → enet*` contract was the right abstraction.

### What Needed Modification

- **Module structure**: The elaborator-network → type-lattice → reduction → metavar-store cycle required extraction. This was a genuine structural issue, not an oversight — the cycle exists because cell operations depend on merge functions which depend on type normalization which depends on meta state.
- **Parser**: `pattern-var?` didn't account for reader sentinels. This is a class bug — any reader sentinel added in the future will collide unless the sentinel set is explicit.
- **`restore-meta-state!`**: Required three architecturally distinct attempts (A4: tagging, A4b: id-map tagging, B1: worldview-aware reads) before the correct mechanism was found. The difficulty was fundamental — synchronous restore and deferred retraction are genuinely different mechanisms.

### Architectural Debt Accepted

1. **Enet-box sync in bridge fire functions**: The box is the bridge between prop-network and elab-network. Correct but not ideal — Track 9 will eliminate this by making all state live in cells.
2. **9 remaining callbacks**: Domain-injection callbacks for lattice merge functions, legacy fallback boxes, central state box. These are architecturally correct — they inject domain-specific logic, not structural indirection.
3. **Module-load-time dual writes**: `macros.rkt` still writes to both cells and parameters for registry functions. Module loading context has no network — can't remove parameter writes until module loading is redesigned.
