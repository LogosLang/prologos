## §1. Stated Objectives

From the design doc (D.4):

1. Make meta-variables cell-primary: `meta-solution` reads from cell via cell-id, not CHAMP
2. Eliminate zonk during elaboration (~650 call sites → cell reads)
3. Reduce zonk.rkt from ~1300 lines to ~200 lines (freeze.rkt)
4. Unify `ground-expr?` (two incompatible definitions → one)
5. Simplify `solve-meta!` (cell write triggers propagators naturally)
6. Remove dual storage (CHAMP solution field → metadata only)

**Performance targets** (from Pre-0 benchmarks):
- `meta-solution`: 150ns → 56ns (2.7×)
- Suite wall time: 244s → ≤235s (~4%)

## §4. What Was Deferred and Why

1. **Phase 5 (defaults at solve-time)**: Architectural cleanup. `default-metas` is a cheap single-pass walk that runs once per boundary. Moving to solve-time requires changing level-meta/mult-meta solve paths + resolution timing. Low impact, high complexity.

2. **Phase 7 (CHAMP fallback removal)**: Genuine dependency on PM Track 10 (module loading on-network). CHAMP fallback needed for module-loading and test contexts that don't have full network infrastructure.

Both are intentional deferrals with clear prerequisites, not scope cuts.

## §5. What Went Well

1. **Pre-0 benchmarks changed the design fundamentally.** 5 findings that redirected the entire track. Original Phase 0 was a no-op (meta-solution already reads from cells). Original Phase 2 scope was 225 sites (actual: 31). Without Pre-0, we'd have built the wrong thing. This is the 4th confirmed instance of "benchmark as design input" (BSP-LE Track 0, SRE Track 1B, SRE Track 2, PM 8F).

2. **D.4 self-critique found the Phase 0/3 chicken-and-egg.** `close-expr` in Phase 0 would break downstream `zonk-at-depth` callers that expect bvars. Resolution: Phase 0 = detection only, Phase 3 = correction atomically. Without D.4, this would have been a mid-implementation surprise.

3. **Custom gen:equal+hash is a performance WIN (2.3× hash ops).** The D.3 critique raised it as a risk; the Pre-0 measurement proved it's a benefit. Integer-only hash gives better distribution than default struct hash.

4. **The REPL hang fix resolves a long-standing velocity drain.** The "pre-existing issue" process smell rule (codified during this track) led to investigating and fixing an issue that cost 2+ hours across sessions.

## §6. What Went Wrong

1. **D.1 estimates were systematically wrong.** 225 zonk sites (actual: 31). typing-core makes ZERO zonk calls (estimated: ~100). Phase 0 was a no-op. Phase 4a (freeze) had zero value. The estimates came from the SRE Research Doc's vision ("elaborator becomes thin walker using structural-relate") which is the END STATE after PM Track 10, not the current architecture. **Lesson**: distinguish the vision from the current migration step.

2. **`zonk-at-depth 0 → zonk` caused timeouts** (Phase 3). Value-equivalent (zero mismatches across 7401 tests) but NOT identity-equivalent. The unify hot loop relies on specific recursion patterns; changing which function recurses causes combinatorial blowup from `perf-inc-zonk!` overhead or different self-recursion targets. Wasted ~2 hours debugging before finding the simpler fix (update `zonk-at-depth`'s own meta-solution call).

3. **Agent bulk-edit confused field accessor with constructor.** The agent changed `(expr-meta-id m)` (accessor) to `(expr-meta (expr-meta-id m) #f)` (wrapping in new constructor) in test-speculation-bridge.rkt. Produced syntax errors caught on next test run, but required manual fix.

4. **`bvar-free?` assertion ran unconditionally** (not gated behind debug flag). `struct->vector` on every `solve-meta!` call added ~1500ns overhead. Should have been gated from the start.

## §7. Where We Got Lucky

1. **The Phase 3 timeout was NOT a subtle correctness bug.** The values matched perfectly (zero mismatches). If they hadn't — if cell-id returned different values from id-map — we'd have had silent wrong unification results. The `zonk-at-depth 0 → zonk` conversion would have introduced an invisible correctness regression. We got lucky that the issue manifested as a timeout (detectable) rather than wrong results (silent).

2. **The pre-existing test failure** (`test-constraint-retry-propagator`) was exposed by our `.zo` cache deletion, not by our code changes. If we hadn't deleted the cache, it would have remained hidden.

## §8. What Surprised Us

1. **typing-core.rkt makes ZERO zonk calls.** The type checker's `infer`/`check` cases don't call zonk — they pass expressions to `unify`, which calls `zonk-at-depth`. The elaborator is already store-agnostic (confirmed by SRE Track 2). The zonk calls live in 3 files: unify.rkt, resolution.rkt, trait-resolution.rkt.

2. **`ground-expr?`'s two definitions are DIFFERENT CONCEPTS**, not duplicate code. global-constraints checks "no logic variables" (logic engine); trait-resolution checks "no unsolved metas" (type checker). Unifying them would be wrong.

3. **The REPL hang root cause was trivial** — `current-prop-net-box = #f`. The fix was 10 lines. It was "pre-existing" for multiple sessions because nobody investigated. The process smell rule (codified during this track) would have caught this earlier.

4. **Cell-id custom equality is FASTER** than default struct equality (43ns vs 52ns). Counter-intuitive — custom code is usually slower. The integer-only hash gives better distribution, making hash table operations 2.3× faster.

## §9. Architecture Assessment

The architecture held up well. The key finding: **meta-solution was ALREADY cell-primary** before PM 8F. The CHAMP path was fallback-only. The real bottleneck was the id-map indirection (82ns per meta-solution call), not the cell/CHAMP split.

The cell-id-in-expr-meta change (Phase 1) is the most architecturally significant: it makes the meta struct carry its own cell identity. This is Data Orientation in action — the data carries its own metadata. The custom `gen:equal+hash` preserves identity semantics while adding the metadata.

The `zonk` infrastructure remains largely unchanged. The vision of "eliminate zonk" requires expressions that don't contain `expr-meta` nodes — which is PM Track 10's end-state, not PM 8F's scope. PM 8F delivered the performance optimization (cell-id fast paths) without the architectural transformation (zonk removal).

## §10. What This Enables

1. **PM Track 10 (convergence)**: Cell-id in expr-meta is the foundation. When module loading is on-network, the CHAMP fallback is removed, and cell-id becomes the ONLY meta access path.
2. **SRE Track 3 (trait resolution)**: Resolution now uses `normalize-for-resolution` without redundant zonk — trait key extraction is leaner.
3. **Future zonk elimination**: When expressions stop containing `expr-meta` (Track 10), zonk.rkt becomes dead code. The cell-id infrastructure makes that transition smooth.
4. **REPL development**: `process-string` now works from bare REPL — enables interactive debugging, acceptance testing, and rapid prototyping.

## §11. Technical Debt Accepted

1. **Phase 5 (defaults at solve-time)**: `default-metas` remains a separate pass. Low impact.
2. **Phase 7 (CHAMP fallback)**: Dual storage persists for module-loading/test contexts.
3. **`fresh-meta` is 1.3× slower** (2,087→2,809ns) from larger expr-meta struct. Acceptable: `fresh-meta` is called ~627 times during elaboration; the ~450ns overhead is ~0.3ms total per command.
4. **13 necessary zonk calls remain** in unify.rkt (5) + resolution.rkt (5) + trait-resolution.rkt (3). These can't be eliminated until expressions don't contain `expr-meta`.

## §12. What Would We Do Differently

1. **Start with benchmarks before writing the design.** D.1 was written without data, then D.2 rewrote it after benchmarks. If Pre-0 came first, D.1 would have been correct from the start.

2. **Gate debug assertions from the start.** The `bvar-free?` assertion should have been behind a flag in Phase 0, not running unconditionally until benchmarks caught it.

3. **Don't assume design estimates are accurate.** 225 zonk sites → 31. ~100 in typing-core → 0. Audit FIRST, then design. The "combined Stage 2/3" format worked but should lead with the audit, not the vision.

## §13. Wrong Assumptions

1. **"~225 zonk call sites during elaboration"** — actual: 31 external, 0 in typing-core.
2. **"Phase 0 (cell-primary meta-solution) is the first step"** — meta-solution already reads from cells. Phase 0 was already done.
3. **"`zonk-at-depth 0` is equivalent to `zonk`"** — value-equivalent but NOT identity-equivalent. The recursion pattern matters.
4. **"Two `ground-expr?` definitions need unification"** — they serve different contexts (type vs logic). Different concepts with the same name.
5. **"The REPL hang is a deep infrastructure issue"** — it was 10 lines of missing network initialization.

## §14. What We Learned About the Problem

The meta-variable system is MORE on-network than we thought. `meta-solution` already reads from cells. The zonk infrastructure is LESS pervasive than estimated (31 sites, not 225). The real bottleneck was a single indirection layer (id-map) that cell-id eliminates.

The "zonk elimination" vision is premature for PM 8F's scope. Zonk can only be eliminated when expressions don't contain `expr-meta` nodes — which requires rethinking expression representation (PM Track 10). PM 8F correctly focused on the achievable: faster meta reads via cell-id.

## §15. Are We Solving the Right Problem?

Yes — but the problem is smaller than we thought. The meta system was already mostly cell-primary. The remaining gap (id-map indirection) was a focused, well-scoped optimization. The broader "zonk elimination" goal is real but belongs to a later track.

The REPL hang fix was NOT part of the stated objectives but delivered more developer velocity improvement than any single phase of PM 8F. The "process smell" rule that led to investigating it was the most valuable process improvement of the track.
