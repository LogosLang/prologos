# Phase 5 Extension: Free Ordering of Declarations

## Overview

Extends the spec pre-scan (Phase 5) to enable maximal flexibility in
definition ordering within a module. Only genuinely cyclic dependencies
produce compiler errors.

## Completed

### Phase 5a: Three-Pass Pre-Registration (commit `53223ca`)

Replaced the single spec pre-scan in `preparse-expand-all` with a
three-pass architecture:

- **Pass 0**: No-dependency declarations — data, trait, deftype, defmacro,
  bundle, property, functor. These only WRITE to registries (idempotent
  via `hash-set`).
- **Pass 1**: Declarations that depend on Pass 0 — spec (reads bundle +
  trait registries) and impl (reads trait registry).
- **Pass 2**: Main processing loop (unchanged).

Verified by the dependency map: 7 of 9 declaration forms have no external
reads; only spec and impl have hard dependencies on Pass 0 outputs.

### Phase 5b: Declaration-First Output Ordering (commit `53223ca`)

Data/trait-generated defs are hoisted before user forms via stable
partition after the main loop. This ensures constructor types and trait
accessor types enter `global-env` before any user `defn`/`def` is
type-checked.

**Key design choice**: impl-generated defs are NOT hoisted because impl
method helpers can reference user-defined functions from the same module
(e.g., `Interval--Lattice--join` calls `interval-join`).

### Idempotency Fix: `register-param-impl!` (commit `53223ca`)

Added idempotency guard checking `param-impl-entry-dict-name` to prevent
double-registration when `process-impl` runs in both Pass 1 and Pass 2.
Without this, parametric impls (e.g., `impl Lattice (FlatVal A)`) would
create duplicate entries causing false overlap warnings.

### Tests

- `test-free-ordering.rkt`: 10 test cases covering defn-before-data,
  match-before-data, multi-constructor, combined orderings
- Full suite: 4638 tests, zero regressions

## What Phase 5a+5b Enables

- `defn` using constructors from a `data` declared later in the module
- `spec` referencing `bundle` or `trait` declared later
- `impl` referencing `trait` declared later (in registries; not yet in
  global-env for generated defn bodies)
- Pattern matching against constructors from later `data` declarations

## Known Limitations

1. **Impl method bodies**: Can't forward-reference user defns from the
   same module (impl defs are not hoisted to avoid breaking helper refs)
2. **Mutual data types**: `data A (mk-a : B)` + `data B (mk-b : A)` —
   within the hoisted bucket, ctor defs of the first data reference the
   second type. Works only if both types are registered in type-meta
   (Pass 0 handles this), but constructor TYPE defs are processed
   sequentially. Would need intra-bucket topological sort.
3. **Cross-module cycles**: Not addressed (deferred to Phase 5c)

## Completed

### Phase 5c: Cross-Module Form-Level Dependency Analysis (commit `b17555c`)

Built `tools/form-deps.rkt` (~300 LOC) — reads all book chapters via the
WS reader, splits at module boundaries, and extracts defines/references
per form. Computes module-level SCCs via `tarjan-scc` from `stratify.rkt`.

**Results**: 121 modules, 907 forms, zero module-level cycles. The stdlib
book's module DAG is clean — no spurious cycles exist.

Added `--analyze` flag to `tangle-stdlib.rkt` (delegates to form-deps).

**ATMS cycle diagnosis**: not needed since no cycles were found. The ATMS
integration point is ready in the architecture if cycles arise as the
library grows.
